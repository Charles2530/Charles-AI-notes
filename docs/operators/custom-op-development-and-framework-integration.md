# 自定义算子与框架集成

很多人第一次写自定义 kernel 时，会把注意力放在“这个 kernel 跑得够不够快”。但在真实工程里，性能只是其中一部分。一个自定义算子想真正进入训练或推理系统，还必须解决：

1. 怎样被框架调用；
2. 怎样与 autograd、图编译和 profiler 兼容；
3. 怎样处理 shape、dtype、device dispatch；
4. 怎样做 fallback、测试和版本演进。

因此“自定义算子开发”其实更像一条完整集成链，而不是孤立的 kernel 编写。

## 1. 为什么很多高性能 kernel 最终没被用起来

常见原因不是它不够快，而是：

1. 集成太难；
2. 只支持少量 shape；
3. autograd 路径缺失；
4. 无法进入 graph compile；
5. fallback 和错误处理太弱；
6. 部署、打包和兼容成本太高。

也就是说，真正让一个 kernel 有工程价值的，不只是性能，而是可接入性。

## 2. 自定义算子的典型组成

一个较完整的 custom op 通常包含：

1. 前端 API；
2. shape / dtype 检查；
3. forward kernel；
4. backward kernel 或 autograd 定义；
5. dispatch 注册；
6. 测试与 benchmark；
7. 可能还要有图编译兼容和 profiler 标记。

### 2.1 为什么这和“写个 CUDA 文件”不是一回事

因为一旦进入真实系统，你还要考虑：

1. contiguous 和 non-contiguous 输入；
2. CPU fallback 或 error path；
3. 多卡与 stream 语义；
4. mixed precision；
5. 编译缓存和运行时加载。

## 3. 与 PyTorch 集成时要想清楚的几件事

### 3.1 接口边界

**这个 op 是**：

1. 完全替换现有算子；
2. 作为 fused op；
3. 只服务某些 shape；
4. 只服务训练或只服务推理。

**不同定位会决定**：

1. API 如何设计；
2. fallback 如何写；
3. 是否需要多个后端实现。

### 3.2 Autograd 边界

如果是训练路径，还必须考虑：

1. backward 是否自己写；
2. 是否可由已有算子组合求导；
3. 中间缓存如何保存；
4. backward 是否也要高性能。

很多 kernel 的集成难点其实不在 forward，而在 backward。

## 4. 与 `torch.compile` / 图编译的关系

自定义算子如果完全绕开图编译器，有时会导致：

1. 断图；
2. 融合机会丢失；
3. 上层优化无法继续穿透；
4. profiler 看到一大片黑盒。

这并不一定意味着不能写 custom op，但意味着你需要清楚：

1. 它是否值得成为黑盒边界；
2. 它是否会阻断更大的图优化；
3. 它是否只在真正热点处使用。

## 5. Triton 自定义算子与 CUDA 自定义算子的差异

### 5.1 Triton

**优点**：

1. 开发快；
2. 更适合快速试 fusion；
3. 与 Python / PyTorch 路径更近。

**缺点**：

1. 某些极致优化场景不如底层实现；
2. 某些复杂同步和特化路径不够灵活。

### 5.2 CUDA

**优点**：

1. 底层控制更强；
2. 可做到更极致特化；
3. 对复杂同步和特定硬件路径支持更充分。

**缺点**：

1. 集成与维护成本更高；
2. 开发调试更慢；
3. 对团队要求更高。

## 6. 什么时候值得做 custom op

**通常需要满足以下几条中的多条**：

1. 热点明显；
2. 默认实现慢；
3. 可融合收益明确；
4. 形状相对稳定；
5. 业务收益足以覆盖维护成本。

如果只是“理论上也许能快一点”，往往不值得。

## 7. 测试与验收不能省

一个可上线的 custom op 至少要经过：

1. 正确性对比；
2. 多 shape 测试；
3. dtype 测试；
4. contiguous / stride 测试；
5. 训练或推理端到端回归；
6. profiler 对照；
7. fallback 验证。

否则一个快 kernel 很容易变成线上隐患。

## 8. 版本演进与兼容

自定义算子不是“一次写完永远不动”的资产。  
**它会持续受以下因素影响**：

1. 新 GPU 架构；
2. 新 PyTorch 版本；
3. 编译器后端变化；
4. shape 分布变化；
5. 模型结构变化。

因此一个工程上成熟的 custom op，还必须有：

1. 版本管理；
2. 持续 benchmark；
3. fallback；
4. 清晰的适用边界。

## 9. 一个形象比喻

自定义算子就像为工厂定制一台专用设备。设备本身可能很快，但如果它接不上现有流水线、维护手册不全、只能处理一种规格的零件、坏了又没有备用流程，那么它的工程价值就会大打折扣。真正有价值的 custom op，不只是“局部最快”，而是能顺滑进入整条生产线。

## 10. 小结

自定义算子开发的关键，不只是 kernel 写得快，而是把性能、集成、autograd、图编译、测试、fallback 和版本演进放在同一张设计图里。只有这样，一个自定义 kernel 才会从 demo 变成真正可维护、可部署、可复用的系统组件。
