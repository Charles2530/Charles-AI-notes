# 检索、重排与向量系统

VLM 在实验室里常被理解成“看图回答问题”的模型，但一旦进入文档、企业知识库、屏幕代理、商品搜索、视频分析等真实系统，它很快就不再是一个独立模型，而是检索、重排与向量基础设施中的一个组成部分。很多线上问题并不是 VLM 不会看，而是“没看到正确证据”“看到了太多无关证据”或者“看到的证据顺序不对”。因此理解 VLM 与检索系统的耦合，是把多模态模型做成可用产品的关键。

## 1. 为什么 VLM 需要检索系统

VLM 的上下文窗口再大，也不可能总把所有文档页、所有屏幕截图、所有商品图和所有操作历史一次性塞进去。系统必须先决定：

1. 哪些视觉证据最值得送入模型；
2. 哪些文本说明、OCR 结果、结构化字段应一同进入；
3. 这些证据的顺序、粒度和表示形式是什么。

**因此整体系统常写成**：

$$
y = f_\theta\big(x_{\text{query}}, \mathcal{R}(x_{\text{query}}, \mathcal{D})\big),
$$

其中 $\mathcal{R}$ 是检索系统，$\mathcal{D}$ 是多模态知识库。若 $\mathcal{R}$ 失败，后面的 VLM 再强也只能在错误证据上推理。

## 2. 多模态检索对象的粒度

VLM 系统里的“检索单元”不是固定的，它可能是：

1. 整张图像
2. 图像区域 / patch
3. 文档页
4. 文档中的版面块
5. OCR 文本行
6. 视频片段
7. 屏幕状态快照
8. GUI 元素树节点

粒度太粗，召回不精确；粒度太细，索引数量爆炸，拼接上下文也更困难。工程上经常需要两级甚至三级索引：先粗召回文档页，再细召回区域或文本块。

## 3. 向量检索的基础形式

最常见的检索是把查询和候选项编码成向量，通过相似度搜索近邻。设查询表示为 $q$，候选表示为 $d_i$，则可按

$$
s_i = \mathrm{sim}(q, d_i)
$$

排序，常见相似度包括点积、余弦相似度和欧氏距离。对归一化向量，余弦与点积等价：

$$
\mathrm{cos}(q, d_i) = \frac{q^\top d_i}{\|q\|\|d_i\|}.
$$

在 VLM 里，$q$ 和 $d_i$ 可以来自同模态，也可以跨模态，例如文本查询检索图像页、图像截图检索历史操作步骤。

## 4. 多模态向量空间的三种组织方式

### 4.1 共享空间

图像、文本、版面块都投到同一向量空间。这种设计最统一，适合跨模态检索，但需要强对齐训练，否则不同模态分布可能彼此挤压。

### 4.2 分模态空间 + 融合重排

先用各自最强的单模态 encoder 做召回，例如文本走 BM25 + text embedding，图像走 CLIP embedding，布局走结构化匹配；之后在重排阶段再统一交给 VLM 判断。它的好处是模块化强，缺点是系统更复杂。

### 4.3 层次式空间

先在粗层面检索“哪一页、哪个视频、哪个应用窗口”，再在细层面检索“哪一块区域、哪一段字幕、哪个 UI 节点”。这类设计对长文档和屏幕系统尤其常见。

## 5. ANN 索引与工程权衡

向量系统在千万级以上样本时，通常依赖近似最近邻（ANN）而非精确搜索。经典结构包括：

1. **IVF**：先聚类到倒排桶，再在桶内搜索；
2. **HNSW**：基于分层图的近邻搜索；
3. **PQ / OPQ**：压缩向量以降低内存；
4. DiskANN 等磁盘友好结构。

核心权衡是召回率、延迟、内存。若索引参数设置为 $(n_{\text{list}}, n_{\text{probe}})$ 一类形式，则通常：

1. 更大 `nprobe` 提升召回，但增加延迟；
2. 更强压缩节省内存，但可能损害排序精度；
3. 动态更新越频繁，索引维护越复杂。

对于屏幕 agent 或文档助手这类时效性高的系统，索引更新成本常常和检索质量一样重要。

## 6. 文本检索仍然不可替代

即便系统是 VLM 驱动，BM25、关键词索引、字段过滤依然很重要。原因有三：

1. 文本关键词能精准命中号码、合同条款编号、股票代码、药品批号等离散符号；
2. OCR 后的文本在一些任务中比视觉特征更稳定；
3. 规则过滤可大幅缩小候选集，提高后续 VLM 重排效率。

例如发票系统里，用户问“税率 13% 的专票”，单纯图像 embedding 可能难以精准区分 9% 与 13%，但 OCR 文本和字段索引能迅速命中目标。

## 7. 重排为什么往往比召回更值钱

召回负责“不漏”，重排负责“不乱”。在多模态系统里，召回通常只保证候选集里“可能有答案”；重排才决定哪些证据真正送进 VLM 上下文。设候选集为 $\{d_i\}_{i=1}^{K}$，重排器输出分数 $r_i$，最终选择 top-$m$：

$$
\mathcal{C}_{\text{final}} = \mathrm{TopM}\{r_i\}_{i=1}^{K}.
$$

如果候选太多、上下文有限，那么重排质量直接决定最终答案质量。

### 7.1 重排模型的几种形式

1. **Cross-encoder**：把查询与候选拼接后联合打分，精度高但慢。
2. Lightweight VLM reranker：输入图像区域 + 文本问题，输出相关性。
3. **任务特定规则重排**：基于字段匹配、时间新鲜度、版面位置、权限等级。
4. **混合打分**：向量分数 + BM25 + 元数据特征 + 业务先验。

实际中，最后一种最常见，因为线上系统很少愿意把全部排序责任交给单一模型。

## 8. 文档理解中的典型检索流水线

以企业合同问答为例，一个常见多阶段流程是：

1. 文档切页并 OCR；
2. 对页、块、字段构建多种索引；
3. 文本查询先走关键词 / BM25 召回；
4. 再用多模态 embedding 补召回视觉相关页；
5. 对候选页中的版面块做细粒度重排；
6. 最终把少量页图和关键文本片段送入 VLM。

这个过程中，VLM 并不承担“全库搜索”，而是承担“基于高质量证据完成理解与推理”。

## 9. 屏幕代理中的检索与历史记忆

屏幕 agent 常需要记住历史操作、界面变化和帮助文档。检索对象可能包括：

1. 过去几步截图
2. DOM / accessibility tree 节点
3. 产品帮助中心文档
4. 历史成功轨迹

当用户说“继续刚才导出报表的流程”，系统要找回的不是一张静态图片，而是“与当前任务最相关的历史子轨迹”。这类检索更像时序检索：

$$
\mathcal{R}(q_t) = \arg\max_{\tau_i \in \mathcal{M}} \mathrm{sim}(g(q_t), h(\tau_i)),
$$

其中 $\tau_i$ 是历史轨迹片段，$\mathcal{M}$ 是记忆库。

## 10. 视频与长时多模态检索

视频系统的难点在于时间。检索单元既可以是关键帧，也可以是片段、字幕、场景摘要、目标轨迹。很多系统采用：

1. 粗粒度视频段召回；
2. 关键帧或字幕重排；
3. 最终选出代表性片段给 VLM。

如果用户问“会议里谁先提出降本方案”，系统需要的不只是“包含降本”字幕的片段，还要知道发言顺序和人物身份。这时仅靠文本检索不够，需要视觉说话人信息和时间线共同参与。

## 11. 向量系统里的元数据过滤

向量检索若完全不加过滤，常会召回看似相似但业务上无效的结果。元数据过滤常用于：

1. **权限控制**：只搜用户可见文档；
2. **时间范围**：只看近 30 天发票或日志；
3. **语言范围**：中文、英文或多语；
4. **文档类型**：合同、发票、邮件、截图说明；
5. **产品版本**：不同 UI 版本对应不同帮助文档。

这一步常能极大降低噪声，并减少后续重排负担。

## 12. 多模态重排的输入拼接问题

把多少图、多少 OCR 文本、多少结构化字段送给重排器或最终 VLM，是一个实际而棘手的问题。可把上下文预算近似写成：

$$
B_{\text{ctx}} = B_{\text{text}} + B_{\text{image}} + B_{\text{meta}}.
$$

预算固定时，给图像更多 token 就意味着给文本更少。文档系统常见的经验是：

1. 若问题依赖布局关系，保留页图更重要；
2. 若问题依赖字段精确匹配，OCR 文本更重要；
3. 若问题依赖时效或权限，元数据必须进入。

因此“最优拼接”不是通用模板，而是任务依赖。

## 13. 线上系统的失效模式

### 13.1 召回对了，重排错了

候选中明明有正确页，但被错误地排到上下文外。这通常发生在长尾版式、复杂图表或弱 OCR 质量样本上。

### 13.2 检索相似页面而非正确答案页

例如合同模板很像，但条款年份或金额不同。embedding 会把它们拉得很近，但业务上完全不能混用。

### 13.3 文本与视觉证据冲突

OCR 文本识别错了一个数字，而图像其实清晰可辨。若重排器完全信文本，最终答案就会偏。

### 13.4 历史记忆污染

屏幕 agent 误把上一次任务的历史轨迹检索回来，导致当前操作被错误引导。这类问题常表现为“系统看起来像记性很好，但总在错的地方很自信”。

## 14. 两个具体例子

### 14.1 发票审核助手

**用户问**：“找出本季度税率 13% 且金额超过 1 万元的专票。”合理系统会：

1. 先用字段索引过滤税率和票种；
2. 用 OCR / 文本索引过滤金额；
3. 再让 VLM 核对图像上盖章、发票类型、异常格式；
4. 最后汇总结果。

如果反过来先用 VLM 对全库逐张看图，成本和延迟都会失控。

### 14.2 屏幕流程助手

**用户说**：“继续完成报表导出，把格式改成 CSV。”系统需要召回与当前软件版本、当前报表页面最相关的历史步骤。如果历史记忆检索没有版本过滤，可能把旧版 UI 的导出按钮位置拿回来，导致 agent 连续点击失败。

## 15. 设计建议

1. 召回和重排分开评估，不要只看最终问答正确率。
2. 对多模态知识库做多粒度切分，尤其是文档和屏幕场景。
3. 文本索引、结构化过滤和向量检索应协同，而不是互斥。
4. 建立“召回命中但未入上下文”的错误集，专门优化重排。
5. 对历史记忆检索加入时间、版本、任务态过滤，避免跨任务污染。

## 16. 小结

VLM 在系统中更像一个多模态推理核心，而不是替代所有检索与排序逻辑的万能模型。向量系统负责把可能有用的视觉与文本证据找回来，重排系统负责在上下文预算内挑出最关键的证据，VLM 才有机会发挥真正的理解能力。脱离检索和重排谈 VLM 落地，往往只会把问题从“模型不够强”误判为“模型还不够大”。
