# 线上事故排查模板

这一页适合在系统线上异常、服务退化或机器人实机异常出现时，快速做分层排查。目标不是完整复盘，而是先把问题压到正确的层，避免团队在最紧急的 30 分钟里朝错误方向狂奔。

## 1. 事件基本信息

```text
事件编号：
发现时间：
负责人：
影响范围：
当前状态：
```

**建议再补**：

```text
影响开始时间：
发现渠道（报警 / 用户反馈 / 值班巡检）：
当前优先级（P0 / P1 / P2）：
是否已触发回滚：
```

这一步的目标，是让后续所有沟通建立在同一张事实表上，而不是不同人各自脑补一套事件版本。

## 2. 现象描述

尽量用客观症状描述，不要先下结论。例如：

- P95 latency 从 900ms 上升到 2.1s
- OCR 正确率突然下降
- 机器人抓取抖动明显增加

### 现象描述的三个原则

1. 先写时间线，再写猜测；
2. 尽量写指标，而不是情绪化形容；
3. 区分“整体退化”和“某些桶退化”。

例如“模型变差了”是低价值描述；“过去 40 分钟内，长 PDF 合同问答的字段抽取错误率从 3% 升到 18%”则是高价值描述。

## 3. 分层排查

### 模型层

**检查**：

- 模型版本是否变化
- 推理配置是否变化
- 量化/采样器是否变化
- prompt 模板是否变化
- 输出长度分布是否突变

### 系统层

**检查**：

- cache 命中率
- queue 长度
- GPU 利用率与显存
- routing / batching 是否异常
- TTFT 与 TPOT 是否同时异常
- 某些 worker 是否单点过载

### 数据层

**检查**：

- 输入分布是否突然变化
- 上游检索结果是否退化
- 标注或模板是否被替换
- OCR / parser / DOM 抽取版本是否变化
- 是否接入了新客户、新场景或新格式数据

### 外部依赖层

**检查**：

- OCR / 检索 / 数据库 / 工具接口
- 机器人传感器或控制器状态
- 外部 API 配额与错误率
- 网络、时钟同步和存储状态

### 安全与业务层

如果是高风险系统，还要补查：

- 是否出现财务、法规、医疗等高风险错误集中暴露
- 是否出现误操作、错误点击或危险动作
- 是否触发安全分类器异常误拦截或漏拦截

## 4. 临时缓解

**记录当前是否已采取**：

- 回退版本
- 限流
- 关闭某项优化
- 切换备用链路
- 关闭高风险功能
- 切到高精度 / 保守模式

### 临时缓解要回答的三个问题

1. 这一步能否立刻止损；
2. 会不会引入新风险；
3. 是否会影响现场证据保留和后续复盘。

有些情况下回滚是最优解；但若回滚会清空关键日志或中断现场证据链，就要同步做快照保留。

## 5. 初步判断

**建议给出**：

- 最可能根因
- 次可能根因
- 仍待验证项

### 不建议直接写“模型不行”

大多数线上事故并不是模型能力突然自己退化，而是：

1. 输入分布漂移；
2. 检索或工具链退化；
3. 路由、缓存、量化或模板变更；
4. 外部依赖异常；
5. 新数据格式或权限版本切换。

“模型不行”通常只是表象，不是根因。

## 6. 下一步动作

**明确列出**：

1. 立即要做什么；
2. 谁负责；
3. 何时回报。

### 推荐动作模板

```text
动作：
负责人：
优先级：
预计完成时间：
验证指标：
```

## 7. 升级条件

建议事先定义哪些情况必须立刻升级：

1. 安全风险上升；
2. 高价值客户或核心链路受影响；
3. 财务、法规、医疗类错误集中出现；
4. 机器人系统出现人机安全边界问题；
5. 无法通过简单回退快速止损。

## 8. 一个实际例子

假设某天企业文档助手的“合同金额抽取”突然大幅退化，比较合理的排查顺序通常不是立刻重训模型，而是：

1. 先看是否接入了新批次扫描件；
2. 再看 OCR 服务版本是否变化；
3. 再看上下文装配模板是否把表格页裁掉；
4. 再看量化或路由策略是否把复杂请求送错模型；
5. 最后才怀疑模型本身参数出问题。

这种顺序更接近真实系统中的因果链，也更能节省时间。

## 9. 事故结束后的最小留档

哪怕还没进入正式复盘，也建议先留下一页最小记录：

1. 发生了什么；
2. 何时开始、何时止损；
3. 初步根因是什么；
4. 采取了什么缓解动作；
5. 哪些证据已经保留。

这能避免团队在几天后再回看时，只剩零散聊天记录和模糊记忆。

## 10. 一句经验

事故排查最怕一上来就把问题归到“模型不行”。先分层，再止损，再定责，再复盘，能大幅减少无效排查。
