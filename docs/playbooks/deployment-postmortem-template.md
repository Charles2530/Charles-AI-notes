# 部署复盘模板

这一页适合在系统上线、灰度或机器人实机部署后做复盘。  
目标不是追责，而是把问题沉淀成下一轮可复用的经验。

真正有效的复盘，至少要做到三件事：

1. 把事件过程还原清楚。
2. 把根因分层拆开，而不是写一句“模型效果不好”。
3. 把结论转成新的数据、指标、脚本或制度。

如果复盘只停留在“那天出了问题”，团队以后仍然会重复犯同样的错。

## 1. 基本信息

```text
项目名称：
部署阶段：
部署环境：
复盘日期：
负责人：
参与角色：
影响范围：
用户影响：
是否触发回退：
```

**建议补充记录**：

- 发布版本号
- 模型版本号
- 配置版本号
- 数据版本号
- 关联监控看板或事故单链接

这些信息后面排查配置漂移时非常重要。

## 2. 上线前预期

**建议写清楚**：

- 目标是什么
- 最重要的成功指标是什么
- 当时预判的主要风险是什么
- 为什么当时认为可以上线

建议不要只写“希望效果更好”，而是给出可验证目标，例如：

- `P95 < 900ms`
- 机器人单任务成功率 `> 92%`
- 文档抽取字段错误率下降 `30%`
- 人工接管率低于 `5%`

## 3. 实际结果

**分别写**：

- 达到的目标
- 未达到的目标
- 超出预期的问题
- 是否出现连锁反应

尽量带数字，例如：

- `P95` 从预期 `800ms` 变成 `1.9s`
- 实机成功率从实验室 `96%` 降到线上 `81%`
- OCR 调用超时率从 `1.2%` 升到 `8.7%`

### 建议拆成三类结果

1. 质量结果
2. 系统结果
3. 业务结果

例如一个模型质量有提升，但系统时延恶化，最终业务成功率未必真的提高。

## 4. 事件时间线

建议尽量写成分钟级或关键操作级时间线。

```text
T0：部署开始
T1：首次异常出现
T2：监控告警
T3：值班同学确认影响
T4：开始定位问题
T5：临时缓解
T6：是否回退
T7：根因确认
T8：修复完成
```

### 时间线里最好额外记录

- 谁观察到异常
- 哪个监控先告警
- 哪个判断导致继续灰度或立即回退
- 哪个动作真正缓解了问题

这能帮助团队识别“技术问题之外的流程问题”。

## 5. 影响面分析

建议单独开一节，不要只在摘要里一句带过。

### 5.1 用户影响

- 哪类用户受影响最重
- 是否只影响某一地区、某一平台、某一模型路线
- 是否影响核心付费链路

### 5.2 系统影响

- 是否拖慢其他共享服务
- 是否造成 GPU / 队列资源挤占
- 是否影响向量库、OCR、浏览器代理等外部链路

### 5.3 业务影响

- 是否造成订单损失
- 是否导致人工接管激增
- 是否增加审核或客服负担

## 6. 根因分析

建议强制从四层归因，而不是一句话归因。

### 6.1 模型层

**可检查**：

- 泛化不足
- 推理配置不合适
- 量化副作用
- CFG 或采样步数过激
- 对齐后行为过度保守或过度啰嗦

### 6.2 系统层

**可检查**：

- cache
- routing
- scheduling
- 延迟问题
- prefix 命中率
- 工具调用编排

### 6.3 数据层

**可检查**：

- 线上分布和测试集不一致
- 长尾场景缺失
- 回流样本不足
- 标注或模板过时

### 6.4 流程层

**可检查**：

- 灰度策略是否合理
- 告警阈值是否过松
- 责任边界是否不清
- 上线 checklist 是否缺项

很多事故其实不只是一层问题。  
例如：

- 模型对长文本更敏感
- 系统又没有给长请求单独路由
- 灰度时还缺少长请求压力回放

这就是典型的多层耦合事故。

## 7. 为什么上线前没发现

这是复盘里极其关键的一段。  
**建议至少回答**：

- 测试集缺了什么
- 指标体系漏了什么
- 监控没覆盖什么
- 灰度策略为什么没及时暴露问题

### 常见原因

- 测试数据太干净
- 只看平均值，不看尾延迟
- 没有记录样本级失败日志
- 只做离线 benchmark，没有真实流量回放
- 没把外部依赖纳入压力测试

## 8. 临时缓解与长期修复

### 8.1 临时缓解

例如：

- 回退模型
- 关闭 speculative
- 缩短最大上下文
- 切手动模式
- 关停高风险工具调用
- 对长请求限流

**临时缓解要记录三件事**：

1. 谁决定做这件事
2. 多快见效
3. 带来了什么副作用

### 8.2 长期修复

例如：

- 增加长尾数据
- 重写路由策略
- 新增安全过滤器
- 调整缓存结构
- 增加回放测试
- 重做 benchmark

**长期修复建议附带**：

- owner
- 优先级
- 截止日期
- 验收标准

## 9. 把复盘结论变成资产

**部署复盘最好最终沉淀成**：

- 新失败样本集
- 新 benchmark
- 新监控指标
- 新 SOP 或模板
- 新回放脚本
- 新告警规则

这是区分“写了篇事故作文”和“真的提升团队能力”的关键。

### 一个生动例子

如果一次机器人抓取失败复盘后，只写了“强反光桌面导致误抓”，那下次还会重演。  
**如果你进一步沉淀出**：

- 一组强反光测试集
- 一条视觉置信度告警
- 一份低照与反光场景 SOP

那这次复盘才真正有长期价值。

## 10. 一个建议的复盘结论模板

**建议固定回答**：

1. 最主要问题是什么？
2. 它更像模型、系统、数据还是流程问题？
3. 下一轮最优先改哪一件事？
4. 哪条经验应写进团队规范？
5. 哪个指标以后必须监控？
6. 哪类样本以后必须进测试集？

```text

本次部署目标：

[时间线]
T0:

T3:

[根因]
模型层：

[为什么没提前发现]

[沉淀资产]
新增数据：

新增监控：
新增规范：
```

## 12. 一句经验

好的复盘不是找到一个锅，而是让团队下次更早看到同类问题。  
如果复盘没有转化成新数据、新指标或新规则，它的长期价值会很有限。
