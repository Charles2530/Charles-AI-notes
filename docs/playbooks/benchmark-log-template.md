# Benchmark 记录模板

这一页适合用于记录模型、采样器、量化方案或系统配置之间的横向比较。  
它的目标是避免出现“结果看起来很漂亮，但其实不可比”的情况。

真正高质量的 benchmark 记录，不只是留下一张表，而是要回答四个问题：

1. 这次比较到底在比什么？
2. 比较是否公平？
3. 结果差异来自哪里？
4. 这些结果是否足以支持工程决策？

如果这四个问题答不清，再漂亮的图表都很难长期复用。

## 1. 基本信息

```text
Benchmark 名称：
日期：
负责人：
任务类型：
数据版本：
评测脚本版本：
硬件环境：
代码提交版本：
模型权重版本：
日志链接：
```

**建议额外记录**：

- 推理框架版本
- CUDA / 驱动版本
- 是否多卡
- 依赖服务版本

因为很多 benchmark 差异并不来自模型本身，而是环境漂移。

## 2. 对比目标

**建议明确写出**：

- 这次到底在比什么
- 为什么值得比
- 想得到什么决策
- 哪些变量是固定的
- 哪些变量是允许变化的

例如：

- 比较 `Euler 20 steps` 和 `DPM-Solver++ 15 steps`
- 比较 `AWQ 4bit` 和 `GPTQ 4bit`
- 比较 `7B` 与 `14B` 在固定延迟预算下的端到端效果
- 比较不同 prompt 模板在工具调用成功率上的影响

### 一个关键提醒

不要同时改太多变量。  
如果一次 benchmark 同时换了：

- 模型
- prompt
- sampling 参数
- 量化方式
- serving 后端

最后很难知道收益到底来自哪里。

## 3. 对比对象

```text
对象 A：
对象 B：
对象 C：
```

**每个对象建议至少记录**：

- 模型版本
- checkpoint 或发布日期
- 推理配置
- 提示词模板
- 量化方式
- 系统开关
- 外部工具配置

```text
对象名称：

```

## 4. 公平性检查

**建议明确标注是否一致**：

- 数据集
- prompt 模板
- 随机种子
- 最大输出长度
- 硬件
- 评测脚本
- 超时设置
- 工具链依赖

如果不一致，必须写清楚原因。

### 为什么这一步这么重要

很多“惊人提升”最后都会被发现来自：

- prompt 更长
- 最大输出上限更高
- 样本过滤方式不同
- timeout 更宽松
- 某些失败样本被 silently 跳过

所以公平性检查必须像实验条件表一样被严肃记录。

## 5. 指标设计：至少拆成三类

按三类拆开最稳。

### 5.1 质量指标

- accuracy
- F1
- success rate
- FID
- pass@k
- CLIP score
- task completion rate

### 5.2 系统指标

- latency
- throughput
- TTFT
- TPOT
- memory
- GPU 利用率
- token/s

### 5.3 稳定性指标

- P95 / P99
- 方差
- 失败率
- 重试率
- 超时率

如果是代理或机器人任务，还建议额外记录：

- 单步成功率
- 中途偏航率
- 人工接管率

## 6. 为什么不能只看均值

设某项指标 \(m_i\) 在样本 \(i\) 上取值，总平均值为：

\[
\bar{m} = \frac{1}{n}\sum_{i=1}^n m_i.
\]

这个均值虽然简单，但会掩盖很多事实，例如：

- 少量极端失败样本
- 长尾场景崩坏
- 某类任务显著退化

因此 benchmark 记录至少应补：

- 分位数
- 方差或标准差
- 分桶结果

例如对输入长度分桶、对任务类型分桶、对场景难度分桶。

## 7. 结果表模板

```text
| 对象 | 质量 | 延迟 | 吞吐 | 显存 | 稳定性 | 备注 |
| ---- | ---- | ---- | ---- | ---- | ------ | ---- |
| A    |      |      |      |      |        |      |
| B    |      |      |      |      |        |      |
| C    |      |      |      |      |        |      |
```

### 更推荐再加两张子表

1. 按任务类型分桶结果
2. 按输入长度或上下文长度分桶结果

因为很多模型只是在平均分上相近，但在长样本或复杂任务上完全不同。

## 8. 样本级分析

**建议至少保留**：

- 代表性成功样本
- 代表性失败样本
- 最容易混淆的边界样本
- 与预期差异最大的样本

总分只能告诉你谁大致更强，样本分析才能告诉你为什么。

### 一个生动例子

假设两个文档 VLM 总体字段准确率只差 `0.8%`。  
**但样本分析发现**：

- 模型 A 在普通打印票据上很好
- 模型 B 在盖章遮挡和歪斜扫描件上明显更稳

如果你的真实业务刚好大量包含扫描件，那么“只差 0.8%”这个结论就会非常误导。

## 9. 分桶分析建议

为了让 benchmark 更能指导工程，建议至少考虑以下分桶：

- 短输入 / 长输入
- 简单任务 / 复杂任务
- 常见样本 / 长尾样本
- 无工具 / 多工具
- 单轮 / 多轮
- 低分辨率 / 高分辨率

**对每个分桶记录**：

- 质量
- 延迟
- 失败类型

这样才能判断某方案是否只是“平均数更好”，还是在关键业务桶里真正更好。

## 10. 结论部分建议固定回答

**建议固定回答**：

1. 哪个方案在什么条件下最好？
2. 哪个方案最稳定？
3. 哪个方案最划算？
4. 当前结果是否足够支持工程决策？
5. 如果要上线，最需要补哪类测试？
6. 这次 benchmark 最大的不确定性是什么？

## 12. 一句经验

一个好的 benchmark 记录，不是表格做得漂亮，而是后来的人一眼就能看出：

- 这次比较到底公不公平
- 这个结论到底值不值得信
- 它是否能真正支持下一步工程决策
