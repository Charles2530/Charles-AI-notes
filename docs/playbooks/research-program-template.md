# 研究计划模板

本模板用于组织一个中长期研究主题，适合扩散模型、VLM、VLA、世界模型、量化、训练系统、推理系统等方向。它不是简单的周报格式，而是一份帮助团队明确问题、拆分假设、安排实验、控制风险和沉淀结论的研究程序模板。使用时建议每个主题单独建一份，按版本迭代。

## 2. 问题定义

请用 3 到 8 句话明确回答以下问题：

1. 你要解决的核心问题是什么？
2. 当前系统或方法为什么不够好？
3. 问题属于能力缺失、成本过高、时延过大、稳定性不足，还是评测盲区？
4. 若问题解决，最直接的收益是什么？

**建议写法**：

`现状`：
现有系统在 `场景 A/B/C` 中表现稳定，但在 `长尾条件` 下出现 `错误类型`。当前瓶颈主要来自 `模块或假设`，导致 `业务后果`。

`目标`：
希望在不超过 `预算约束` 的条件下，将 `关键指标` 从 `当前值` 提升到 `目标值`，并验证该提升主要来自 `核心机制`。

## 3. 假设树

把大问题拆成可验证假设，每条假设应尽量可证伪。

**示例结构**：

1. `H1`：增加 `数据或结构` 能提升 `指标`，因为 `机制解释`。
2. `H2`：当前退化主要由 `失效模式` 引起，而不是 `其他因素`。
3. `H3`：在固定成本下，`方法 A` 比 `方法 B` 更优。

建议同时记录每条假设的风险等级：

1. **高风险**：前提可能不成立，实验成本高。
2. **中风险**：可快速验证，但收益不确定。
3. **低风险**：大概率有效，主要是工程实现问题。

## 4. 指标定义

不要只写一个总指标。建议至少拆成四层：

1. 训练层指标
2. 离线任务指标
3. 长尾 / 安全 / 稳定性指标
4. 成本 / 时延 / 资源指标

**模板**：

`主指标`：
例如 `Recall@5`、`最终任务成功率`、`P95 TTFT`、`闭环碰撞率`。

`次指标`：
例如 `恢复成功率`、`长尾桶准确率`、`单位 token 成本`。

`拒绝指标`：
哪些指标一旦退化就直接否决方案，例如 `财务数字错误率`、`安全事件数`、`P99 超时率`。

## 7. 实验矩阵

**建议把实验分成三层**：

### 7.1 验证性实验

**目标**：最低成本验证核心假设是否成立。

例如：

1. 小规模数据子集实验
2. 低分辨率或短上下文快速试验
3. 单模块离线对照

### 7.2 扩展性实验

**目标**：验证是否能扩展到目标规模、目标硬件、目标任务。

例如：

1. 更长上下文
2. 更复杂场景
3. 多模型 / 多节点部署

### 7.3 稳定性与边界实验

**目标**：测试长尾、失败恢复、分布偏移和异常输入。

## 9. 阶段里程碑

不要只设一个终点。建议按阶段定义：

1. `M1`：核心假设最低成本验证通过
2. `M2`：在标准集上达到基线以上
3. `M3`：在长尾集与成本约束下仍成立
4. `M4`：具备上线 / 集成条件

每个里程碑都写清楚通过条件和失败后的分支动作。

## 10. 风险清单

**建议至少列**：

1. **技术风险**：假设不成立、训练不稳定、推理过慢
2. **数据风险**：污染、长尾不足、标注不一致
3. **系统风险**：runtime 不支持、显存不足、难以部署

## 11. 结果记录规范

**建议每次实验至少记录**：

1. 代码版本
2. 配置版本
3. 数据版本
4. 主要超参
5. 主指标与分桶指标
6. 失败样例
7. 成本与时延

**尤其要记录**：

1. 为什么做这个实验
2. 结果支持或反驳了哪条假设
3. 下一步如何调整

## 12. 结论模版

每个阶段结束后，建议用以下结构写结论：

`结论摘要`：
本轮结果表明 `方法 / 假设` 在 `哪些条件` 下有效，在 `哪些条件` 下无效。

`最关键证据`：
1. 指标变化
2. 分桶变化
3. 成本变化
4. 失败案例

`保留问题`：
尚未回答的问题有哪些？

`下一步`：
继续推进、缩小范围、转方向，还是终止？

## 13. 一个示例：VLA 闭环恢复研究

如果主题是“提升 VLA 在真实仓库拣选中的恢复能力”，这个模板可以具体化为：

1. **问题**：首次成功率尚可，但失败后无法恢复，导致实际吞吐低。
2. **假设**：显式失败检测 + 恢复策略比单纯加大模型更有效。
3. **指标**：首次成功率、最终成功率、平均重试次数、安全事件率。
4. **实验**：先离线回放失败轨迹，再小规模实机灰度。
5. **风险**：失败检测误报导致过度保守。

## 14. 使用建议

1. 一份模板只服务一个核心问题，不要把多个主题混在一起。
2. 写“为何做”和“如何判失败”比写实验数量更重要。
3. 定期删掉已经被证伪的假设，保持研究计划可读。
4. 让数据、系统、算法和产品相关方都能看懂。

## 15. 小结

好的研究计划不是把可能的实验都列出来，而是把问题、假设、评测、预算和止损条件组织成一个可执行程序。模板的价值，不在于格式本身，而在于强迫团队持续回答：我们究竟在验证什么、为什么值得继续投入、以及什么证据足以改变方向。
