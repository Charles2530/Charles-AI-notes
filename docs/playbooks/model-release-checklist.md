# 模型发布检查清单

模型训练完成并不等于可以发布。真正的发布意味着：你知道它在什么数据上训练、在哪些场景有效、哪些风险已经验证、哪些风险还未覆盖、遇到问题如何回滚、线上应该监控什么，以及这一版本为何值得取代上一版本。本清单适用于基础模型、VLM、VLA、世界模型、量化模型和推理服务版本发布。

## 1. 版本身份

**发布前先记录最基本信息**：

1. 模型名称与版本号
2. 代码 commit
3. 配置版本
4. 数据版本
5. 训练时间区间
6. 负责人

如果这一步都不完整，后续任何问题都难追踪。

## 2. 目标与变更说明

**明确本次发布解决什么问题**：

1. 质量提升
2. 成本降低
3. 时延下降
4. 新模态支持
5. 新数据回流
6. 风险修复

**并写清**：

1. 与上版相比改了什么；
2. 哪些变化最可能带来副作用；
3. 这次发布的判断依据是什么。

## 3. 训练与数据检查

**发布前必须确认**：

1. 数据来源合规且可追溯；
2. 无明显评测污染；
3. 去重与清洗规则记录完整；
4. 关键长尾集被覆盖；
5. 若有合成数据，其占比和用途清楚。

对于多模态和具身系统，还要确认：

1. 模态对齐关系是否稳定；
2. 轨迹和指令是否错配；
3. OCR / grounding 等外部组件版本是否锁定。

## 4. 离线评测门禁

**至少检查**：

1. 主指标是否达到发布阈值；
2. 长尾桶是否无明显回归；
3. 拒绝指标是否全部通过；
4. 与上版的差异是否超出噪声区间。

**建议结果汇总成三类**：

1. `通过`
2. `通过但有观察项`
3. `不通过`

## 5. 成本与时延评测

模型发布不仅是质量问题。需明确：

1. 平均成本
2. P95 / P99 延迟
3. 显存与并发占用
4. 是否需要新硬件或新 runtime
5. 是否影响其他模型共享资源

若新版本更强但推理成本显著上升，要明确上线范围和路由策略，而不是默认全量替换。

## 6. 安全与风险检查

**按系统类型关注**：

### 6.1 文本 / 多模态模型

1. 幻觉风险
2. 数字与事实错误
3. 敏感内容处理
4. 引用与证据可靠性

### 6.2 屏幕 agent / 工具系统

1. 错误点击或误操作
2. 权限边界
3. 死循环
4. 工具参数错误

### 6.3 机器人 / VLA / 世界模型

1. 危险动作
2. 恢复失败
3. 不确定性时的回退
4. 人工接管链路

## 7. 回归检查

**发布前必须跑**：

1. smoke set
2. standard set
3. long-tail set
4. 关键业务集

**并记录**：

1. 新增错误类型
2. 已修复错误是否复现
3. 是否出现过去不存在的奇怪失败

## 8. 可观测性配置

模型上线前，必须知道要监控什么。建议确认：

1. 关键质量代理指标
2. 时延与错误率指标
3. 模型路由与版本标签
4. 日志采样与回放机制
5. 线上报警阈值

如果没有观测方案，发布后等于盲飞。

## 9. 灰度与回滚方案

**明确**：

1. 首次灰度比例
2. 灰度观察窗口
3. 触发回滚的指标条件
4. 回滚操作步骤
5. 回滚后如何保留日志做复盘

灰度不是形式流程，而是给系统留一条后路。

## 10. 文档与交接

**发布前还应完成**：

1. 版本说明
2. 评测报告
3. 变更摘要
4. 风险备注
5. 运维与产品知会

特别是多团队协作时，发布失败往往不是模型问题，而是信息没同步清楚。

## 11. 特殊场景补充项

### 11.1 量化版本

1. 是否仅特定硬件可用；
2. 长上下文与长尾任务是否专项验证；
3. 是否保留高精度回退路径。

### 11.2 多模型路由版本

1. 路由器是否单独评测；
2. 新路由是否改变整体流量分布；
3. 是否会让某些模型过载。

### 11.3 机器人部署版本

1. 实机小流量测试是否通过；
2. 急停、限位和人工接管是否验证；
3. 新版本是否改变 ODD 边界。

## 12. 发布结论模版

**建议每次发布前写一段简明结论**：

`本版本在 [关键能力] 上相较上一版提升 / 持平 / 退化 [幅度]，在 [长尾或风险项] 上通过 / 未通过 检查。推理成本 [变化]，计划以 [灰度策略] 发布，若出现 [触发条件] 将立即回滚。`

这段文字会逼迫负责人把真正重要的信息说清楚。

## 13. 一个例子：文档 VLM 发布

若新版本主要提升复杂表格解析，应检查：

1. 表格长尾桶是否稳定提升；
2. 普通页问答是否无回归；
3. 关键财务数字错误率是否保持安全阈值内；
4. OCR 依赖链路是否版本兼容；
5. 线上是否能回放新错误样本。

## 14. 一个例子：VLA 恢复版本发布

若新版本加入失败恢复模块，应重点检查：

1. 最终成功率是否提升；
2. 首次成功率是否明显退化；
3. 平均恢复时长是否可接受；
4. 是否增加危险尝试动作；
5. 线下和实机灰度结果是否一致。

## 15. 小结

模型发布检查清单的核心，不是增加流程负担，而是让“是否值得上线”从个人感觉变成有证据、有门禁、有回滚的工程决策。模型越复杂、链路越长，这份清单就越不是形式，而是降低事故与回归风险的最后防线。
