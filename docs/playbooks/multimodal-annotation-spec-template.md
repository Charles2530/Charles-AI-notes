# 多模态标注规范模板

这一页适合在构建 VLM、文档理解、屏幕代理或多模态检索数据集时，统一标注规范。目标是减少“同一任务不同标注员理解不一致”带来的系统噪声。

## 1. 项目基本信息

```text
数据项目名称：
任务类型：
负责人：
标注版本：
适用数据范围：
```

**建议再补**：

```text
标注平台：
交付格式：
质检负责人：
更新时间：
```

## 2. 标注目标

**明确写出**：

- 要标什么
- 为什么标
- 下游模型会如何使用这些标签

例如：

- 文档字段抽取
- 图文匹配标签
- 屏幕元素定位
- 多轮视觉问答答案

### 标注目标要尽量写到可评测

不要只写“给模型训练用”，更建议明确：

1. 它影响哪个模型头；
2. 它对应哪个评测指标；
3. 如果这个标签错了，会影响哪类业务结果。

这样标注、算法和评测团队更容易保持一致。

## 3. 输入定义

**记录输入形式**：

- 单图
- 多图
- 视频
- 文档页
- OCR 文本
- 对话上下文

### 还应补充输入上下文规则

例如：

1. 多页文档是否允许跨页理解；
2. 视频是按帧、片段还是整段标注；
3. UI 标注是否保留 DOM / accessibility tree；
4. OCR 文本以原始结果为准还是以人工修正版为准。

这些规则对标签一致性影响极大。

## 4. 输出定义

明确标签结构，例如：

\[
y = \{c, b, t, r\}
\]

**其中可能包括**：

- 类别 \(c\)
- 框 \(b\)
- 文本答案 \(t\)
- 关系标签 \(r\)

### 输出定义最好补字段说明

**建议给每个字段写清**：

1. 数据类型；
2. 是否必填；
3. 空值何时允许；
4. 单位、大小写、时间格式是否标准化；
5. 是否允许多标签或多值。

## 5. 标注规则

**建议逐条写**：

- 正例定义
- 负例定义
- 边界例如何处理
- 遮挡与模糊如何处理
- 不确定样本如何上报

### 建议增加“禁止猜测”规则

多模态标注中最危险的噪声之一，是标注员在看不清、读不准、关系不明确时凭常识补全。建议显式规定：

1. 看不清时如何标；
2. OCR 与视觉冲突时以什么为准；
3. 图文不一致时如何处理；
4. 能否进入待仲裁池，而不是强行给结论。

## 6. 一致性要求

**建议规定**：

- 文本大小写是否统一
- 单位和时间格式是否统一
- 框选边界标准
- 多实体排序规则

### 一致性要求最好配示例

例如：

1. 日期统一为 `YYYY-MM-DD`；
2. 金额统一保留两位小数；
3. 多实体按从左到右、从上到下排序；
4. 框选边界是否包含阴影、边框和空白区域。

纯文本规则远不如带示例有效。

## 7. 质检流程

**建议至少有**：

- 抽样复核
- 双标一致性检查
- 冲突样本仲裁

### 推荐增加质量门槛

例如：

1. 一致性低于阈值要返工；
2. 高风险字段必须二次复核；
3. 难例进入专家仲裁；
4. 每一轮标注都要更新错因库。

## 8. 常见错误样例

**最好专门列**：

- 易混淆正负例
- OCR 错字导致的误判
- 遮挡和部分可见实体
- 多模态关系错配

### 建议把错误样例做成错因库

**每个样例最好包含**：

1. 原样本；
2. 错误标签；
3. 正确标签；
4. 出错原因；
5. 以后如何避免。

这种结构比一句“请注意一致性”有用得多。

## 9. 标注输出格式

**明确**：

- JSON / CSV / 数据库表结构
- 字段名
- 必填字段
- 空值规则

### 最好附一个最小 JSON 示例

这样下游训练、清洗和评测团队能更快接入，也能减少字段语义误解。

## 10. 一个具体例子

如果你在做屏幕代理元素标注，一个真正有用的规范不应只写“标记按钮位置”，还应写清：

1. 弹窗遮挡时是标可见部分还是完整元素；
2. 同名按钮多个实例如何区分；
3. hover、禁用、选中等状态是否单独标；
4. 图标按钮和文案按钮是否共用类别体系。

这些规则不清，后续 grounding 模型通常会非常不稳。

## 11. 一句经验

多模态标注最怕“标签看起来很多，但语义不一致”。真正有价值的规范，是让不同标注员对边界例也尽量给出一致判断。
