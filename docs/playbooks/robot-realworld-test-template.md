# 机器人实机测试模板

这一页适合在机器人策略进入真实环境测试前后使用。目标是把“实机测试”从拍视频 demo，变成一套可复现、可比较、可回流的数据流程。

## 1. 基本信息

```text
测试项目：
日期：
负责人：
机器人平台：
软件版本：
模型/策略版本：
```

**建议再补**：

```text
机械臂 / 底盘配置：
末端执行器：
控制频率：
安全负责人：
日志存储位置：
```

## 2. 测试目标

建议明确写成可判断的句子，例如：

- 验证抓取策略在 20 个新物体上的成功率
- 验证 VLA 在 5 种新光照条件下是否稳定
- 验证新安全过滤器是否能减少抖动和急停

### 目标应尽量可证伪

不要只写“看看是否更好”，更建议写清：

1. 要提升哪个指标；
2. 在什么条件下提升；
3. 允许什么副作用；
4. 哪些结果算测试失败。

## 3. 场景配置

**记录**：

- 环境布局
- 光照
- 物体种类
- 相机位置
- 是否有人类干扰
- 是否有背景噪声或遮挡

### 建议增加环境版本化

若条件允许，最好给每套场景一个稳定 ID，并记录：

1. 桌面 / 工位编号；
2. 物体集合版本；
3. 相机外参；
4. 背景干扰物；
5. 是否为长期固定或临时变化场景。

## 4. 测试任务列表

```text
任务 1：
任务 2：
任务 3：
```

**每个任务最好写清**：

- 成功定义
- 失败定义
- 最大允许时长

### 对长任务建议补阶段定义

**比如“开抽屉取物”可拆成**：

1. 接近把手；
2. 成功打开；
3. 成功识别目标物；
4. 成功抓取；
5. 成功放置。

这样失败归因会清楚很多。

## 5. 安全检查清单

**实机测试前建议逐项确认**：

- 急停可用
- 关节限位正常
- 安全工作空间正确
- 人机隔离措施到位
- 日志采集正常

### 建议增加三项

1. 人工接管链路已演练；
2. 异常停止后可安全复位；
3. 现场参与人员已知晓风险和停止手势。

## 6. 记录指标

**建议至少记录**：

- 成功率
- 平均完成时间
- 人工接管率
- 急停次数
- 失败类型分布

### 若是多步骤任务，建议额外记录

1. 首次成功率；
2. 最终成功率；
3. 平均重试次数；
4. 恢复成功率；
5. 平均恢复耗时。

这些指标比单个总成功率更接近真实可部署性。

## 7. 失败分类

**建议固定分类**：

- 感知失败
- 动作执行失败
- 路径规划失败
- 接触失败
- 安全触发
- 外部环境干扰

### 失败分类最好能直接回流

**即每一类失败最好都能映射到**：

1. 训练回流数据；
2. benchmark hard set；
3. 系统报警或安全规则；
4. 控制器、感知或规划模块责任归属。

## 8. 样本回流

**明确记录**：

- 哪些失败样本进入回流池
- 是否需要人工标注
- 是否进入 benchmark 或 hard set

### 建议增加回流优先级

并不是所有失败都值得立即处理。可以按：

1. 安全相关；
2. 高频复现；
3. 长尾高价值；
4. 低优先级偶发

四级做排序。

## 9. 结论

**建议固定回答**：

1. 当前策略最稳的是哪类任务？
2. 最主要失败来自哪一层？
3. 下一轮更该改模型、控制器，还是数据？

### 推荐的结论模板

```text
本轮总体判断：
可进入下一阶段的能力：
暂不建议继续放量测试的场景：
最优先改动方向：
需要复测的条件：
```

## 10. 一个示例

如果一轮家庭机器人实机测试中发现：

1. 常规杯子抓取成功率高；
2. 透明杯与反光餐具错误多；
3. 被人打断后恢复率很低；
4. 安全触发主要发生在靠近台面边缘时；

那下一轮最该优先做的，通常不是“继续录成功视频”，而是：

1. 采透明与反光物体失败数据；
2. 建被打断场景恢复 benchmark；
3. 校正边缘接近时的控制与安全阈值。

## 11. 一句经验

实机测试最怕“只记成功视频，不记失败结构”。真正能推动系统变强的，往往是那些被分类清楚、可回流、可复现的失败样本。
