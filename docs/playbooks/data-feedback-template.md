# 数据回流模板

这一页适合在系统上线后，把失败样本、长尾样本和新场景样本结构化回流到训练链路中。目标是避免“收了一堆日志，但没有形成可训练资产”。

## 1. 回流任务基本信息

```text
回流批次：
时间范围：
负责人：
来源系统：
目标模型/任务：
```

**建议再补**：

```text
回流目标（补训练 / 补评测 / 补监控）：
是否涉及人工标注：
是否涉及敏感数据：
版本归属：
```

## 2. 回流触发条件

建议明确写出触发规则，例如：

- 任务失败
- 人工接管
- 低置信输出
- 工具调用失败
- 机器人碰撞边缘事件

### 触发条件最好配优先级

不是所有触发样本都一样值钱。建议按优先级记录：

1. `高优先级`：直接影响用户结果、安全或高价值业务；
2. `中优先级`：暴露长尾能力边界；
3. `低优先级`：主要用于监控分布变化。

这样可以避免回流池被大量“有日志但没价值”的样本淹没。

## 3. 样本筛选逻辑

如果存在自动筛选，可写成：

\[
v(x) = \alpha \cdot \text{failure}(x) + \beta \cdot \text{novelty}(x) + \gamma \cdot \text{uncertainty}(x)
\]

并说明高 \(v(x)\) 样本如何进入标注或复查池。

### 如何理解 novelty

很多系统只看 failure，但 novelty 同样重要。比如：

1. 模型这次答对了，但场景是新的；
2. 模态组合从未出现过；
3. 工具链走出了新路径；
4. 机器人第一次见到某种新物体或新布局。

这类样本现在未必失败，但常常是未来回归的前兆。

## 4. 样本分类

**建议至少按以下类别拆分**：

- 明确失败样本
- 边界样本
- 新场景样本
- 伪成功样本

### 推荐增加的两类

1. `高风险正确样本`：结果暂时对，但过程很脆弱；
2. `标注冲突样本`：不同 judge 或人工意见不一致。

这两类样本对完善评测与 judge 模型非常重要。

## 5. 标注与加工

**记录**：

- 需要补什么标签
- 是否需要二次清洗
- 是否要脱敏
- 是否要做难例重权重采样

### 标注前建议先做两件事

1. 去掉明显重复和低价值日志；
2. 先做失败分类，再决定进入哪条标注流程。

这样可以显著降低标注成本，并提高后续训练价值。

## 6. 回灌计划

**明确写**：

- 回灌到哪个数据集版本
- 是否单独做 hard set
- 是否进入 benchmark
- 是否进入下一轮训练或评测

### 回灌不应只有一个出口

成熟回流系统通常至少有三条出口：

1. 进入训练集；
2. 进入 benchmark / hard set；
3. 进入监控和报警规则。

并不是所有样本都适合训练。有些样本更适合专门盯回归，而不是直接喂给模型。

## 7. 结论

**最终应能回答**：

- 这批回流数据最大的价值是什么
- 它更像补数据、补评测，还是补监控
- 下一轮模型最该借它改什么

### 推荐增加的结论字段

```text
回流价值判断：
建议进入的资产层：
预计下轮收益点：
本轮不处理的样本原因：
```

## 8. 一个实际例子

假设你在做企业合规问答 agent，本周回流了 1200 条异常请求。经过筛选后，可能会发现：

1. 300 条是法规版本冲突；
2. 180 条是长文档上下文装配过长导致关键信息被淹没；
3. 90 条是工具超时导致的伪失败；
4. 40 条是高风险财务数字解释错误；
5. 其余大多是重复或低价值样本。

**这时最合理的回灌可能是**：

1. 财务数字错误进入 hard set；
2. 版本冲突样本进入检索一致性专项集；
3. 工具超时样本进入系统监控，而不是模型训练。

如果没有结构化回流模板，这几类样本很容易被混在一起，最后既浪费标注，又没有真正改变模型边界。
