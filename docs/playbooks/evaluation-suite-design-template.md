# 评测体系设计模板

本模板用于设计一套长期可维护的评测体系，适合模型、检索、推理系统、VLM/VLA、世界模型、量化与具身场景。评测体系不是若干 benchmark 的简单拼接，而是一套帮助团队持续发现回归、识别长尾问题、支持上线决策的结构化资产。以下模板强调“分层、分桶、可追溯、可扩展”。

## 1. 评测对象

`系统名称`：

`评测版本`：

`覆盖范围`：
例如 `基础问答`、`图文检索`、`屏幕代理`、`机器人抓取`、`长上下文推理`。

## 2. 评测目标

**请明确这套体系要支持什么决策**：

1. 研究迭代比较
2. 上线前质量门禁
3. 线上回归监控
4. 长尾问题发现
5. 成本与时延权衡

若不写清楚，评测很容易变成“什么都测一点，但没有一个指标真正能拍板”。

## 3. 分层结构

**建议至少分成以下四层**：

### 3.1 核心能力层

测模型最基本能力，例如分类、检索、问答、动作成功率。

### 3.2 结构与场景层

测特定结构任务，如文档表格、图表、UI 元素 grounding、长时多步骤任务。

### 3.3 鲁棒与长尾层

测模糊、遮挡、分布偏移、稀有对象、异常版式、失败恢复。

### 3.4 系统与业务层

测时延、成本、可观测性、人工接管、用户体验等。

## 4. 主指标与拒绝指标

**为每一层定义**：

`主指标`：
用于总体排序或版本比较。

`辅助指标`：
解释原因和分解变化。

`拒绝指标`：
任何版本只要超过阈值就不能上线。

**例如文档 VLM**：

1. **主指标**：字段抽取 F1
2. **辅助指标**：表格结构 IoU、数字一致性错误率
3. **拒绝指标**：财务数字严重错误率、P99 超时率

## 5. 分桶设计

平均分不够，必须事先定义桶。模板建议至少从以下角度考虑：

1. 输入长度
2. 难度等级
3. 模态类型
4. 场景类型
5. 长尾类型
6. 风险等级

例如：

1. **文档**：单页 / 多页 / 跨页表格 / 低质扫描
2. **推理**：短上下文 / 长上下文 / 工具调用 / 多轮对话
3. **机器人**：单步技能 / 多步骤任务 / 遮挡 / 恢复场景

## 6. 数据集来源

**请分别记录**：

1. 公开 benchmark
2. 自建专项集
3. 线上回流挑战集
4. 长尾失败集
5. 不可触碰 holdout

**并记录**：

1. 样本量
2. 更新频率
3. 标注责任人
4. 污染风险

## 7. 标签与判分规则

**每个评测集都应有明确判分标准**：

1. 完全匹配还是归一化匹配？
2. 多答案如何计分？
3. 图文定位如何算正确？
4. 机器人任务成功由谁判定？
5. 恢复任务是看首次成功还是最终成功？

若判分规则模糊，版本之间很难比较。

## 8. 自动评测与人工评测分工

并非所有任务都适合全自动。建议为每个子集注明：

1. 可自动评测
2. 需人工抽样复核
3. 必须人工主评

例如：

1. 数值字段抽取可自动比对；
2. 复杂图表解释需人工审读；
3. 机器人安全与舒适度通常需人工观察。

## 9. 成本与时延评测

若系统会部署，评测体系必须包含：

1. 平均成本
2. P50/P95/P99 时延
3. GPU / CPU / 内存占用
4. QPS / token 吞吐
5. 关键路径分解耗时

**模板建议记录公式**：

$$
\text{Net Utility}
=
\alpha \cdot \text{Quality}
- \beta \cdot \text{Latency}
- \gamma \cdot \text{Cost}.
$$

用于帮助版本选择。

## 10. 回归测试设计

评测体系要支持快速发现回归。建议建立：

1. **smoke set**：少量高价值样本，快速跑
2. **standard set**：日常主评估
3. **long-tail set**：周评或版本发布前跑
4. **holdout set**：关键里程碑或上线前跑

这样既保证速度，也保证覆盖。

## 11. 错误分类体系

对失败样本要有 taxonomy，而不是统一记为“错”。模板建议记录：

1. 感知错误
2. 推理错误
3. 检索错误
4. 格式错误
5. 系统错误
6. 安全错误
7. 恢复失败

每类错误都应能回流到相应团队或模块。

## 12. 版本治理

**每次评测结果必须绑定**：

1. 模型版本
2. 数据版本
3. 配置版本
4. 推理引擎版本
5. 索引 / 检索版本

否则结果不可复现。

## 13. 变更门禁规则

**建议事先写清**：

1. 何种变化允许直接发布
2. 何种变化必须人工审查
3. 何种变化必须灰度
4. 何种指标一旦下降即阻断合并

这能避免评测只停留在报告，而不真正影响决策。

## 14. 示例：屏幕代理评测体系

**可按如下设计**：

1. **核心层**：页面元素 grounding、单步点击正确率
2. **场景层**：登录、导出、筛选、报表配置
3. **长尾层**：弹窗遮挡、缩放变化、深浅色主题、旧版 UI
4. **系统层**：每步时延、重试率、人工接管率

**错误 taxonomy 可分为**：

1. 看错元素
2. 误解任务状态
3. 错误点击后未恢复
4. 工具调用失败

## 15. 示例：机器人 VLA 评测体系

**可按如下设计**：

1. **核心层**：单步技能成功率
2. **场景层**：多步骤任务、语言条件任务
3. **长尾层**：遮挡、反光、透明物体、抓取失败恢复
4. **系统层**：执行时间、安全事件、接管率、恢复耗时

**并区分**：

1. 首次成功率
2. 最终成功率
3. 恢复成功率
4. 危险动作比例

## 16. 维护建议

1. 每月复盘一次哪些集合已被“刷穿”。
2. 定期从线上回流新难例。
3. 长尾集宁可小而精，也不要大而杂。
4. 保持至少一部分 holdout 长期不可见。
5. 评测报告要面向决策，而不是只堆表格。

## 17. 输出格式建议

**每次评测报告至少包含**：

1. 摘要结论
2. 主指标变化
3. 分桶变化
4. 成本与时延变化
5. 关键回归样例
6. 发布建议

## 18. 小结

评测体系设计的核心，是把“模型好不好”拆成可持续回答的一组问题：基础能力是否稳定、长尾是否可控、系统成本是否合理、上线风险是否可接受。模板的价值在于迫使团队在版本迭代之前就想清楚这些问题，而不是等问题在线上爆发后再临时补救。
