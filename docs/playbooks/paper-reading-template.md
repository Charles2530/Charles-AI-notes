# 论文阅读模板

这一页适合用于系统做论文笔记。目标不是重复摘要，而是把一篇论文放回技术主线里，回答它真正改变了哪一层、为什么值得投入时间、以及它对你的项目意味着什么。

## 1. 基本信息

```text
标题：
作者：
年份：
领域：
链接：
```

**建议再补**：

```text
阅读日期：
阅读目的（入门 / 复现 / 选型 / 路线判断）：
与当前项目关系：
```

## 2. 问题定义

**建议用一句明确的话写出**：

- 它解决的旧问题是什么
- 为什么旧方法不够

### 建议再补两个问题

1. 这个问题在方法链里属于哪一层瓶颈？
2. 如果不解决它，真实系统会受什么限制？

因为很多论文看起来都在“提精度”或“提速度”，但真正有用的判断是：它到底在解哪个卡点。

## 3. 它改了哪一层

**推荐从下面几类中选**：

- 训练目标
- 参数化
- 推理求解器
- 生成路径
- 系统结构
- 数据组织
- 部署策略

这一步能帮你避免“看了很多论文，但不知道它们到底是不是一类工作”。

### 如果它同时改了多层怎么办

**建议分成**：

1. `主贡献层`
2. `次贡献层`

例如某篇论文主贡献是求解器，次贡献是训练重加权；或者主贡献是数据引擎，次贡献是 judge 模型。这种写法比笼统说“它做了很多事”更清楚。

## 4. 核心方法

**建议只记**：

- 最关键的公式
- 最关键的模块
- 和上一代方法的最小差异

### 更实用的记录方式

**可以按下面结构写**：

1. `输入`：它吃什么；
2. `输出`：它产什么；
3. `关键改动`：比上一代多了哪个关键部件；
4. `训练变化`：损失、数据、日程变了什么；
5. `推理变化`：速度、成本、部署方式变了什么。

这能让你几个月后依然快速恢复理解。

## 5. 最值得带走的直觉

这部分用自然语言写。例如：

- “它真正重要的不是新损失，而是把采样统一到 ODE solver 视角。”
- “它的价值不在更大模型，而在更合理的数据组织。”

### 为什么这一步最重要

因为一篇论文真正值钱的，常常不是公式本身，而是它让你换了一个看问题的角度。能把这层直觉写出来，才说明你真正吸收了这篇工作。

## 6. 实验与边界

**建议回答**：

- 它在哪些 setting 下强
- 它在哪些 setting 下不稳
- 它有没有真的解决论文一开始提出的问题

### 建议额外看四件事

1. 基线是否公平；
2. 预算是否对齐；
3. 提升主要发生在哪些桶；
4. 是否依赖某些强工程条件。

这四个问题常常决定一篇论文是“有意思”，还是“可落地”。

## 7. 它在主线中的位置

用一句话定位，例如：

- “它是 DDPM 到 solver 时代之间的桥梁。”
- “它把多模态对齐推进到可服务检索系统。”

### 推荐再加一项

`它依赖哪些前置工作才能成立？`

因为很多论文不是孤立突破，而是建立在前一代工作已经把问题压缩到一定程度之后才有意义。

## 8. 你的判断

**最后写**：

1. 这篇论文最重要的贡献是什么？
2. 哪部分可能被高估？
3. 如果要复现，最先做哪一块？
4. 如果要继续做后续工作，最值得补哪一点？
5. 若把它用到我的系统，最可能卡在哪一环？
6. 它更像研究增量，还是工程增量？

这六问比简单写“很有启发”更能帮助后续路线判断。

## 9. 一个示例结构

如果你在读一篇扩散加速论文，最终笔记最好能压缩成：

1. **它解决的瓶颈**：采样太慢；
2. **它改的层**：求解器；
3. **关键直觉**：把逆过程当 ODE 解；
4. **主要证据**：低步数下质量保持；
5. **局限**：强依赖教师模型或特定噪声日程；
6. **你的判断**：适合服务端预览，不一定适合最高质量离线渲染。

如果一篇笔记写不到这个压缩程度，通常说明还没真正抓到主干。

## 10. 横向联系

建议最后补一个小节，写出：

1. 它与哪两篇前作最接近；
2. 它和哪一条主线发生交叉；
3. 它之后最自然的下一步研究问题是什么。

这会帮助你逐渐建立路线图，而不是一篇篇孤立记忆。

## 11. 一句经验

好的论文笔记，不是换一种说法抄摘要，而是让几个月后的你还能快速判断：

- 它到底改了哪
- 它为什么重要
- 它有没有真解决问题
- 它值不值得放进你的方法栈里
