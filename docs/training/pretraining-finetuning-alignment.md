# 预训练、微调与对齐

今天的大模型几乎都不是“一次训练出来”的。  
**它们通常经历三层递进过程**：

1. `预训练`
2. `监督微调`
3. `偏好对齐`

这三层做的事情并不一样。  
如果把它们混成一个概念，就很容易在模型设计和问题诊断时判断失真。

## 1. 预训练：决定模型“能不能”

语言模型最常见的预训练目标是下一个 token 预测：

\[
\mathcal{L}_{\text{LM}} = - \sum_t \log p_\theta(x_t \mid x_{<t})
\]

这件事看起来简单，但它迫使模型从海量文本中学到：

- 语法
- 常识
- 事实共现
- 代码模式
- 长程依赖
- 推理痕迹

也就是说，预训练主要塑造的是**能力边界**。

### 一个直观例子：学一门外语

预训练像是长期浸泡式学习。  
它不教你怎么当客服，也不教你怎么拒答敏感请求，但它决定了你是否真的“会这门语言”，以及你懂多少世界知识。

## 2. 为什么预训练损失低，不代表下游一定强

因为 \(\mathcal{L}_{\text{LM}}\) 本质上是在优化平均 token 预测。  
**而真实任务可能关心的是**：

- 严格格式输出
- 长链推理
- 工具使用
- 安全性

这些能力与 token 级平均损失之间并非一一对应。

**因此一个很重要的判断是**：  
预训练打底，但并不直接定义交互行为。

## 3. SFT：决定模型“怎么做”

**监督微调通常是条件交叉熵**：

\[
\mathcal{L}_{\text{SFT}} = - \sum_t \log p_\theta(y_t \mid y_{<t}, x)
\]

其中 \(x\) 是输入，\(y\) 是高质量目标回答。  
SFT 的作用通常不是重建整个世界知识，而是：

- 让模型学会遵循指令
- 学会按任务格式回答
- 学会在给定上下文中如何组织输出

### 一个例子：客服机器人

预训练让它“会说中文”，但未必会：

- 先确认问题再回答
- 给出工单编号
- 避免情绪化表述

**SFT 则会教它**：

- 开场怎么说
- 遇到退款问题怎么归类
- 输出必须遵循什么模板

## 4. SFT 的本质：行为克隆

从训练角度看，SFT 非常像“语言行为克隆”。  
你给模型看一堆理想示范，它学习复现这种示范分布。

因此 SFT 的收益高度依赖数据质量。  
**如果示范数据本身**：

- 模板僵硬
- 逻辑不严谨
- 过于冗长
- 含有错误拒答或过度保守

模型会一并学进去。

## 5. 偏好对齐：决定模型“倾向怎么选”

即使经过 SFT，模型仍可能在多个合理答案之间做出不符合人类偏好的选择。  
因此对齐阶段常引入偏好数据，例如一对回答：

\[
(y^+, y^-)
\]

其中 \(y^+\) 是更优回答，\(y^-\) 是较差回答。

以 DPO 为例，常见目标写成：

\[
\mathcal{L}_{\text{DPO}} = - \log \sigma\left(\beta \log \frac{\pi_\theta(y^+ \mid x)}{\pi_{\text{ref}}(y^+ \mid x)} - \beta \log \frac{\pi_\theta(y^- \mid x)}{\pi_{\text{ref}}(y^- \mid x)} \right)
\]

**它的直觉是**：

- 提高优选回答相对参考模型的概率
- 压低劣选回答相对参考模型的概率

因此，对齐阶段主要塑造的是：

- 风格
- 风险偏好
- 拒答边界
- 冗长与简洁倾向

## 6. RLHF、DPO、RLAIF 的区别直觉

虽然实现细节不同，但它们都在回答同一个问题：  
模型已经会做题了，怎么让它**更像人类偏好的样子**？

### RLHF

通过奖励模型 + 强化学习更新策略。  
灵活，但训练更复杂。

### DPO

直接从偏好对做优化，不显式训练在线 RL 环节。  
工程上更简单。

### RLAIF

偏好来源不一定是人，也可能是更强模型或规则系统。

## 7. 三层训练各自塑造什么

**可以粗略记成**：

- `预训练`：塑造知识与表示能力
- `SFT`：塑造任务行为
- `对齐`：塑造偏好和风险边界

### 一个更生动的比喻

**把模型看作一个新员工**：

- 预训练像多年教育背景和阅读积累
- SFT 像岗位培训
- 对齐像企业文化、服务规范和风险制度

如果这三层任何一层出问题，表现都不同：

- **预训练弱**：能力上限低
- **SFT 弱**：不会按要求做事
- **对齐弱**：会做事，但做得不让人放心

## 8. 为什么很多问题不能靠对齐补救

如果模型本身没有能力，对齐阶段很难凭空创造出来。  
例如：

- 没有数学推导能力
- 没有代码修复能力
- 没有长上下文建模能力

这类能力短板更可能源自预训练不足或数据分布不对，而不是“再做一点 DPO”就能解决。

## 9. 反过来，为什么很多体验问题又主要是对齐问题

如果模型基础能力很强，但用户抱怨：

- 太啰嗦
- 太爱说教
- 拒答边界奇怪
- 容易自信但不确定

**那这些问题通常更像**：

- SFT 风格问题
- 偏好对齐问题

而不是预训练问题。

## 10. 三个具体场景

## 10.1 客服机器人

预训练决定它是否理解自然语言和常见业务常识。  
SFT 决定它是否按客服流程和模板输出。  
DPO 决定它是否更礼貌、更稳妥、减少胡说。

## 10.2 代码助手

预训练决定它是否知道语法、库用法和常见设计模式。  
SFT 决定它是否会按“解释 + 修复 + 代码块”结构回答。  
对齐决定它是更保守还是更激进地建议改动。

## 10.3 医疗问答

预训练决定底座是否理解一般医学术语。  
SFT 决定是否按“症状分析 + 风险提示 + 就医建议”组织输出。  
对齐则决定它是否在不确定时更愿意提醒就医，而不是给出冒进建议。

## 11. 常见失败模式

## 11.1 预训练不足，却试图靠 SFT 硬拉

**结果往往是**：

- 模板学会了
- 深层能力没有提升

## 11.2 SFT 数据太模板化

模型会变得很会按套路说话，但开放任务泛化变差。

## 11.3 对齐过强，压制能力

如果拒答或保守偏好过重，模型可能：

- 能答也不答
- 答案过于空泛
- 工具使用变保守

## 11.4 三阶段目标相互冲突

例如：

- 预训练鼓励信息尽量完整
- SFT 鼓励结构化回答
- 对齐鼓励保守和简短

若平衡不好，模型风格会显得别扭甚至不稳定。

## 12. 工程判断

当你要诊断一个模型问题时，最好先判断它属于哪一层：

### 能力边界问题

优先回看预训练数据和模型规模。

### 任务执行问题

优先看 SFT 数据与格式设计。

### 风格、安全和用户体验问题

优先看偏好对齐。

这是一个很实用的排障框架。

## 13. 总结

预训练、微调与对齐并不是重复劳动，而是三种不同的塑形机制。  
预训练让模型“会”，SFT 让模型“按要求做”，对齐让模型“做得更像人希望的那样”。  
理解这条分工线，才能真正知道该把数据和算力花在哪一层，而不是把所有问题都模糊地归结为“再训一下模型”。

## 14. 三阶段其实对应三种不同的监督信号

若把训练样本抽象成 \((x,y,\pi)\)，可以粗略理解为：

1. 预训练主要用无标签序列分布 \(p(x)\)；
2. SFT 主要用条件监督 \(p(y\mid x)\)；
3. 偏好对齐主要用相对偏好 \(\pi(y_a \succ y_b \mid x)\)。

这三类监督信号提供的信息量并不一样：

1. 预训练告诉模型“世界里什么模式常出现”；
2. SFT 告诉模型“在给定任务里标准答案长什么样”；
3. 偏好对齐告诉模型“多个可行答案之间，人更喜欢哪一个”。

### 14.1 为什么这三类信号不能简单互相替代

1. 只有预训练，没有 SFT，模型常常会“会但不会按要求做”；
2. 只有 SFT，没有强预训练，模型只会模仿表层模板；
3. 只有偏好对齐，没有足够好的 SFT，优化对象会很不稳定；
4. 只做对齐想补底座能力，通常成本高且效果有限。

## 15. 预训练阶段真正要解决哪些问题

### 15.1 表示能力

模型要学到语言、代码、视觉或多模态的通用统计结构。

### 15.2 长程依赖

模型要在足够长的上下文中维持一致性，而不是只会局部续写。

### 15.3 可迁移性

预训练不是为了在网页文本上拿最低 loss，而是为了让模型在后续任务中仍有可塑性。

### 15.4 能力边界

很多基础能力是否存在，往往在预训练阶段就已经定型，包括：

1. 数学符号处理习惯；
2. 代码结构理解；
3. 多语种覆盖；
4. 文档和表格理解；
5. 工具说明书式文本的解析能力。

### 15.5 预训练并不只是“更大数据”

成熟团队在预训练阶段实际要同时设计：

1. 数据去重和时间切分；
2. 混合比例和 curriculum；
3. token budget 与长度上限；
4. 优化器和 batch 组织；
5. 多模态对齐方式；
6. 训练栈，如 `Megatron-LM`、`DeepSpeed`、FSDP、ZeRO、TP/PP/EP 组合。

因此“预训练”本质上已经是一整套大规模系统工程。

## 16. SFT 的真正难点不在公式，而在示范分布

SFT 的目标写起来简单，但它本质上是在做行为克隆。于是关键问题变成：

1. 示范是谁写的；
2. 示范代表什么风格；
3. 示范覆盖哪些边界；
4. 示范是否在奖励错误行为。

### 16.1 SFT 最常见的三类目标

1. `instruction following`：让模型学会按输入任务组织输出；
2. `format alignment`：让模型稳定输出 schema、JSON、步骤、引用、工具调用；
3. `domain adaptation`：让模型贴合特定业务、产品或角色。

### 16.2 SFT 数据为什么经常把模型“训硬”

如果 SFT 数据过于模板化，模型会出现：

1. 回答结构固定；
2. 明明问题简单却过度展开；
3. 面对开放任务泛化变差；
4. 工具场景中只会复述模板而不会灵活调用。

这也是为什么很多团队会在 SFT 数据里主动保留风格多样性和不同长度示范。

### 16.3 SFT 不是越干净越好

“干净”并不等于“真实”。如果所有示范都来自过度理想化的标注环境，模型上线后会遇到：

1. 用户提问不完整；
2. 指令互相矛盾；
3. 工具返回值脏乱；
4. 上下文缺字段；
5. 多轮纠错和反问。

因此高质量 SFT 数据并不只是规范文本，还应包含真实产品形态下的高价值边界样本。

## 17. 对齐阶段常见路线如何取舍

### 17.1 RLHF

**优点**：

1. 可以显式建奖励模型；
2. 灵活处理复杂偏好和长链行为；
3. 适合多目标奖励组合。

**代价**：

1. 训练系统复杂；
2. 采样、回放和稳定性要求高；
3. 容易出现 reward hacking。

### 17.2 DPO/IPO 一类直接偏好优化

**优点**：

1. 工程实现更直接；
2. 不需要在线 RL 闭环；
3. 更容易和现有 SFT 管线衔接。

**代价**：

1. 偏好信号表达力受样本构造影响很大；
2. 如果参考模型选择不好，结果会偏；
3. 对偏好分布外样本的泛化未必稳定。

### 17.3 RLAIF 与模型判分

**优点**：

1. 可扩展性强；
2. 能把人工难覆盖的长尾场景快速铺开；
3. 易与 judge 体系结合。

**代价**：

1. 容易把模型偏差闭环放大；
2. 对高风险任务很难完全替代人工；
3. 若评价模型不校准，训练方向会整体偏移。

## 18. 什么时候该把预算花在预训练，什么时候该花在后训练

**一个实用判断是**：

### 18.1 更像底座能力缺口时，优先看预训练

例如：

1. 数学推导链条本身断裂；
2. 长上下文无法维持主题；
3. 代码修复能力明显不足；
4. 多语言或多模态表征很弱。

### 18.2 更像行为和风格缺口时，优先看 SFT / 对齐

例如：

1. 输出格式不稳定；
2. 太啰嗦或太说教；
3. 风险边界异常；
4. 工具选择保守或激进；
5. 解释顺序不符合产品要求。

### 18.3 很多问题其实是中间层问题

现实中还有大量问题既不是单纯底座，也不是单纯偏好，而是：

1. 数据混合配方问题；
2. SFT 示例覆盖不对；
3. 对齐目标和产品目标不一致；
4. 评测口径把问题看错了。

## 19. 工具使用和 agent 场景下的三阶段重分工

对工具调用系统，三阶段分工会变得更明显：

1. 预训练决定模型能否理解 API 文档、错误日志、网页结构和自然语言说明；
2. SFT 决定模型能否按协议输出调用参数、解析工具返回；
3. 对齐决定模型何时该调用、何时该停止、何时该承认不确定。

### 19.1 为什么工具使用很怕对齐过强

如果偏好对齐过度奖励“看起来稳妥”，模型会：

1. 不愿尝试工具；
2. 不愿提出中间假设；
3. 遇到模糊任务时直接给模棱两可的保守答案；
4. 明明需要浏览或检索，却选择空泛解释。

因此工具场景的对齐目标，往往不是“更保守”，而是“更会在恰当时刻外化不确定并调用外部系统”。

## 20. 多模态和世界模型场景中的三阶段变化

### 20.1 多模态 SFT 更像接口对齐

对 VLM 来说，很多 SFT 数据不是在教授世界知识，而是在教授：

1. 如何引用图像区域；
2. 如何处理 OCR、图表、版面；
3. 如何把视觉证据组织成回答。

### 20.2 世界模型或 VLA 的后训练更像行动对齐

对机器人和世界模型路线，后训练常常不再只是文本偏好，而是：

1. 动作轨迹偏好；
2. 恢复策略偏好；
3. 风险敏感性；
4. 人类演示风格；
5. 成功/失败回放的相对排序。

这使“对齐”从纯文本风格问题扩展成了“行为选择问题”。

## 21. 长推理、深度思考与三阶段训练

如果团队要做更强 reasoning 模型，一个常见误区是直接把大量“长 CoT”示范堆进 SFT。更稳的路线通常是：

1. 先在预训练中保证数学、代码、文档、逻辑的底座覆盖；
2. 再在 SFT 中引入高质量多步推理示范，而不是模板化唠叨；
3. 最后在偏好对齐中约束推理展开时的帮助性、简洁性和错误恢复。

### 21.1 为什么长推理特别怕示范污染

**如果 SFT 数据中的长推理包含**：

1. 虚假的中间步骤；
2. 自圆其说但不正确的解释；
3. 过长但信息密度低的模板；

模型会很快学会“像在认真思考”，却不一定真正更会推理。

## 22. 三阶段训练和数据引擎如何闭环

成熟团队不会把三阶段看成一次性的串联流水，而会做闭环：

1. 线上发现失败；
2. 判断失败更像底座问题、任务问题还是偏好问题；
3. 把样本分别送回预训练补充池、SFT 池、偏好池；
4. 再做专项回归评测。

### 22.1 为什么这很重要

因为如果所有失败都只回流到 DPO 或 SFT，系统会长期陷入“行为补丁式优化”，而底层能力短板一直不解决。

## 23. 常见失效模式再往下拆

### 23.1 预训练不足的症状

1. 对新领域迁移很差；
2. 长链推理一开始就断；
3. 结构化文档理解困难；
4. 多语言或多模态覆盖不均。

### 23.2 SFT 失配的症状

1. 模型会但不按格式做；
2. 工具参数经常漏字段；
3. 明明懂问题却回答组织混乱；
4. 不同 prompt 风格下行为不稳定。

### 23.3 对齐失配的症状

1. 过度保守；
2. 过度冗长；
3. 喜欢做安全说教；
4. 工具触发阈值错误；
5. 对高风险任务的拒答边界摇摆。

## 24. 一个更实用的资源分配框架

如果只有有限预算，可以用如下顺序判断：

1. 先确认问题主要属于哪一层；
2. 如果是底座能力，优先加预训练覆盖和长度；
3. 如果是任务行为，优先加高质量 SFT 和接口约束；
4. 如果是用户体验和风险，优先做偏好对齐与 judge 校准；
5. 如果三层都不像，就回去查数据混合、评测口径和系统实现。

## 25. 小结

预训练、SFT 与对齐不是三段可随意互换的流水线，而是三套不同的能力塑形机制。预训练负责把世界压进参数里，SFT 负责把任务接口和行为模式压进模型里，对齐负责把偏好、风险和产品边界压进决策倾向里。理解这三套机制的分工，才能知道该在哪一层加数据、加算力、加人审，而不是把所有问题都模糊地归结为“再训一下”。
