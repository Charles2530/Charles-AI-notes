# 输入管线、Packing 与训练吞吐治理

很多训练团队在模型、优化器、并行和混合精度上花了大量时间，最后却发现 GPU 利用率仍然不高。原因常常不是模型本身，而是输入管线：数据切片太碎、tokenizer 太慢、packing 太差、shuffle buffer 不合理、长样本把有效 token 利用率拖垮、恢复训练后数据游标错位。输入管线、packing 与吞吐治理，正是训练系统中最容易被低估、但最直接决定 GPU 是否被喂满的一层。

**这页重点讨论**：

1. 为什么输入管线经常成为大模型训练的隐形瓶颈；
2. packing、bucketing、streaming 与 tokenizer 各自解决什么问题；
3. 长上下文和多模态训练为什么会把输入系统难度抬高一个量级；
4. 如何把“吞吐变差”拆成可定位、可治理的问题。

## 1. 输入管线真正要解决的不是“把数据读出来”

训练输入系统至少同时承担五件事：

1. 正确读取和解码数据；
2. 保持足够高吞吐；
3. 维持采样和混合配方的统计正确性；
4. 支持断点恢复与可复现；
5. 为不同长度、模态和任务组织有效 batch。

任何一件做不好，最后都会体现在：

1. GPU 等数据；
2. 有效 token 利用率低；
3. curriculum 与 mixture 实际曝光失真；
4. 恢复训练后重复样本或跳样；
5. 长上下文训练成本被无效 padding 放大。

## 2. 为什么 padding 经常比想象中更贵

如果 batch 内样本长度差异很大，padding 会浪费大量 token 位。设一个 batch 内第 \(i\) 个样本长度为 \(l_i\)，最大长度为 \(L\)，则有效 token 利用率可写成：

\[
\eta = \frac{\sum_i l_i}{B \cdot L}.
\]

当 \(\eta\) 较低时，系统虽然“看起来 batch 很大”，但实际 GPU 有相当一部分在算 padding。

### 2.1 为什么长上下文场景尤其严重

因为：

1. 长样本和短样本差距更大；
2. attention 成本对长度更敏感；
3. packing 和 bucketing 稍微做差，浪费会被平方级或更高代价放大。

## 3. Bucketing 是输入系统的第一层基本功

最常见的做法，是先按长度做 bucket，再在 bucket 内组 batch。这样可以显著提升：

1. 有效 token 利用率；
2. kernel shape 稳定性；
3. profile 的可解释性；
4. 混合长度下的吞吐稳定性。

### 3.1 bucket 太少和太多都不好

1. **太少**：长度差异仍大，padding 浪费明显；
2. **太多**：调度复杂、统计不稳、shuffle 粒度变小。

**一个实用思路是围绕**：

1. 常见上下文长度；
2. 模型 kernel 友好形状；
3. 数据真实分布；

共同选择 bucket。

## 4. Packing 解决的是“碎样本太多”问题

对于语言建模和文档类训练，大量样本其实比最大长度短很多。若逐条对齐成固定长度，会浪费大量位置。于是常把多个短样本 pack 到同一序列中。

### 4.1 Packing 的收益

1. 提高有效 token 占比；
2. 降低 padding 浪费；
3. 提升吞吐；
4. 让 global batch token 更可控。

### 4.2 Packing 的代价

1. 需要正确构建 attention mask；
2. 不同任务可能需要边界隔离；
3. 多模态场景下文本和视觉 token 对齐更复杂；
4. 恢复训练时要重建 pack 状态。

### 4.3 为什么 packing 会影响方法学

因为同样的“训练了 1T token”，如果一种管线 padding 浪费高，另一种 pack 很好，那么二者实际喂给模型的有效信号并不等价。

## 5. Streaming 与随机性如何平衡

在超大数据集上，很多系统采用流式读取，而不是把全量样本提前落到本地。  
**这带来一组新问题**：

1. 如何保持足够随机性；
2. shuffle buffer 多大合适；
3. 如何支持多机一致性；
4. 如何恢复到精确数据位置。

### 5.1 shuffle buffer 太小的问题

它会让实际训练顺序过于局部，导致：

1. mixture 配方实际失真；
2. curriculum 切换不平滑；
3. 长期采样出现偏置。

### 5.2 shuffle buffer 太大的问题

1. 内存占用高；
2. 恢复状态更重；
3. 多 worker 协同更复杂。

因此输入管线里的随机性设计，本质上也是资源与统计正确性的平衡。

## 6. tokenizer 经常是被忽略的吞吐瓶颈

**尤其在**：

1. 原始文本多样；
2. 文档切片复杂；
3. 多语种；
4. 多模态需要 OCR 或视觉前处理；

的场景里，tokenizer 和前处理常常比大家想象得更慢。

### 6.1 一个常见误区

团队会在 GPU 侧做大量优化，但 CPU 侧 tokenizer 或文档解析只跑出很低吞吐，最终 GPU 一直在等。

### 6.2 常见优化方向

1. 预 tokenize 或缓存 tokenize 结果；
2. 把重前处理离线化；
3. 用更适合批处理的数据格式；
4. 对多模态做前处理分层缓存。

## 7. 多模态输入管线为什么更难

因为它不仅要处理文本长度，还要处理：

1. 图像分辨率；
2. 页数；
3. 视频帧数；
4. OCR 结果；
5. 模态对齐关系。

### 7.1 这意味着 batch 不再只有一个长度维度

很多多模态 batch 的代价取决于：

1. 文本 token 数；
2. 视觉 token 数；
3. 编码器前端计算；
4. 不同模态混合比例。

因此多模态 packing 和 bucketing 不能简单沿用纯文本逻辑。

## 8. 长上下文训练会把输入系统推到极限

当最大长度从 `4k` 提到 `32k` 或更高时，输入系统要重新回答：

1. 长样本是否真有足够供给；
2. 是否需要专项 long-context 数据层；
3. packing 是否仍然值得；
4. bucket 如何重新设计；
5. sampler 如何避免长样本占比和 wall-clock 同时爆炸。

### 8.1 为什么长上下文不只是模型问题

如果输入管线不能稳定提供足够多、足够好的长样本，那么所谓长上下文训练很可能只是：

1. 架构支持更长；
2. 实际训练中却没见够；
3. 结果上线后只能“理论支持，实际不稳”。

## 9. 输入系统的可复现性要求

要做到高质量恢复和实验对比，输入系统必须记录：

1. 数据快照版本；
2. 过滤版本；
3. tokenizer 版本；
4. bucket 配置；
5. pack 逻辑版本；
6. sampler 随机状态；
7. 每个 worker 的游标。

### 9.1 为什么恢复训练很容易在这里出错

如果这些状态不完整，恢复后就会出现：

1. 重复样本；
2. 跳过样本；
3. mixture 统计变化；
4. curriculum 阶段错位。

很多“恢复后 loss 变怪”的问题，根源都不在模型，而在输入系统状态不连续。

## 10. 如何监控输入系统

**建议至少单独监控**：

1. dataloader 等待时间；
2. tokenizer 吞吐；
3. bucket 占比；
4. pack 后有效 token 利用率；
5. 不同数据源真实曝光率；
6. 损坏样本率；
7. 恢复后数据位置连续性。

### 10.1 为什么“只看 token/s”不够

同样的 token/s，可能来自：

1. padding 很多但 GPU 很忙；
2. 有效 token 很多且输入系统健康；

两者对训练价值完全不同。

## 11. 一个更实用的排障顺序

如果训练吞吐低，建议先按下面顺序检查：

1. 先看 GPU 是不是在等数据；
2. 再看 bucket 和 padding 效率；
3. 再看 tokenizer 和前处理；
4. 再看 pack 逻辑是否造成额外开销；
5. 最后才判断是否真的需要更改模型或更复杂并行。

这比一开始就怀疑硬件不够更有效。

## 12. 常见失效模式

### 12.1 看似并行没问题，实际是输入层没喂满

GPU 利用率不高，但大家继续加机，只会把空转放大。

### 12.2 packing 提高了吞吐，却破坏了任务边界

mask 或分段逻辑错误会让模型学到不该跨样本传播的信息。

### 12.3 bucket 做得太细，随机性反而变差

训练统计看起来更整齐，泛化却未必更好。

### 12.4 只优化文本管线，忽略多模态前处理

结果是文本 worker 空闲，OCR 或图像处理一直堆积。

## 13. 设计建议

1. 输入系统优化先看有效 token 利用率，而不是只看样本条数；
2. bucketing、packing、streaming 和 tokenizer 要一起设计；
3. 长上下文和多模态训练单独建设数据层与监控；
4. 恢复训练必须同时恢复 sampler、pack 和 tokenizer 相关状态；
5. 把输入管线当作训练系统的一等模块，而不是后台细节。

## 14. 小结

输入管线、packing 与吞吐治理，决定了训练系统究竟是在高效吸收有效信号，还是在昂贵地处理 padding、等待和前处理抖动。模型越大、上下文越长、多模态越复杂，这一层就越不能被当作附属脚手架。真正成熟的训练团队，不只是会调优化器和并行，也会把输入系统做成可监控、可恢复、可解释的生产管线。

## 15. 输入系统也应做专项回归

**每次改**：

1. tokenizer；
2. pack 逻辑；
3. bucket 配置；
4. streaming 路径；
5. 长样本采样比例；

**都应单独回归**：

1. 有效 token 利用率；
2. 数据源真实曝光率；
3. 恢复训练后的连续性；
4. 长度桶分布；
5. 训练短跑曲线。

### 15.1 为什么这很关键

因为很多训练退化并不是模型问题，而是输入系统悄悄改变了模型看到的数据统计结构。

## 16. 小结

输入管线不是后台搬运工，而是训练统计分布和系统吞吐的共同塑形者。只要团队开始做长上下文、多模态和复杂 mixture，packing、bucket、streaming、tokenizer 与恢复状态就都不再是细节，而是需要被正式治理和回归验证的主系统。

## 17. Megatron-LM 风格数据集到底解决了什么

很多开源训练栈最后都会回到一类相似设计：**先离线把原始文本或多模态样本转成可顺序读取、可随机切片、可恢复的索引格式**，再在训练时尽量避免做昂贵的在线解析。Megatron-LM 一类 indexed dataset 的价值，不只是“读得快”，而是同时解决四类问题：

1. **样本定位**。给定全局 sample id，系统能快速找到其字节偏移、token 区间或模态片段；
2. **恢复训练**。当 global step 恢复时，sampler 可以重新落回相同顺序，而不是重新扫描整个语料；
3. **并行切分**。不同 data parallel rank 可以在不互相通信的情况下定位自己应该读取的切片；
4. **统计治理**。样本长度、来源、模态、质量标签可以作为旁路元数据参与采样。

### 17.1 为什么“直接读 JSONL”常常撑不起大训练

小实验时，团队常直接从 `jsonl`、`parquet` 或 `webdataset tar` 在线读取，这在几十亿 token 以内往往还行；但数据一旦上到真正的预训练规模，问题就开始堆叠：

1. **解析成本高**。重复做 UTF-8 解码、JSON 解析和字段抽取会吞掉 CPU；
2. **随机访问差**。想跳到某个样本时，往往只能从前面顺序扫；
3. **恢复成本高**。重启后需要重新定位 shuffle 状态和样本游标；
4. **多 worker 行为不稳定**。不同 worker 对文件切片的边界理解不一致，容易重复样本或漏样；
5. **难以监控**。当训练退化时，很难快速回答“到底读到了哪些源、哪些长度、哪些过滤规则下的样本”。

### 17.2 一个更实用的数据格式判断标准

选数据格式时，不应只问“峰值吞吐多少”，而应同时问：

1. **是否支持 deterministic resume**；
2. **是否支持长度、来源、质量标签等附加索引**；
3. **是否支持 object store 和本地 NVMe 两种路径**；
4. **是否方便在线重打包与增量追加**；
5. **是否能为多模态样本维护统一 sample id**。

只看第一条，往往会把整个输入系统做成一条一次性脚本，而不是长期可运营的数据平面。

## 18. WebDataset、Parquet、Indexed Dataset 的实际取舍

这几类格式没有绝对优劣，关键在于你的数据形态和运维方式。

### 18.1 `WebDataset`

**优点**：

1. **天然适合对象存储**，分 shard 拉取，便于流式训练；
2. **对图像、视频、音频等文件型样本友好**；
3. **易于并行预处理和重打包**；
4. **能把多文件样本打成统一 key 前缀**。

**缺点**：

1. **样本级随机访问较弱**；
2. **shard 粒度过粗时，恢复训练和局部重放不够优雅**；
3. **tar 内部顺序会影响 worker 行为和 cache 命中**。

### 18.2 `Parquet`

**优点**：

1. **列式压缩和 schema 管理成熟**；
2. **对结构化特征、标签和统计字段很友好**；
3. **适合做数据分析、过滤和离线治理**。

**缺点**：

1. **对超长文本或原始二进制模态文件不一定最优**；
2. **训练时若频繁读取大列，实际可能退化成“读整块再拆字段”**；
3. **在线读取时受 row group 和远端存储布局影响很大**。

### 18.3 `Indexed Dataset`

**优点**：

1. **训练时吞吐稳定**；
2. **随机切片和恢复语义清楚**；
3. **适合基于 token 长度和 sample id 做细粒度控制**。

**缺点**：

1. **离线构建链路更重**；
2. **对多模态复杂嵌套 schema 的表达能力不如通用表格式自然**；
3. **如果原始数据更新频繁，重建成本可能偏高**。

### 18.4 一种常见的折中架构

成熟团队常会形成分层设计：

1. **治理层**用 `Parquet` 或数据库维护元信息；
2. **原始文件层**用对象存储保存图片、视频、音频和大文档；
3. **训练消费层**则重打包为 `WebDataset` 或 `Indexed Dataset`；
4. **监控层**单独记录样本曝光、长度分布和过滤日志。

这样做的好处是：**治理格式服务离线分析，训练格式服务高吞吐消费**，两者不必强行统一。

## 19. Pretokenization 不是纯离线优化，而是统计契约

很多人把 pretokenization 理解成“省 tokenizer 时间”，这太窄了。**更重要的是，它把训练时的 token 边界、特殊符号策略和长度统计固定下来**，从而减少训练期间的隐式漂移。

### 19.1 Pretokenization 最该固定的内容

1. **tokenizer 版本**；
2. **normalization 策略**，如 NFC/NFKC、大小写和空白折叠；
3. **特殊 token 注入方式**，如 `BOS/EOS/system/user/assistant`；
4. **多轮对话拼接模板**；
5. **模态占位符和边界 token**；
6. **是否保留原始偏移映射**。

如果这些策略一半离线、一半在线，训练会出现一种非常危险的情况：**loss 曲线还在变，但数据分布已经悄悄变了**。

### 19.2 Pretokenization 也有代价

1. **tokenizer 升级困难**。一旦词表或分词规则变更，整套数据都要重建；
2. **prompt 模板演进受限**。如果数据格式写死了模板，后续实验切换代价大；
3. **多语言和多任务复用困难**。同一原始数据可能需要多个 tokenization 视角；
4. **存储放大**。尤其是保留原文、token、offset 和模态元数据时。

### 19.3 一个更稳妥的实践

建议把数据分成两层：

1. **基础层**保留清洗后的 canonical text / multimodal references；
2. **训练层**根据具体 tokenizer 和模板生成消费格式。

这样一来，团队既能获得稳定吞吐，也不会把所有上游语料都绑死在某一版 tokenizer 上。

## 20. Object Store Streaming 的关键不是能不能流，而是稳不稳定

对象存储让训练摆脱了本地磁盘容量限制，但也引入新的系统问题。真正难的不是把文件从远端读下来，而是让它在长时间训练中保持**稳定、可恢复、可预测**。

### 20.1 流式读取最常见的四类抖动

1. **首字节延迟抖动**。远端对象冷启动、DNS、鉴权链路都会放大 tail；
2. **worker 同步放大**。多个 rank 在同一时刻去抢热点 shard；
3. **局部热点**。某些热门 shard 因采样权重高被反复命中；
4. **小文件风暴**。数据切得太碎，HTTP/TLS 开销压过真实 payload。

### 20.2 为何单看平均带宽没有意义

训练更怕的是**最慢的那几个 worker**。因为 data parallel 往往要等最慢 rank 喂完 batch，单个 worker 的尾延迟就会被全局同步放大。所以对象存储链路应重点看：

1. **P95/P99 fetch latency**；
2. **per-rank fetch skew**；
3. **缓存命中率**；
4. **重试率与 429/5xx 比例**；
5. **被迫降级到本地 fallback 的频次**。

### 20.3 实用架构：远端冷数据 + 本地热缓存

更稳的设计往往是：

1. **对象存储**存完整数据；
2. **节点本地 NVMe**做 shard cache；
3. **rank 内 worker**优先复用本地热 shard；
4. **后台异步 prefetch**按未来若干 step 的采样预测补货。

这类架构的关键不是 cache 存在，而是**cache eviction 和恢复语义是否清楚**。否则一旦节点重启，训练又会被冷启动抖动砸中。

## 21. Deterministic Resume 需要恢复哪些状态

很多团队只恢复模型、优化器和学习率，然后以为训练已可恢复。对大规模训练来说，这远远不够。输入系统至少还要恢复：

1. **sampler 状态**；
2. **每个数据源的游标**；
3. **shuffle buffer 随机种子与当前位置**；
4. **packing buffer 内尚未消费完的短样本**；
5. **对象存储 shard 分配状态**；
6. **多模态对齐状态**，如视频 clip 的窗口位置、字幕偏移；
7. **在线过滤器状态**，如重复样本跳过计数、失败重试白名单。

### 21.1 为什么只恢复“step”不够

因为很多输入系统都不是 `sample_id = f(step)` 这么简单。只要你启用了：

1. **按长度分桶**；
2. **混合数据源**；
3. **packing**；
4. **动态过滤**；
5. **流式读取失败重试**；

同一个 `global_step` 下，真实曝光的样本集合都可能不同。

### 21.2 一个常被忽略的问题：恢复后的曝光偏移

恢复训练后，曲线看起来可能很正常，但如果某些高权重数据源被重复曝光，模型会出现两类隐蔽偏差：

1. **表面上 loss 更低**，实则是重复见过更多近似样本；
2. **下游泛化退化**，因为 mixture 实际分布被篡改。

所以 deterministic resume 的验收，不能只看“能否跑起来”，而要看**恢复前后采样分布是否一致**。

## 22. 多模态 Packing 比文本 Packing 更容易出事故

文本 packing 的主要难点是 mask 和边界；多模态 packing 则还要额外面对**模态长度不齐、时序对齐、占位 token 扩张和前处理异步性**。

### 22.1 典型复杂点

1. **一段视频对应多个 frame token**，而字幕或动作标签长度又不同；
2. **图像 patch token 数与分辨率相关**，动态分辨率会影响 pack 行为；
3. **音频帧、文本 token、控制 token 处于不同时间尺度**；
4. **多图多轮对话里图像位置不是固定槽位**；
5. **动作条件模型**中，动作 token 既是监督目标又是未来条件。

### 22.2 一个重要原则：先定义 segment contract，再写 packer

对于多模态样本，团队应先定义清楚：

1. **每种模态的边界 token**；
2. **跨模态 attention mask 规则**；
3. **哪些模态之间允许信息流动**；
4. **哪些 supervision 只在局部 segment 内有效**；
5. **padding 对各模态分别如何处理**。

如果 contract 不先写清楚，后面的 packer 很容易在“吞吐优化”中悄悄破坏监督语义。

## 23. 数据加载器也应有自己的 SLO

训练团队常给 GPU 集群、存储和网络设 SLO，却忽略了 data loader 自己也应该被当成服务治理对象。

### 23.1 建议至少追的 SLO 指标

1. **batch ready latency**；
2. **data stall ratio**，即 GPU 因等数据而空转的比例；
3. **有效 token 利用率**；
4. **平均 pack 填充率与 P95 pack 填充率**；
5. **per-source 曝光偏差**；
6. **prefetch 队列饱和度**；
7. **worker 异常重启率**。

### 23.2 为什么输入层也要有 error budget

如果 data loader 每 1000 step 就抖一次，GPU 平均利用率看起来可能还行，但长跑训练会出现：

1. **吞吐抖动难以归因**；
2. **多机同步放大**；
3. **线上 scheduler 误判节点异常**；
4. **checkpoint 周期与吞吐预算被打乱**。

把输入系统纳入 error budget 后，团队才会认真处理“偶发卡顿”和“恢复后短暂抖动”这类之前被忽略的问题。

## 24. 长上下文训练里的 Packing 不只是装满，而是控制算子形状

在长上下文训练里，packing 还会直接影响后端 kernel 形状。**同样的有效 token 数，如果 segment 切法不同，attention、rope、mask 构造和 position 处理的代价可能完全不同。**

### 24.1 形状友好的 packing 目标

1. **尽量把大多数 batch 落进有限几类 bucket**；
2. **减少极端尾部长度**，避免少量超长样本拖垮整批；
3. **让后端算子更容易复用已有 kernel specializations**；
4. **控制 prefix / suffix 切分策略，减少奇怪边界**。

### 24.2 为什么“理论填充率最高”不一定最好

如果某个 packing 策略虽然把 token 装得很满，却导致：

1. **shape 高度离散**；
2. **mask 构造复杂**；
3. **position 计算和段边界处理更重**；
4. **编译缓存和 kernel cache 命中下降**；

那它未必比一个稍微浪费一点 token、但形状更稳定的策略更快。

## 25. 输入系统监控最好做成四层

### 25.1 数据源层

1. **各语料源权重与实际曝光率**；
2. **源内长度分布**；
3. **解析失败率**；
4. **清洗规则命中率**；
5. **对象存储读延迟**。

### 25.2 样本层

1. **token 长度分布**；
2. **多模态 segment 分布**；
3. **超长样本占比**；
4. **被 pack 的短样本个数分布**；
5. **重复样本概率估计**。

### 25.3 Batch 层

1. **padding ratio**；
2. **pack fill ratio**；
3. **bucket 命中分布**；
4. **batch 构造时间**；
5. **worker 间 skew**。

### 25.4 训练层

1. **GPU 空转时间**；
2. **step time 抖动**；
3. **有效 token/s**；
4. **恢复训练后的分布偏移**；
5. **输入异常与 loss 异常的相关性**。

## 26. 一个更完整的输入系统验收模板

要把一个输入系统从“实验能跑”升级成“长期能养”，建议至少通过以下验收：

1. **功能正确性**：mask、segment、标签对齐、模态边界全部正确；
2. **吞吐稳定性**：在 8 小时以上长跑中，step time 无明显周期性抖动；
3. **恢复一致性**：checkpoint 恢复后，采样分布与 pack 行为与恢复前一致；
4. **混合正确性**：数据源权重、长度桶分布、任务混合比例均符合设计；
5. **观测完备性**：故障时能快速定位到数据源、worker、shard 或 packer 级别。

### 26.1 为什么要把验收写成模板

因为输入系统的很多退化都不是一眼能看见的。它们常常以“loss 变慢一点”“吞吐抖动一点”“长尾任务变差一点”的方式出现。只有把验收标准写成模板，团队才能在每次更换 tokenizer、加入新数据源、上线新 pack 逻辑时做真正可比较的回归。

## 27. 小结

真正成熟的训练输入系统，不只是把数据喂进模型，而是把**样本统计契约、吞吐契约、恢复契约和监控契约**统一起来。只要开始进入多源混合、长上下文、多模态和对象存储流式训练阶段，输入系统就不再是脚手架，而是训练平台最容易决定上限的一层。
