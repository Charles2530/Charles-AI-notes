# 输入管线、Packing 与训练吞吐治理

很多训练团队在模型、优化器、并行和混合精度上花了大量时间，最后却发现 GPU 利用率仍然不高。原因常常不是模型本身，而是输入管线：数据切片太碎、tokenizer 太慢、packing 太差、shuffle buffer 不合理、长样本把有效 token 利用率拖垮、恢复训练后数据游标错位。输入管线、packing 与吞吐治理，正是训练系统中最容易被低估、但最直接决定 GPU 是否被喂满的一层。

**这页重点讨论**：

1. 为什么输入管线经常成为大模型训练的隐形瓶颈；
2. packing、bucketing、streaming 与 tokenizer 各自解决什么问题；
3. 长上下文和多模态训练为什么会把输入系统难度抬高一个量级；
4. 如何把“吞吐变差”拆成可定位、可治理的问题。

## 1. 输入管线真正要解决的不是“把数据读出来”

训练输入系统至少同时承担五件事：

1. 正确读取和解码数据；
2. 保持足够高吞吐；
3. 维持采样和混合配方的统计正确性；
4. 支持断点恢复与可复现；
5. 为不同长度、模态和任务组织有效 batch。

任何一件做不好，最后都会体现在：

1. GPU 等数据；
2. 有效 token 利用率低；
3. curriculum 与 mixture 实际曝光失真；
4. 恢复训练后重复样本或跳样；
5. 长上下文训练成本被无效 padding 放大。

## 2. 为什么 padding 经常比想象中更贵

如果 batch 内样本长度差异很大，padding 会浪费大量 token 位。设一个 batch 内第 \(i\) 个样本长度为 \(l_i\)，最大长度为 \(L\)，则有效 token 利用率可写成：

\[
\eta = \frac{\sum_i l_i}{B \cdot L}.
\]

当 \(\eta\) 较低时，系统虽然“看起来 batch 很大”，但实际 GPU 有相当一部分在算 padding。

### 2.1 为什么长上下文场景尤其严重

因为：

1. 长样本和短样本差距更大；
2. attention 成本对长度更敏感；
3. packing 和 bucketing 稍微做差，浪费会被平方级或更高代价放大。

## 3. Bucketing 是输入系统的第一层基本功

最常见的做法，是先按长度做 bucket，再在 bucket 内组 batch。这样可以显著提升：

1. 有效 token 利用率；
2. kernel shape 稳定性；
3. profile 的可解释性；
4. 混合长度下的吞吐稳定性。

### 3.1 bucket 太少和太多都不好

1. **太少**：长度差异仍大，padding 浪费明显；
2. **太多**：调度复杂、统计不稳、shuffle 粒度变小。

**一个实用思路是围绕**：

1. 常见上下文长度；
2. 模型 kernel 友好形状；
3. 数据真实分布；

共同选择 bucket。

## 4. Packing 解决的是“碎样本太多”问题

对于语言建模和文档类训练，大量样本其实比最大长度短很多。若逐条对齐成固定长度，会浪费大量位置。于是常把多个短样本 pack 到同一序列中。

### 4.1 Packing 的收益

1. 提高有效 token 占比；
2. 降低 padding 浪费；
3. 提升吞吐；
4. 让 global batch token 更可控。

### 4.2 Packing 的代价

1. 需要正确构建 attention mask；
2. 不同任务可能需要边界隔离；
3. 多模态场景下文本和视觉 token 对齐更复杂；
4. 恢复训练时要重建 pack 状态。

### 4.3 为什么 packing 会影响方法学

因为同样的“训练了 1T token”，如果一种管线 padding 浪费高，另一种 pack 很好，那么二者实际喂给模型的有效信号并不等价。

## 5. Streaming 与随机性如何平衡

在超大数据集上，很多系统采用流式读取，而不是把全量样本提前落到本地。  
**这带来一组新问题**：

1. 如何保持足够随机性；
2. shuffle buffer 多大合适；
3. 如何支持多机一致性；
4. 如何恢复到精确数据位置。

### 5.1 shuffle buffer 太小的问题

它会让实际训练顺序过于局部，导致：

1. mixture 配方实际失真；
2. curriculum 切换不平滑；
3. 长期采样出现偏置。

### 5.2 shuffle buffer 太大的问题

1. 内存占用高；
2. 恢复状态更重；
3. 多 worker 协同更复杂。

因此输入管线里的随机性设计，本质上也是资源与统计正确性的平衡。

## 6. tokenizer 经常是被忽略的吞吐瓶颈

**尤其在**：

1. 原始文本多样；
2. 文档切片复杂；
3. 多语种；
4. 多模态需要 OCR 或视觉前处理；

的场景里，tokenizer 和前处理常常比大家想象得更慢。

### 6.1 一个常见误区

团队会在 GPU 侧做大量优化，但 CPU 侧 tokenizer 或文档解析只跑出很低吞吐，最终 GPU 一直在等。

### 6.2 常见优化方向

1. 预 tokenize 或缓存 tokenize 结果；
2. 把重前处理离线化；
3. 用更适合批处理的数据格式；
4. 对多模态做前处理分层缓存。

## 7. 多模态输入管线为什么更难

因为它不仅要处理文本长度，还要处理：

1. 图像分辨率；
2. 页数；
3. 视频帧数；
4. OCR 结果；
5. 模态对齐关系。

### 7.1 这意味着 batch 不再只有一个长度维度

很多多模态 batch 的代价取决于：

1. 文本 token 数；
2. 视觉 token 数；
3. 编码器前端计算；
4. 不同模态混合比例。

因此多模态 packing 和 bucketing 不能简单沿用纯文本逻辑。

## 8. 长上下文训练会把输入系统推到极限

当最大长度从 `4k` 提到 `32k` 或更高时，输入系统要重新回答：

1. 长样本是否真有足够供给；
2. 是否需要专项 long-context 数据层；
3. packing 是否仍然值得；
4. bucket 如何重新设计；
5. sampler 如何避免长样本占比和 wall-clock 同时爆炸。

### 8.1 为什么长上下文不只是模型问题

如果输入管线不能稳定提供足够多、足够好的长样本，那么所谓长上下文训练很可能只是：

1. 架构支持更长；
2. 实际训练中却没见够；
3. 结果上线后只能“理论支持，实际不稳”。

## 9. 输入系统的可复现性要求

要做到高质量恢复和实验对比，输入系统必须记录：

1. 数据快照版本；
2. 过滤版本；
3. tokenizer 版本；
4. bucket 配置；
5. pack 逻辑版本；
6. sampler 随机状态；
7. 每个 worker 的游标。

### 9.1 为什么恢复训练很容易在这里出错

如果这些状态不完整，恢复后就会出现：

1. 重复样本；
2. 跳过样本；
3. mixture 统计变化；
4. curriculum 阶段错位。

很多“恢复后 loss 变怪”的问题，根源都不在模型，而在输入系统状态不连续。

## 10. 如何监控输入系统

**建议至少单独监控**：

1. dataloader 等待时间；
2. tokenizer 吞吐；
3. bucket 占比；
4. pack 后有效 token 利用率；
5. 不同数据源真实曝光率；
6. 损坏样本率；
7. 恢复后数据位置连续性。

### 10.1 为什么“只看 token/s”不够

同样的 token/s，可能来自：

1. padding 很多但 GPU 很忙；
2. 有效 token 很多且输入系统健康；

两者对训练价值完全不同。

## 11. 一个更实用的排障顺序

如果训练吞吐低，建议先按下面顺序检查：

1. 先看 GPU 是不是在等数据；
2. 再看 bucket 和 padding 效率；
3. 再看 tokenizer 和前处理；
4. 再看 pack 逻辑是否造成额外开销；
5. 最后才判断是否真的需要更改模型或更复杂并行。

这比一开始就怀疑硬件不够更有效。

## 12. 常见失效模式

### 12.1 看似并行没问题，实际是输入层没喂满

GPU 利用率不高，但大家继续加机，只会把空转放大。

### 12.2 packing 提高了吞吐，却破坏了任务边界

mask 或分段逻辑错误会让模型学到不该跨样本传播的信息。

### 12.3 bucket 做得太细，随机性反而变差

训练统计看起来更整齐，泛化却未必更好。

### 12.4 只优化文本管线，忽略多模态前处理

结果是文本 worker 空闲，OCR 或图像处理一直堆积。

## 13. 设计建议

1. 输入系统优化先看有效 token 利用率，而不是只看样本条数；
2. bucketing、packing、streaming 和 tokenizer 要一起设计；
3. 长上下文和多模态训练单独建设数据层与监控；
4. 恢复训练必须同时恢复 sampler、pack 和 tokenizer 相关状态；
5. 把输入管线当作训练系统的一等模块，而不是后台细节。

## 14. 小结

输入管线、packing 与吞吐治理，决定了训练系统究竟是在高效吸收有效信号，还是在昂贵地处理 padding、等待和前处理抖动。模型越大、上下文越长、多模态越复杂，这一层就越不能被当作附属脚手架。真正成熟的训练团队，不只是会调优化器和并行，也会把输入系统做成可监控、可恢复、可解释的生产管线。

## 15. 输入系统也应做专项回归

**每次改**：

1. tokenizer；
2. pack 逻辑；
3. bucket 配置；
4. streaming 路径；
5. 长样本采样比例；

**都应单独回归**：

1. 有效 token 利用率；
2. 数据源真实曝光率；
3. 恢复训练后的连续性；
4. 长度桶分布；
5. 训练短跑曲线。

### 15.1 为什么这很关键

因为很多训练退化并不是模型问题，而是输入系统悄悄改变了模型看到的数据统计结构。

## 16. 小结

输入管线不是后台搬运工，而是训练统计分布和系统吞吐的共同塑形者。只要团队开始做长上下文、多模态和复杂 mixture，packing、bucket、streaming、tokenizer 与恢复状态就都不再是细节，而是需要被正式治理和回归验证的主系统。
