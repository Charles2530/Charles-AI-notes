# 评测与消融方法学

训练系统做大之后，最容易出现一种表面繁荣：实验很多、曲线很多、表格也很多，但结论并不真正可靠。模型变好了，是因为新目标函数有效，还是因为 batch 变大了？某个技巧看起来提升 1.2 分，是统计显著还是随机波动？数据混合方式改了之后效果更好，是采样策略起作用，还是恰好新数据本身更多？如果没有严格的评测与消融方法学，系统迭代很容易变成经验堆砌。

## 1. 评测方法学为什么重要

可以把一次训练改动看成对系统函数 $f$ 的一次扰动。我们真正想知道的是：

$$
\Delta = \mathbb{E}[M(f_{\text{new}})] - \mathbb{E}[M(f_{\text{old}})],
$$

其中 $M$ 是某个指标。但现实中我们看到的只是有限样本上的 noisy estimate。方法学的任务，就是把“看起来变了”与“确实因这个改动而变了”尽量区分开。

## 2. 先定义清楚评测对象

**评测对象可以是**：

1. 预训练表示质量
2. 微调后的任务质量
3. 系统级行为，如时延、成本、稳定性
4. 业务级收益，如任务完成率、人工纠正率

不同对象需要不同指标，混在一起会让结论失焦。比如一个新训练配方让 perplexity 更好，但工具调用成功率下降；这不能简单说“模型更强了”。

## 3. 离线评测与在线评测要分层

一个完整评测体系通常至少有三层：

1. **训练内指标**：loss、梯度、收敛速度、稳定性。
2. **离线任务指标**：准确率、F1、Recall@K、pass@k、成功率等。
3. **在线业务指标**：点击、转化、工单关闭率、人工反馈。

若只看训练内指标，容易优化代理目标而不是实际收益；若只看线上，定位问题又太慢、太贵。

## 4. 消融的核心原则：一次只问一个清楚问题

好的消融不是“列十个版本”，而是每组实验都回答一个明确问题。例如：

1. 仅增加困难负样本是否提升检索？
2. 固定总算力，增加真实回流数据是否比增加训练步数更值？
3. 把蒸馏从 4 步改成 2 步，质量损失主要来自哪里？

如果一次同时改模型、数据、超参和训练时长，结果通常不可解释。

## 5. 控制变量的重要性

设结果指标为

$$
Y = \alpha + \beta_1 X_1 + \beta_2 X_2 + \cdots + \epsilon,
$$

其中 $X_1$ 可能是“是否加入新损失”，$X_2$ 是“训练 token 数”，$X_3$ 是“batch 大小”。若你只比较两个配置，但这些变量同时变化，那么观察到的差异根本不能归因到某一个改动。

现实中未必能完全控制变量，但至少应：

1. 明确哪些变量保持不变；
2. 记录不可避免变化；
3. 在结果解释时诚实区分因果与相关。

## 6. 消融常见形态

### 6.1 Remove-one

从完整系统中移除一个组件，看性能下降多少。适合验证某组件是否必要，但不一定反映它的独立上限。

### 6.2 Add-one

从基线开始，每次增加一个组件。更适合看增益路径，但成本可能较高。

### 6.3 Factorial design

同时考察多个因素的交互。例如数据清洗和 schedule 是否有相乘效应。虽然更严谨，但实验数量增长快。

### 6.4 Budget-matched comparison

固定算力、参数量或训练 token，比的是“同等预算下谁更值”，这通常比无预算约束比较更有工程意义。

## 7. 方差与随机种子

大模型训练昂贵，很多团队只跑一次。但若指标波动显著，单次结果可能误导结论。若同一配置在不同随机种子下结果为 $y_1,\dots,y_n$，则均值和标准误差分别为

$$
\bar{y}=\frac{1}{n}\sum_i y_i,
\qquad
\mathrm{SE} = \frac{s}{\sqrt{n}}.
$$

并不是所有实验都能多种子复现，但至少对关键改动和小提升结论，最好做重复验证。

## 8. 显著性与效应量

即使统计上显著，也不意味着工程上重要。一个改动在百万级样本上提升 0.1%，可能统计显著，但若推理成本上升 30%，就未必值得。

**因此建议同时报告**：

1. 平均提升幅度
2. 置信区间
3. 成本变化
4. 长尾桶影响

## 9. 平均分会掩盖什么

总体分数常常掩盖长尾退化。设测试集分为常见桶 $D_h$ 与长尾桶 $D_t$：

$$
M = \alpha M_h + (1-\alpha)M_t.
$$

只要 $\alpha$ 足够大，一个对长尾有害的改动仍可能提高总分。对文档、具身和安全相关任务，这类掩盖尤其危险。

## 10. 分桶评测

好的评测体系应预先设计桶，例如：

1. 输入长度桶
2. 难度桶
3. 语言桶
4. 版式桶
5. 对象类别桶
6. 长时任务桶

分桶可以让你知道提升来自哪里，损失又发生在哪里。

## 11. 训练过程中不只看最终分数

很多方法最终分数接近，但训练轨迹差异极大。应记录：

1. 收敛速度
2. 是否早期不稳定
3. 恢复 checkpoint 后是否一致
4. 是否出现 loss spike
5. 是否更易受数据噪声影响

有些方法最终略好，但训练风险很高；有些方法均值略低，却极其稳定。这在生产环境中可能是更好的选择。

## 12. 预算对齐为何重要

如果方法 A 比方法 B 多训练 30% token、多用两倍显存，再比较最终分数，很难说是方法更好还是资源更多。因此应明确对齐方式：

1. 同参数规模
2. 同训练 token
3. 同 wall-clock
4. 同 GPU 小时
5. 同推理成本

不同问题下，对齐标准也不同。

## 13. 在线评测中的因果噪声

线上系统中，用户群体、流量时段、内容热点变化都可能影响指标。A/B 测试之外，还应做：

1. 分层随机化
2. 时间切片分析
3. 异常流量剔除
4. 版本锁定和灰度回退

否则一次“效果提升”可能只是撞上了更容易的流量。

## 14. 评测套件如何维护

评测集不是一次性静态资产。成熟团队会维护：

1. 标准公开 benchmark
2. 自建业务专项集
3. 长尾故障集
4. 线上回流挑战集
5. 不可触碰的 holdout 集

其中最后一类尤其重要，用来防止团队长期在公开或常用集合上过拟合。

## 15. 消融表的常见误区

### 15.1 同时改太多变量

表格漂亮，但无法解释。

### 15.2 基线过弱

新方法赢了一个未调好的基线，没有意义。

### 15.3 不报告失败实验

只展示成功配置会造成错误归因。

### 15.4 没有成本列

仅报告精度，不报告吞吐、显存、训练时长，结论不完整。

## 16. 一个例子：文档 VLM 的版面增强

团队加入新版面增强后，整体准确率提升 0.8%。但进一步分解发现：

1. 清晰扫描页提升 0.3%
2. 复杂表格页提升 4.2%
3. 手写批注页下降 1.7%

如果只看总分，会误以为方法普遍有效；如果再看成本，发现训练时间增加 25%，则还需判断这是否值得。

## 17. 另一个例子：机器人恢复策略

加入失败恢复模块后，首次成功率下降 1%，最终成功率上升 6%。这说明系统更保守，但整体更可靠。若你只看首次成功率，可能会误删一个对部署极其有价值的功能。

## 18. 设计建议

1. 每个消融实验都写清楚想回答的问题。
2. 对关键结论尽量做多种子或重复验证。
3. 所有结果同时报告平均分、分桶、成本和稳定性。
4. 建立长期 holdout，防止 benchmark 过拟合。
5. 在线实验要锁版本、控变量、做灰度。

## 19. 小结

评测与消融方法学的核心，是让实验结论变得可信、可解释、可复用。它要求团队从“做了很多实验”提升到“每个实验都回答了一个明确问题，并且证据足够支持结论”。在模型、数据和系统都越来越复杂的时代，这种方法学本身就是竞争力。 

## 20. 评测对象至少要拆成四层

一个成熟训练团队通常不会只维护一个总分，而是把评测对象拆成四层：

1. `建模层`：loss、perplexity、bits-per-byte、稳定性；
2. `能力层`：知识、推理、代码、长上下文、视觉理解、工具使用；
3. `系统层`：吞吐、显存、恢复一致性、服务成本、尾延迟；
4. `业务层`：任务完成率、人工修正率、用户满意度、转化收益。

### 20.1 为什么四层必须同时存在

1. 建模层更接近底层健康指标，但不等于最终可用性；
2. 能力层说明“会不会做”，却不能说明“值不值得部署”；
3. 系统层决定方法能否在同等预算下跑出来；
4. 业务层最贴近价值，但最难定位问题来源。

评测方法学真正要构建的，不是一个万能分数，而是一条从训练到上线的证据链。

## 21. 预训练评测和后训练评测不要混在一起

### 21.1 预训练阶段该看什么

1. validation loss 的绝对值和下降斜率；
2. 各数据域 held-out loss；
3. 长度桶 loss；
4. 多语言或多模态桶 loss；
5. checkpoint 恢复前后一致性；
6. nan、loss spike、梯度爆炸和收敛抖动。

### 21.2 后训练阶段该看什么

1. 指令遵循率；
2. 格式遵循率；
3. 偏好胜率；
4. 工具调用成功率；
5. 长尾失败集修复率；
6. judge 与人工的一致性。

如果把两者混成一张表，团队很容易把行为风格的改变误判成底座能力的大幅跃迁。

## 22. benchmark 组合比单一榜单更重要

完整评测体系通常至少需要五类集合：

1. 社区公开 benchmark；
2. 自建业务专项集；
3. 长尾故障集；
4. 线上回流挑战集；
5. 严格不可触碰的 holdout 集。

### 22.1 公共 benchmark 的价值

1. 方便和社区对齐；
2. 容易发现基础能力缺口；
3. 更适合做方法趋势判断。

### 22.2 公共 benchmark 的局限

1. 容易污染；
2. 不一定覆盖真实业务；
3. 很多任务已经被过度优化，边际信息量下降。

### 22.3 为什么自建集常常更值钱

真正决定产品竞争力的，通常是系统在：

1. 企业文档乱排版；
2. 工具轨迹空转；
3. 多轮记忆丢失；
4. 数字和引用不一致；
5. 安全误拒或过度保守；

这些真实场景里表现如何。

## 23. 数据污染与 benchmark 泄漏

评测失真的高风险来源之一，是 contamination。尤其在开放网络数据和大规模合成数据并存时，泄漏风险会不断上升。

### 23.1 常见污染来源

1. benchmark 原题直接出现在预训练网页中；
2. 题目被改写后在博客、论坛和教程中流传；
3. synthetic data 把 benchmark 风格间接蒸馏进训练集；
4. SFT、偏好集或线上回流集中混入测试近邻样本；
5. 数据去重只做精确匹配，漏掉语义近邻。

### 23.2 污染控制手段

1. n-gram 和 hash 近似去重；
2. 语义近邻召回后 spot-check；
3. 对核心 benchmark 维护 contamination watchlist；
4. 用严格 holdout 集作为最终裁决。

### 23.3 为什么“完全无污染”不现实

现实目标通常不是宣称零污染，而是：

1. 控制高风险污染；
2. 记录污染假设和检查方法；
3. 用更多自建真实集降低对单一公开 benchmark 的依赖。

## 24. 分桶评测应当是主结构，不是附表

平均分太粗时，最先被掩盖的是长尾退化。建议至少维护：

1. 长度桶；
2. 难度桶；
3. 语言桶；
4. 任务桶；
5. 模态桶；
6. 风险桶；
7. 业务价值桶；
8. 失败类型桶。

### 24.1 长度桶示例

例如：

1. `0-512`
2. `512-2k`
3. `2k-8k`
4. `8k-32k`
5. `32k+`

**每个桶都应单独报告**：

1. 主指标；
2. 延迟或成本；
3. 截断率；
4. 引用正确率；
5. 典型失败样本。

### 24.2 失败类型桶更直接指导改进

对真实系统，更有价值的常是：

1. 数字不一致；
2. 证据不支撑结论；
3. JSON schema 破损；
4. 工具参数错误；
5. 早拒答；
6. 多轮状态遗忘。

这些桶与训练改进方向的映射更直接。

## 25. 校准和置信度也是评测对象

很多系统输出的不只是答案，还有：

1. 置信度；
2. 是否调用工具；
3. 是否升级到更强模型；
4. 是否拒答；
5. 是否需要人工复核。

因此校准误差同样影响整体系统收益。

### 25.1 为什么校准重要

如果模型在错误时依然过度自信，那么下游流程会：

1. 错误跳过人工复核；
2. 错误走便宜模型而不升级；
3. 在高风险场景中做出过度激进动作。

### 25.2 可看哪些指标

1. reliability diagram；
2. 置信度分桶后真实正确率；
3. ECE 一类校准误差；
4. 路由阈值变化下的质量-成本曲线。

## 26. Judge 模型进入评测后，judge 本身也要被评测

后训练越来越依赖 judge，但最危险的不是它会出错，而是团队忘了它也只是一个模型。

### 26.1 Judge 至少要做三类验证

1. 与人工的一致性；
2. 对长度、风格、语言的偏差分析；
3. 对高风险样本的错误类型拆解。

### 26.2 Judge 常见偏差

1. 偏爱更长回答；
2. 偏爱格式整齐但事实普通的回答；
3. 低估代码执行失败；
4. 对安全边界判断摇摆。

### 26.3 更稳的做法

**把 judge 拆成多个维度**：

1. factuality；
2. helpfulness；
3. format；
4. safety；
5. execution success。

最后再按业务权重合成，而不是一开始就做一个大总分。

## 27. 高质量消融表应当包含什么

一个真正可复用的消融表，不应只包含模型名和结果。建议至少包含：

1. 实验问题；
2. 基线配置；
3. 唯一变化项；
4. 训练预算；
5. 评测集与分桶；
6. 平均分；
7. 成本；
8. 稳定性；
9. 失败样本摘要。

### 27.1 一个建议模板

| 实验ID | 变更项 | 固定项 | 预算 | 主指标 | 长尾桶 | 成本变化 | 备注 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| A0 | baseline | - | 1.0x | 71.3 | 48.2 | 1.0x | - |
| A1 | 长样本权重 +3% | 其余固定 | 1.0x | 71.5 | 51.8 | 1.04x | TTFT 无关 |
| A2 | reasoning synthetic +5% | 其余固定 | 1.0x | 72.0 | 49.1 | 1.02x | judge 偏长 |

**这种表格能让人直接看出**：收益来自哪里，代价是什么，是否只是某个桶被拉起来。

## 28. 对训练系统实验，必须明确预算口径

比较不同方法时，至少要说明你对齐的是：

1. 同参数规模；
2. 同训练 token；
3. 同 wall-clock；
4. 同 GPU-hour；
5. 同峰值显存；
6. 同上线推理成本。

### 28.1 为什么预算口径决定结论

**因为现实里经常出现**：

1. 某方法同 token 更强，但每步更慢；
2. 某并行方案同 wall-clock 更优，但每 token 更贵；
3. 某对齐方法离线分高，但上线成本不可接受。

不先说清楚预算，争论往往没有意义。

## 29. 系统改动也需要消融，而不是只看吞吐

训练系统里的很多改动都可能改变最终质量：

1. loader 重写；
2. packer 策略调整；
3. tokenizer 版本变化；
4. checkpoint 恢复逻辑变化；
5. mixed precision 路径变化；
6. fused kernel 或 compiler pass 变化。

### 29.1 为什么不能只看吞吐

一个新的 fused kernel 也许：

1. 吞吐更高；
2. 但数值误差略大；
3. 长跑后收敛曲线轻微偏移；
4. 在特定长度桶上放大 instability。

**因此系统变更至少应做**：

1. 数值一致性检查；
2. 短跑 loss 对齐；
3. 长跑 spot-check；
4. 下游核心评测回归。

## 30. 在线评测要防止“伪提升”

**线上实验最容易出现几类假象**：

1. 回答更长，用户主观满意感提高，但任务完成率没变；
2. 过度拒答，事故率下降，但帮助性明显下降；
3. 更激进调用工具，短期正确率升高，但成本和失败调用暴涨；
4. 更换路由阈值后，更多请求落到大模型，总分升了但经济性变差。

### 30.1 在线实验的最小记录字段

1. 模型版本；
2. prompt 版本；
3. 检索和索引版本；
4. 工具版本；
5. 路由版本；
6. 采样参数；
7. 请求长度与模态；
8. 是否命中缓存；
9. 是否升级或回退。

没有这些字段，线上复盘很难做出可信结论。

## 31. 失败实验也应该进入方法学资产

成熟团队不会只保存成功实验，因为失败实验能说明：

1. 哪些直觉是错的；
2. 哪些交互项危险；
3. 训练配方的边界在哪里；
4. 未来哪些方向不值得重复浪费预算。

### 31.1 失败实验应记录什么

1. 原始假设；
2. 具体配置差异；
3. 观察到的负面结果；
4. 失败类型；
5. 可能原因；
6. 是否值得在其他预算口径下重试。

## 32. 一个完整评测流程示例

如果你要验证“提升长上下文文档问答能力”的训练改动，可以采用如下流程：

1. 固定总 token 预算，提升长样本配比；
2. 先看不同长度桶上的 held-out loss；
3. 再看文档问答专项集中的跨页引用正确率；
4. 再看公开 benchmark，确认通用能力没有被局部优化换掉；
5. 再看长尾失败集中的表格、多附件、脏 OCR 场景；
6. 同时记录服务侧 TTFT、prefill 成本和 KV 压力；
7. 小流量线上灰度，追踪任务完成率与人工纠正率；
8. 抽样人工复核，确认 judge 没被“更长更像样”的回答骗到。

这种流程的价值在于，你最终得到的不是一个数字，而是一整条可解释的证据链。

## 33. 设计建议

1. 先定义问题，再设计评测，而不是先找榜单；
2. 公共 benchmark、自建集、长尾故障集、holdout 集同时维护；
3. 所有关键实验都显式记录预算口径；
4. 对 judge、自定义评分器和路由置信度做独立校准；
5. 把失败实验、污染风险和方法边界也文档化。

## 34. 小结

评测与消融方法学不是实验报告的装饰，而是训练系统的证据基础设施。它决定你看到的提升到底是因果还是巧合，是普遍收益还是局部补丁，是能上线的真实改进还是 benchmark 上的幻觉。模型越大、系统越复杂，这套方法学就越不能靠临场经验，而必须被设计成一套长期维护的制度和工具链。

## 35. 评测集不应只有公共榜单，还要有组织私有桶

公共 benchmark 重要，但它们不能覆盖组织自己的真实任务边界。成熟团队通常会维护多层评测桶。

### 35.1 建议的评测桶层次

1. **公共 benchmark**：便于外部对齐和学术比较；
2. **内部主线任务集**：覆盖核心业务场景；
3. **长尾失败集**：积累高价值失效样本；
4. **回归保护集**：防止已修复问题再次出现；
5. **发布门槛集**：只有少量但极关键的上线判定样本。

### 35.2 为什么只靠公共榜单不够

因为公共榜单更擅长反映通用能力，而真正决定业务价值的常常是：

1. **长文档边界场景**；
2. **多轮工具调用一致性**；
3. **罕见语言或模态组合**；
4. **组织特有的格式、流程和风险要求**。

## 36. Judge 模型本身也需要被评测

越来越多训练与评测环节依赖 judge model 或自动评分器，但很多团队对 judge 的信任远超它应得的程度。

### 36.1 Judge 需要单独验证什么

1. **与人工偏好的相关性**；
2. **对长度、格式和措辞偏差的敏感性**；
3. **在不同任务桶上的一致性**；
4. **是否偏爱某类模型风格**；
5. **是否易被 prompt hacking 或格式操控欺骗**。

### 36.2 为什么这一步不可省

若 judge 本身偏置明显，那你在训练中看到的所有提升都可能只是“更会讨好评分器”，而非真正能力提升。

## 37. 消融实验最该问的是“收益归因”，不是“堆表格”

很多消融做成了参数排列组合，但并没有真正回答收益来自哪里。

### 37.1 更高质量的消融问题

1. **这个收益来自更多数据、更多 compute，还是方法本身**；
2. **来自某个单独模块，还是几个模块交互后才成立**；
3. **收益是否集中在少数任务桶，而非整体提升**；
4. **收益是否建立在更高成本、更低稳定性之上**。

### 37.2 为什么这比“多做几个 ablation”更重要

因为组织真正需要的不是更多表，而是能指导下一轮资源投入的因果判断。

## 38. 离线评测和在线评测需要明确接口

离线分数不该和线上指标割裂。更稳的做法是让离线评测直接服务线上决策。

### 38.1 常见接口方式

1. **把线上失败案例回流为离线长尾桶**；
2. **把离线关键分数映射到上线门槛**；
3. **让 canary 和线上实验复用离线样本分类法**；
4. **对 judge、route 和 fallback 的效果做线上线下一致口径分析**。

### 38.2 为什么这能减少“离线很好，线上一般”

因为一旦评测口径和业务口径脱节，训练优化就很容易朝错误方向卷。

## 39. 统计显著性与预算口径必须一起写

当实验预算昂贵时，团队更容易忽略统计纪律，拿一次跑出来的好结果当结论。

### 39.1 建议至少记录

1. **随机种子数**；
2. **每次实验的 compute/token 预算**；
3. **是否做了多次重复**；
4. **收益是否超出自然波动区间**；
5. **是否有早停或选择性汇报风险**。

### 39.2 为什么这很重要

因为大模型实验看起来昂贵，越昂贵越容易让人想少跑几次；但正因昂贵，更不能接受证据质量太差。

## 40. 评测结果应按风险桶和长度桶切开看

一个整体平均分常常掩盖了最关键的信息。

### 40.1 典型切法

1. **按输入长度桶**；
2. **按输出长度桶**；
3. **按任务类型桶**；
4. **按风险等级或失败类型桶**；
5. **按模态组合桶**。

### 40.2 为什么这比总分更有指导意义

因为很多方法的真实收益往往只出现在：

1. **长上下文**；
2. **复杂推理**；
3. **高风险边界样本**；
4. **某类多模态组合**。

如果不切桶，你会不知道这份收益到底值不值得系统复杂度。

## 41. 发布门槛集要小而硬

发布门槛集不应追求覆盖一切，而应聚焦最不能退化的能力。

### 41.1 门槛集常见内容

1. **关键业务任务**；
2. **高风险失败样本**；
3. **监管、许可或安全相关场景**；
4. **组织最在意的长尾问题**。

### 41.2 为什么要保持小

因为门槛集的作用是快速、稳定地阻止明显有问题的版本，而不是取代完整评测。

## 42. 评测工具链也需要版本化

很多实验对比不可信，根因不在模型，而在：

1. **评分脚本变了**；
2. **judge prompt 变了**；
3. **数据切分变了**；
4. **去重或过滤规则变了**。

### 42.1 应该版本化什么

1. **评测数据 manifest**；
2. **评分器和 prompt**；
3. **解析与后处理逻辑**；
4. **置信度和校准配置**。

## 43. 一份更工程化的评测方法学清单

1. **是否同时维护公共榜单与私有桶**；
2. **judge 是否经过独立验证**；
3. **消融是否真正回答收益归因**；
4. **是否明确记录统计显著性与预算口径**；
5. **结果是否按关键桶切开分析**；
6. **是否设有小而硬的发布门槛集**；
7. **评测工具链是否完整版本化**。

## 44. 小结

评测与消融方法学的真正任务，不是生成更多表格，而是建立一套**可比较、可归因、可上线决策**的证据体系。只有这样，训练团队才能把“看起来有提升”的实验，筛成真正值得进入主线配方和生产系统的改进。
