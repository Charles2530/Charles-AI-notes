# 训练中的 Scaling、课程学习与数据混合

很多训练讨论在谈模型结构，但真正决定大模型成长曲线的，往往是另外三件事：

1. `Scaling`
2. `课程学习`
3. `数据混合`

它们共同决定模型在有限计算预算下，会把能力长在哪、短板留在哪。很多团队以为自己在调模型，实际上是在调“资源如何被分给哪些样本、哪些阶段、哪些能力”。

## 1. Scaling 讨论的到底是什么

在最粗的层面，训练表现可以看作模型规模 \(N\)、数据规模 \(D\)、计算规模 \(C\) 的函数：

\[
\mathcal{L} \approx f(N,D,C)
\]

很多 scaling 工作最重要的贡献，不是给出某个具体指数，而是让大家意识到：

- 参数更大不自动意味着更优
- 数据太少时，大模型会浪费
- 计算预算有限时，规模之间需要平衡

这使训练从“拍脑袋堆大”变成更系统的预算分配问题。

### 1.1 为什么 scaling 规律不能机械照搬

真实系统里，\(f(N,D,C)\) 只是一个近似。它至少会受到以下因素扰动：

1. 数据质量并不均匀；
2. 训练目标会从预训练切换到后训练、偏好对齐或工具学习；
3. 多模态数据并不是同质 token；
4. 推理成本会反过来约束模型规模；
5. 系统实现会限制可到达的最优点。

因此 scaling law 更像一张“地形图”，而不是直接可执行的路线图。它告诉你往哪个方向可能更值，但不告诉你每一步该怎么走。

### 1.2 compute-optimal 和 product-optimal 经常不是一回事

某个模型规模可能从训练损失角度最优，但如果上线后显存翻倍、推理尾延迟恶化、路由和缓存复杂度上升，它就未必是产品层最优。这也是为什么成熟团队会同时看：

1. 训练损失下降速度；
2. 下游业务收益；
3. 服务成本与系统复杂度。

如果只看训练曲线，很容易把“更会拟合训练分布”误判成“更适合落地”。

## 2. 为什么数据混合比是训练中的隐形主旋律

**设训练分布为**：

\[
p(x) = \sum_{k=1}^{K}\alpha_k p_k(x),\qquad \sum_k \alpha_k = 1
\]

这里 \(\alpha_k\) 决定了不同数据源在训练中的权重。这不是纯工程细节，它会直接塑造模型能力：

- 网页文本多，知识广但噪声大
- 代码多，代码能力强但对话风格可能变硬
- 指令数据多，交互更顺，但底层语言建模可能被冲淡

### 2.1 数据混合比本质上在定义模型人格

如果把模型理解成一名长期接受训练的学生，那么数据混合比就是它的课程表。课程表不同，最后学出来的不是同一个系统：

1. 网页和百科多，模型更像通才；
2. 代码和仓库文档多，模型更像工程助手；
3. 指令和偏好数据多，模型更像对话产品；
4. 私域文档和流程数据多，模型更像企业内部专家。

因此数据混合并不只是决定“会不会做某个任务”，还决定系统说话方式、风格边界和擅长的知识域。

### 2.2 为什么混合比常被低估

因为它不像新架构那样显眼，也不像更多 GPU 那样容易被量化。但在很多训练迭代中，真正带来能力变化的不是模型结构，而是某类数据终于被正确放大、某类噪声终于被压低、某类长尾终于被保留了下来。

## 3. 一个直观例子：训练企业代码助手

**如果你把**：

- GitHub 代码
- 文档
- FAQ
- 工单记录

混在一起训，而不控制比例，模型可能变成：

- 会写代码，但不会解释企业内部流程
- 会聊天，但代码风格飘
- 知道很多通用知识，却不熟你的私域 API

这说明数据混合比在本质上是在定义模型的“技能画像”。

### 3.1 如果继续加入事故复盘和内部规范

企业代码助手真正有用的部分，往往不是“会写一段排序算法”，而是：

1. 理解内部服务依赖；
2. 知道上线流程；
3. 知道历史故障教训；
4. 会提醒你某个库在内部环境下不该这样用。

这类能力通常来自复盘文档、发布规范、工单和内部 FAQ，而不是来自通用 GitHub 代码。若这些数据在混合中权重过低，模型就会长期停留在“通用开发者助手”，而不是“懂这家公司系统的助手”。

## 4. 课程学习不是老概念，而是大模型训练里依然重要

课程学习可理解为让训练分布随时间变化：

\[
p_t(x) = \sum_{k=1}^{K}\alpha_k(t)p_k(x)
\]

**这意味着**：

- 早期更多基础、稳定数据
- 后期增加复杂、长链或高质量数据

**它适合的场景包括**：

- 长上下文训练
- 复杂推理能力注入
- 多域混合训练

### 4.1 课程学习并不只有“从简单到困难”

现实里的 curriculum 常沿三条轴展开：

1. `长度轴`：先短上下文，再长上下文；
2. `噪声轴`：先高质量数据，再逐步放开噪声更大的开放域数据；
3. `任务轴`：先基础预测，再引入推理、工具、偏好和对齐。

对多模态和具身系统，这三条轴经常同时存在。

### 4.2 多模态中的课程价值更高

VLM 和 VLA 往往面对更异质的数据。如果一开始就把低质 OCR、复杂图表、长文档、多轮视觉问答、遥操作轨迹全混在一起，模型早期表征会非常不稳定。此时合理的课程安排，往往比单纯再堆参数更有效。

## 5. 为什么课程学习有时有效

因为模型早期还在建立基本表征，如果一开始就塞大量难样本，可能会导致：

- 训练不稳
- 学习信号被噪声淹没

而先从更基础、更干净的数据开始，后面再提高复杂度，有时能得到更稳的优化路径。

### 5.1 一个优化视角的解释

课程学习可以理解为让优化器先在更平滑、更低噪的目标地形上找到合理 basin，再逐步把任务难度拉高。若反过来，模型一开始就会被高方差梯度和复杂样本拖着跑，很容易进入不稳定区间。

### 5.2 课程学习特别适合三种任务

1. **长上下文**：先让模型学会短文本逻辑，再逐步理解更长依赖；
2. **工具使用**：先学单步调用格式，再学多步工具规划；
3. **机器人控制**：先学短技能，再学长任务和恢复。

## 6. 但课程学习也不是免费午餐

**它会带来新的问题**：

- 设计难
- 调参难
- 容易引入人为偏置

如果课程划分不好，反而会让模型过度适应某种阶段性分布。

### 6.1 两类典型坏课程

第一类坏课程是“过于理想化”，只在早期喂最干净、最标准的数据，导致模型后期一接触真实分布就受到巨大冲击。第二类坏课程是“主观划分”，团队凭直觉定义什么叫简单和困难，但这种定义与真实任务结构不一致，最后把训练重点引到了错误方向。

### 6.2 课程学习最常见的副作用

不是模型完全不收敛，而是能力长偏：

1. 某些结构化任务突然很好；
2. 某些开放问题却明显退化；
3. 长尾能力一直上不来；
4. 阶段切换时训练曲线出现剧烈抖动。

## 7. 三个典型场景

### 7.1 通用语言模型

**更关注**：

- 数据多样性
- 去重
- 长尾知识
- 长上下文课程

这里常见问题不是“数据不够”，而是“高质量多样性不够”。

### 7.2 代码模型

**更关注**：

- 仓库级去重
- 文档与代码比例
- 长文件和项目级结构
- API 与注释一致性

代码模型若只提高代码占比，不一定就更强，因为项目文档、issue、测试和构建配置同样会影响其真实工程能力。

### 7.3 多模态模型

**更关注**：

- 图文质量
- 高质量配对比例
- 图像分辨率与文本长度混合
- 结构化视觉内容比重

多模态模型里，数据混合比本身经常决定模型最终更偏“自然图像聊天”，还是更偏“文档、图表、屏幕和结构化场景”。

### 7.4 机器人与 VLA

对 VLA，课程学习常体现在：

1. 先学单步技能；
2. 再学多步骤任务；
3. 先学规整环境；
4. 再引入遮挡、反光、动态变化；
5. 最后再加入失败恢复和真实长尾。

如果这一顺序反了，模型往往会表现成“什么都见过，但什么都不稳”。

## 8. 数据混合如何变成工程系统

成熟团队一般不会只维护一个“全量语料池”，而会维护：

1. 数据分层和标签；
2. 各源配比配置；
3. 采样器版本；
4. 不同阶段的 curriculum 配方；
5. 每轮训练后的能力变化报表。

也就是说，数据混合比应当是版本化资产，而不是脚本里一个随手改掉的参数。

### 8.1 一个简单的工程表示

**你可以把每轮训练的配方写成**：

\[
\boldsymbol{\alpha}^{(t)}
=
(\alpha_{\text{web}}, \alpha_{\text{code}}, \alpha_{\text{private}}, \alpha_{\text{inst}}, \alpha_{\text{longctx}})
\]

再把这组参数与 benchmark 变化、服务代价变化对应起来。哪怕不是严格因果分析，这也会比“凭感觉加一点代码数据”更可解释。

## 9. 工程建议

如果你在搭建训练数据配方，建议至少回答：

1. 当前预算更缺参数、数据，还是训练步数？
2. 数据混合比是否与最终任务一致？
3. 是否需要在不同阶段引入不同难度样本？
4. 是否为长尾或高价值数据保留更高权重？
5. 是否能用评测证明某次混合比变化真的有效？
6. curriculum 的切换点是否有明确理由和观测信号？

## 10. 一个更完整的判断

如果把训练系统想成一支球队，那么：

1. scaling 决定你能负担多大阵容；
2. 数据混合决定球员结构和位置配置；
3. 课程学习决定训练顺序和阶段重点。

只看阵容规模，不看结构和训练方式，很难得到真正强队。模型训练也是一样。

## 11. 一个总判断

Scaling、课程学习和数据混合，表面上像“训练配方细节”，实际上它们决定了模型能力最终长成什么形状。很多看似是“模型架构问题”的现象，最后都会回到这三件事上。真正成熟的训练系统，通常不是先问“有没有更大的模型”，而是先问“预算、数据结构和课程安排是否已经对齐目标能力”。 

## 12. 把数据配方写成真正可执行的版本资产

很多团队说自己有 mixture，但真正落地时仍停留在口头描述，比如“这轮把代码和长文本再加一点”。更稳妥的做法，是把配方写成可审计、可复现、可比较的配置对象。例如：

```yaml
recipe: mix_v23_longctx_code_balance
phase: pretrain_main
global_batch_tokens: 8388608
sampler:
  type: token_budgeted_temperature
  temperature: 0.72
  min_examples_per_bucket: 1024
sources:
  - name: web_text
    weight: 0.38
    max_length: 4096
    quality: medium
  - name: code_repo
    weight: 0.19
    max_length: 8192
    quality: high
  - name: docs_and_forums
    weight: 0.12
    max_length: 4096
    quality: medium
  - name: reasoning_math
    weight: 0.08
    max_length: 6144
    quality: high
  - name: long_context_docs
    weight: 0.11
    max_length: 32768
    quality: gated
  - name: multilingual_web
    weight: 0.12
    max_length: 4096
    quality: medium
curriculum:
  length_ramp:
    start_step: 0
    end_step: 120000
    start_len: 2048
    end_len: 8192
  longctx_boost:
    start_step: 150000
    end_step: 260000
    start_weight: 0.03
    end_weight: 0.11
```

这个配置对象的价值不在格式，而在于它强迫团队回答：

1. 哪些数据真的被用了，哪些只是盘点时写上了名字；
2. 某类数据的权重变化到底从哪一步开始；
3. 长样本和高质量样本是否被单独 gate；
4. 每次训练结果变化，是模型结构导致，还是采样器悄悄变了。

### 12.1 为什么配方必须版本化

因为训练退化最难查的一类问题，就是“明明模型和优化器都没变，但结果变了”。很多时候根源在于：

1. 某个数据池被重抓了一次；
2. tokenizer 版本更新后样本长度分布变了；
3. packer 逻辑调整，长样本曝光率变化；
4. curriculum 切换点改了，但实验文档没写清楚。

如果配方没有独立版本号，这些变化就会被埋在脚本和日志里。

## 13. Scaling 在工程里真正对应什么

论文里讨论 scaling，常写成参数规模、数据规模和计算预算之间的函数关系。但对训练负责人来说，它最终会变成几项很具体的决策：

1. 参数规模定到多少；
2. 总 token 预算给多少；
3. 最大上下文长度定多少；
4. 能接受多大的分布式复杂度；
5. 上线后的推理成本能否回收训练投入。

### 13.1 参数、token 和长度不是三个互不相关的旋钮

把参数做大，常常意味着：

1. 单步计算更贵；
2. 优化器状态更大；
3. 通信压力更高；
4. 为了装下模型，需要更多 TP、PP、EP 或 ZeRO 分片。

把最大长度拉长，又会意味着：

1. attention 成本上升；
2. packing 逻辑变化；
3. 长样本 token 利用率下降；
4. 激活 checkpoint 和 sequence parallel 变得更关键。

所以很多“只是把 context 从 8k 拉到 32k”的改动，实际上改动了整条训练栈。

### 13.2 compute-optimal 和 product-optimal 之间的缝隙

即便某个规模在训练损失上最优，也未必是产品最优。因为产品侧还要看：

1. 推理 TTFT；
2. 单请求显存；
3. 多副本部署成本；
4. 量化后性能保持；
5. 尾延迟和缓存压力。

**更现实的优化目标往往接近**：

\[
\text{Utility}
=
\lambda_1 \cdot \text{quality}
+ \lambda_2 \cdot \text{product gain}
- \lambda_3 \cdot \text{train cost}
- \lambda_4 \cdot \text{serve cost}
- \lambda_5 \cdot \text{risk}.
\]

这就是为什么训练规划必须和推理、容量规划一起讨论。

## 14. 数据混合不只是混来源，还在混长度、噪声和难度

很多文档把 mixture 理解成“网页 40%、代码 20%、指令 10%”。这只是第一层。更完整的表达往往应写成：

\[
p(x)=\sum_{k=1}^{K}\sum_{b=1}^{B}\alpha_{k,b} p_{k,b}(x),
\]

其中 \(k\) 是来源，\(b\) 是长度桶、难度桶或质量桶。

### 14.1 长度混合

如果长样本占比太低，模型的长上下文能力常常只是“架构允许”，而不是“训练学会”。但如果长样本占比太高，又会：

1. 拉低有效 token 吞吐；
2. 让早期训练信号变稀疏；
3. 提高 padding 和 packing 成本；
4. 过早暴露不稳定的长依赖样本。

所以常见做法不是一开始就把长样本权重拉满，而是分阶段逐步提高。

### 14.2 难度混合

**难度可以用很多 proxy 近似**：

1. 输入长度；
2. 稀有词或符号比例；
3. 是否需要跨段引用；
4. 是否涉及工具或多步推理；
5. 是否来自历史失败集。

如果完全不做难度感知，模型会把太多预算花在早已熟练的简单模式上。

### 14.3 噪声混合

开放域语料的价值，往往在于覆盖面大；它的风险，也往往在于噪声大。更稳妥的做法常是：

1. 先用高质量小池打底；
2. 中期放开更大规模开放数据；
3. 后期再提高高质量池权重，重新压住噪声；
4. 对 synthetic 和爬虫数据单独做 gate。

## 15. 课程学习在大模型里的几种具体形态

### 15.1 长度课程

这是目前最常见的一类。典型流程是：

1. 早期以 `2k` 或 `4k` 为主；
2. 中期逐步引入 `8k`、`16k`；
3. 后期才加大 `32k+` 的比例；
4. 必要时再做单独的 long-context phase。

这里要注意的不是“max length 有没有改大”，而是：

1. 训练栈能否支撑；
2. 长样本是否真有足够供给；
3. 短上下文能力是否被拖累；
4. 评测是否覆盖真实长依赖而不是只看长度。

### 15.2 任务课程

很多能力不适合一开始混着来训，例如：

1. 先基础语言和知识；
2. 再格式和指令；
3. 再工具使用；
4. 再多步规划；
5. 最后才是高风险拒答和复杂 agent 轨迹。

如果一上来就把复杂轨迹和 noisy reasoning 样本大量混入，模型经常只会学到表面模板。

### 15.3 模态课程

对 VLM、VLA、世界模型尤其重要。一个更稳的顺序通常是：

1. 单图像和短文本；
2. 多页、长文档、结构化图表；
3. 视频、多帧、多视角；
4. 动作轨迹、交互恢复和高风险场景。

### 15.4 噪声课程

不是所有高覆盖数据都值得在所有阶段等权进入。很多团队会在后期把高质量数据重新提高权重，本质上是在做一种“末期净化 curriculum”。

## 16. 为什么 curriculum 很难复现

因为你以为你改的是“样本顺序”，实际可能同时改了：

1. 样本长度分布；
2. token budget；
3. pack 后实际利用率；
4. optimizer 历史；
5. validation 分布；
6. 恢复 checkpoint 的统计量路径。

**因此课程学习实验必须额外记录**：

1. 切换步数；
2. 各 phase 的有效 token 数；
3. 长度桶占比；
4. 每 phase 的 validation 指标；
5. 系统吞吐变化。

## 17. Synthetic data 应该如何进入 mixture

合成数据越来越重要，但它不应简单替代真实数据。更可控的方案通常是：

1. 给 synthetic 单独池和单独配额；
2. 只在目标缺口处补，如工具轨迹、复杂结构化输出、罕见错误恢复；
3. 用 judge、执行结果或人工检查做 gating；
4. 记录 teacher 版本和生成模板版本。

### 17.1 三种常见 synthetic 用法

1. `coverage fill-in`：补某类业务或 benchmark 缺口；
2. `trajectory densification`：把工具调用轨迹和中间步骤写完整；
3. `counterfactual augmentation`：围绕 near-miss 样本生成条件变化的对照例。

### 17.2 Synthetic 的主要风险

1. 风格闭环，自我蒸馏出模板化回答；
2. judge 和 teacher 同源，偏差相互放大；
3. 高质量表象掩盖事实空洞；
4. 表面 reasoning token 增多，但真实正确率没涨。

## 18. 多语言和跨域 mixture 的特殊问题

### 18.1 多语言不是按 token 均匀就公平

如果简单按语料量采样，低资源语言很容易被淹没。因此往往需要：

1. 上采样高质量低资源语料；
2. 单独监控语言桶；
3. 对机器翻译扩增设置上限；
4. 分清“会寒暄”和“具备专业能力”。

### 18.2 跨域混合最怕 schema 不一致

企业文档、代码仓库、工单、FAQ 混合时，最常见的问题并不是主题不同，而是：

1. 时间版本冲突；
2. 同一实体命名不一致；
3. 结构化字段被展平后语义丢失；
4. 内部规范和公开文档矛盾。

这类问题如果不在数据治理阶段解决，后续任何 curriculum 都只是在更稳定地学习矛盾。

## 19. 数据混合与训练框架强耦合

这也是为什么 `Megatron-LM`、`DeepSpeed` 一类训练栈会反过来塑造配方设计。很多 mixture 看起来只是数据问题，但它能否落地，取决于训练栈是否支持：

1. 高效 packing；
2. token-based batch scheduling；
3. sequence/context parallel；
4. 长样本下稳定 checkpoint；
5. ZeRO、TP、PP、EP 组合下维持吞吐。

例如 long-context curriculum 若把吞吐打到太低，团队最后就会被迫减少长样本占比，形成“理论上想学，系统上喂不进去”的局面。

## 20. 数据混合的典型失效模式

### 20.1 领域挤占

提升某类权重，往往会挤占别的能力。例如：

1. 代码占比过高，对话风格变硬；
2. 偏好数据过高，基础 factual recall 下降；
3. reasoning synthetic 过高，形式化推理痕迹增多但真正确率没有提升。

### 20.2 高损失错觉

高损失样本不一定高价值，它也可能只是：

1. 标签错误；
2. OCR 乱码；
3. prompt 模板损坏；
4. 数据重复拼接导致不可学。

### 20.3 过度课程化

phase 切得过细时，会带来：

1. 超参数爆炸；
2. 解释困难；
3. 阶段切换点抖动；
4. 回归验证成本过高。

## 21. 一个训练配方设计清单

设计新的 scaling 与 mixture 策略时，建议至少回答：

1. 总 token 预算和 wall-clock 上限是多少；
2. 每个 phase 的能力目标是什么；
3. 长度桶、难度桶、质量桶如何定义；
4. synthetic 和低质量开放域如何 gate；
5. 是否需要单独的 long-context 或 reasoning phase；
6. 哪些评测桶会用来验证收益；
7. 训练栈能否支撑目标长度和吞吐。

## 22. 小结

`Scaling`、课程学习和数据混合，本质上都是资源分配问题：在有限预算下，把多少训练信号、在什么阶段、以什么长度和质量，分给模型的哪些能力。真正成熟的训练团队不会把它们当作附属调参项，而会把它们配方化、版本化、评测化，并和训练框架、后训练闭环、推理成本一起联合设计。只有这样，scaling law 才不会停留在论文图表上，而会变成能落地的训练生产线。

## 23. Scaling 先决定“预算切几段”，再谈细节配方

很多团队把 scaling 简化成“给更多 token”。更工程化的做法是先问：**总预算应该分成几个阶段，每个阶段的主要能力目标是什么。**

### 23.1 常见阶段划分

1. **基础语言/视觉建模阶段**；
2. **高质量数据强化阶段**；
3. **长上下文或复杂推理专项阶段**；
4. **多模态或动作条件专项阶段**；
5. **后训练或对齐准备阶段**。

### 23.2 为什么分阶段比统一混合更好

1. **不同阶段需要不同长度分布**；
2. **不同阶段对数据质量敏感度不同**；
3. **不同阶段的系统代价不同**；
4. **有些能力若过早注入，收益并不高，反而拖慢吞吐**。

## 24. Curriculum 的关键不是“由易到难”这么简单

课程学习常被口号化，真正落地时至少要明确三类 curriculum：

1. **长度 curriculum**；
2. **质量 curriculum**；
3. **能力目标 curriculum**。

### 24.1 长度 curriculum

先用更短上下文打基础，再逐步抬高长上下文比例。好处是：

1. **早期吞吐更高**；
2. **系统更稳**；
3. **长上下文预算可留给更后期更有价值的阶段**。

### 24.2 质量 curriculum

不是简单“先低后高”或“先高后低”，而是看目标：

1. **早期高质量有助于稳定基础表征**；
2. **中期引入更广覆盖数据有助于泛化**；
3. **后期再拉高高质量权重有助于收口**。

### 24.3 能力目标 curriculum

例如：

1. **先基础语言，再代码/数学**；
2. **先静态图文，再视频时序**；
3. **先观察建模，再动作联合建模**。

## 25. Data Mixture 应该被看成“预算分配器”

数据混合不是配方装饰，而是训练预算的真正分配器。

### 25.1 它至少在决定

1. **哪类数据被看到更多**；
2. **哪类能力更早被学到**；
3. **哪类风险被更充分覆盖**；
4. **训练吞吐会不会被长样本或重模态拉低**。

### 25.2 一个常见误区

把 mixture 固定成一组权重后一路跑到底。现实中更有效的常常是：

1. **阶段性调 mixture**；
2. **针对失败桶定向加权**；
3. **针对新能力目标临时建专项 phase**。

## 26. 长上下文阶段最好单独做容量与收益核算

长上下文很贵，因此不应只是“多加一些长样本”。更好的做法是把它当成一个单独 phase 核算。

### 26.1 需要单独核算什么

1. **额外 GPU 小时**；
2. **吞吐下降比例**；
3. **数据系统和 packer 的复杂度增加**；
4. **对目标评测桶的真实收益**；
5. **是否影响基础短上下文能力**。

### 26.2 为什么这样做

因为很多团队会在长上下文上花很大预算，却没有验证收益是否足够覆盖系统成本。

## 27. Synthetic Data 在 curriculum 里更像“专项加速器”

合成数据通常不适合作为无差别主粮，更适合作为明确能力目标的加速器。

### 27.1 典型适用场景

1. **结构化推理格式**；
2. **工具使用轨迹**；
3. **长尾边界样本扩增**；
4. **世界模型或控制里的风险分支扩充**。

### 27.2 为什么要放进专门 phase

因为如果长期高权重注入 synthetic，模型可能会：

1. **学到模板偏置**；
2. **忽略真实噪声结构**；
3. **在离线评分器上变好但真实泛化一般**。

## 28. Mixture Governor 应该有正式职责

在大团队里，数据混合不应是谁想改就改。更成熟的组织会把 mixture governor 视为正式职责。

### 28.1 这个角色需要回答

1. **每次权重变更的目标是什么**；
2. **会影响哪些能力桶与吞吐指标**；
3. **是否已有回滚条件**；
4. **哪组评测桶负责验证收益**；
5. **是否会影响污染、许可或成本风险**。

### 28.2 为什么这有必要

因为 mixture 是最容易产生“看起来只是改了点权重，实际上整个训练分布都变了”的地方。

## 29. Scaling Law 真正有用时，通常要和系统成本一起看

很多 scaling 曲线只讨论 loss 与 token 或参数的关系，但工程上更值钱的是：

1. **同样多预算，投给更大模型还是更多高质量 token**；
2. **长上下文阶段是否真值得占这么多预算**；
3. **某种复杂能力是否必须靠额外 phase，还是主训练已足够**；
4. **系统瓶颈是否会让理论 scaling 收益落不下来**。

### 29.1 为什么这能改变策略

因为理论最优不一定是系统最优。若长上下文阶段把吞吐打得太低，或者某类高质量数据清洗成本极高，最佳预算分配可能和纯论文结论不同。

## 30. 评测门槛应嵌在 phase 切换点

每个训练 phase 切换前，最好都有清晰门槛，而不是凭直觉“差不多可以进下一阶段了”。

### 30.1 典型门槛

1. **基础桶是否已稳定**；
2. **长上下文专项是否已达到最小收益阈值**；
3. **多模态专项是否没有明显伤到通用能力**；
4. **吞吐与稳定性是否仍在预算内**。

### 30.2 为什么 phase gating 重要

它能防止团队把大量预算投进尚未准备好的下一阶段，或者在上一阶段已经边际收益很低时继续盲烧卡。

## 31. 一份更工程化的 scaling 清单

1. **总预算是否已切成明确阶段**；
2. **每个阶段是否有能力目标与系统预算**；
3. **mixture 是否支持阶段性调整**；
4. **长上下文和 synthetic 是否被单独核算**；
5. **phase 切换是否有明确评测门槛**；
6. **scaling 判断是否和真实系统成本一起看**。

## 32. 小结

`Scaling`、课程学习和数据混合真正要解决的，不是“多加一些 token”这么简单，而是**怎样把有限预算按阶段、长度、质量和能力目标做最优分配**。一旦把这件事做成正式的预算治理与 phase 管理，训练配方才会从经验调参升级为可迭代、可审计的生产策略。
