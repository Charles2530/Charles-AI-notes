# 训练稳定性、数值异常与故障排查

大模型训练里最贵的失败，往往不是“效果差一点”，而是训练跑到中后期突然 `nan`、loss spike、恢复后曲线漂移、低精度路径悄悄失真，最后一整段 GPU 小时白白损失。训练稳定性、数值异常与故障排查，正是把这些高代价问题拆成可观测、可定位、可复现的流程。

**这页重点讨论**：

1. 大模型训练中常见的数值与稳定性故障；
2. 为什么很多异常看起来像优化器问题，根因却在数据、并行或恢复路径；
3. 如何建立一套更有效的 triage 顺序；
4. 如何把稳定性问题变成长期可积累的工程资产。

## 1. 训练不稳定不只是一种现象

**常见异常至少可以分成几类**：

1. 立即爆炸型：刚起跑不久就 `nan`；
2. **中途尖刺型**：loss 长时间正常，某一步突然 spike；
3. **缓慢漂移型**：训练没挂，但质量越来越不对；
4. **恢复后失真型**：checkpoint 恢复后曲线和之前不连续；
5. **低精度隐性型**：没有立刻报错，但数值逐渐恶化。

### 1.1 为什么先分类很重要

因为不同异常的高概率根因完全不同。  
不先分类，团队就很容易把所有问题都归咎于“学习率太大”。

## 2. 最常见的数值故障源

### 2.1 数据异常

例如：

1. tokenizer 损坏；
2. 空样本或极端长样本；
3. label mask 错位；
4. 多模态对齐错误；
5. 乱码或非法值进入前处理。

### 2.2 优化器与 schedule

例如：

1. warmup 太短；
2. batch 变化但学习率没改；
3. 恢复后 scheduler 状态错位；
4. 某阶段切换目标函数时没有重新校准超参。

### 2.3 低精度路径

例如：

1. loss scaling 失控；
2. FP8/FP16 统计量不同步；
3. fused kernel 数值边界差；
4. master weight 与实际权重路径不一致。

### 2.4 分布式通信与并行

例如：

1. 某个 rank 梯度异常但未及时暴露；
2. all-reduce 后出现隐性不同步；
3. pipeline stage 间状态错位；
4. EP/CP 路径下的边界条件问题。

## 3. Loss spike 最常见的几类根因

### 3.1 数据分布突变

curriculum 切换、数据版本变化、长样本比例突然提高，都可能让 loss 在切换点剧烈抖动。

### 3.2 恢复状态不完整

**尤其是**：

1. optimizer moment 丢失；
2. grad scaler 丢失；
3. sampler 错位；
4. 学习率调度器恢复错误。

### 3.3 低精度与 fused kernel 的边界问题

这种问题最麻烦，因为：

1. 平时可能很稳定；
2. 只在特定 shape 或特定 batch 组合触发；
3. 很容易被误判成随机波动。

## 4. `nan / inf` 不一定来自“模型太大”

**现实中更常见的是**：

1. 输入里有非法值；
2. mask 逻辑错误；
3. 某一层输出被异常放大；
4. 低精度范围溢出；
5. 某个 rank 先坏，再在同步中扩散。

### 4.1 为什么要做逐层与逐阶段缩圈

如果一出现 `nan` 就直接全局改学习率，排障效率会很低。更稳妥的做法通常是：

1. 先定位出问题是在输入、前向、反向还是更新后；
2. 再定位是哪个模块、哪种 shape、哪一类样本更容易触发；
3. 最后才改超参或实现。

## 5. 恢复后曲线漂移是一类单独问题

这类问题特别常见，因为表面上训练还能继续，但其实统计已经变了。

### 5.1 常见原因

1. sampler 状态没恢复；
2. RNG 状态变化；
3. scheduler 和 global step 对不上；
4. 某些并行配置变化；
5. 数据快照、pack 或 tokenizer 版本变化。

### 5.2 为什么这类问题危险

因为团队很容易误以为“恢复成功了”，而实际上后续实验已经不再可比。

## 6. 如何建立一套更有效的 triage 顺序

**推荐顺序通常是**：

1. 先看是不是新数据、新 checkpoint、新并行配置引入的；
2. 再看异常是即时爆炸、阶段切换、还是恢复后出现；
3. 再拆输入、前向、反向、优化器更新四个阶段；
4. 再看是否和特定长度桶、特定模态或特定 rank 相关；
5. 最后才决定是改超参还是改实现。

### 6.1 为什么“先缩圈”比“先调参”更重要

因为大量稳定性问题本质上是实现或数据错误。  
如果不先缩圈，改学习率只是在给错误实现打补丁。

## 7. 需要哪些监控才能真正排障

**建议至少单独记录**：

1. loss 与 grad norm；
2. grad scaler 或 FP8 统计；
3. 每层激活和梯度的极值；
4. 每个 rank 的异常计数；
5. 数据源和长度桶曝光率；
6. 恢复事件时间线；
7. checkpoint 写入与加载日志。

### 7.1 为什么按 rank 监控重要

有些分布式问题只会先在个别 rank 上出现。  
如果所有监控只看全局平均，异常会被稀释，直到彻底爆炸才暴露。

## 8. 一个更实用的故障类型表

### 8.1 刚起跑就炸

**优先怀疑**：

1. 输入异常；
2. mask 错；
3. 学习率过大；
4. 低精度初始化路径。

### 8.2 跑很久后 spike

**优先怀疑**：

1. 数据切换；
2. 恢复问题；
3. 某类 rare sample；
4. checkpoint 恢复后的隐性状态错位。

### 8.3 不挂但效果越来越怪

**优先怀疑**：

1. 数据漂移；
2. mixture 失真；
3. optimizer 或 schedule 与阶段不匹配；
4. 输入系统统计悄悄变化。

## 9. 稳定性问题如何沉淀成团队资产

**每次高价值故障都建议记录**：

1. 症状；
2. 触发条件；
3. 根因；
4. 修复方式；
5. 如何监控提前发现；
6. 是否需要加入回归测试。

### 9.1 为什么回归测试重要

如果同一类问题修完不进回归，团队在下一轮更复杂训练里很可能还会再遇到。

## 10. 设计建议

1. 把稳定性问题先分类，再排障；
2. 数据、恢复、低精度、并行四类路径分别监控；
3. 对恢复训练单独做一致性校验；
4. 每次高价值故障都进入 playbook 和回归测试；
5. 不要把所有异常都粗暴归结为“学习率太大”。

## 11. 小结

训练稳定性和数值异常排查的核心，不是更快找一个万能超参，而是把高代价故障拆成可观察、可缩圈、可回归的问题。模型越大、训练越长、分布式栈越复杂，这套 triage 能力就越像训练系统的安全壳。谁能把故障排查做成制度化能力，谁就更能在大规模训练中稳定积累有效进展，而不是不断把 GPU 小时烧在同一类可避免问题上。


## 12. BF16、FP16、FP8 的故障画像并不相同

很多团队会把混合精度问题统一叫成“低精度不稳定”。这会让排障过于粗糙。实际上，不同精度路径的故障画像明显不同。

### 12.1 `FP16` 常见问题更像“动态范围不够”

`FP16` 路径下，常见现象是：

1. 梯度或激活容易溢出，导致 `inf` / `nan`；
2. loss scaling 配置不合适时，训练在某些阶段反复振荡；
3. 某些 fused kernel 内部累加路径过短，误差在长序列或大 batch 下放大；
4. 恢复训练后 loss scaler 状态没衔接好，导致恢复点前后数值环境突变。

因此，`FP16` 排障通常要重点看：**loss scale、异常层定位、累加精度、grad clip 触发频率和恢复点前后的 scaler 行为**。

### 12.2 `BF16` 更稳，但不是“天然无脑稳”

`BF16` 因为指数位更大，通常比 `FP16` 稳定得多，所以很多团队会把大量训练默认迁到 `BF16`。但它常见的问题会换一种形态出现：

- 不是频繁爆 `nan`，而是**更隐蔽的精度退化**；
- 某些 layernorm、softmax、reduction 路径仍需要 FP32 累加；
- 某些 fused optimizer 或自定义 kernel 在 `BF16` 下逻辑正确但数值误差偏大；
- 训练可以跑通，但长程收敛略差，直到对比充分实验才发现。

**所以 `BF16` 的关键不是“会不会炸”，而是“会不会悄悄变差”。** 这要求监控不仅看故障，还要看长期质量漂移。

### 12.3 `FP8` 问题通常更像“校准链路不成熟”

`FP8` 的价值主要在吞吐和显存，但排障画像又不同：

1. `amax` 或 scale 更新节奏不合适，导致某些层长期量化失真；
2. 前几千步统计还不稳定时，模型更容易出现不必要的早期波动；
3. 不同实现的 scale 粒度、累加路径和 fallback 策略差异很大；
4. 某些层适合 `FP8`，某些层必须保留高精度，混合策略不当会造成局部崩坏。

因此，`FP8` 排障除了看 loss 曲线，还要看：

- 各层 `amax` 分布；
- scale 更新频率；
- fallback 层比例；
- 异常 batch 出现时是否集中在特定模块；
- 恢复训练后量化统计是否正确续接。

## 13. 优化器状态损坏和分布式状态错乱

很多数值问题表面看是 loss spike，实质却来自优化器状态、master weight 或梯度聚合状态不一致。

### 13.1 这类问题为什么难查

因为模型参数本身可能看起来完全正常。问题出在：

- 某个 rank 的 Adam `m/v` 没有正确恢复；
- 某个 shard 的 master weight 被错误覆写；
- ZeRO/FSDP 在 gather/scatter 过程中混入旧状态；
- 分布式 grad clip 的范数统计不一致；
- 恢复后某些参数组学习率调度不同步。

于是你看到的症状往往是：**没有立刻报错，但几百步后曲线慢慢偏离**。

### 13.2 识别信号通常是什么

| 信号 | 更像什么问题 |
| --- | --- |
| **恢复后几百步内效果稳步变坏** | optimizer/scheduler 状态未完全恢复 |
| **只有部分 rank 报溢出或梯度极大** | shard 状态错乱或局部数据异常 |
| **训练能跑但评测明显退化** | 低精度或 master weight 精度路径有问题 |
| **同配置重跑结果波动异常大** | RNG / sampler / optimizer 状态未对齐 |

### 13.3 更好的排查顺序

1. 先确认权重是否对齐；
2. 再确认 optimizer state、scheduler step、loss scale；
3. 再确认 sampler、数据版本、packing 状态；
4. 最后才去怀疑更深层的 kernel 或模型结构问题。

这样能显著减少“其实是恢复链路问题，却花了几天调超参”的无效劳动。

## 14. 长上下文、MoE、多模态三类系统的特殊稳定性问题

### 14.1 长上下文训练为什么更容易出现“局部正常、全局异常”

因为长上下文会把数值问题放大到多个层面：

- attention softmax 的数值范围更大；
- activation checkpoint 重算路径更长；
- 某些 batch 的有效 token 数波动显著；
- 显存峰值更高，更容易触发 allocator 抖动；
- CP/SP 或 paged attention 相关路径可能引入新的同步与 layout 问题。

这类场景里，**必须按长度 bucket 维度单独看故障分布**。如果只看全局平均，很容易把长序列 bucket 的系统性问题淹没掉。

### 14.2 MoE 稳定性问题往往绕不过路由

MoE 模型常见的不稳定，很多并不是来自专家本身，而是来自：

1. 路由分布塌缩；
2. load balance loss 权重不合适；
3. 某些专家长期过载；
4. all-to-all 通信和路由噪声交织；
5. 专家参数更新频率不均衡。

所以 MoE 的稳定性监控至少要额外记录：**每个专家 token 数、专家负载熵、丢 token 比例、路由概率分布、专家梯度范数**。

### 14.3 多模态训练容易把数据问题伪装成数值问题

多模态系统经常同时涉及图像、文本、视频、动作等多种输入，于是以下问题会互相混淆：

- 图像坏样本或解码失败；
- 文本模板变更；
- 视频长度分布漂移；
- 动作标签尺度变化；
- 不同模态 loss 权重失衡。

最终症状可能是 loss 抖动、梯度爆炸或某个分支几乎不学习，但根因其实在数据和标签语义层，而不是纯数值层。**因此多模态排障必须先把数据语义核对完，再谈 kernel 或 precision。**

## 15. 一个更可靠的数值缩圈流程

如果训练出了稳定性问题，建议按下面顺序缩圈，而不是先改一堆超参：

### 15.1 第一步：确认问题属于哪一类

先判断是：

1. **立刻崩溃型**：几步内 `nan/inf/OOM`；
2. **中途爆炸型**：跑一段时间后突然 spike；
3. **隐性退化型**：训练不挂但质量越来越差；
4. **恢复偏移型**：从 checkpoint 恢复后明显不对；
5. **只在特定 bucket 触发型**：长序列、多模态、特定专家或特定数据源才触发。

分类之后，排障效率会高很多。

### 15.2 第二步：把变量减到最少

常见缩圈动作包括：

- 固定一个最小数据片段；
- 关闭部分数据源；
- 降到单节点或更小并行配置；
- 关闭部分 fused kernel；
- 关闭 activation checkpoint 或某种 overlap；
- 暂时回退到更高精度。

**注意**：缩圈的目的不是证明小模型能跑，而是找出哪一类机制一旦去掉，问题就消失。

### 15.3 第三步：建立逐层与逐阶段观测

真正高效的排查常依赖以下观测：

| 观测层 | 典型指标 |
| --- | --- |
| **输入层** | 长度分布、pack ratio、模态缺失率、坏样本率 |
| **前向层** | 激活均值/方差、amax、异常层输出 |
| **反向层** | grad norm、per-layer grad 异常、overflow 位置 |
| **优化层** | Adam m/v 范围、weight update ratio、loss scale |
| **分布式层** | rank 间范数差异、通信超时、bucket 异常 |
| **恢复层** | 恢复前后 step 级对比、sampler 状态、一致性 hash |

## 16. 数值测试不能只靠“和 PyTorch 对一对”

很多自定义 kernel 或训练路径，只做了简单的前向误差对比。这远远不够。

### 16.1 更完整的数值测试矩阵

建议至少覆盖：

1. **前向误差**：不同 shape、不同范围、不同 dtype；
2. **反向误差**：梯度与参考实现对齐；
3. **边界输入**：全零、超大值、超小值、极端长度；
4. **随机回归**：固定 seed 的多轮结果稳定性；
5. **训练级 smoke test**：短跑若干步，看 loss 是否合理下降；
6. **恢复级测试**：中途存档、恢复、再跑，比较轨迹是否连续。

### 16.2 为什么恢复测试必须单独做

因为很多问题根本不会出现在“从头连跑”的测试里，而只会在：

- state_dict 序列化 / 反序列化；
- distributed shard 重组；
- optimizer state 类型转换；
- loss scaler 恢复；
- 数据顺序恢复

这些路径上暴露。

## 17. 预防性设计比事后排查便宜得多

### 17.1 一个实用的训练前稳定性清单

在正式长跑前，建议至少做：

1. **短跑单卡 / 单节点验收**；
2. **短跑分布式验收**；
3. **插入一次 checkpoint 恢复验收**；
4. **最长序列或最大图像分辨率 stress test**；
5. **坏样本和异常输入扫描**；
6. **关键 fused kernel 的 reference 对比**；
7. **主要监控项和报警阈值确认**。

### 17.2 监控项要“够早发现”而不是“够多”

最值钱的监控通常不是数量最多的，而是最能提前预警的。例如：

- 某些层的 `amax` 或 activation variance 开始漂；
- pack ratio 突然降低；
- 个别 rank 的 step time 长尾抬升；
- 某些专家长期过载；
- 恢复后学习率步数与 global step 不一致。

这些信号往往在 loss 真正崩掉之前就已经给出提示。

## 18. 故障复盘模板建议

把稳定性问题真正沉淀成团队资产时，可以统一记录以下字段：

| 字段 | 建议内容 |
| --- | --- |
| **症状** | spike、nan、恢复漂移、吞吐抖动、质量退化 |
| **首次出现时间** | 第几步、第几个 checkpoint 后、哪次版本升级后 |
| **触发范围** | 所有任务还是特定模型/数据源/长度桶 |
| **临时止血方案** | 回退精度、关闭某个 fused kernel、降序列长度 |
| **根因** | 数据、数值、通信、恢复、调度、代码缺陷 |
| **永久修复** | 代码修复、配置变更、测试补充、监控补充 |
| **回归测试** | 哪个 smoke test / unit test / replay test 能覆盖 |

**这份复盘一旦标准化**，后续再遇到类似问题时，排障速度会明显提升。

## 19. 扩展小结

训练稳定性本质上是一个跨层问题：上接数据和任务定义，下接低精度、kernel、分布式状态和恢复链路。谁把它简单理解为“调学习率”，谁就会在大模型、长上下文、MoE 和多模态训练里不断重复踩坑。更成熟的做法，是把稳定性当作系统工程的一条主线：按故障类型分类、按状态语义做恢复校验、按负载场景做专项监控、按复盘模板沉淀资产。这样训练系统才会越来越稳，而不是每次换模型规模或换任务形态就重新裸奔。

## 20. BF16、FP16、FP8 的稳定性治理不应混为一谈

虽然都属于低精度训练，但三条路径的故障触发方式和最佳治理手段并不相同。

### 20.1 `BF16`

**优势**：

1. **动态范围相对更宽**；
2. **许多场景下比 FP16 更稳**；
3. **对大模型训练更友好**。

**常见问题**：

1. **不是所有 kernel 都在 BF16 路径上同样成熟**；
2. **某些归约或优化器状态若仍处理不当，照样会出数值问题**；
3. **团队容易因“BF16 很稳”而放松监控**。

### 20.2 `FP16`

**典型风险**：

1. **更容易 overflow/underflow**；
2. **更依赖 loss scaling**；
3. **对极端激活和梯度更敏感**。

### 20.3 `FP8`

**真正难点**：

1. **不仅是表示位宽更低**，还涉及 scaling metadata 与校准策略；
2. **不同层、不同算子对 FP8 的可承受度差异更大**；
3. **训练和推理用的 FP8 路径可能并不完全一致**。

### 20.4 为什么必须分开治理

因为一个在 BF16 下稳定的模型，切到 FP8 后暴露的问题往往不是“把学习率调小一点”就能解决，而需要重新审视：

1. **哪些层可降精度**；
2. **scale 的更新逻辑**；
3. **fused kernel 的数值路径**；
4. **监控指标是否足够细**。

## 21. MoE 稳定性是一类单独的故障体系

MoE 训练里的不稳定，往往同时来自数值和系统两层。

### 21.1 常见触发点

1. **router logits 极端化**；
2. **expert 负载严重失衡**；
3. **capacity factor 太紧导致 token drop 波动大**；
4. **稀有 expert 的优化器状态长期不稳定**；
5. **dispatch / gather 路径在极端 token 分布下出错**。

### 21.2 为什么 MoE 更难排

因为异常可能表现成：

1. **loss spike**；
2. **部分 rank 吞吐突然下降**；
3. **少数 expert 权重漂移异常**；
4. **训练不挂但泛化明显变差**。

这要求团队额外监控：

1. **expert load 分布**；
2. **router entropy**；
3. **token drop 比例**；
4. **各 expert 的 grad norm / activation 范围**。

## 22. 长上下文训练会放大一切数值边界

很多本来在 4k 或 8k 序列长度下罕见的问题，到 32k、64k 甚至更长时会显著增加。

### 22.1 原因

1. **attention 和 reduction 链路更长**；
2. **rope、位置编码和 mask 逻辑更复杂**；
3. **长样本更容易携带异常格式或长尾模式**；
4. **activation 和 checkpoint 重算成本更高，使排障迭代更慢**。

### 22.2 需要单独看的指标

1. **按长度桶统计 loss 与 grad norm**；
2. **长样本下的激活极值**；
3. **长样本占比变化与吞吐/异常的相关性**；
4. **长上下文恢复后的曲线连续性**。

## 23. 多模态训练里的异常通常先伪装成“优化器问题”

多模态训练中，图像、视频、音频和文本一旦对齐出错，loss 曲线常常会表现得像：

1. **学习率太大**；
2. **某个模态难学**；
3. **batch 太小**。

### 23.1 实际可能的根因

1. **图像与文本配对错位**；
2. **视频 clip 时间窗错一两帧**；
3. **OCR / ASR 结果与图像裁剪版本不一致**；
4. **模态占位 token 与标签区间错位**。

### 23.2 为什么特别危险

因为这些问题通常不会立刻 `nan`，而是让模型在错误监督下缓慢偏离。若没有样本级可视化和对齐检查，很容易长期误判。

## 24. Rank-local 观测必须足够细

很多大作业里，异常首先出现在少数 rank，但全局平均会把它们稀释掉。

### 24.1 建议按 rank 跟踪

1. **grad norm**；
2. **overflow / scaler 事件**；
3. **每层激活极值**；
4. **step time**；
5. **数据源与长度桶分布**；
6. **通信错误计数与重试**。

### 24.2 为什么 rank-local 视角重要

1. **某一张坏卡或某个坏节点只会先影响局部**；
2. **某个数据 worker 的问题也常只投射到相关 rank**；
3. **EP/CP 路径下部分 rank 负担更特殊，更易先暴露问题**。

## 25. 故障回放能力往往比更多监控更值钱

监控很重要，但真正能加速排障的，常常是**把同一批输入和同一 checkpoint 重新回放**的能力。

### 25.1 一个可回放系统至少需要

1. **问题发生前后的样本 id 或 batch id**；
2. **对应 checkpoint 或最近一致性状态点**；
3. **当时的配置、代码版本与并行配置**；
4. **可重放的随机种子**。

### 25.2 为什么这能显著缩短排障周期

因为很多数值问题不是静态看日志能推出来的，而必须在最接近出事的状态下逐层缩圈。

## 26. 故障 triage 更适合用“三段式”

当训练异常发生时，建议把排查拆成三段：

1. **先止血**；
2. **再定位**；
3. **后归因和制度化**。

### 26.1 先止血

优先判断是否要：

1. **停止作业**；
2. **切回更稳精度或禁用某个 fused path**；
3. **降低序列长度或关闭某个新数据源**；
4. **回滚到上一稳定 checkpoint**。

### 26.2 再定位

1. **缩小到输入、前向、反向、更新哪一段**；
2. **判断是全局问题还是 rank-local 问题**；
3. **判断是否与新变更、恢复或数据切换相关**。

### 26.3 后归因和制度化

1. **形成复盘**；
2. **补监控**；
3. **补回归测试**；
4. **补上线门槛**。

## 27. 什么情况下应果断停训练

不是所有异常都值得带病继续跑。一个更工程化的判断是：

1. **若出现持续 `nan/inf` 且定位不清，应停**；
2. **若恢复后关键指标明显不连续，应停**；
3. **若只有少数 step spike 但可归因于可控样本桶，可边隔离边观察**；
4. **若异常来自某个新 kernel、精度路径或数据版本，应优先回退而非硬扛**。

### 27.1 为什么这很关键

因为团队最常见的高成本错误之一，就是在异常已经明显存在时还继续烧很多 GPU 小时，希望“也许后面会自己好起来”。

## 28. 稳定性门槛应进入实验发布流程

大训练实验上线前，不应只检查功能和吞吐，还应检查稳定性门槛。

### 28.1 建议门槛项

1. **固定 smoke run 无异常**；
2. **恢复训练短跑一致**；
3. **关键长度桶和关键模态桶无明显异常**；
4. **低精度统计量在正常范围**；
5. **新增 kernel / 新数据源 / 新并行路径已有专项回归**。

### 28.2 为什么要制度化

否则每次新实验都会重新用生产规模去验证最基本稳定性，成本极高。

## 29. 一份更实用的稳定性资产清单

团队如果想把稳定性变成平台能力，而不是个人经验，建议沉淀：

1. **故障类型字典**；
2. **按精度路径划分的排障手册**；
3. **按模型族和任务族划分的已知高风险配置**；
4. **标准化回放脚本**；
5. **恢复一致性测试**；
6. **故障后必填复盘模板**。

## 30. 小结

训练稳定性不是“出了问题再去调”的附属工作，而是训练平台必须长期经营的核心能力。只要把**精度路径、模型结构、数据制度、分布式状态、恢复语义和故障制度化**一起治理，团队就能把许多本来要靠运气避免的事故，提前变成可监控、可回放、可回归的问题。
