# 分布式训练与 Checkpoint

当模型规模从几亿参数走向几十亿、上百亿甚至多模态长序列系统时，训练问题很快不再只是“调学习率”，而是“如何把算力、显存、通信、容错和恢复组织成一个持续可运行的系统”。分布式训练与 checkpoint 管理，正是现代大模型工程的主干。

如果把训练看成一个完整生产系统，那么这页讨论的就不是“模型学什么”，而是“训练能不能真的跑完、跑稳、跑快、跑得可恢复”。这也是为什么到了大模型阶段，`Megatron-LM`、`Megatron Core`、`DeepSpeed`、`ZeRO`、`FSDP`、流水线调度、异步 checkpoint、弹性恢复这些名字，会迅速变成主角。

本页从并行维度、系统栈、通信瓶颈、存储设计与故障恢复几个层面展开。

## 1. 为什么单卡思维会失效

单卡训练里，人们更关心损失函数和 batch size；分布式训练里，额外出现至少六个现实约束：

1. 参数放不下；
2. 激活放不下；
3. 优化器状态占用巨大；
4. 通信成为瓶颈；
5. 任意一台机器故障都会拖垮作业；
6. 保存和恢复 checkpoint 的时间越来越贵。

设模型参数量为 \(P\)，每个参数字节数为 \(b_p\)，优化器状态倍数为 \(c_{\text{opt}}\)，则仅模型状态所需显存约为

$$
M_{\text{state}} \approx P \cdot b_p \cdot (1 + c_{\text{opt}}).
$$

对 Adam 而言，若参数和梯度使用 BF16，动量与二阶矩常用 FP32，则 \(c_{\text{opt}}\) 通常远大于 1。参数本身不是问题的全部，优化器状态才是真正的大头之一。

### 1.1 大模型训练真正受限的通常不是 FLOPs 本身

很多人初看会以为“只要 GPU 算力够强就能训起来”，但真实瓶颈更常出在：

1. 通信带宽跟不上；
2. 显存碎片化；
3. 激活峰值过高；
4. pipeline bubble 太大；
5. checkpoint 落盘太慢；
6. 集群故障率随着规模上升而上升。

因此分布式训练不是“把模型拆一拆”这么简单，而是需要同时平衡：

1. 单卡内存；
2. 节点内互联；
3. 跨节点网络；
4. I/O 子系统；
5. 作业调度与恢复。

## 2. 并行维度的统一视角

现代大模型训练不再只靠一种并行。更常见的统一写法是把总 world size 分解为：

$$
N_{\text{world}}
=
N_{\text{DP}}
\times
N_{\text{TP}}
\times
N_{\text{PP}}
\times
N_{\text{CP}}
\times
N_{\text{EP}}.
$$

这里：

1. `DP` 是数据并行；
2. `TP` 是张量并行；
3. `PP` 是流水线并行；
4. `CP` 是上下文并行；
5. `EP` 是专家并行。

并不是每个系统都会同时用到这些维度，但这个写法很有帮助，因为它提醒我们：训练规模不是单一轴扩展，而是多维度共同构成的拓扑设计问题。

### 2.1 为什么“用了很多并行”不等于系统更强

并行维度越多，理论容量越大，但代价也越大：

1. 进程组变复杂；
2. 通信模式更碎片化；
3. 调试和恢复更困难；
4. checkpoint 更难迁移；
5. 某一维配置不合理会拖垮整体效率。

真正好的分布式设计，不是把所有并行都打开，而是让并行维度和硬件拓扑、模型结构、序列长度及 batch 组织相匹配。

## 3. 数据并行：最基础也最常见

数据并行（DP）把不同 mini-batch 切到不同设备，每个设备都有一份完整模型副本。前向和反向独立执行，最后通过 AllReduce 汇总梯度。

设总 batch 为 \(B\)，设备数为 \(N\)，单设备 batch 为 \(B/N\)。梯度同步后，更新等价于在大 batch 上训练。其优点是概念简单、扩展性好；缺点是模型副本完整复制，显存占用高。

通信开销粗略可写成

$$
T_{\text{comm}} \propto \frac{P \cdot b_g}{\text{bandwidth}},
$$

其中 \(b_g\) 是梯度字节数。随着参数量增大，AllReduce 很快成为瓶颈。

### 3.1 数据并行的工程细节

1. **梯度累积**：通过多步累积模拟更大 batch，减少同步频率。
2. **Bucket 化**：把梯度分桶，边反传边通信，重叠计算和通信。
3. **混合精度**：降低传输字节数。
4. **ZeRO/FSDP**：进一步分片参数与优化器状态。

### 3.2 数据并行为什么常被误解为“最便宜的扩展方式”

它实现最直接，但并不总是最便宜。因为一旦模型副本、梯度和优化器状态都完整复制，显存浪费会非常明显。很多系统最后不是算不动，而是内存先炸，或者通信先把吞吐打下来。

## 4. 张量并行：把一个算子拆到多卡

当单层矩阵乘本身太大时，可使用张量并行（TP）。例如线性层

$$
Y = XW
$$

可以按列或按行切分 \(W\)，让不同 GPU 负责不同子矩阵。这样单卡显存压力下降，尤其适合超大隐藏维度模型。

TP 的代价是层间通信频繁。因为一次前向中的矩阵乘结果常需要在多卡间拼接或聚合，所以它通常更依赖高速互联，如 NVLink 或同机箱拓扑。跨节点 TP 代价往往较高。

### 4.1 TP 解决的是什么问题

TP 主要解决的是“单层太宽，单卡算不下或效率差”的问题，而不是整体模型层数过深的问题。它适合：

1. 超大 hidden size；
2. 超大前馈层；
3. 注意力投影矩阵过大；
4. 单层 GEMM 本身就需要拆。

### 4.2 TP 的代价为什么集中在通信上

TP 每经过一个大算子，就常需要：

1. Reduce-Scatter；
2. All-Gather；
3. All-Reduce；
4. 层间中间结果同步。

因此 TP 的有效性高度依赖节点内高带宽低延迟互联。很多实践里，TP 尽量限制在单机或同一高速互联域内，就是为了避免跨节点通信把收益吃掉。

## 5. Sequence Parallel 与 Context Parallel

当序列长度变长时，只拆参数还不够，因为激活和注意力内存会迅速膨胀。这时就会出现与序列维相关的并行方法。

### 5.1 Sequence Parallel

Sequence parallel 可以理解为：在某些张量并行场景下，把原本在序列维上重复保存的一部分激活或中间状态进一步分摊到不同 rank。它常和 TP 一起出现，目的是减少激活内存和部分通信压力。

从工程角度看，可以把它理解成“给 TP 配套的激活分摊机制”。没有它，TP 在长序列和某些 MoE 场景下会更容易出现内存或效率问题。

### 5.2 Context Parallel

Context parallel 更偏向把长上下文直接切到多个设备上，让不同设备处理不同上下文片段，再在注意力等关键阶段做必要通信。它主要服务超长上下文训练和推理统一化需求。

**其收益在于**：

1. 更长序列能放得下；
2. 注意力激活峰值下降；
3. 某些长上下文模型不必单纯依赖更大 activation recompute。

**其代价在于**：

1. 通信模式更复杂；
2. 注意力实现需要配合；
3. 对 kernel、拓扑与实现细节更敏感。

### 5.3 为什么长序列训练离不开这类技术

当序列长度从 `4k` 走到 `32k`、`128k` 甚至更长时，训练不再是“多开几张卡”就能解决。真正难的是：

1. 激活按序列长度线性甚至更差地增长；
2. 注意力相关算子和缓存压迫显存；
3. checkpoint 重算成本也被放大。

因此长上下文训练通常是 TP、CP、activation checkpointing、flash attention 和内核优化的联合问题。

## 6. Pipeline 并行：按层切网络

Pipeline parallelism（PP）按层把模型切成多段，不同设备负责不同段。微批次像流水线一样依次流经各段。其好处是每张卡只保存部分层；坏处是存在 bubble，即某些时刻有设备闲置。

若 pipeline 段数为 \(K\)，微批次为 \(m\)，理想化利用率近似为

$$
\text{utilization} \approx \frac{m}{m + K - 1}.
$$

因此想提高利用率，通常需要足够多的微批次。但微批次过多又会影响优化和显存。

### 6.1 1F1B 与 interleaving

PP 的核心不只是“切层”，还包括调度。常见思路包括：

1. all-forward-all-backward：实现简单，但 bubble 大；
2. `1F1B`：一前一后交替，减少流水线空泡；
3. **interleaved pipeline**：把一个 stage 再虚拟切分，进一步减少 bubble。

**这类调度设计直接影响**：

1. 显存峰值；
2. 激活保留时间；
3. 利用率；
4. 调试复杂度。

### 6.2 PP 为什么非常依赖 micro-batch 设计

DeepSpeed 官方流水线说明里就强调，pipeline parallelism 是通过 micro-batch 和 gradient accumulation 抽取出来的。一个全局 batch 会拆成多个 micro-batch，前向和反向在不同 stage 上交错执行。  
如果 micro-batch 太少，pipeline 气泡很大；如果太多，调度复杂度、激活管理和优化行为都会受影响。[DeepSpeed Pipeline](https://www.deepspeed.ai/tutorials/pipeline/)

## 7. 专家并行与 MoE

当模型进入 Mixture-of-Experts 路线时，还会出现专家并行（EP）。它的核心是：并不是所有 token 都走所有 FFN，而是由 router 把 token 分配给部分专家，专家本身分布在不同设备上。

### 7.1 EP 解决的是什么

EP 主要解决的是“参数量可以继续上去，但单 token 激活计算不能等比例上去”的问题。也就是把模型容量增长和单次计算增长部分解耦。

### 7.2 EP 的代价是什么

1. token dispatch / gather 通信复杂；
2. 负载均衡很难；
3. 容量因子和 drop 策略影响训练稳定；
4. 常需要和 TP、DP、PP 联合设计。

因此 MoE 训练系统比稠密模型更依赖高质量的并行与通信实现。

## 8. 3D / ND 并行：现代大模型训练的主流组合

现实系统常把 DP、TP、PP 组合，形成所谓 3D parallelism。若再叠加上下文并行、专家并行等，系统复杂度会继续上升。

**大致思路是**：

1. TP 解决单层太大问题；
2. PP 解决模型层数太深问题；
3. DP 负责横向扩展吞吐；
4. ZeRO/FSDP 进一步压缩冗余状态；
5. CP 解决超长上下文；
6. EP 解决 MoE 容量扩展。

关键不是“用了多少种并行”，而是你的硬件拓扑、模型结构和通信能力是否匹配。

### 8.1 一个常见的拓扑思路

**很多生产系统会优先让**：

1. TP 留在单节点；
2. PP 跨若干节点扩展；
3. DP 再在更外层横向复制；
4. ZeRO/FSDP 在 DP 组内分片状态。

**这种分法背后的直觉是**：  
越频繁的通信，越应该放在高带宽局部域；越粗粒度的复制，越适合跨更大拓扑范围做。

## 9. `Megatron-LM` / `Megatron Core` 在训练栈里扮演什么角色

如果从工程定位来理解，`Megatron-LM` 更像“大规模 Transformer 训练参考系统”，`Megatron Core` 更像“可组合的 GPU 优化训练库”。NVIDIA 当前官方 README 也把二者明确区分开：前者偏参考脚本和研究实践，后者提供 Transformer 构件、TP/PP/DP/EP/CP 等并行策略、混合精度和更模块化的训练基础设施。[Megatron-LM](https://github.com/NVIDIA/Megatron-LM)

### 9.1 为什么 Megatron 重要

Megatron 路线的影响力主要在于：

1. 它把 TP 和 PP 的工程实现做成了大模型训练基石；
2. 它长期代表“大规模 Transformer 该如何高效拆分”的主流实践；
3. 它与内核优化、通信重叠、长上下文和 MoE 扩展结合得很深。

### 9.2 Megatron 更适合解决哪类问题

**更适合**：

1. 训练超大稠密 Transformer；
2. 多维并行组合；
3. 对吞吐与 MFU 有较高要求；
4. 需要和底层 GPU 拓扑、内核和通信深度耦合的场景。

**相对没那么适合的是**：  
只想用最少系统复杂度快速把中等规模模型训起来的团队。因为它的收益往往来自更重的系统设计与实现成本。

### 9.3 Megatron 里真正重要的不只是 TP/PP

官方 README 现在也明确提到，它不仅覆盖 TP、PP、DP，还覆盖 EP、CP，以及通信重叠、混合精度、checkpoint 与容错等完整训练管线能力。[Megatron-LM](https://github.com/NVIDIA/Megatron-LM)

这意味着 Megatron 不应被简单理解成“一个模型并行库”，而应理解成“大规模训练系统参考实现”。

## 10. `DeepSpeed` 在训练栈里扮演什么角色

如果说 Megatron 更强调超大模型的并行切分和 GPU 优化，那么 `DeepSpeed` 更像一个更偏训练运行时与内存优化的平台。它的代表能力包括：

1. ZeRO 系列状态分片；
2. pipeline parallel runtime；
3. offload 到 CPU / NVMe；
4. 混合并行训练引擎；
5. checkpoint 与训练脚手架集成。

### 10.1 DeepSpeed 为什么常和 ZeRO 绑定出现

因为 ZeRO 是它最具标志性的能力之一。DeepSpeed 官方教程把 ZeRO 分成三个经典阶段：

1. `Stage 1`：分片优化器状态；
2. `Stage 2`：再分片梯度；
3. `Stage 3`：再分片参数；

并且在 `ZeRO-3` 之上进一步支持 CPU / NVMe offload，也就是常说的 ZeRO-Infinity 路线。[DeepSpeed ZeRO](https://www.deepspeed.ai/tutorials/zero/)

### 10.2 DeepSpeed 更适合解决哪类问题

**更适合**：

1. 单纯 DP 已经放不下，但又不想立刻进入更重 TP/PP 设计；
2. 想先靠状态分片和 offload 把大模型训起来；
3. 需要比较完整的训练运行时配置体系；
4. 希望把 pipeline、ZeRO、激活 checkpoint 等系统能力统一在一个运行时里管理。

## 11. `Megatron + DeepSpeed` 为什么经常一起出现

两者之所以常被组合，是因为它们擅长的层面不同：

1. Megatron 擅长模型并行和大模型结构切分；
2. DeepSpeed 擅长运行时、ZeRO、offload 和训练引擎组织。

DeepSpeed 官方 pipeline 教程也直接提到，其 pipeline engine 可以与 Megatron-LM 这类 model parallelism 组合，形成混合 3D parallelism。[DeepSpeed Pipeline](https://www.deepspeed.ai/tutorials/pipeline/)

### 11.1 一个粗暴但实用的理解

**可以把二者关系理解成**：

1. Megatron 负责“模型怎么拆”；
2. DeepSpeed 负责“拆完以后怎么更省内存、更好调度、更能跑”。

当然，这只是帮助建立直觉的简化说法。真实系统里二者边界会互相渗透，尤其是现在 Megatron Core 本身也在不断补齐训练系统能力。

## 12. ZeRO 与 FSDP 的核心思想

**数据并行最大的浪费是**：每张卡都完整保存参数、梯度和优化器状态。ZeRO 通过分阶段分片减少冗余：

1. **Stage 1**：分片优化器状态；
2. **Stage 2**：再分片梯度；
3. **Stage 3**：再分片参数。

FSDP 与其思路相近，更强调 module 级全分片与按需 gather。一个简化理解是：平时大家只存各自那一小片，前向或反向经过某层时，临时把需要的权重 gather 回来，用完再释放。

### 12.1 什么时候更偏向 ZeRO

**更常见于**：

1. 已经在 DeepSpeed 运行时体系中；
2. 想用成熟的 zero stage 配置快速调参；
3. 想结合 CPU / NVMe offload；
4. 希望对大模型内存问题做更渐进式处理。

### 12.2 什么时候更偏向 FSDP

**更常见于**：

1. 更贴近 PyTorch 原生生态；
2. 希望减少对特定训练运行时的依赖；
3. 模型和框架定制较多；
4. 希望更自然地接入原生 distributed stack。

### 12.3 它们真正的共同代价

无论 ZeRO 还是 FSDP，本质都是拿更多 gather/scatter 与状态管理复杂度换显存。真正决定成败的不是名字，而是：

1. 分片粒度是否合理；
2. 参数 wrap 策略是否合适；
3. 通信是否能重叠；
4. checkpoint 是否可恢复且可迁移。

## 13. Activation Checkpointing 与重计算

除了模型状态，激活也是显存大户。Activation checkpointing 的做法是：前向时不保存所有中间激活，只在若干边界点存储；反向时再局部重算。

如果一个 block 的前向计算代价为 \(C_f\)，不 checkpoint 时显存为 \(M_f\)，checkpoint 后可把显存压到更低水平，但代价是反向额外重算，训练总算力需求上升。工程上常把它看作“用计算换显存”的手段。

对长上下文 Transformer 而言，这几乎是标配。

### 13.1 Activation checkpointing 与流水线/ZeRO 是耦合问题

很多新手会把这些技巧分别看待，但真实系统里它们互相影响：

1. checkpointing 会改变每层前向/反向时序；
2. 这会影响 pipeline stage 的显存峰值；
3. 也会影响 ZeRO-3 的参数 gather 节奏；
4. 最终还会影响 checkpoint 落盘时的状态组织。

所以它通常不是单独调一个开关，而是整个训练拓扑联调的一部分。

## 14. 通信重叠与拓扑意识

到了大规模训练阶段，通信优化几乎和算子优化同等重要。常见思路包括：

1. grad reduce 与反向计算重叠；
2. param gather 与前向重叠；
3. TP 通信与 kernel 执行细粒度重叠；
4. PP 点对点传输与计算重叠；
5. 尽量让高频通信留在节点内。

Megatron 当前公开 README 也明确强调了 DP、TP、PP 通信重叠相关优化，例如 `--overlap-grad-reduce`、`--overlap-param-gather`、`--tp-comm-overlap` 等能力。[Megatron-LM](https://github.com/NVIDIA/Megatron-LM)

### 14.1 为什么“网络够快”仍然不等于系统就快

因为通信性能不只由名义带宽决定，还取决于：

1. 通信粒度；
2. collective 类型；
3. 是否频繁跨节点；
4. kernel 调度是否打断通信；
5. 微批次大小是否让重叠成为可能。

一个带宽很高但粒度不合理的系统，仍然可能表现很差。

## 15. Checkpoint 到底应该保存什么

很多人以为 checkpoint 就是模型权重文件，实际上一个可恢复训练状态至少包括：

1. 模型参数；
2. 优化器状态；
3. 学习率调度器状态；
4. 随机数状态；
5. 数据加载进度 / sampler 状态；
6. 全局 step 和 epoch；
7. 混合精度缩放器状态；
8. 并行拓扑相关元数据；
9. tokenizer / vocab 或数据管线相关关键配置。

如果缺少其中某些项，你恢复后的训练轨迹很可能不连续，尤其在大规模长跑任务中会造成隐藏偏差。

### 15.1 为什么分布式训练对元数据要求更高

单卡恢复最多是“优化器状态是否一致”；  
**分布式恢复还要问**：

1. 当前 world size 是多少；
2. TP/PP/DP/EP 分组是什么；
3. sharded state 该怎么重组；
4. 数据流该从哪个 shard 的哪个位置继续；
5. 混合精度和 loss scaling 是否延续。

缺了这些，恢复出来的可能不是“续训”，而是“表面续训，实际重新开始另一条轨迹”。

## 16. Checkpoint 存储成本为什么越来越大

设参数量为 \(P\)，参数存 BF16，优化器状态存 FP32，额外有梯度主副本，则一次完整 checkpoint 的体量可能近似为

$$
S_{\text{ckpt}} \approx P \cdot (b_{\text{param}} + b_{\text{master}} + b_{\text{opt1}} + b_{\text{opt2}}).
$$

对几十亿参数模型，这往往是数百 GB 到 TB 级。若每小时保存一次，就会很快给对象存储或并行文件系统带来巨大压力。

### 16.1 ZeRO/FSDP 让 checkpoint 更复杂而不是更简单

虽然状态分片降低了单卡内存，但也带来了 checkpoint 组织复杂度：

1. state_dict 不再天然完整；
2. 保存时可能需要 gather；
3. 不同拓扑恢复时需要重映射；
4. 离线导出 full weights 可能额外昂贵。

DeepSpeed 官方 ZeRO 教程就专门提醒，ZeRO-3 下 `state_dict` 默认并不包含完整权重，如需导出完整 16-bit 权重，需要显式打开 gather 相关保存选项。[DeepSpeed ZeRO](https://www.deepspeed.ai/tutorials/zero/)

## 17. Checkpoint 的几种策略

### 17.1 Full checkpoint

保存一切，恢复最完整。优点是稳，缺点是大。

### 17.2 Sharded checkpoint

每个 rank 只保存自己负责的状态分片。恢复时再按当前拓扑重组。它更适合 ZeRO/FSDP，但要求良好的元数据管理。

### 17.3 Async checkpoint

训练主线程只把状态快照交给后台线程或独立进程落盘，尽量减少停顿。难点在于：

1. 一致性保证；
2. I/O 峰值控制；
3. 元数据提交顺序；
4. 后台写入失败时的回滚。

### 17.4 Incremental checkpoint

只保存相对上一个 checkpoint 的变化块。理论上节省空间，但实现复杂，对恢复链条完整性要求高。

### 17.5 双层 checkpoint

工程上常见的不是只选一种，而是双层策略：

1. 高频保存轻量 step metadata 或 sharded checkpoint；
2. 低频保存全量、可迁移、可归档的大 checkpoint。

这样既兼顾恢复速度，又兼顾长期可追溯性。

## 18. 保存频率如何权衡

若 checkpoint 间隔为 \(\Delta t\)，平均故障恢复损失近似可看成 \(\Delta t / 2\) 的训练时间；而保存本身耗时为 \(T_{\text{save}}\)。因此总成本可粗略写为

$$
\text{Cost}(\Delta t)
\approx
\frac{T_{\text{train}}}{\Delta t} T_{\text{save}}
+ \lambda \frac{\Delta t}{2},
$$

其中第二项代表故障导致的重复训练损失。大集群里，机器越多，故障率越高，往往就越需要更频繁但更高效的 checkpoint。

### 18.1 为什么大集群更偏好异步和分层保存

**因为节点越多**：

1. 平均无故障运行时间越短；
2. 保存窗口越难和训练节奏完全对齐；
3. 任意一次大停顿都会放大集群成本。

所以“少存点，坏了再补跑”在大集群上通常不是好策略。

## 19. 恢复训练时容易忽略的坑

### 19.1 数据重复或跳样

如果 sampler 状态没保存，恢复后可能重新读到之前数据，或者跳过一部分数据。这在严格配比训练和 curriculum learning 中影响很大。

### 19.2 学习率调度不连续

warmup、cosine decay、restart 若恢复错误，会导致 loss 曲线突然抖动。很多“恢复后训练发散”并不是模型坏了，而是 scheduler 状态不对。

### 19.3 全局 batch 变化

若恢复时设备数变化，全局 batch size、梯度累积和学习率缩放可能都要重新推导。不能简单地“接着跑”。

### 19.4 并行拓扑变化

比如从 `8 x TP, 8 x PP, 16 x DP` 恢复成另一种组合时，最难的往往不是文件能不能读，而是：

1. 参数分片布局是否兼容；
2. 优化器状态能否正确重映射；
3. pipeline 切层是否保持一致；
4. 数据并行组内统计量是否还可信。

### 19.5 以为“权重恢复成功”就等于训练状态恢复成功

这是一种非常常见的误判。权重只是训练状态的一部分。真正的训练连续性还依赖：

1. optimizer moment；
2. RNG；
3. sampler；
4. grad scaler；
5. 全局 step；
6. 并行配置元数据。

## 20. 容错设计

在大规模集群上，节点故障是常态而不是例外。容错策略包括：

1. 定期 checkpoint；
2. 自动重试作业；
3. 节点剔除与弹性重排；
4. 训练状态校验；
5. 损坏 checkpoint 的回退链；
6. 恢复后关键指标自检。

**一个成熟系统至少要能回答**：

1. 某个 rank 挂了怎么办？
2. 恢复是否允许 world size 变化？
3. checkpoint 不完整时能否检测并回滚？
4. 恢复后如何验证状态一致？
5. 是否需要短时间 shadow 监控恢复后的损失和吞吐？

## 21. `Megatron-LM`、`DeepSpeed`、`FSDP` 该怎么选

没有单一标准答案，但可以用一个工程化判断框架：

### 21.1 更偏 `Megatron`

**当你需要**：

1. 超大 Transformer 训练；
2. TP/PP/CP/EP 等多维并行；
3. 极高吞吐与通信重叠优化；
4. 更贴近 GPU 拓扑和底层 kernel 的控制。

### 21.2 更偏 `DeepSpeed`

**当你需要**：

1. 以 ZeRO 为核心先解决内存问题；
2. 快速接入 pipeline runtime 与 offload；
3. 倾向通过配置驱动训练运行时；
4. 希望训练脚手架更完整。

### 21.3 更偏原生 `FSDP`

**当你需要**：

1. 更贴近 PyTorch 主生态；
2. 中到大规模模型的全分片训练；
3. 较少依赖专门训练框架；
4. 更自然地和自定义模型代码整合。

### 21.4 为什么很多团队最后是混合路线

因为真实系统通常不会纯粹。常见形态包括：

1. Megatron Core 做模型并行，外层接入其他运行时；
2. DeepSpeed 负责 ZeRO 和 pipeline，模型结构借鉴 Megatron；
3. PyTorch FSDP 为主，只在特定模型或阶段引入更重的并行系统。

## 22. 一个更具体的系统例子

假设训练一个 `70B` 模型，采用 `8-way TP`、`8-stage PP`、`16-way DP`，同时使用 ZeRO-1。每次保存 full checkpoint 都会造成明显停顿。更现实的设计可能是：

1. 训练中每 `30` 分钟保存 lightweight step metadata；
2. 每 `2` 小时做一次异步 sharded checkpoint；
3. 每天做一次全量可迁移 checkpoint；
4. 关键版本额外复制到异地对象存储；
5. 恢复后自动跑一小段校验，确认 loss、吞吐和数据读取正常。

这样既降低恢复成本，也兼顾长期归档。

## 23. 一个形象比喻

可以把分布式训练想成修一条横跨多个城市的铁路。数据并行像让多支施工队平行铺设不同地段，张量并行像多人合力吊装同一段超大桥梁，流水线并行像不同工序依次推进，专家并行像按列车类型把不同维修工派往不同站点，而 checkpoint 则像定期把施工现场的进度、图纸、材料和机器状态全部登记归档。

没有归档，一次暴雨或设备事故后，你可能根本不知道该从哪里恢复，更糟的是恢复后还以为自己接上了之前的进度，实际上已经悄悄偏航。

## 24. 小结

分布式训练不是若干并行名词的堆叠，而是一个在显存、通信、拓扑、存储和故障率之间持续平衡的系统工程。`Megatron-LM` 代表的是大模型多维并行与高吞吐训练的主流路线，`DeepSpeed` 代表的是以 ZeRO、pipeline runtime 和 offload 为核心的训练运行时路线，`FSDP` 则代表更原生的全分片路径。它们不是简单互斥关系，而是不同工程约束下的不同取舍。

Checkpoint 也不是“顺手存个权重”，而是决定训练是否可持续、可恢复、可追溯的关键基础设施。模型越大，这两件事越不是附属问题，而是主问题。

## 25. 训练前的系统验收应该检查什么

在真正启动大作业前，建议做一轮系统验收，至少包括：

1. 并行拓扑是否和设计一致；
2. 单步吞吐是否达到预期区间；
3. checkpoint 写入与恢复是否打通；
4. sampler 与数据版本是否可追踪；
5. loss、grad norm、学习率在短跑中是否稳定；
6. 恢复后短跑曲线是否与连续跑一致。

### 25.1 为什么短跑验收值钱

**它能提前暴露**：

1. 错误的并行组；
2. checkpoint 元数据缺失；
3. 学习率和 batch 推导错误；
4. 数据顺序和 curriculum 配置不一致；
5. 混合精度路径的隐藏不稳定。

## 26. 一个更实用的分布式排障顺序

如果大训练出了问题，更稳妥的定位顺序通常是：

1. 先判断是吞吐问题、显存问题、数值问题还是恢复问题；
2. 若是吞吐问题，拆数据、通信、kernel 和 pipeline bubble；
3. 若是显存问题，拆参数状态、激活、长序列中间态和碎片；
4. 若是数值问题，先看混合精度、梯度缩放、恢复点和 fused optimizer；
5. 若是恢复问题，重点看 sampler、scheduler、RNG 和并行元数据。

## 27. 小结

大规模分布式训练真正困难的地方，不在于知道多少并行术语，而在于能否把拓扑、通信、checkpoint、恢复和实验元数据组织成可验证、可恢复、可解释的系统。谁能把这些流程做成稳定机制，谁才有资格把更大模型和更长训练真正跑成资产，而不是一次性豪赌。


## 28. 让并行设计落地：从逻辑网格到物理机器

教材里的并行设计通常画成一个规整张量：某个维度做 TP，某个维度做 PP，某个维度做 DP，再乘上 CP 或 EP，看起来像把总 GPU 数分解成几个整数因子。真正把它放到集群里时，还必须回答：这些维度具体落在哪些 GPU、哪些节点、哪些机架，以及跨维度通信是否会在真实网络上互相打架。

### 28.1 一个从逻辑到物理的映射模板

可以把映射过程拆成四步：

1. **先确定最怕延迟的通信维度**：例如 TP 的 all-reduce、某些 EP 的 all-to-all；
2. **再确定最怕泡沫的串行维度**：例如 PP stage 之间的发送接收；
3. **然后看最适合横向复制的维度**：通常是 DP；
4. **最后再处理长序列与多模态引入的 CP/SP 或视觉分支并行。**

如果不经过这一步，团队很容易出现一种现象：逻辑配置表面“可整除”，但真实 job 在某些节点之间频繁跨交换机通信，吞吐和抖动都非常差。

### 28.2 一个简化的映射对照表

| 场景 | 更适合放节点内的维度 | 可接受跨节点的维度 | 主要原因 |
| --- | --- | --- | --- |
| **纯文本 dense 模型** | TP、部分 SP | DP、部分 PP | TP 同步频繁，DP 最易扩展 |
| **长上下文 dense 模型** | TP、CP、SP | DP、部分 PP | CP 对链路稳定性更敏感 |
| **MoE 模型** | TP、热路径 EP | DP、PP | all-to-all 易受网络拓扑影响 |
| **多模态模型** | 视觉塔内部并行、主干 TP | DP、PP | 多分支长度差异更大，局部链路更宝贵 |

**这张表不是规则，而是起点。** 真正的落点还要看节点 GPU 数、NVLink/NVSwitch 结构、IB 带宽、交换机 oversubscription 与作业调度策略。

### 28.3 为什么小规模验证很容易误导大规模设计

因为小规模验证时：

- 网络跳数少；
- 坏节点概率低；
- checkpoint 体量小；
- 数据抖动对全局影响有限；
- pipeline bubble 容易被平均值掩盖。

一旦扩到几十上百节点，原本几乎不可见的链路抖动、对象存储抖动、数据尾延迟、坏节点降频，都会进入主路径。所以大规模分布式设计的关键不是“把小规模方案等比放大”，而是承认扩容后约束变了。

## 29. Checkpoint 生命周期要按产品资产来设计

很多团队把 checkpoint 当作训练中途顺手保存出来的副产物。更成熟的做法，是把 checkpoint 生命周期当作模型资产生命周期的一部分来设计。

### 29.1 一个完整 checkpoint 生命周期通常包含什么

| 阶段 | 关键动作 | 主要风险 |
| --- | --- | --- |
| **生成** | 周期性保存 full/sharded state | 写放大、训练阻塞 |
| **验证** | 立即做可读性与恢复性检查 | 只写不验，损坏文件未被发现 |
| **归档** | 记录版本、数据、并行、指标元数据 | 元数据缺失导致日后无法复用 |
| **迁移** | 拷贝到对象存储或异地机房 | 大文件复制失败或文件集不完整 |
| **恢复** | 在原拓扑或新拓扑上继续训练 | 状态不一致、兼容性断裂 |
| **转换** | 转成推理格式或别的框架格式 | 权重顺序、分片语义、rope/embedding 细节出错 |

**只有把这六步都定义清楚**，checkpoint 才能真正进入平台化管理。

### 29.2 为什么“保存成功”不等于“可恢复成功”

分布式环境下尤其如此。常见原因包括：

1. 某个 rank 文件实际没落盘完整；
2. 元数据和 shard 文件版本不一致；
3. 优化器状态保存成功，但 scheduler / sampler 未保存；
4. sharded save 时并行组定义变化，读回时无法还原；
5. 某些异步 save 在后台失败，但主训练进程没感知。

**因此建议默认做两件事**：

- 每次大 checkpoint 之后做一次轻量恢复 smoke test；
- 定期做一次“随机抽检恢复”，验证不是只有最近一个 checkpoint 可读。

### 29.3 Full、Sharded、Portable 三种目标并不完全相同

可以把 checkpoint 目标拆成三类：

- **Full checkpoint**：优先强调完整恢复和长时间归档；
- **Sharded checkpoint**：优先强调大规模训练效率和可扩展保存；
- **Portable checkpoint**：优先强调跨框架、跨并行拓扑迁移。

现实里很难用一种格式同时把三者做到最好，所以成熟平台往往会采用**双格式策略**：训练过程中以 sharded save 为主，关键里程碑额外导出一份更 portable 的模型资产。

## 30. 异步保存、对象存储与训练主链的边界

当 checkpoint 变大后，异步保存几乎会成为默认选择。但异步并不是免费午餐，它把“保存时间”从同步阻塞问题，改写成后台资源竞争与失败可观测性问题。

### 30.1 异步保存的真实收益来自哪里

| 收益来源 | 解释 |
| --- | --- |
| **减少主训练阻塞** | 计算线程可以更快返回下一 step |
| **更灵活的存储分层** | 先落本地 NVMe，再异步传对象存储 |
| **可做批量聚合写入** | 减少小文件过多带来的元数据开销 |
| **更容易做错峰复制** | 避免训练高峰期和存储高峰期直接重叠 |

### 30.2 异步保存最容易漏掉的风险

1. **后台线程失败无告警**；
2. **本地临时盘被旧 checkpoint 占满**；
3. **对象存储最终一致性导致目录状态短暂不完整**；
4. **训练继续跑，但最近几个 checkpoint 其实都不可用**；
5. **恢复脚本默认从“最新目录”读取，结果读到半成品**。

因此，异步 checkpoint 要想可靠，必须额外有：**manifest 文件、原子性完成标记、后台错误上报、空间水位监控、回收策略和恢复端的完整性校验**。

### 30.3 建议把 checkpoint 目录当数据库记录来设计

也就是说，不要只存一堆张量文件，而要显式记录：

- 训练 step；
- git commit / 配置 hash；
- 数据快照版本；
- tokenizer 版本；
- 并行拓扑；
- shard 列表和校验和；
- 是否包含 optimizer / scheduler / sampler / RNG；
- 与上一里程碑的关系。

这样恢复工具、转换工具和审计工具才能共享同一份事实来源，而不是靠目录命名猜状态。

## 31. 弹性训练与故障域设计

真正的大集群训练迟早会遇到抢占、坏节点、网络抖动、局部文件系统失败、驱动异常和单机重启。此时重要的不只是“能否重启”，而是**重启成本是否被控制在业务可以接受的区间**。

### 31.1 可以把故障域拆成四层

| 故障域 | 典型问题 | 更关心什么 |
| --- | --- | --- |
| **进程级** | 单 rank 崩溃、CUDA OOM、assert | 快速定位与可重试性 |
| **节点级** | GPU 掉卡、驱动异常、磁盘故障 | 节点隔离与作业迁移 |
| **网络级** | 某个交换机抖动、局部 RDMA 异常 | 拓扑重建与超时策略 |
| **控制面级** | 调度器抢占、对象存储访问异常 | 作业编排与状态一致性 |

不同故障域对应不同恢复策略。如果所有问题都一律“全作业从上个 checkpoint 重启”，在大规模下会浪费大量 GPU 小时。

### 31.2 弹性恢复最怕什么

最怕的不是显式失败，而是**半坏状态**：

- 某个节点速度明显慢但没挂；
- 某个 rank 丢了一次通信但后续又恢复；
- 某些数据 worker 卡住但主训练仍能继续拿到部分 batch；
- 某些 checkpoint 分片晚到但系统没有报错。

这类问题会让训练“继续运行”，却造成吞吐严重下滑或状态变脏。为此，系统要具备**一致性阈值**与**熔断规则**，而不是只依赖硬故障检测。

### 31.3 故障恢复不是只有技术问题

还涉及策略问题：

1. **什么情况下立即回滚**；
2. **什么情况下允许跳过一次保存**；
3. **什么情况下允许降配继续跑**；
4. **何时判定当前 job 继续跑已不经济**。

这些决策如果没有预案，就会在值班时变成人肉拍脑袋。

## 32. 数据顺序、packing 与恢复一致性

很多团队第一次真正掉进“恢复不一致”这个坑，不是因为权重文件坏了，而是因为**数据顺序在恢复前后悄悄变了**。

### 32.1 为什么这件事比想象中更常见

因为现代训练常常同时使用：

- 多数据源 mixture；
- 长度 bucket；
- dynamic packing；
- curriculum；
- 断点续跑；
- 分布式 sampler；
- 在线数据清洗或过滤。

只要其中任一层的状态没被记录，恢复后样本序列就可能不同。短跑看起来差异不大，长跑累计后可能对收敛和评测造成明显偏移。

### 32.2 需要记录的往往不只是随机种子

更完整的恢复一致性信息通常包括：

1. 当前 epoch / shard / file offset；
2. 多数据源采样器的内部计数；
3. packing 缓冲区尚未消费的样本；
4. curriculum 阶段与阈值；
5. 过滤器版本和启停状态；
6. tokenizer 或 template 版本。

**只记录随机种子**经常不够，因为很多输入管线早已不是纯函数式顺序读取。

### 32.3 一个实用的恢复校验方法

恢复后，不要只看 loss 是否大致连续。还应比对：

- 前几百步的有效 token 数；
- 各数据源采样占比；
- 平均序列长度与 pack ratio；
- 每步 wall time 分布；
- 梯度 norm 与 loss scale 行为。

这些信号往往比单点 loss 更快暴露输入系统是否发生了语义漂移。

## 33. 通信预算表要写在设计文档里

很多系统设计文档会写“这里有 all-reduce”“那里有 all-to-all”，但没有量化这件事。更成熟的做法，是给关键阶段做一张**通信预算表**。

### 33.1 最少要量化哪些问题

| 问题 | 典型量化方式 |
| --- | --- |
| **每步每种 collective 的字节量** | 按 hidden size、seq len、tp/cp/ep degree 估算 |
| **是否能 overlap** | 用时间线验证而不是只看代码路径 |
| **跨节点比例** | 统计 collective 中跨节点 rank 占比 |
| **尾延迟敏感性** | 观察 p95 / p99 通信耗时而非均值 |
| **在坏节点/抖动下的退化曲线** | 做受控压测或回放 |

### 33.2 为什么预算表能改变方案选择

因为很多并行方案在纸面 FLOPs 上差不多，但**通信形状**完全不同。预算表一旦写出来，你会更容易发现：

- 某个 TP 方案虽然显存漂亮，但 all-reduce 太频繁；
- 某个 PP 方案泡沫并不大，反而比跨节点 TP 更稳；
- 某个 EP 设置对平均吞吐友好，但对长尾风险很差；
- 某个 CP 方案只有在特定 bucket 配置下才成立。

## 34. 训练到推理、到后训练、到评测的接口

分布式训练和 checkpoint 不是终点。模型之后往往还要进入：

1. 离线评测；
2. 后训练对齐；
3. 量化与推理编译；
4. 在线灰度；
5. 数据回流。

### 34.1 为什么接口设计要前置考虑

如果训练阶段完全不考虑下游接口，就容易出现：

- checkpoint 只能被原训练框架读取；
- rope scaling、位置编码、tokenizer 模板信息没有随权重携带；
- 视觉分支、LoRA 分支、MoE 专家权重格式分裂；
- 评测工具拿不到完整配置与数据版本。

更好的做法是：在训练平台里把**模型卡、配置卡、数据卡、评测卡**一起沉淀，而不是只沉淀一份权重目录。

### 34.2 训练系统设计为什么要有“资产意识”

因为一旦模型进入组织内的多阶段流水线，真正复用的单位就不再是“某次 job 成功了”，而是“某份可追踪、可恢复、可评测、可转换的训练资产”。

## 35. 扩展小结

分布式训练与 checkpoint 设计真正难的地方，不在于并行术语本身，而在于你能否把**逻辑并行、物理拓扑、状态语义、数据顺序、故障域和下游接口**组织成一套长期稳定机制。做到这一步后，训练才不仅是把更大模型跑起来，而是把整个研发组织的试验能力变成可继承、可迁移、可审计的系统资产。
