# 数据引擎与自我改进

世界模型的真正长期价值，不只在于一次训练出一个“会想象未来”的模型，而在于它能否成为持续收集数据、发现盲点、生成新经验并反哺策略的核心引擎。换句话说，世界模型不仅是一个预测器，也可能是数据引擎和自我改进循环的中枢。

这也是为什么很多世界模型论文看起来讨论的是预测或规划，真正落地时却会迅速进入另一个问题：模型如何帮助决定“下一批最值得采的数据是什么”。只要这个问题出现，世界模型就不再只是策略的辅助模块，而开始扮演数据调度器、失败分析器、样本筛选器和反事实生成器。

本页讨论世界模型如何与主动采样、失败回流、近失误挖掘、仿真生成、规划评估、人工反馈和门禁验证结合，形成持续演化的学习系统。

## 1. 为什么世界模型天然适合数据引擎

相比纯策略模型，世界模型多了一层内部环境表示。这让它可以做几件很关键的事：

1. 评估哪些状态区域自己最不确定；
2. 想象不同动作会导向什么未来；
3. 在离线环境中筛选高价值轨迹；
4. 从失败轨迹中定位错误传播起点；
5. 生成反事实轨迹，分析“如果当时换个动作会怎样”；
6. 把“看起来异常”和“对决策真的重要”区分开来。

因此它不只是消费数据，还能帮助决定“下一批最值得采的数据是什么”。

### 1.1 和普通日志回流有什么不同

很多系统也会做日志分析和失败回流，但世界模型的独特之处在于：  
它不仅能告诉你“哪里失败了”，还可能告诉你：

1. 哪一步开始进入危险分支；
2. 哪类观测状态最容易漂移；
3. 哪种动作会显著降低风险；
4. 哪些数据即使没真实采到，也值得先在内部想象里筛一遍；
5. 哪些近失误样本虽然没出事故，却最值得优先回流。

这使它天然适合作为数据引擎的“评分器”和“选样器”。

### 1.2 为什么纯策略模型很难承担这个角色

纯策略模型通常只直接输出动作或价值，它当然也能给出失败案例，但很难系统性回答：

1. 模型具体在哪个状态转移上理解错了；
2. 某类错误是观测问题还是动力学问题；
3. 采一条新数据究竟会修正哪部分世界理解；
4. 哪些反事实分支对未来收益影响最大。

世界模型之所以更适合做数据引擎，不是因为它“更复杂”，而是因为它显式学习了状态演化结构。只要有状态演化结构，就更容易围绕“哪里不会、为什么不会、该补什么数据”建立闭环。

## 2. 数据引擎的闭环结构

**一个典型闭环可以写成**：

1. 收集真实交互轨迹 \(\mathcal{D}_t\)；
2. 训练或更新世界模型 \(M_t\)；
3. 用 \(M_t\) 评估不确定性、价值或失败模式；
4. 选择下一批采样任务或生成候选轨迹；
5. 获得新数据 \(\Delta \mathcal{D}\)；
6. 合并更新到 \(\mathcal{D}_{t+1}\)。

可以形式化为

$$
\mathcal{D}_{t+1} = \mathcal{D}_t \cup \mathcal{A}(M_t, \mathcal{D}_t),
$$

其中 \(\mathcal{A}\) 是主动数据采集策略。

### 2.1 关键点不在“闭环”二字，而在选择机制

**很多系统形式上也有闭环**：  
线上出错，收集日志，再训一次。  
但真正的数据引擎会更进一步回答：

1. 哪些日志最值得回流；
2. 哪些失败最有信息量；
3. 哪些新任务值得去真实采；
4. 哪些想象轨迹值得人工复核；
5. 哪类数据应该被拒绝进入训练集。

如果没有这个选择机制，所谓“自我改进”往往只是在堆更多相似数据。

### 2.2 实际上通常是三个相互缠绕的闭环

工程里更真实的情况，不是一个闭环，而是至少三个并行闭环：

1. **模型闭环**：世界模型预测哪里差，就去补哪里；
2. **策略闭环**：策略在哪些任务上退化，就去补对应行为数据；
3. **安全闭环**：系统在哪些边界条件接近失控，就重点挖掘 near-miss 数据。

这三个闭环的优先级未必一致。一个区域可能模型误差不大，但策略一到那里就会做错；另一个区域模型误差很高，却暂时不在产品主路径上。成熟的数据引擎必须学会给不同来源的需求排优先级。

### 2.3 在线闭环与离线闭环并不一样

**很多团队会把数据引擎分成两层**：

1. **离线闭环**：从历史日志、仿真和回放中筛选样本；
2. **在线闭环**：根据当前系统行为动态触发采样、回退和人工审核。

**离线闭环更适合做**：

1. 批量误差分析；
2. 模型版本对比；
3. 大规模近失误挖掘；
4. 课程式数据重采样。

**在线闭环更适合做**：

1. 高不确定时触发额外观察；
2. 高风险样本打标回流；
3. 控制是否允许真实执行新策略；
4. 实时决定是否进入人工接管或保守模式。

## 3. 主动采样：把昂贵试错花在刀刃上

在机器人和自动驾驶里，真实交互昂贵，因此要优先采那些“最可能提升模型”的数据。世界模型可基于：

1. 预测不确定性；
2. 规划分歧；
3. 长尾失败风险；
4. 奖励估计误差；
5. counterfactual 敏感性；
6. 版本间行为差异。

来选样本。若定义某状态-动作对的信息价值为 \(I(s,a)\)，则可优先采

$$
(s^\star, a^\star)
=
\arg\max_{s,a} I(s,a).
$$

这比随机采样更有望提高样本效率。

### 3.1 信息价值可以从哪些信号近似

现实里很难直接算真信息增益，因此通常会用代理分数：

1. 模型 ensemble 分歧；
2. rollout 误差预测；
3. 价值函数分歧；
4. 成功率边界附近的高敏感区；
5. 风险头或终止头的不稳定性；
6. 不同 planner 给出的行动建议冲突；
7. 新旧模型在同一状态上的预测偏移。

这些分数不一定完美，但至少比“随机多采一些”更有方向。

### 3.2 不同目标会导向不同采样策略

如果目标是提升平均成功率，采样会倾向于成功率边界附近的样本；  
如果目标是降低高风险事故，采样会更偏向 near-miss 与高不确定区域；  
如果目标是提升恢复能力，则更应该采失败后仍可恢复的那部分轨迹，而不是采已经彻底不可逆的灾难样本。

因此数据引擎不能只问“哪里错得多”，还要问“补这个地方的样本，对当前系统收益到底大不大”。

### 3.3 主动采样也需要预算分配

即使知道哪些区域值得采，现实里也要面对预算约束。设不同数据源集合为 \(\{\mathcal{S}_i\}\)，每种数据源单位成本为 \(c_i\)，预期收益为 \(u_i\)，则一个概念上的预算分配问题可写成：

$$
\max_{\{n_i\}}
\sum_i n_i u_i
\quad
\text{s.t.}
\quad
\sum_i n_i c_i \le B.
$$

**这里的数据源可能包括**：

1. 真实机器人采样；
2. 驾驶 shadow mode 日志；
3. 高保真仿真器；
4. 世界模型 imagined rollouts；
5. 人工演示与修正。

真正成熟的数据引擎，不只是“会挑样本”，还要会在成本约束下挑样本。

## 4. 失败回流为何特别重要

成功轨迹多告诉你系统的舒适区，失败轨迹才真正暴露盲点。世界模型在失败回流中有两个独特作用：

1. 判断失败是从哪一步开始不可逆偏离；
2. 在失败附近生成反事实轨迹，比较“如果当时换个动作会怎样”。

例如仓库机器人抓箱失败，世界模型可帮助判断问题更可能来自目标位置误判、夹爪姿态不对，还是后续抬升路径碰撞。这样数据回流就能更有针对性。

### 4.1 为什么世界模型比单纯日志更适合做失败分层

普通日志告诉你“最后失败了”；  
**世界模型还能帮你追溯**：

1. 第一个错误状态是什么；
2. 当时系统其实有没有更安全的动作；
3. 失败是观测误差、动力学误差，还是规划误差；
4. 失败是逐步积累，还是某一个关键转折点触发。

**这会直接影响你到底该补**：

1. 感知数据；
2. 接触数据；
3. 规划边界数据；
4. 恢复轨迹数据；
5. 还是安全接管数据。

### 4.2 失败回流最好不是二元的

现实里很多高价值轨迹并不是“成功/失败”二元，而是更适合分层：

1. 完全成功；
2. 成功但过程不稳；
3. near-miss；
4. 可恢复失败；
5. 不可恢复失败；
6. 被人工接管的危险状态。

在数据引擎里，往往 `3`、`4`、`6` 类最有价值，因为它们离系统当前能力边界最近。

## 5. 近失误挖掘为什么特别重要

在高风险系统里，真正最值钱的数据往往不是事故本身，而是 near-miss。也就是：

1. 差一点碰撞；
2. 差一点并线失败；
3. 差一点抓空；
4. 差一点把杯子碰翻；
5. 差一点越过安全边界但最终被回退或人工纠正。

**这些样本同时具备三个特点**：

1. 风险高；
2. 可学习；
3. 数量往往比真实事故多得多。

### 5.1 为什么 near-miss 比纯失败更适合训练

纯失败有时已经太晚，系统可能完全脱离正常分布；  
near-miss 则常处在“还来得及修正”的边界带上。  
对学习系统来说，这种边界带尤为重要，因为它更容易揭示：

1. 哪些观测线索正在被忽略；
2. 哪些动作改一点就能把结果翻回来；
3. 哪些风险头应该更早亮灯；
4. 哪些 planner 分支需要更早被剪掉。

### 5.2 near-miss 可以怎么挖

**常见信号包括**：

1. 最小碰撞距离过小；
2. 紧急制动或急停触发；
3. 人工接管前若干秒；
4. 风险头高但最终侥幸成功；
5. 规划器分歧突然变大；
6. 世界模型预测与真实执行轨迹出现显著偏差。

这些都比只看“是否真正失败”更能抓到高价值样本。

## 6. 反事实数据为什么有价值

真实系统里的高价值问题常常不是“系统做了什么”，而是“它本来还能怎么做”。  
**世界模型让反事实分析变得自然**：

$$
p(\tau' \mid s_t, a'_{t:T})
$$

可以和真实轨迹 \(\tau\) 比较：

$$
\Delta R = R(\tau') - R(\tau).
$$

如果某类失败轨迹附近总存在明显更优的反事实路径，那说明：

1. 系统已经接近成功区；
2. 补充这些边界数据很可能高回报；
3. 恢复策略或技能修正可能比重做整个策略更划算；
4. 这个区域尤其适合做示范补充或人工干预学习。

### 6.1 反事实最适合回答什么问题

在工程里，反事实最有价值的用途通常不是“生成一个更好看的替代故事”，而是回答：

1. 如果早一步减速，是否能避免风险；
2. 如果抓取角度稍微不同，是否能避免碰撞；
3. 如果先观察再执行，是否能显著降低不确定性；
4. 如果 planner 选了第二候选，结果会不会更稳。

这类问题天然就是数据选择问题，因为只要发现某类反事实经常更优，就说明应该补对应数据。

## 7. Imagined data 何时有用，何时危险

世界模型最诱人的能力之一，是在内部生成 imagined rollouts。理论上这能显著扩大训练信号，但必须警惕 model bias。设真实轨迹分布为 \(p_{\text{env}}\)，世界模型生成分布为 \(p_M\)，若二者差距过大，则用 imagined data 训练策略会积累偏差。

**一个概念性风险量可写作**：

$$
\text{Bias}
=
\mathbb{E}_{\tau \sim p_M}[R(\tau)]
- \mathbb{E}_{\tau \sim p_{\text{env}}}[R(\tau)].
$$

imagined data 的价值，在于补充真实数据稀疏区；其风险，则是把错误世界当真世界。

### 7.1 为什么 imagined data 很诱人

**因为它看上去能同时解决**：

1. 数据少；
2. 长尾稀有；
3. 真实试错贵；
4. 训练信号不足；
5. 某些风险样本几乎无法主动真实采集。

**但它真正危险的地方也在这里**：  
一旦世界模型错得有结构性，imagined data 不是“多一些噪声”，而是可能把整个策略朝错误方向推。

### 7.2 imagined data 最容易在哪些地方出错

高风险区域通常恰恰是 imagined data 最不可信的地方。典型问题包括：

1. 覆盖不足区域被模型过度平滑；
2. 接触或碰撞细节被错误简化；
3. 多主体交互的关键分支被洗成均值未来；
4. 模型对规则边界错误自信；
5. 罕见动作导致的后果被错误乐观化。

所以 imagined data 的第一原则不是“尽量多用”，而是“先判断哪里可以用”。

## 8. 选择性使用 imagined rollouts

更稳妥的做法通常不是全盘采用 imagined data，而是选择性使用：

1. 只在高置信度区域生成；
2. 只用于值函数预训练或候选过滤；
3. 用于探索方向建议，而非直接监督最终策略；
4. 用作困难场景扩增，而不取代真实样本；
5. 让 imagined 样本先经过规则或人工筛查。

这种保守策略在机器人和驾驶系统中更现实。

### 8.1 一个实用的 imagined data 使用顺序

**通常从低风险用途开始最稳**：

1. 先做离线分析；
2. 再做 candidate reranking；
3. 再做值函数或 reward model warmup；
4. 再做 curriculum 扩增；
5. 最后才考虑直接训练 policy。

这样能把 model bias 的伤害控制在更小范围内。

### 8.2 imagined data 最好有门禁函数

可以把 imagined 样本的可接纳性写成一个门禁分数：

$$
g(\tau)=\mathbf{1}\{
u(\tau) < \epsilon_u,
\;
\text{risk}(\tau) < \epsilon_r,
\;
\text{consistency}(\tau) > \epsilon_c
\},
$$

也就是只有当不确定性低、风险估计稳定、与真实分布一致性足够高时，想象样本才允许进入训练池。这个公式只是概念表达，但它提醒我们：imagined data 的关键不是生成，而是验收。

## 9. 世界模型驱动的 curriculum

世界模型还能帮助生成课程学习顺序。若某类任务当前预测稳定，说明系统已大致掌握；若某类任务 rollout 误差和规划 regret 都高，则可把它提升为下一阶段训练重点。

定义任务难度估计 \(d(\tau)\)，数据采样权重可设为

$$
w(\tau) \propto \exp(\alpha d(\tau)).
$$

这样训练分布会自动聚焦于“尚未学会但又不是完全无从下手”的区域。

### 9.1 难度不是越高越好

如果一类场景完全超出当前能力，反而可能浪费采样预算。  
**最有价值的区域往往是**：

1. 已经快会了但总差一点；
2. 成功和失败都经常出现；
3. 对上线安全或业务效果影响很大；
4. 一旦补齐就能带动整片任务族进步。

这类区域最适合做课程式推进。

### 9.2 curriculum 也可以按误差类型组织

不是所有“难”的任务都难在同一处。数据引擎完全可以按误差类型来排课程，例如：

1. 先补观测遮挡；
2. 再补接触边界；
3. 再补恢复策略；
4. 最后补长时组合任务。

这样比把所有困难样本混在一起重采样更容易得到稳定收益。

## 10. 规划分歧作为采样信号

若多个规划器、不同温度下的 rollout 或不同世界模型版本在同一状态下给出分歧很大的建议，这常意味着该区域不确定或多模态。设不同规划器输出价值估计为 \(\{V_i(s)\}\)，则分歧可定义为

$$
\sigma_V(s) = \mathrm{Var}\{V_i(s)\}.
$$

\(\sigma_V\) 高的状态可优先纳入数据采样和人工审核。

### 10.1 为什么规划分歧往往比重建误差更有用

因为最终系统真正关心的是“决策会不会变”。  
有时观测误差不大，但对动作选择影响极大；  
也有时画面预测不太准，却不影响高层决策。  
规划分歧更贴近后者。

### 10.2 版本分歧同样是重要信号

如果新旧模型在同一日志上给出完全不同的 rollout、风险估计或动作排序，这通常意味着：

1. 某些区域仍然不稳定；
2. 新模型可能学到了新能力，也可能引入了新偏差；
3. 这些片段值得进入回放、人工审核和定向评测。

因此数据引擎不只服务下一轮训练，也服务模型版本上线前的差异分析。

## 11. 在线/离线数据引擎的典型架构

实际系统中，数据引擎常被拆成几个模块：

1. **采集层**：日志、传感器流、控制命令、接管事件；
2. **评分层**：世界模型输出不确定性、风险、分歧和回报估计；
3. **挖掘层**：从海量日志中选 near-miss、失败和高价值反事实种子；
4. **验证层**：规则、人工审核或高保真仿真器做门禁；
5. **训练层**：决定哪些数据进入哪一种训练任务；
6. **回放层**：比较新旧模型与策略在同一数据上的差异。

### 11.1 离线架构通常更像“批处理工厂”

离线系统会周期性扫历史数据仓库，做：

1. 模型误差重打分；
2. 近失误候选挖掘；
3. 反事实种子选择；
4. 训练集版本打包；
5. 新旧模型对比。

这类架构的优点是稳定、可控、便于复现实验；缺点是反馈慢。

### 11.2 在线架构通常更像“门禁与报警系统”

在线系统更像一个实时监听器，当检测到以下信号时触发动作：

1. 不确定性异常升高；
2. planner 分歧显著；
3. 风险头突然跳变；
4. 人工接管；
5. 安全壳介入；
6. 真实执行与世界模型预测明显偏离。

在线闭环的价值不只是多收数据，更重要的是及时把高价值样本打上标签并保存上下文。

## 12. 与仿真器和生成式视频模型结合

在更复杂系统里，世界模型不必独自承担一切。它可以与规则仿真器、传统模拟器、视频生成器配合：

1. 规则仿真器保证物理和交通规则底线；
2. 世界模型补充数据驱动的人类行为与长尾视觉变化；
3. 视频生成器提供更丰富的观测层多样性；
4. 高保真回放器承担最终验收。

这种多源数据引擎比单一学习型模拟器更稳健。

### 12.1 一个常见分工

#### 高保真仿真器

负责物理或规则正确性。

#### 世界模型

负责从真实数据中学到难以手写建模的行为分布和交互细节。

#### 生成式视频模型

负责视觉层多样性、风格变化和长尾观测扰动。

这类组合系统的价值，在于把“物理正确”“行为真实”“视觉丰富”拆给不同模块。

## 13. 自我改进不等于自发变强

“Self-improvement” 常被浪漫化，但在工程上它通常意味着受控闭环：

1. 模型发现盲点；
2. 系统采集新证据；
3. 人类或规则验证；
4. 训练后再次评估；
5. 只有通过门禁的数据才真正回流。

如果缺少验证，自我改进很容易变成自我强化偏差。

### 13.1 一个危险模式

世界模型若本来就在某类状态上过度自信，又拿自己生成的 imagined data 去补自己，那可能会形成闭环幻觉：  
模型越来越确信一个本来就错的世界。

所以自我改进的关键，从来不是“闭环”本身，而是闭环里有没有校验器。

### 13.2 另一个危险模式：只补容易样本

很多系统表面上也在不断回流，但回流进来的数据主要来自：

1. 容易任务；
2. 频繁场景；
3. 低风险日志；
4. 模型已经会的区域。

这样做的结果通常是平均指标继续上升，但长尾失败几乎不变。数据引擎如果不能显式压制“易样本惯性”，就很难真正改善边界能力。

## 14. 人类反馈在闭环中的角色

世界模型能指出“哪里不确定”，但未必能判断“什么是最值得修正的错误”。人类反馈在这里可起几个作用：

1. 提供高价值错误的优先级排序；
2. 对 imagined data 或反事实建议做可信性过滤；
3. 给恢复策略和任务边界加上业务语义；
4. 决定哪些场景必须真实采集，不能只靠模型合成；
5. 决定哪些 near-miss 已经接近不可接受风险。

特别在高风险场景里，自我改进必须嵌入人工审核环。

### 14.1 人工审核最适合插在哪些位置

人工并不一定要审核全部数据，更现实的插入点包括：

1. 高分 near-miss 候选确认；
2. imagined 样本抽检；
3. 新旧模型高分歧片段复核；
4. 恢复轨迹标签校验；
5. 新策略真实部署前的关键样本门禁。

这能把人力花在真正高杠杆的位置。

## 15. 自动驾驶中的数据引擎示例

世界模型可在离线驾驶日志中扫描：

1. 哪些交通场景 rollout 误差高；
2. 哪些交互对象行为难预测；
3. 哪些闭环规划结果与真实驾驶员动作差异大；
4. 哪些接管前数秒呈现出典型风险演化模式。

**然后驱动下一轮**：

1. 特定场景回放；
2. 受控路测采样；
3. 危险边界场景合成；
4. 人工标注强化；
5. shadow mode 的专项验证。

这比盲目增加路测公里数更高效。

### 15.1 驾驶场景里最有价值的数据常常不是事故本身

真正值得采的往往是 near-miss：

1. 差一点碰撞；
2. 差一点并线失败；
3. 差一点误判行人意图；
4. 差一点错过施工导流；
5. 差一点因为礼让判断错误导致急刹。

**因为这些样本同时具备**：

1. 高风险；
2. 可学习；
3. 比真正事故更容易大规模获取。

### 15.2 驾驶里为什么门禁尤其重要

自动驾驶中，很多数据不能因为“模型觉得有趣”就直接拿去增强主策略。更现实的顺序往往是：

1. 先进入离线风险分析；
2. 再进入 planner reranking 评估；
3. 通过规则和仿真门禁后，才进入更强的训练闭环；
4. 真正影响线上行为前，还要经过 shadow mode 和人工审核。

这类多层门禁，是避免自我强化偏差的关键。

## 16. 机器人中的数据引擎示例

对于桌面操作机器人，世界模型可发现：

1. 透明物体的预测误差高；
2. 抽屉半开状态下的接触结果不稳定；
3. 侧向相机视角与顶视角对隐状态估计分歧大；
4. 某些失败动作其实在两步前就已经注定高概率失败。

于是下一轮数据采集重点就可以转向透明物体、抽屉操作、多视角示范和失败恢复，而不是继续采集大量容易的抓取成功轨迹。

### 16.1 机器人里为什么恢复数据尤其值钱

成功抓取教会系统“怎么做对”；  
恢复轨迹教会系统“做错后怎么回来”。  
世界模型特别适合帮助找这类样本，因为它能估计：

1. 哪些失败仍可恢复；
2. 哪些失败已不可逆；
3. 哪个恢复动作最可能把系统带回成功区；
4. 哪些恢复策略值得由人工示范补强。

### 16.2 机器人里为什么要显式记录接管与重试

如果只保留最终成败标签，很多高价值边界信息会丢掉。真实机器人系统应尽量记录：

1. 第一次失败前的状态；
2. 人工是如何接管的；
3. 自动恢复是否有效；
4. 重试是否成功；
5. 重试成功前做了哪些关键修正。

这些数据对世界模型来说，比单纯多采几条成功轨迹更有辨识力。

## 17. 数据门禁与版本化评估

自我改进系统若没有门禁，最终常常会把数据仓库变成噪声仓库。比较稳妥的做法是把样本分层入池，例如：

1. 原始日志池；
2. 候选高价值池；
3. 通过规则校验池；
4. 通过人工抽检池；
5. 可进入训练池；
6. 可进入高风险训练池。

### 17.1 为什么版本化很重要

每一轮数据回流都可能改变模型偏好，因此必须记录：

1. 样本来自哪一轮模型；
2. 由哪个门禁通过；
3. 是否是 imagined data；
4. 是否经过人工修订；
5. 它最终影响了哪些训练任务。

没有版本化，很难回答“这次性能变化到底是因为新模型结构，还是因为某批 near-miss 数据进了训练集”。

### 17.2 训练集不应该是一锅粥

同一批数据不一定适合喂给同一目标。更合理的做法通常是按用途分桶：

1. 真实高置信数据喂给主世界模型；
2. near-miss 数据优先喂风险头和 planner 评估；
3. 恢复轨迹喂恢复策略和行为先验；
4. imagined 数据优先喂 value warmup 或辅助任务；
5. 被质疑质量的数据先放灰区池，等待后续验证。

这样比“所有东西混成一个大训练集”更符合工程现实。

## 18. 评估数据引擎是否有效

不能只看数据量变多，还要看：

1. 新数据是否提高长尾成功率；
2. 训练成本是否可接受；
3. 世界模型的不确定性是否更校准；
4. 失败分布是否发生实质性收缩；
5. 新增数据是否来自真正困难而高价值的区域；
6. 近失误是否更早被识别；
7. 人工接管是否减少；
8. 新版本是否在同等预算下变得更稳。

如果新增数据主要来自容易区域，自我改进循环可能只是表面繁忙。

### 18.1 一个更具体的评估问题

每一轮回流后，最好都问：

1. 哪类失败下降了；
2. 哪类失败没动；
3. 是否出现了新的失败模式；
4. 模型是否变得更保守，还是更聪明；
5. planner 是否只是因为更怂而减少了风险；
6. 线上收益是否真的匹配离线指标变化。

否则“成功率提升”可能只是把系统变得更怂。

### 18.2 数据引擎本身也要被评测

很多团队只评测模型，却不评测数据选择器本身。但如果数据选择器长期偏向错误区域，整个闭环都会被带偏。因此也应追踪：

1. 被选样本最终有多少真的带来收益；
2. 哪种打分信号最有预测力；
3. 哪些门禁最能过滤低价值 imagined data；
4. 人工审核花费主要消耗在哪些样本类型上。

## 19. 一个形象比喻

把世界模型想成一位飞行教官兼模拟器管理员。它不仅教飞行员怎么飞，还能观察学员在哪些天气、哪些机型、哪些紧急情况中总出错，并据此安排下一轮训练课表，甚至在模拟器里生成更贴近真实事故边界的演练场景。

真正高效的训练体系，不是每天重复最简单的起降，而是持续把有限训练时间投入最能减少事故的地方。世界模型驱动的数据引擎，本质上也是在做同样的事：把有限真实试错预算投向最能缩短能力边界的区域。

## 20. 小结

世界模型的数据引擎价值，在于它把“学习世界”与“选择下一步学习什么”联系起来。只有当模型能发现自己的盲点、指导数据采集、生成有价值的反事实、挖掘 near-miss，并在验证框架下回流训练，世界模型才真正从一个预测器变成持续自我改进系统的中枢。

真正成熟的自我改进循环，不是让模型自己给自己加数据，而是让世界模型成为真实数据、仿真、反事实、人工审核、门禁验证和策略训练之间的调度层。也只有在这种受控闭环下，“self-improvement” 才不是一个听起来好听的口号，而是一个可被验证、可被上线、可被持续优化的工程系统。

## 21. 数据引擎本身也要有回归测试

每次改动采样器、门禁或 imagined data 比例后，都应单独验证：

1. 新选样本是否更接近真实高价值错误；
2. 人工审核命中率是否提高；
3. 训练后长尾收益是否真的增加；
4. 是否引入了新的模式偏差。

### 21.1 为什么这一步重要

因为数据引擎一旦偏航，后续所有训练收益都会被一起拉偏。  
不评测数据选择器，等于默认“选哪些数据”永远正确，这在大规模闭环里几乎从不成立。

## 22. 小结

世界模型驱动的数据引擎，真正难的不是生成更多样本，而是持续证明自己正在把采样预算和人工预算投入最值钱的区域。只有当选样、门禁、回流和回归测试都形成制度，这条自我改进链路才会越跑越准，而不是越跑越偏。
