# WM、WAM、VAM 与动作条件世界建模

近两年“世界模型”讨论里，一个越来越常见的现象是：很多论文都在谈未来预测、动作建模、视频生成、控制和仿真，但它们关注的对象已经不完全相同。为了区分这些方向，很多研究与工程讨论开始使用一组并不绝对统一、但很有阅读价值的缩写：

1. `WM`：World Model
2. `WAM`：World-Action Model
3. `VAM`：Video-Action Model

这组术语不是社区里唯一、严格、不可变的官方标准，但它们非常适合拿来做阅读地图。尤其当你同时关注经典 latent dynamics、视频生成路线、机器人控制和动作联合建模时，它能帮助你快速判断一篇工作到底在强调什么。

本页的目标不是争论术语定义，而是回答四个更实际的问题：

1. `WM / WAM / VAM` 各自到底想建模什么；
2. 它们的接口、目标函数和应用重点有什么不同；
3. 为什么这些路线正在互相靠近，而不是各走各路；
4. 在机器人、自动驾驶、具身 agent 和数据引擎里，该如何理解它们的工程价值。

## 1. 一个统一问题：未来如何随动作变化

三条路线都围绕同一个基本问题展开：

\[
\text{给定历史观测和动作，未来会怎样演化？}
\]

只是不同路线把这个问题展开成了不同建模对象：

1. `WM` 更强调环境状态或潜状态如何演化；
2. `WAM` 更强调动作和世界未来作为联合对象如何一起生成；
3. `VAM` 更强调视频表示与动作表示之间的统一建模，尤其重视视频先验对动作泛化的价值。

## 2. `WM`：最经典也最宽的总类

`WM` 是范围最大的词。只要一个模型在学习：

\[
p(s_{t+1:t+H}, o_{t+1:t+H} \mid o_{\le t}, a_{t:t+H-1}),
\]

**或者在潜空间中学习**：

\[
p(z_{t+1:t+H} \mid z_t, a_{t:t+H-1}),
\]

它通常就可以被归入广义的 world model。

### 2.1 `WM` 的核心目标

1. 学到可 rollout 的环境动态；
2. 在不直接上真实环境试错的情况下做 imagined planning；
3. 支持 reward、risk、termination 等辅助预测；
4. 为后续控制、规划和数据回流提供内部环境表征。

### 2.2 `WM` 的典型实现

1. `RSSM` / `Dreamer` 一类 latent dynamics；
2. 预测未来观测的生成式模型；
3. 融合 reward head、value head、risk head 的环境预测器；
4. 结合几何或对象状态的结构化世界表示。

### 2.3 `WM` 的核心问题意识

它最关心的通常不是“动作应该怎么生成”，而是：

1. 世界的最小可决策表示是什么；
2. 如何稳定地长时 rollout；
3. 如何把环境模型和规划器接起来；
4. 在部分可观测下如何维持记忆。

## 3. `WAM`：把 world 和 action 放到一个联合模型里

`WAM` 可以理解成从经典 `WM + policy` 进一步往前走的路线。  
**它不再满足于**：

1. 先学一个世界模型；
2. 再在这个模型上单独学一个 policy。

**而是试图直接学习**：

\[
p(o_{t+1:t+H}, a_{t:t+H-1} \mid o_{\le t}, g),
\]

其中 \(g\) 可以是语言目标、任务条件、地图、规则或高层意图。

### 3.1 `WAM` 想解决什么

**它的核心直觉是**：

1. 世界如何演化和动作如何生成，本来就强耦合；
2. 如果完全把两者拆开，可能损失 joint modeling 的好处；
3. 对复杂控制任务，动作先验本身就是世界结构的一部分。

### 3.2 为什么 `WAM` 对机器人很有吸引力

在机器人里，很多动作根本不是抽象数字，而是一段和视觉未来强耦合的行为。比如：

1. 接近把手；
2. 抓住杯子；
3. 拉开抽屉；
4. 绕开障碍重定位。

这类动作是否合理，往往必须通过未来世界状态来判断。因此让动作和未来观测一起建模，会显得更自然。

### 3.3 `WAM` 和经典 `WM + planner` 的区别

**一个很实用的区分是**：

1. 经典 `WM` 更强调“给动作，预测世界”；
2. `WAM` 更强调“给目标和历史，同时生成动作与世界未来”。

这意味着 `WAM` 更接近“统一时空生成模型”，而不只是“环境模拟器”。

## 4. `VAM`：把视频建模和动作建模绑得更紧

`VAM` 通常更明确地强调视频表示的中心地位，可以把它近似理解成：

\[
p(v_{t+1:t+H}, a_{t:t+H-1} \mid v_{\le t}, g),
\]

这里 \(v\) 是视频观测，而不只是抽象 latent。

### 4.1 `VAM` 的典型动机

1. 视频天然包含丰富时空先验；
2. 视频预训练可能比静态图像更利于动作泛化；
3. 互联网和模拟器中更容易拿到海量视频，而不总能拿到高质量状态标签；
4. 视频和动作的联合建模可能成为机器人泛化的更强底座。

### 4.2 `VAM` 为什么常出现在具身和机器人语境

**因为它天然适合处理**：

1. 遮挡与显现；
2. 物体连续运动；
3. 接触前后的视觉变化；
4. 多视角和多帧历史；
5. 通过视频先验迁移动作知识。

### 4.3 `VAM` 不等于“会生成视频”

真正重要的不是能不能生成漂亮视频，而是：

1. 这些视频 latent 是否有利于动作生成；
2. 视频历史是否帮助模型理解环境可供性；
3. 视频未来是否帮助筛掉危险动作；
4. 视频表示是否能迁移到新任务和新物体。

## 5. 三者的最短区别

**如果必须用最短的一句话概括**：

1. `WM`：重点是世界动态建模；
2. `WAM`：重点是动作和世界未来的联合建模；
3. `VAM`：重点是利用视频表示把动作建模做得更强、更统一。

### 5.1 但它们并不是互斥集合

现实里，一篇工作完全可能同时具备：

1. `WM` 的 latent dynamics 核心；
2. `WAM` 的动作联合生成接口；
3. `VAM` 的视频先验和视频 latent。

**因此更准确的理解是**：这三者像是一个坐标系，而不是互相排斥的标签。

## 6. 从建模对象看三者差异

### 6.1 `WM`

**更可能建模**：

1. 潜状态；
2. 奖励；
3. 终止；
4. 风险；
5. 条件世界演化。

### 6.2 `WAM`

**更可能联合建模**：

1. 未来动作 token；
2. 未来观测 token；
3. 目标条件；
4. 技能或 latent action。

### 6.3 `VAM`

**更可能重点放在**：

1. 视频 tokenizer；
2. 视频 latent；
3. 视频-动作对齐；
4. 大规模视频先验迁移；
5. 通过视频支撑动作泛化。

## 7. 从目标函数看三者差异

### 7.1 `WM` 的典型目标

1. 观测重建；
2. latent dynamics 预测；
3. reward / value / risk 头；
4. imagined rollout 稳定性。

### 7.2 `WAM` 的典型目标

1. 联合序列建模；
2. 动作与观测的一致性；
3. 目标条件下的成功轨迹建模；
4. latent action 与 world latent 的联合优化。

### 7.3 `VAM` 的典型目标

1. 视频 token 或 latent 重建；
2. 视频未来预测；
3. 动作与视频变化的一致性；
4. 视频先验迁移到动作生成或控制。

### 7.4 为什么目标函数差异会影响部署价值

因为：

1. 更偏 `WM` 的模型常更适合规划与风险评估；
2. 更偏 `WAM` 的模型常更适合统一动作生成和未来模拟；
3. 更偏 `VAM` 的模型常更适合利用大规模视频做泛化和迁移。

## 8. 从系统接口看三者差异

### 8.1 `WM` 的接口

**常是**：

1. 当前状态或观测；
2. 候选动作；
3. rollout 未来；
4. 再由 planner 选动作。

### 8.2 `WAM` 的接口

**常是**：

1. 当前历史；
2. 高层目标；
3. 直接输出未来动作和世界变化；
4. 再由筛选或规划器做选择。

### 8.3 `VAM` 的接口

**常是**：

1. 视频历史；
2. 动作条件或目标条件；
3. 输出未来视频 latent 与动作分支；
4. 再由 policy head 或规划器消费。

## 9. 为什么这三条路线会汇合

过去的差异更大，是因为：

1. latent world model 更像 RL 社区路线；
2. 视频生成更像生成模型社区路线；
3. 动作建模更像机器人 policy 路线。

现在它们开始汇合，原因在于：

1. 视频生成模型变强，开始足以承载世界预测；
2. 动作建模越来越需要视频和大规模视觉先验；
3. 规划越来越希望统一在生成模型或 token 模型里完成；
4. 数据引擎和仿真需求推动这些模块互相借力。

## 10. 在机器人里怎么理解三者

### 10.1 `WM` 视角

**重点是**：

1. 想象动作后果；
2. 做 risk-aware planning；
3. 提升样本效率；
4. 支持失败恢复。

### 10.2 `WAM` 视角

**重点是**：

1. 把技能和未来世界作为联合轨迹建模；
2. 利用 joint modeling 学到动作先验；
3. 让目标、世界和动作在一个模型中对齐。

### 10.3 `VAM` 视角

**重点是**：

1. 让视频历史和视频未来直接参与动作学习；
2. 利用海量视频预训练提升泛化；
3. 处理复杂遮挡、接触和时空交互。

## 11. 在自动驾驶里怎么理解三者

### 11.1 `WM`

**更强调**：

1. 多主体世界演化；
2. 风险评估；
3. near-miss 模拟；
4. 规划辅助。

### 11.2 `WAM`

**更强调**：

1. 把驾驶动作和环境未来一起建模；
2. 让规划像条件生成；
3. 对目标轨迹、驾驶意图和环境反馈联合建模。

### 11.3 `VAM`

**更强调**：

1. 视频观测对动作意图的支撑；
2. 长时视觉变化；
3. 从大规模驾驶视频中学习先验。

## 12. 在数据引擎里怎么理解三者

即使模型暂时不上闭环控制，这三条路线都可能在数据引擎中有很大价值。

### 12.1 `WM`

**适合**：

1. 风险前瞻；
2. 反事实 rollout；
3. near-miss 发现；
4. 主动采样。

### 12.2 `WAM`

**适合**：

1. 候选动作生成；
2. 失败恢复轨迹构造；
3. 目标条件下的数据扩展。

### 12.3 `VAM`

**适合**：

1. 视频未来合成；
2. 视角增强；
3. 视频先验迁移；
4. 复杂交互片段的样本筛选。

## 13. 三条路线各自的典型风险

### 13.1 `WM` 的风险

1. latent 漂移；
2. 单步准、长时崩；
3. 视频看起来不强，难说服人；
4. 规划器和环境模型接口不佳。

### 13.2 `WAM` 的风险

1. 联合建模目标太复杂；
2. 动作分支质量不够会污染世界分支；
3. 训练数据要求高；
4. 很容易变成“什么都想统一，什么都不够稳”。

### 13.3 `VAM` 的风险

1. 视频生成质量和控制价值脱节；
2. 模型过重，难以进主环；
3. 小接触误差被视频平均化；
4. 大规模视频预训练不一定带来可执行动作先验。

## 14. 一个更实用的判断框架

当你看一篇论文或设计一个系统时，可以先问：

1. 这个模型最终输出的核心对象是什么，状态、视频，还是动作与世界联合轨迹；
2. 它主要优化的是真实控制收益、视频质量、还是动作泛化；
3. 它的接口更像 simulator、planner、policy，还是 data engine；
4. 它最需要的数据是动作轨迹、视频，还是结构化状态。

这四个问题通常比“它到底算不算世界模型”更有用。

## 15. 为什么 `WM / WAM / VAM` 这组术语值得保留

尽管术语不绝对统一，但它们有三个实际好处：

1. 让阅读时能快速识别重心；
2. 避免把所有未来预测工作都模糊归为同一类；
3. 方便把经典 RL 世界模型、视频生成和动作建模放到同一张地图里。

## 16. 一条从 `WM` 到 `WAM` 再到 `VAM` 的阅读路径

**可以粗略这样理解方法演进**：

1. 从经典 latent dynamics world model 出发；
2. 逐步引入更强观测建模，尤其是视频；
3. 再引入 latent action、技能和 planning as inference；
4. 最后走向更统一的 world-action 或 video-action 联合建模。

### 16.1 这条路径说明了什么

它说明社区不是在抛弃旧路线，而是在把旧路线往更统一、更大规模、更贴近真实数据的方向推进。

## 17. 设计建议

1. 不要把 `WM / WAM / VAM` 当成僵硬标签，而要把它们当成方法重心坐标；
2. 若目标是规划与风险评估，优先看 `WM` 是否真的 rollout 稳定；
3. 若目标是统一动作生成和未来模拟，重点看 `WAM` 的 joint modeling 是否真的有收益；
4. 若目标是用大规模视频提升控制泛化，重点看 `VAM` 的视频先验是否真正进入动作分支；
5. 无论哪条路线，都要最终回到闭环任务成功率、风险事件和数据引擎收益上。

## 18. 小结

`WM`、`WAM`、`VAM` 最有价值的地方，不是给论文贴标签，而是帮助我们理解：今天的世界模型研究已经不再只是“学一个未来预测器”。它同时在走向更强的视频建模、更紧的动作联合建模、更统一的生成式规划接口，以及更现实的数据引擎和部署目标。把这三条线放在一起看，才能真正读懂现代世界模型为什么会同时吸收生成模型、强化学习、机器人和系统工程的语言。

## 19. 先把三类模型的“接口对象”分清楚

理解 `WM / WAM / VAM` 最容易混乱的地方，在于很多论文都写“预测未来”，但它们真正建模的接口对象并不一样。

### 19.1 `WM` 更强调什么

`WM` 往往把重点放在：

1. **隐状态如何概括过去观测**；
2. **给定动作后，未来状态怎样演化**；
3. **奖励、终止、风险等头如何从隐状态中读出**；
4. **如何在 latent space 中做 imagined rollout 与规划**。

它的核心接口是：**状态、动作、未来分布与价值信号之间的关系**。

### 19.2 `WAM` 更强调什么

`WAM` 更进一步，把动作也放进统一生成对象里。它常会同时学习：

1. **给定过去，未来观测如何生成**；
2. **给定过去，合理动作如何生成**；
3. **动作和未来是否可以在同一个 latent 轨迹中联合建模**；
4. **反事实动作替换后，未来是否会一致改变**。

它的核心接口是：**观测和动作不再是两张表，而是同一条轨迹的一体两面**。

### 19.3 `VAM` 更强调什么

`VAM` 常引入更强的视频先验与视觉生成能力，重点可能变成：

1. **高维视频未来能否被逼真预测**；
2. **视频中的交互结构能否帮助动作生成**；
3. **视频 latent 是否可作为更强的控制上下文**；
4. **生成未来视频是否能服务 planning、risk analysis 或 data engine**。

它的核心接口是：**视觉未来不仅是监督信号，也是动作与规划的中间载体**。

## 20. Action Interface 决定世界模型能不能接到真实系统

很多世界模型论文重点讲 latent，很少花大篇幅讲动作接口。但工程上，**动作接口的设计常常直接决定 `WM / WAM / VAM` 是否可用**。

### 20.1 常见动作表示

1. **离散动作 token**，适合游戏、网页操作、部分规划任务；
2. **连续低维控制向量**，适合机械臂、导航、无人车底层控制；
3. **动作 chunk / skill token**，适合把高频控制压成较低频抽象；
4. **层级动作接口**，高层给意图、低层给控制轨迹。

### 20.2 为什么动作表示会改变模型难度

1. **离散动作便于建模，但表达精度有限**；
2. **连续动作更贴近执行，但生成与评测都更难**；
3. **chunk 化降低了 rollout 长度，却会掩盖 chunk 内细节失稳**；
4. **层级动作更灵活，但 credit assignment 更复杂**。

### 20.3 一个很重要的判断

如果世界模型输出的动作接口和真实执行器不兼容，或者需要一层很重的 adapter 才能落地，那么论文里的 rollout 再漂亮，也很难稳定进入系统。

## 21. 状态表示要同时服务三件事

世界模型里的 latent state 不是“压缩得越狠越好”，而是至少要同时服务：

1. **观测重建或未来预测**；
2. **动作条件建模**；
3. **价值、风险或规划读出**。

### 21.1 为什么单靠重建目标不够

一个只擅长重建视频细节的隐状态，未必保留了动作决策真正需要的信息，例如：

1. **接触是否即将发生**；
2. **障碍物是否会在几秒后进入碰撞路径**；
3. **隐藏物体是否仍然存在**；
4. **任务相关物体是否已被遮挡但仍需跟踪**。

### 21.2 工程上更关心的 latent 特性

1. **是否支持 belief state**，在部分可观测环境中维护不确定性；
2. **是否支持 action sensitivity**，换动作后 latent rollout 会不会明显变化；
3. **是否可线性读出奖励、风险、接触等关键量**；
4. **是否足够紧凑，能在 real-time 或 near-real-time 预算内运行**。

## 22. 训练目标不能只看一个 NLL

`WM / WAM / VAM` 的训练常是多头、多目标、多尺度的。简单说一句“最小化未来预测损失”，会遮蔽掉最关键的设计权衡。

### 22.1 常见目标族

1. **重建或未来观测 NLL / diffusion loss**；
2. **latent dynamics consistency**；
3. **action modeling loss**；
4. **reward / terminal / risk prediction**；
5. **contrastive / consistency / BYOL-style 表示目标**；
6. **planner-friendly auxiliary heads**，如可达性、接触事件、关键点状态。

### 22.2 为什么会互相拉扯

1. **过强视觉重建** 可能牺牲控制关键信息密度；
2. **过强动作拟合** 可能让模型更像行为克隆器，而不是可反事实推演的世界模型；
3. **过多辅助头** 会增加训练复杂度，甚至让 latent 学成折中而不专精；
4. **过强抽象** 又可能让 rollout 缺乏可解释性和可视化验证能力。

## 23. 数据制度：机器人、驾驶、游戏并不是一类问题

世界模型常被统一讨论，但它依赖的数据制度其实差异很大。

### 23.1 机器人数据的典型特征

1. **采集贵**；
2. **动作连续且带执行器延迟**；
3. **近距离接触与遮挡多**；
4. **多机位视频与本体传感器需要严格对齐**；
5. **domain shift 来自光照、物体摆放、机械磨损等现实因素**。

### 23.2 自动驾驶数据的典型特征

1. **量大但尾部稀有事件极重要**；
2. **多传感器同步复杂**；
3. **安全约束强**；
4. **反事实评测需求高**；
5. **世界模型常需支持 long-horizon risk analysis，而不是只做短期控制**。

### 23.3 游戏与仿真数据的典型特征

1. **可大量采样**；
2. **动作接口离散或结构清晰**；
3. **环境可复现、可 counterfactual**；
4. **闭环评测成本低**；
5. **更容易把模型收益直接归因到规划能力上**。

### 23.4 这意味着什么

同一个 `WM / WAM / VAM` 架构，在不同数据制度下的成败因素可能完全不同。不要轻易把某一域里的成功经验直接套到另一域。

## 24. `WM` 往往是更稳的系统起点

如果团队刚开始搭世界模型链路，通常更稳的做法是先从偏 `WM` 的设计起步。

### 24.1 为什么

1. **状态、动作、奖励、终止之间的接口更清楚**；
2. **更容易分层调试**；
3. **更容易把 rollout 用到 planner 或 risk assessor**；
4. **在部分可观测和风险敏感任务中，更强调 belief 与 uncertainty 的表达**。

### 24.2 它的局限

1. **动作接口可能外接 policy，统一程度不足**；
2. **对高维视觉细节的表达可能不如大视频模型强**；
3. **如果下游主要目标是视觉生成式模拟，纯 `WM` 表达会显得抽象不足**。

## 25. `WAM` 的真正挑战是“联合建模是否真的联合”

很多 `WAM` 风格方法都主张统一建模动作和世界，但工程上必须追问：这份统一究竟带来了什么。

### 25.1 需要验证的关键问题

1. **动作分支是否真的利用了世界结构，而不是只是学了行为克隆**；
2. **未来分支是否真的对动作变化敏感，而不是只会生成平均未来**；
3. **联合 latent 是否比解耦的 `world + policy` 更有压缩效率或泛化能力**；
4. **统一训练是否让规划与控制接口更简单，还是只是增加了复杂度**。

### 25.2 最容易发生的伪收益

1. **模型容量更大了，所以一切都变好**；
2. **额外引入了更多视频数据，所以动作也变好**；
3. **训练目标更多了，所以某些离线指标提高，但闭环无增益**。

如果不做收益归因，`WAM` 很容易被过度神化。

## 26. `VAM` 的价值常在于“视频先验进入动作系统”

`VAM` 真正有吸引力，不是因为它能生成很像真的视频，而是因为**大规模视频先验可能让模型学到更强的物理直觉、交互先验和长程时空结构**，这些再反哺到动作生成。

### 26.1 但这条路有三道硬门槛

1. **视频 tokenizer 或 latent 必须保留控制相关信息**；
2. **动作头必须真正接入视频 latent，而不是只共享一个很表面的 backbone**；
3. **推理成本必须被约束，否则模型只能做离线分析器**。

### 26.2 一个更现实的落地位置

很多 `VAM` 风格模型短期更适合：

1. **离线生成式评估**；
2. **低频 planning proposal**；
3. **风险分支模拟**；
4. **数据引擎中的 near-miss 与 counterfactual 回放**。

而不一定马上进入高频控制主环。

## 27. 世界模型里的不确定性不是“可选附属件”

只要任务涉及部分可观测、长时规划和安全风险，不确定性就必须进入主设计，而不是做个额外头部点缀一下。

### 27.1 为什么

1. **模型会遇到多模态未来**；
2. **训练数据不可能覆盖所有风险场景**；
3. **高维观测里常有遮挡、传感器丢失和稀有事件**；
4. **闭环部署要求系统知道自己何时不该自信**。

### 27.2 不确定性至少要服务哪些动作

1. **回退与人工接管**；
2. **规划时的风险惩罚**；
3. **数据回流优先级选择**；
4. **评测时的 stress bucket 构造**。

## 28. 一份更实用的设计选择表

如果要在 `WM / WAM / VAM` 之间选路线，可先问：

1. **你是更需要稳态规划，还是更需要统一动作生成**；
2. **你有没有足够多且高质量的视频数据**；
3. **你的动作接口是离散、连续还是 chunk 化**；
4. **你的模型会进入高频闭环，还是先做低频分析与数据引擎**；
5. **你最需要提升的是任务成功率、风险识别，还是视觉泛化**。

### 28.1 一个粗略经验

1. **重规划和风险评估**：先看 `WM`；
2. **重统一建模与动作生成**：重点评估 `WAM`；
3. **重视频先验与视觉未来模拟**：重点评估 `VAM`；
4. **如果三者都想要**：通常意味着系统要分层，而不是强行一体化。

## 快速代码示例

```python
def rollout_by_mode(model, obs, actions, mode="wm"):
    if mode == "wm":
        return model.predict_state(obs, actions)
    if mode == "wam":
        return model.predict_state_action(obs, actions)
    if mode == "vam":
        return model.predict_video_and_action(obs, actions)
    raise ValueError(f"unknown mode: {mode}")
```

这段代码把 `WM / WAM / VAM` 的接口差异收敛到同一个调用入口，便于在同一评测框架里对比不同路线。实践中通常再补充统一的风险指标和回退策略，避免只比较生成效果。

## 29. 小结

`WM / WAM / VAM` 的真正差异，不在名字，而在**状态接口、动作接口、训练目标、数据制度和部署位置**。把这些接口真正讲清楚后，世界模型路线才不会停留在口号层面，而能变成一套面向真实任务的设计空间。
