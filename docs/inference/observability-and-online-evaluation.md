# 可观测性与在线评测

模型上线后，如果你只能看到“QPS 正常、GPU 没挂、用户偶尔抱怨”，那这个系统在工程上仍然是盲飞。推理系统的真正复杂度不只是模型本身，而是请求分布、缓存命中、工具链路、内容安全、业务目标和人工反馈共同作用的结果。可观测性回答“系统现在发生了什么”，在线评测回答“这些变化到底是好是坏”。两者缺一不可。

## 1. 为什么离线 benchmark 不能替代在线评测

离线测试集通常固定、干净、可重复，而真实流量会不断变化：

1. 用户问题分布会漂移；
2. 上下文长度会波动；
3. 工具调用可能失败；
4. 某类提示词可能被大规模复用；
5. 长尾错误会集中出现在高价值用户身上。

因此系统上线后的质量函数更像

$$
Q_{\text{online}}
=
\mathbb{E}_{x \sim p_t(x)}[u(f_\theta(x))],
$$

其中 $p_t(x)$ 随时间变化，$u(\cdot)$ 是业务效用函数。离线高分并不保证在线高效用。

## 2. 可观测性的三层对象

### 2.1 系统层

包括延迟、吞吐、错误率、CPU/GPU 使用率、显存、网络、缓存、队列长度等。

### 2.2 模型层

包括 token 数、TTFT、TPOT、上下文长度、拒答率、工具调用率、采样参数、输出长度分布、不确定性信号、路由结果等。

### 2.3 业务层

包括点击率、任务完成率、人工修正率、用户满意度、会话留存、转化率、工单关闭率等。

如果只监控系统层，你只能知道“机器很忙”；若只监控业务层，你又看不出问题源头在模型、缓存还是工具链。

## 3. 请求级 tracing

现代推理服务越来越需要端到端 tracing。一次请求通常会经历：

1. 网关接入
2. 检索或路由
3. prompt 构造
4. 模型 prefill
5. decode
6. 工具调用
7. 后处理
8. 返回客户端

把这条链拆开记录，才能在尾延迟变差时定位到底是 RAG 变慢、模型排队变长，还是外部 API 卡住。对 agent 系统尤其如此，因为一次用户请求可能触发多轮模型调用。

## 4. 在线质量代理指标

不是所有场景都能实时拿到“正确答案”。因此需要代理指标（proxy metrics），例如：

1. 用户是否继续追问同一问题
2. 是否复制答案
3. 是否点击推荐文档
4. 是否人工接管
5. 是否触发安全拦截
6. 工具调用成功率

这些代理指标不是最终质量，但能提供快速反馈。

## 5. 在线实验设计

### 5.1 A/B 测试

最经典的方式。随机把流量分配到控制组和实验组，比较关键指标差异。若指标为 $Y$，处理变量为 $T \in \{0,1\}$，则关注

$$
\Delta = \mathbb{E}[Y \mid T=1] - \mathbb{E}[Y \mid T=0].
$$

但对模型系统而言，A/B 还要防止：

1. 用户跨设备或跨会话污染
2. 流量分层不均
3. 工具版本或检索索引同时变化
4. 提示词缓存共享导致干扰

### 5.2 Interleaving

在检索或排序场景中，可把两个系统的结果交织展示，利用用户点击偏好比较优劣。它比传统 A/B 更节省流量，但适用范围有限。

### 5.3 Canary / 灰度

先把新版本放到小比例流量上观察关键风险指标，如报错率、空响应率、安全拦截率、极端长响应。灰度不是为了得出最终结论，而是为了先确认“不会立刻出事故”。

## 6. 置信区间与样本量

很多线上实验失败，不是模型没变好，而是样本量不足导致噪声淹没信号。若观测到某比例指标 $\hat{p}$，样本量为 $n$，其近似标准误差为

$$
\mathrm{SE}(\hat{p}) \approx \sqrt{\frac{\hat{p}(1-\hat{p})}{n}}.
$$

若想检测微小提升，就需要更大样本。对高方差任务完成率或长会话指标尤其如此。

## 7. 模型特定的在线监控项

### 7.1 输入分布漂移

包括：

1. prompt 长度分布变化
2. 语言分布变化
3. 工具类型变化
4. 多模态输入比例变化

这些变化会直接影响延迟和质量。

### 7.2 输出行为漂移

包括：

1. 平均输出长度变化
2. 拒答率变化
3. 幻觉关键词增多
4. 格式遵循率下降
5. 特定模板化回答增多

### 7.3 路由与缓存行为

多模型系统中，某个路由器参数微调，就可能让更多请求落到慢模型或贵模型上。没有监控就很难察觉。

## 8. 在线评测中的人类反馈

自动指标永远有盲区，因此高价值场景往往需要抽样人工评审。评审可以关注：

1. 正确性
2. 相关性
3. 完整性
4. 风险性
5. 工具使用是否合适

**工程上可采用双层方案**：海量请求用代理指标，少量高价值样本用人工精审。人工结果再回流为训练和评测数据。

## 9. RAG 与 agent 场景的特殊问题

RAG 系统如果答案变差，问题可能来自：

1. 检索召回退化
2. 重排器退化
3. prompt 拼接过长
4. 生成模型未正确利用证据

**agent 系统则还要额外监控**：

1. 计划步数是否异常增长
2. 工具调用是否空转
3. 工具参数错误率
4. 死循环或重复调用

因此在线评测不能只看最终回答，还要把中间链路纳入观测。

## 10. 错误分型

成熟系统会把线上错误拆成类型，而不是简单记作“失败”。例如：

1. 检索失败
2. 推理超时
3. 工具调用失败
4. 幻觉式回答
5. 安全策略误拦截
6. 多轮记忆错误

有了错误 taxonomy，后续改进才能对症下药。

## 11. 线上回放与复现实验

当线上出现长尾问题时，第一步往往不是再跑一个通用 benchmark，而是把问题请求及其上下文回放到离线环境。一个好的系统应支持：

1. 请求快照
2. 模型版本绑定
3. 索引版本绑定
4. 工具结果回放或 mock
5. 路由决策回放

否则你看到的只是“用户说错了”，却无法稳定重现。

## 12. 数据回流闭环

在线评测的真正价值，不只是报警，而是形成闭环：

1. 发现问题请求
2. 分类与抽样
3. 人工评审
4. 进入训练 / 规则 / 检索索引修复
5. 再上线验证

如果没有回流机制，可观测性最终会沦为 dashboard 装饰。

## 13. 两个具体例子

### 13.1 企业知识助手

某助手上线后，用户满意度总体稳定，但工单关闭率下降。Tracing 发现平均模型延迟没变，真正变差的是检索阶段：新接入的一批 PDF 文档 OCR 质量差，导致召回片段错误。若只看模型 token 级指标，根本定位不到问题。

### 13.2 屏幕操作 agent

另一个系统升级后，任务完成率从 68% 下降到 61%。进一步看日志发现模型生成步数更多，但工具成功率下降，原因是 UI 元素 grounding 模型版本更新后坐标尺度变了。这里离线语言评测完全无法反映真实退化。

## 14. 常见误区

### 14.1 只看均值，不看尾部

P50 正常不代表 P99 没问题，而尾部问题往往决定用户体验和事故风险。

### 14.2 把安全拦截率下降当好事

有时拦截率下降是因为分类器失效，不是系统更安全了。

### 14.3 同时改太多变量

如果一次同时更新模型、检索、提示词和缓存策略，线上实验结论几乎不可解释。

### 14.4 没有保底回滚

没有清晰回滚机制的灰度，本质上不是实验，是赌博。

## 15. 一个形象比喻

可观测性像汽车仪表盘，在线评测像真实路测。只有仪表盘而不路测，你知道发动机转速，却不知道这辆车在雨天急弯是否打滑；只有路测而没有仪表盘，你知道车“不太对劲”，却不知道是胎压问题、刹车过热还是油路堵塞。大模型系统上线后也一样，必须同时知道“发生了什么”和“结果好不好”。

## 16. 小结

可观测性与在线评测的本质，是把模型服务从“离线实验品”变成“可运营系统”。它要求同时看系统链路、模型行为和业务效果，并通过 tracing、灰度、人工评审和数据回流形成闭环。没有这套机制，任何上线后的优化都很容易变成盲调。 

## 17. 可观测性不是 dashboard 数量，而是因果定位能力

很多团队表面上有很多图，但真出问题时仍然不知道根因在哪。原因在于他们监控的是“现象”，而不是“链路”。

### 17.1 真正有用的可观测性要回答什么

1. 问题发生在哪一层；
2. 问题是从什么时候开始的；
3. 哪一类请求受影响最大；
4. 是否与某次版本变化强相关；
5. 如何在离线环境稳定复现。

如果这些问题回答不了，再多面板也只是装饰。

## 18. Tracing 应该贯穿整条请求路径

一条高价值 trace 通常至少覆盖：

1. 网关进入；
2. 路由决策；
3. 检索与 rerank；
4. prompt 构造；
5. prefill；
6. decode；
7. 工具调用；
8. 后处理；
9. 返回客户端。

### 18.1 为什么要保留版本信息

**每一步都应尽量绑定**：

1. 模型版本；
2. prompt 模板版本；
3. 检索索引版本；
4. reranker 版本；
5. 工具版本；
6. 路由器版本。

否则你只能知道“今天变差了”，却不知道是模型、检索还是工具链变了。

## 19. 指标要按请求类型切，而不是只看全局均值

全局均值很容易掩盖结构性退化。至少应按以下维度切开：

1. 长度桶；
2. 模态类型；
3. 是否带附件；
4. 是否工具调用；
5. 用户等级；
6. 路由模型；
7. 租户或业务线。

### 19.1 一个典型误判

如果普通 FAQ 流量占 80%，而文档问答只占 20%，那么即使文档问答严重退化，全局平均也可能几乎不动。  
这就是为什么分桶不是可选项，而是可观测性的主结构。

## 20. 在线评测需要同时看质量、成本和稳定性

模型系统最怕“离线质量提升但线上总效用下降”。因此线上实验不应只看质量 proxy，还要同时看：

1. 平均和分位延迟；
2. 工具调用次数；
3. 升级到大模型的比例；
4. 缓存命中率；
5. GPU 成本和外部 API 成本；
6. 错误恢复率。

### 20.1 为什么成本指标必须同行

如果一个新路由策略把更多请求送去大模型，质量可能略升，但成本和排队也会上升。  
没有成本视角，就会把“更贵的正确”误判成“更好的系统”。

## 21. 在线质量代理指标该如何选

**好的 proxy 指标应尽量满足**：

1. 与真实业务目标相关；
2. 反馈及时；
3. 难以被表面风格作弊；
4. 能按请求类型分桶解释。

### 21.1 常见 proxy 指标

1. 用户是否继续追问同一问题；
2. 是否点击引用文档；
3. 是否复制回答；
4. 是否转人工；
5. 是否完成工具流程；
6. 是否触发安全拦截。

### 21.2 proxy 的风险

1. 更长回答可能提高复制率，但未必真正更有帮助；
2. 更保守回答可能降低事故率，但也可能伤害任务完成；
3. 用户不继续追问不一定是满意，也可能是放弃。

因此 proxy 需要与人工评审和业务最终指标一起使用。

## 22. 人工评审在线上体系中的位置

自动指标解决覆盖，人工评审解决可信度。高价值场景常用双层机制：

1. 海量流量靠自动指标做粗筛；
2. 少量高价值样本做人审精查。

### 22.1 什么样本最值得人工评审

1. 多模型分歧最大；
2. judge 置信度最低；
3. 高价值业务请求；
4. 新功能和新版本相关请求；
5. 线上投诉或事故触发请求。

## 23. 错误 taxonomy 要服务回流，而不是只服务报表

错误分类的目标不是图表好看，而是决定后续动作。一个实用 taxonomy 通常要映射到：

1. 训练数据回流；
2. prompt 修复；
3. 检索索引修复；
4. 工具链修复；
5. 路由阈值调整；
6. 安全策略调整。

### 23.1 例如

1. `retrieval_miss` 应回流到检索和文档处理链；
2. `tool_arg_error` 应回流到工具 SFT 和 schema 校验；
3. `factual_hallucination` 可能回流到后训练和证据拼接；
4. `over_refusal` 应回流到偏好和安全边界数据。

## 24. Canary 和灰度不只是防事故，也是在收集证据

灰度的作用不仅是“先上 1% 看有没有炸”，还应被设计成一套高价值观测流程：

1. 看哪类请求先出问题；
2. 看问题是否集中在某个租户或某个模型路径；
3. 看是否存在系统级退化但全局均值还没报警；
4. 收集第一批高价值回流样本。

### 24.1 灰度阶段最该盯什么

1. 错误率；
2. P95/P99；
3. 升级率或路由比例变化；
4. 工具失败率；
5. 安全拦截率异常变化；
6. 高价值用户的任务完成情况。

## 25. 线上回放系统应该如何设计

为了让问题可复现，线上回放至少要尽量固定：

1. 请求内容；
2. 上下文；
3. 模型版本；
4. 检索索引版本；
5. prompt 模板；
6. 工具返回或 mock；
7. 路由决策。

### 25.1 为什么很多团队回放无效

因为他们只保存了用户问题，却没保存：

1. 当时检索到了什么；
2. 当时命中了哪个模型；
3. 当时工具是否超时；
4. 当时使用的是哪个 prompt 模板。

结果是复盘时“好像重现了问题”，其实环境已经不同。

## 26. 线上监控也要看行为漂移

除了系统指标，还要看模型行为有没有悄悄变化，例如：

1. 回答长度越来越长；
2. 拒答率突然升高；
3. 某类模板话术变多；
4. 工具调用率异常变化；
5. 输出格式遵循率下降。

### 26.1 为什么行为漂移危险

因为很多线上退化不会第一时间表现成明显错误，而是先表现成：

1. 更贵；
2. 更慢；
3. 更啰嗦；
4. 更保守；
5. 更不稳定。

如果只盯错误率，很容易错过这些早期信号。

## 27. 一个实践化在线评测闭环

1. 实时采集请求级 trace；
2. 对核心指标按请求类型和模型路径分桶；
3. 对异常桶做自动告警和样本抽取；
4. 用 judge 和程序规则做初筛；
5. 人工复核高价值样本；
6. 把问题映射到训练、检索、工具或路由修复；
7. 下一版本上线后验证同类错误是否下降。

### 27.1 这套闭环的关键

不是“多监控几个面板”，而是保证每一次异常都能落回可执行的修复动作。

## 28. 设计建议

1. 监控先服务定位，再服务展示；
2. 所有核心指标都按任务类型、模型路径和租户分桶；
3. 在线实验同时看质量、成本和稳定性；
4. 对 judge、路由器和安全分类器做独立校准与回放；
5. 建立从异常样本到训练/系统修复的标准回流路径。

## 29. 小结

可观测性与在线评测真正要建设的，是一种把“线上异常”快速翻译成“可定位、可复现、可修复问题”的能力。没有这套能力，模型上线后所有优化都像在黑箱里试探；有了这套能力，训练、推理、检索、工具和产品目标才能真正被一条闭环连接起来。

## 30. 观测模型系统时，最怕“高基数但无结构”

现代模型服务会产生大量维度：

1. **租户**；
2. **任务类型**；
3. **模型版本**；
4. **量化版本**；
5. **路由路径**；
6. **输入长度桶**；
7. **工具调用类型**。

### 30.1 为什么高基数会让观测失效

如果把所有标签直接打进监控系统，会出现：

1. **查询成本高**；
2. **面板难读**；
3. **告警规则失控**；
4. **真正关键的模式反而被淹没**。

### 30.2 更稳妥的做法

1. **保留少数核心维度做实时聚合**；
2. **把高基数原始细节放进 trace / 日志**；
3. **对长度、风险、任务类型预先做分桶**；
4. **在面板上只展示对决策有用的切面**。

## 31. 先定义 SLI 树，再建面板

很多团队先堆面板再想指标含义，这常常导致“看得见很多数字，但不知道哪个真重要”。

### 31.1 更有效的顺序

1. **先定义最重要的 SLI**；
2. **再定义这些 SLI 由哪些底层指标支持**；
3. **最后才设计仪表盘与告警**。

### 31.2 一个常见的 SLI 树

1. **用户体验 SLI**：任务完成率、P95 响应时延、人工接管率；
2. **模型行为 SLI**：拒答率、工具成功率、格式正确率、judge 分数；
3. **系统资源 SLI**：GPU 利用率、KV 占用、队列时延、错误率；
4. **成本 SLI**：单位请求成本、单位 token 成本、高峰期扩容成本。

## 32. Trace Schema 应提前设计，而不是事后补

许多线上问题之所以难查，是因为 trace 一开始没有设计好。

### 32.1 建议的核心字段

1. **request id / session id**；
2. **tenant / product / task type**；
3. **input length / output length bucket**；
4. **model route / runtime route / fallback route**；
5. **cache 命中信息**；
6. **工具调用链和耗时**；
7. **judge / policy / safety 结果**；
8. **最终业务结果或反馈 proxy**。

### 32.2 为什么这一步要前置

因为出了事再补 trace 字段，通常已经错过了最关键的现场信息。

## 33. 在线评测要区分“持续监控”和“实验验证”

持续监控和实验验证看似都在看指标，但目的不同。

### 33.1 持续监控关注

1. **系统有没有退化**；
2. **某个桶是否出现异常漂移**；
3. **告警是否应触发**。

### 33.2 实验验证关注

1. **新版本是否比旧版本更好**；
2. **收益是不是因果性的**；
3. **收益是否值得引入额外成本和复杂度**。

### 33.3 为什么两者不能混用

若用实验面板替代持续监控，日常异常会被看漏；若用日常聚合面板判断实验结论，往往又缺少统计纪律。

## 34. Replay 系统是线上观测闭环的核心

真正成熟的推理平台，通常都会建设 replay 能力，把一批真实请求或代表性流量在离线环境中重放。

### 34.1 Replay 有什么用

1. **复现线上问题**；
2. **比较新旧模型或 runtime**；
3. **评估路由和 cache 策略变化**；
4. **做风险评估而不直接伤真实用户**。

### 34.2 Replay 至少要保留什么

1. **输入样本**；
2. **上下文和附件**；
3. **工具返回或其 mock**；
4. **原始路由信息**；
5. **当时关键配置和模型版本**。

## 35. Shadow Evaluation 适合看“如果换版本会怎样”

并不是所有线上验证都要真正影响用户。很多时候更稳的做法是 shadow：

1. **真实用户只看到主路径输出**；
2. **新版本在后台同步跑**；
3. **比较两者时延、格式、judge 分数和工具行为**。

### 35.1 Shadow 的价值

1. **更安全**；
2. **能快速积累对照样本**；
3. **适合做高风险模型或路由策略预验证**。

### 35.2 它的限制

1. **无法直接看到真实用户交互反馈**；
2. **工具调用和副作用场景需要额外隔离**；
3. **某些系统成本会明显增加**。

## 36. 质量代理指标需要分层校准

线上很多时候拿不到 gold label，因此会依赖：

1. **judge 分数**；
2. **用户重试率**；
3. **人工接管率**；
4. **工具成功率**；
5. **任务完成率 proxy**。

### 36.1 为什么这些代理要校准

因为它们不一定真正等价于质量。例如：

1. **用户不追问，不代表答案正确**；
2. **工具调用成功，不代表选择正确**；
3. **judge 高分，也可能只是格式更讨喜**。

### 36.2 更好的做法

定期抽样人工复核，建立：

1. **proxy 与人工质量的一致性曲线**；
2. **不同任务桶上的偏差图**；
3. **不同模型版本间 proxy 稳定性对照**。

## 37. 线上安全评测不能只看拦截率

安全与合规场景里，光看拦截率很危险。

### 37.1 还应看

1. **误杀率**；
2. **漏检率**；
3. **不同语言和任务类型的偏差**；
4. **高价值用户是否被过度保守地拒绝**；
5. **安全模型和主模型、路由器的交互副作用**。

### 37.2 为什么这对业务很重要

因为“更安全”若通过过高误杀换来，业务体验可能同样不可接受。

## 38. 告警规则要从“绝对阈值”升级到“分桶异常”

纯绝对阈值告警在模型系统里常不够用。更有效的是：

1. **按任务桶看漂移**；
2. **按长度桶看异常**；
3. **按租户或地区看局部退化**；
4. **按 route path 看新版本异常**。

### 38.1 一个典型例子

整体 P95 延迟可能没变，但长上下文企业请求已经显著变慢。若没有分桶告警，这类高价值退化会被整体平均值掩盖。

## 39. 面板设计应服务定位，而不是只服务汇报

很多面板做得很漂亮，但出了事仍然帮不上忙。更好的面板应支持：

1. **从总览一键下钻到任务桶**；
2. **从任务桶下钻到模型/路由/缓存路径**；
3. **从路径下钻到 trace 样本**；
4. **从异常样本回到 replay 或回流系统**。

### 39.1 为什么“下钻路径”比图表数量更重要

因为观测系统真正的价值，不是展示系统有多复杂，而是让工程师在几分钟内把问题从“这里不对”缩小到“就是这类请求、这条链路、这次改动”。

## 40. 线上评测要把成本一起纳入结论

很多线上实验只汇报：

1. **质量提升**；
2. **时延变化**；

却不看成本。这会导致很多错误结论。

### 40.1 应同时汇报

1. **单位请求成本**；
2. **单位 token 成本**；
3. **高峰期容量消耗**；
4. **因质量变化带来的重试或人工成本变化**。

### 40.2 为什么这是必需项

因为某些实验虽然质量略好，但成本和容量代价过高，长期并不值得进入主线。

## 41. 一份更工程化的在线评测闭环

1. **线上监控发现异常或机会**；
2. **trace 与样本回收进入 replay**；
3. **shadow / A/B / canary 验证改动**；
4. **抽样人工校准代理指标**；
5. **高价值失败样本进入训练或系统修复回流**；
6. **回归集更新并纳入下一轮发布门槛**。

### 41.1 为什么闭环必须完整

如果观测只能发现问题，却不能把问题变成后续可修复资产，那它仍然只是报警器，不是系统能力。

## 42. 一份可观测性与在线评测清单

1. **是否有结构化 SLI 树**；
2. **trace schema 是否足以串起请求到业务结果**；
3. **是否有 replay 和 shadow 能力**；
4. **代理指标是否定期做人审校准**；
5. **告警是否支持分桶异常**；
6. **面板是否支持快速下钻**；
7. **实验结论是否同时看质量、时延和成本**；
8. **异常样本是否能顺畅回流到修复系统**。

## 43. 小结

可观测性与在线评测真正要建设的，不只是更多日志和面板，而是一套**能定位、能验证、能回放、能回流**的闭环系统。谁能把这套闭环做扎实，谁就能更快、更稳地把线上问题转成下一轮模型和系统迭代的有效燃料。

## 44. Session 级观测常常比单请求更贴近真实体验

很多指标都是按请求记，但不少业务体验实际上发生在 session 级。

### 44.1 为什么 session 级重要

1. **用户会连续追问和修正**；
2. **一次工具调用失败可能影响后续多轮**；
3. **模型若前后不一致，单请求指标不一定能暴露问题**；
4. **用户的“是否满意”往往是整段会话级判断**。

### 44.2 建议的 session 指标

1. **会话完成率**；
2. **平均追问次数**；
3. **同题重试率**；
4. **多轮工具成功闭环率**；
5. **会话级人工接管率**。

## 45. 合成探针流量是线上观测的“哨兵”

除了真实流量，还建议持续注入少量合成探针请求。

### 45.1 探针流量适合覆盖

1. **关键业务模板**；
2. **高风险安全样本**；
3. **长上下文基准样本**；
4. **典型工具链路样本**；
5. **缓存和路由敏感样本**。

### 45.2 为什么探针值钱

1. **它稳定可比较**；
2. **能在真实流量漂移时提供参照系**；
3. **适合做版本灰度和日常回归**；
4. **出现异常时能快速判断问题是系统退化还是用户分布变了**。

## 46. 异常检测应区分“业务漂移”和“系统退化”

线上指标变差时，不一定是系统坏了，也可能是流量结构变了。

### 46.1 更稳妥的判断方式

1. **先看探针流量是否也变差**；
2. **再看输入长度、任务类型、附件大小等分布是否漂移**；
3. **再看同分桶下旧版本与新版本是否有差异**；
4. **最后才决定是回滚系统还是调整流量假设**。

### 46.2 为什么要这样分

因为如果把业务漂移误判成系统退化，团队会做出错误回滚；反过来若把系统退化误判成流量变化，也会拖长故障时间。

## 47. 线上观测系统要有“样本采样策略”

不是所有请求都值得同等保留。更成熟的系统会设计结构化采样策略。

### 47.1 建议重点采样

1. **高不确定性样本**；
2. **高成本样本**；
3. **高价值租户样本**；
4. **长尾失败样本**；
5. **路由和 judge 分歧大的样本**；
6. **高风险安全样本**。

### 47.2 为什么采样策略重要

如果只做随机采样，系统往往会保留大量普通样本，却错过最能帮助定位和回流的边界样本。

## 48. 隐私与脱敏必须是观测管线的一部分

模型系统的 trace 和 replay 往往包含丰富上下文，若不处理好隐私，观测系统本身会成为风险源。

### 48.1 至少应治理

1. **用户标识脱敏**；
2. **正文和附件中的敏感字段掩码**；
3. **仅对高权限人员开放原始样本回放**；
4. **对 replay 数据做分级存储与访问审计**。

### 48.2 为什么这一步不能后补

因为一旦观测系统广泛落地，再补隐私治理的代价会非常高，而且容易引发合规问题。

## 49. 发布治理需要和在线评测直接联动

发布流程如果和观测系统脱节，团队就会在灰度期靠人工盯图判断，很难稳定扩展。

### 49.1 更成熟的发布联动

1. **灰度开始前自动比对探针和回归桶**；
2. **灰度中按关键桶自动判断是否继续放量**；
3. **超过回滚阈值自动触发回退或冻结**；
4. **灰度结束后自动生成质量/时延/成本对照报告**。

### 49.2 为什么自动化值得投入

因为模型与 runtime 迭代会越来越频繁，纯人工发布评估无法长期撑住复杂度。

## 50. 多模型系统要做“路径级归因”

在多模型、多 runtime、多路由路径并存的系统里，线上问题很容易变成“到底是哪条路径出问题”。

### 50.1 路径级归因需要回答

1. **哪类请求走了哪条模型路径**；
2. **哪条路径的成本最高**；
3. **哪条路径的失败率或人工接管率更高**；
4. **同类请求换路径后会不会更好**。

### 50.2 为什么这对优化顺序关键

因为很多时候不需要重写整个系统，只要把一类请求从错误路径移走，就能获得比内核优化更大的收益。

## 51. 线上评测要有“负向实验档案”

团队常记录成功上线的实验，却不系统记录失败实验。长期看这会造成重复犯错。

### 51.1 负向档案应记录

1. **实验假设**；
2. **最终失败的主要指标**；
3. **失败集中在哪些桶**；
4. **是否因成本、时延、安全或质量被否决**；
5. **后续是否可能在其他场景重试**。

### 51.2 为什么这很值钱

因为线上实验成本高。若不保留失败经验，组织会反复为同一类错误付费。

## 52. 观测平台本身也应有 owner 和版本

很多组织对模型版本很敏感，却对观测平台的变更缺少治理。

### 52.1 应版本化的内容

1. **关键指标定义**；
2. **trace schema**；
3. **judge proxy 口径**；
4. **仪表盘与告警规则**；
5. **采样和脱敏策略**。

### 52.2 为什么这一步重要

否则你可能会在不知不觉中改变统计口径，导致前后版本的线上数据不可比。

## 53. 一份更高级的可观测性清单

1. **是否具备 session 级指标**；
2. **是否有长期运行的合成探针流量**；
3. **异常检测是否能区分业务漂移与系统退化**；
4. **是否有面向高价值样本的采样策略**；
5. **隐私与脱敏是否进入观测主流程**；
6. **发布治理是否与在线评测自动联动**；
7. **多模型系统是否支持路径级归因**；
8. **失败实验是否被系统记录**；
9. **观测平台本身是否版本化**。

## 54. 小结

高级可观测性建设的目标，不是让系统拥有更多监控名词，而是让线上模型平台具备**持续感知、自动比较、快速归因、合规回放和制度化学习**的能力。只有当观测平台本身也被当作正式产品来设计和治理时，模型迭代速度才不会被线上不确定性持续拖住。

## 55. Oncall 视角下的观测设计要回答“下一步做什么”

很多观测系统能告诉值班同学“这里红了”，却不能告诉他“下一步该查哪里”。这会显著拉长故障恢复时间。

### 55.1 一个更实用的 oncall 面板

1. **先给出当前受影响的关键 SLI**；
2. **再给出最可疑的分桶**；
3. **再给出最近变更和灰度状态**；
4. **再提供直接跳转到 trace、replay、路由看板和回滚开关**。

### 55.2 为什么 oncall 视角重要

因为线上事故不是论文阅读题。最重要的是在几分钟内缩小问题面，决定：

1. **回滚还是继续观察**；
2. **扩容还是限流**；
3. **通知模型团队还是平台团队**。

## 56. Golden Signals 需要为模型系统重写

传统服务强调 latency、traffic、errors、saturation，但模型系统需要扩展版本。

### 56.1 模型系统的 golden signals

1. **Latency**：TTFT、TPOT、P95/P99；
2. **Traffic**：请求数、token 数、附件数、工具调用数；
3. **Errors**：HTTP 错误、空响应、格式错误、工具失败、安全拦截异常；
4. **Saturation**：GPU、KV cache、queue、OCR/检索/工具链饱和度；
5. **Quality**：judge 分数、任务完成率 proxy、人工接管率。

### 56.2 为什么要加第五项 Quality

因为模型系统就算系统层完全健康，也可能因为路由、prompt、judge、cache 或模型版本变化而在质量上悄悄退化。

## 57. Feature Flag 和线上指标必须可关联

随着灰度和实验越来越多，很多线上退化并不是“新模型上线”这么简单，而是多个 feature flag 叠加造成。

### 57.1 常见 flag 类型

1. **新模型版本**；
2. **新 runtime 或新 kernel path**；
3. **新路由规则**；
4. **新缓存策略**；
5. **新安全策略或工具策略**。

### 57.2 为什么要与观测直接关联

如果看板里看不到某个时间点到底开了哪些 flag，就很难回答：

1. **异常是否由该 flag 触发**；
2. **多个 flag 是否有交互副作用**；
3. **关掉哪一个最可能快速止血**。

## 58. 长期存档策略决定了你能不能做“月级回看”

很多组织的观测数据只能保留几天到几周，这会让很多慢性问题难以看清。

### 58.1 哪些数据值得长期保留

1. **核心聚合指标**；
2. **关键灰度和发布期的 trace 样本**；
3. **高价值失败样本和 replay 数据**；
4. **发布报告与线上实验结论**；
5. **重大事故前后的观测快照**。

### 58.2 为什么长期回看有价值

1. **能识别季节性或产品节奏带来的流量变化**；
2. **能看清某些慢性成本漂移**；
3. **能为模型和系统的长期演化提供证据**。

## 59. 观测所有权必须明确到团队

模型系统跨越平台、模型、检索、工具、产品多个团队，如果没有明确 owner，线上问题很容易互相甩锅。

### 59.1 更清晰的 ownership 划分

1. **平台团队**负责资源、queue、runtime、cache、trace 基础设施；
2. **模型团队**负责模型行为、judge、prompt、路由策略效果；
3. **产品团队**负责业务成功率定义、价值分层和上线门槛；
4. **安全/合规团队**负责风险指标、脱敏和审计要求。

### 59.2 为什么 owner 明确比面板更多更重要

因为真正的闭环不是“谁都能看见问题”，而是“有人必须对问题的定位、修复和回归负责”。

## 60. 一份面向组织的观测清单

1. **oncall 面板是否能直接指导下一步动作**；
2. **golden signals 是否覆盖了质量项**；
3. **feature flag 是否与线上指标可关联**；
4. **是否有足够长的长期存档策略**；
5. **每类指标是否有明确 owner 和回滚责任人**。

## 61. 小结

真正成熟的线上观测，不只是技术系统，而是一套**让值班、灰度、发布、复盘和长期演化都能依赖的组织机制**。当 oncall、golden signals、feature flags、长期存档和 ownership 全部接起来时，观测平台才算真正具备了支撑大规模模型服务持续进化的能力。

## 62. 冷启动观测要单独设计

很多系统在稳态下表现不错，但一到新模型刚上线、新池子刚扩容或新缓存刚清空时，就会出现明显退化。

### 62.1 冷启动常见问题

1. **编译或权重加载开销**；
2. **prefix / KV cache 尚未形成热点**；
3. **routing policy 对新路径缺少历史统计**；
4. **工具链和外部依赖尚未进入稳态缓存**。

### 62.2 为什么要单独监控冷启动

因为如果把冷启动和稳态混在一起，团队既看不清真实稳态表现，也很难知道新版本到底是“稳态更差”还是“只是热身更慢”。

## 63. 地域、语言和租户分桶会显著改变线上结论

一个全球化或多行业系统中，整体指标常常掩盖局部退化。

### 63.1 建议重点分桶

1. **地域或部署区**；
2. **语言**；
3. **行业或产品线**；
4. **关键租户与普通租户**；
5. **免费流量与付费流量**。

### 63.2 为什么这些桶必须长期保留

因为很多局部问题不会先在全局均值上体现，但会先伤到最高价值用户。

## 64. 事故归档应和观测快照绑定

线上事故复盘最怕只留下文字总结，没有现场证据。

### 64.1 更完整的事故归档

1. **关键时间窗的指标快照**；
2. **相关 trace 样本**；
3. **当时生效的 feature flags**；
4. **回滚和止血动作时间线**；
5. **后续修复与回归验证链接**。

### 64.2 为什么这很重要

因为半年后再遇到类似问题时，真正能帮助团队加速定位的往往不是结论本身，而是当时的证据链。

## 65. 发布后首周应有强化观测模式

很多版本在灰度通过后，放量一周内才暴露问题，因此建议对新版本设置强化观测窗口。

### 65.1 强化观测窗口重点看

1. **关键高价值桶**；
2. **成本与容量是否缓慢漂移**；
3. **冷启动与热启动差异**；
4. **异常样本是否持续增加**；
5. **人工接管和工单是否上升**。

### 65.2 为什么这一周最关键

因为很多副作用只有在真实规模和真实用户分布下才会显现。

## 66. 一份最终的线上观测准则

1. **稳态和冷启动分开看**；
2. **全局和高价值局部分开看**；
3. **事故文字和证据快照一起存**；
4. **新版本首周进入强化观测**；
5. **所有线上异常都能回到 trace、replay 和回流链路**。

## 67. 小结

线上观测真正成熟的标志，不是图表越来越多，而是系统能在**冷启动、局部高价值流量、事故复盘和发布后演化**这些最容易失真的阶段仍然保持可解释。做到这一点后，观测平台才真正成为模型服务的长期稳定器。

## 68. 观测口径变更必须公告

当团队修改 judge、指标公式、bucket 边界或采样策略时，本质上是在改变“系统如何被看见”。

### 68.1 为什么要公告

1. **避免前后版本不可比**；
2. **避免灰度和发布结论被误读**；
3. **让相关 owner 知道哪些面板和告警需要同步调整**。

## 69. 面向高价值客户的白名单观测值得单独做

并不是所有用户都需要同等级别的观测深度。对高价值租户或关键业务流，单独保留更精细的面板、trace 与人工巡检通常是值得的。

### 69.1 这类白名单观测可重点覆盖

1. **专属时延与质量桶**；
2. **特定工具链成功率**；
3. **升级后的体验变化**；
4. **人工接管与工单关联**。

## 70. 最终小结

可观测性与在线评测做到最后，拼的不是图表数量，而是**统计口径稳定、高价值流量可见、异常样本可追、发布变更可解释**。这几件事连起来，线上系统才真正能支撑高频迭代而不失控。

## 71. 观测也要服务容量预测

线上观测不只用于排障，也应反哺容量规划。

### 71.1 应反哺的信号

1. **长度分布漂移**；
2. **高峰时段的任务结构变化**；
3. **prefix 命中与工具调用的季节性变化**；
4. **高价值桶的增长趋势**。

## 72. 回流优先级最好由观测直接打分

不是所有异常样本都值得立刻进入训练。观测系统可根据频率、风险、业务价值和可复现性为样本打优先级分。

### 72.1 这样做的好处

1. **训练回流更聚焦**；
2. **人审资源更高 ROI**；
3. **线上问题不会在回流链路里再次失焦**。

## 73. 收口小结

当观测能同时服务**排障、发布、回流和容量预测**时，它才真正从辅助系统变成主系统。这也是大规模模型平台最后必须补上的一层治理能力。

## 74. 观测面板应保留“最近一次变更”栏

很多异常发生时，最先该看的就是最近有哪些变更进入了系统。把最近一次模型、runtime、路由、prompt、judge、缓存策略变更直接显示在面板上，往往能显著缩短定位时间。

## 75. 高风险桶应有固定巡检节奏

对长上下文、工具调用、安全敏感、高价值租户等高风险桶，建议设固定巡检，而不是只等告警触发。这样能更早发现那些尚未穿透全局门槛、但已经在局部恶化的问题。

## 76. 终局小结

线上观测最终拼的是两件事：**异常能否足够早地被看见，问题能否足够快地被翻译成下一步动作**。只要把这两件事做扎实，模型系统的高频迭代才会真正可控。

## 77. 一份可直接落地的线上观测执行模板

如果团队要把这页内容真正落到平台里，建议按下面顺序推进：

1. **先定核心 SLI**：
   把 TTFT、P95、任务完成率 proxy、人工接管率、单位请求成本定成最小核心集合。
2. **再定分桶**：
   至少切出长度桶、任务桶、高价值租户桶、高风险桶和多模态桶。
3. **然后补 trace schema**：
   request id、route path、cache 命中、工具链、judge 结果、feature flags 一次补齐。
4. **再建 replay 与 shadow**：
   没有这两层，后续很多线上异常只能看见，不能验证。
5. **最后建回流**：
   把高优先级异常样本自动沉淀到训练、评测和复盘链路。

### 77.1 为什么这个顺序更稳

因为很多团队会倒着做：

1. 先堆很多 dashboard；
2. 再补几个告警；
3. 出问题后才发现没有 trace、没有 replay、没有高价值样本回流。

这样最终只能得到一套“看起来很全、但真正定位时不够用”的观测系统。

### 77.2 执行模板的验收标准

一套线上观测体系至少应让团队在出现重大异常时，能在较短时间内回答：

1. **受影响的是哪类用户和哪类请求**；
2. **最可能是哪条模型/路由/runtime 路径出了问题**；
3. **是否与最近变更直接相关**；
4. **是否需要立刻回滚、限流或切换 fallback**；
5. **哪些样本应该进入 replay 与后续回流**。

只要这五个问题还答不快，观测体系就还没有真正成型。

## 78. 观测体系的长期维护建议

线上观测不是一次性工程，真正难的是长期维护不失真。建议团队固定做三件事：

1. **季度口径审计**：
   检查关键 SLI、judge proxy、桶定义、采样策略是否已经偏离当前产品和模型形态。
2. **月度高价值样本复查**：
   从高风险桶、高价值租户桶、人工接管桶里抽样，确认观测指标仍能真实反映问题。
3. **版本后复盘归档**：
   每次重要模型、runtime、路由升级后，沉淀一份质量、时延、成本和异常分桶的对照报告。

### 78.1 为什么维护比搭建更难

因为模型系统变化极快：

1. **模型会换**；
2. **prompt 会换**；
3. **judge 会换**；
4. **路由和缓存会换**；
5. **业务流量也会换**。

如果观测体系不跟着一起治理，它很快就会变成一套还在“看旧系统”的面板。到那时，团队以为自己在做数据驱动，其实只是在用过时口径判断新问题。

## 79. 最后一句实践判断

如果一个团队能够在一次线上退化发生后，**当天定位、当天回放、当天给出回滚或修复方向，并在下一轮训练或系统版本里吸收这次异常样本**，那它的可观测性体系就已经进入成熟区间。反过来，如果每次线上问题都要花很久才能确认影响范围、还原请求路径和找到可复现样本，那么无论面板有多漂亮，观测体系都还没有真正建立起来。

## 80. 极简验收表

1. **能否看到问题**；
2. **能否定位路径**；
3. **能否快速回放**；
4. **能否决定回滚或修复**；
5. **能否把样本回流到下一轮迭代**。

只要这五项里有两三项做不到，线上观测体系就还不能算成熟。

同时还应保证：**指标口径稳定、关键桶长期可追、异常样本可复盘、发布变更可回看**。少了任何一项，观测都会在高频迭代中逐渐失真。

更直接地说，真正可用的线上观测，不是让团队“知道系统很复杂”，而是让团队在复杂度持续上升时，仍然能把问题压回到**具体请求、具体路径、具体变更、具体动作**这四个层面。

当一套系统真的能把这四件事稳定打通时，线上优化、灰度发布、事故止血、训练回流和容量规划才会第一次共享同一套事实基础。

这也是线上观测最终最难、但最值得做成平台能力的一步。

没有这一步，平台就始终不算闭环。

也难以在组织内长期复用与审计。
