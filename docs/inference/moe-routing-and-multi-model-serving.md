# MoE 路由与多模型服务

随着模型越来越大，单一 dense 模型并不是唯一选择。Mixture of Experts（MoE）通过“参数很多，但每次只激活一部分”的方式扩大容量；而在线服务又常常不只跑一个模型，而是基础模型、推理模型、视觉模型、reranker、embedding、工具模型共同组成一个系统。MoE 路由与多模型服务的共同主题，是如何把“哪一个模型 / 哪一部分参数该为当前请求工作”这件事做对。

## 1. 为什么需要路由

路由的本质是资源分配。假设总参数容量为 $P$，若对每个请求都使用全部参数，则成本高；若能根据输入选择性激活子模块，就可能在相似成本下获得更大有效容量。

在 MoE 中，路由选择 expert；在多模型服务中，路由选择模型、版本或精度路径。两者都在做：

$$
r(x) \rightarrow \text{resource subset}.
$$

## 2. MoE 的基本形式

设输入 token 表示为 $h$，有 $E$ 个 experts，gate 网络输出分数 $g(h)$。Top-k 路由常写为：

$$
\alpha = \mathrm{TopK}(\mathrm{softmax}(W_g h)),
$$

输出为

$$
y = \sum_{e \in \mathcal{K}(h)} \alpha_e \cdot \mathrm{Expert}_e(h).
$$

其中 $\mathcal{K}(h)$ 是被选中的 experts 集合。这样总参数量可以很大，但单 token 的活跃计算较少。

## 3. 路由为何带来新问题

### 3.1 负载不均

若某些 experts 被频繁选中，就会热点拥塞，其余 experts 闲置。

### 3.2 专家塌缩

gate 可能学会过度依赖少数 experts，导致多样性下降。

### 3.3 通信复杂

专家分布在多 GPU 或多节点时，token 需跨设备交换。

### 3.4 调试难

某类请求性能突然变差，问题可能来自 gate、expert、调度或数据分布偏移。

## 4. 负载均衡目标

为避免专家塌缩，训练中常加入 load balancing loss，鼓励 token 分配更均匀。概念上可让实际 token 占比 $f_e$ 与 gate 概率均值 $p_e$ 接近均匀：

$$
\mathcal{L}_{\text{lb}}
=
E \sum_{e=1}^{E} f_e p_e.
$$

不同实现细节不同，但核心都是别让 experts 使用极端偏斜。

## 5. Capacity factor 与 token drop

每个 expert 在一个 batch 中有容量上限，常写作

$$
C = \left\lceil \text{capacity factor} \cdot \frac{kN}{E} \right\rceil,
$$

其中 $N$ 是 token 数，$k$ 是 top-k。若某 expert 超载，多出来的 token 会被丢弃、重路由或延迟处理。训练和服务中，这都会影响稳定性与吞吐。

## 6. 推理阶段的 MoE 与训练阶段不同

训练时关注收敛和均衡；推理时更关注时延和通信。尤其在 decode 阶段，每步 token 少而通信频繁，若 experts 分布不当，MoE 的理论效率可能被跨卡开销吃掉。

因此线上 MoE 部署常问两个问题：

1. experts 如何映射到设备拓扑？
2. 当前请求分布会不会造成热点 expert？

## 7. Expert parallel 与系统拓扑

MoE 常采用 expert parallel，把不同 experts 放在不同设备。其效率高度依赖互联。若 token routing 需要大量跨节点 all-to-all，延迟会迅速上升。很多系统宁愿牺牲部分参数效率，也尽量把常协同 experts 放在同一节点或高速互联域内。

## 8. 多模型服务与 MoE 的相似性

在线系统的多模型路由其实也类似：

1. 简单请求 -> 小模型
2. 难请求 -> 大模型
3. 检索任务 -> embedding + reranker
4. 屏幕理解 -> VLM
5. 长推理 -> reasoning model

**可以把它看成系统级 MoE**：不是 token 路由到 expert，而是请求路由到最合适的模型子系统。

## 9. 请求路由的目标函数

多模型路由往往同时考虑质量、延迟和成本，可抽象成：

$$
\min_{m \in \mathcal{M}}
\; \lambda_1 \cdot \text{Latency}(m,x)
+ \lambda_2 \cdot \text{Cost}(m,x)
- \lambda_3 \cdot \text{Quality}(m,x).
$$

这里的难点在于质量常无法事先精确知道，只能通过 proxy、历史统计或分类器估计。

## 10. 常见路由策略

### 10.1 规则路由

按输入长度、模态类型、用户等级、是否需要工具等规则分流。优点是简单可控，缺点是僵硬。

### 10.2 分类器路由

训练一个 lightweight router 判断任务类型、难度或风险，再选择模型。适合中大型系统。

### 10.3 多臂赌博机 / 在线学习

让系统边服务边学习不同路由策略的收益。更灵活，但线上风险更高。

### 10.4 级联路由

先用小模型尝试，置信度低或失败再升级到大模型。这是很多企业系统的主流设计。

## 11. 置信度驱动的升级

一个简单策略是先由便宜模型输出答案及置信度 $c$，若 $c < \tau$，则升级到更强模型：

$$
\text{model}(x)=
\begin{cases}
m_{\text{small}}, & c(x)\ge \tau \\
m_{\text{large}}, & c(x)< \tau
\end{cases}
$$

关键在于置信度必须校准，否则升级策略会失真。

## 12. 多模型系统里的缓存与共享前缀

若多个模型共享 tokenizer、embedding 或系统提示词，缓存策略会影响整体成本。尤其在级联路由中，若小模型失败后升级大模型，却无法复用前缀编码，延迟会变得更差。设计时应考虑：

1. 前缀是否可共享；
2. 检索结果是否跨模型复用；
3. 多模型间的中间表示是否可传递。

## 13. 失效模式

### 13.1 热点模型 / 热点 expert

流量偏到某一条路，导致局部排队严重。

### 13.2 错误分流

本应走大模型的难题被送到小模型，导致质量大幅下降。

### 13.3 级联震荡

很多请求先走小模型再升级，反而两头都排队，整体时延劣化。

### 13.4 版本耦合

更换一个 reranker 或路由阈值，就改变了整个多模型负载分布。

## 14. 例子：企业多模态助手

**一个企业助手同时服务**：

1. FAQ 文本问答
2. PDF 合同理解
3. 屏幕操作指导
4. 图表解析

**合理设计可能是**：

1. 普通 FAQ 走小语言模型；
2. 命中 PDF 或截图附件则走 VLM；
3. 高风险财务问题直接走大模型 + 检索增强；
4. 结构化抽取请求先走轻量 extractor，再决定是否升级。

这比“一切都走最强模型”更经济，也更容易扩展。

## 15. 例子：MoE 在线部署

假设一个 8-expert MoE 模型在线解码时某两类 expert 因代码和数学请求过多而长期热点。解决方式可能包括：

1. 调整 gate 温度和负载均衡；
2. 把热点 experts 复制到更多设备；
3. 对这类请求直接路由到专门模型；
4. 在系统层做流量整形。

这说明 MoE 优化不只是模型训练问题，还包含服务策略。

## 16. 设计建议

1. 对 MoE 训练同时监控 gate 分布、expert 负载和 token drop。
2. 多模型路由要同时看质量、成本和排队，不要只看单模型准确率。
3. 级联系统要验证升级路径的端到端时延，而不是各段单独时延。
4. 为高风险任务设置保守路由，宁可多花算力，也不要错误分流。
5. 把路由器视为可观测和可回滚的核心组件，而不是隐藏逻辑。

## 17. 小结

MoE 路由与多模型服务，本质上都在回答“面对当前请求，哪部分系统最值得被激活”。前者在模型内部做稀疏激活，后者在系统层面做资源分配。两者要成功，都必须处理同样的问题：负载均衡、置信度、延迟、成本和错误路由。理解这一点，有助于把模型设计与服务架构放在同一张资源优化地图上。 

## 18. MoE 在线服务和 dense 模型服务最大的区别是什么

很多人会把 MoE 看成“大模型但更省算”。在线服务里，真正的区别是：

1. dense 模型主要在层内做固定形态计算；
2. MoE 模型会在 token 级别产生动态路由；
3. 路由结果会直接改变跨卡通信和设备负载；
4. 同一批请求的成本不再只由长度决定，还由 expert 热度决定。

### 18.1 为什么这会让 tail latency 更难控

因为即使两条请求长度一样，它们也可能因为命中的 experts 不同而表现出不同延迟。  
一旦某类 experts 长期热点，就会出现：

1. 某些请求稳定很快；
2. 某些请求稳定很慢；
3. 平均值看似还行，但 P95/P99 明显恶化。

## 19. MoE 路由要同时优化三个层面

### 19.1 模型层

让 gate 不塌缩，确保 experts 真能学出分工。

### 19.2 系统层

让 experts 映射到合理设备拓扑，避免热点通信。

### 19.3 产品层

理解哪些请求类型会稳定命中高成本路径，并决定是否需要系统级分流。

如果只在模型层做 load balancing loss，但不看线上热点流量分布，最后仍然可能出现“训练均衡、部署拥塞”。

## 20. Expert placement 是线上成功的关键

即使 gate 很好，如果 experts 摆放不合理，all-to-all 成本也会把收益吃掉。常见设计考虑包括：

1. 高频协同 experts 是否放在同一节点；
2. 热点 experts 是否复制到多个设备；
3. decode 阶段是否允许不同 expert 共享部分前缀状态；
4. 不同租户是否共享同一 MoE 副本。

### 20.1 什么时候复制 experts 比重新训练更有效

如果线上热点集中在少数 expert，而这些热点并非由 gate 失衡导致，而是由真实流量分布决定，那么复制热点 experts 到更多设备，常常比在训练侧继续调 load balancing 更直接。

## 21. 多模型路由其实是系统级的稀疏激活

从资源视角看，多模型服务和 MoE 很像：

1. 都不是每次都激活全部资源；
2. 都依赖一个 router 决定用哪部分容量；
3. 都必须处理错误分流和热点问题；
4. 都需要在质量、延迟、成本之间做权衡。

因此可以把多模型服务理解成一种“粗粒度系统级 MoE”。

## 22. 路由器的输入设计

一个更成熟的路由器，不会只看“输入长度”一个特征，而会综合：

1. 任务类型；
2. 是否带附件；
3. 检索证据密度；
4. 历史失败率；
5. 用户等级；
6. 模型置信度；
7. 线上拥塞情况。

### 22.1 为什么拥塞状态也该进入路由

因为路由目标本身就是多目标优化。某个模型也许质量更高，但如果当前队列已经拥塞，而另一个模型在质量损失极小的前提下更空闲，那么动态路由就应该把系统状态也纳入决策。

## 23. 级联系统最怕“回头路”

**一个常见的多模型设计是**：

1. 小模型先试；
2. 不确定时升级到大模型；
3. 必要时再调用 VLM 或工具链。

问题在于，如果升级后很多计算无法复用，就会形成：

1. 先花一次小模型成本；
2. 再花一次大模型完整成本；
3. 还增加一次队列等待；
4. 最终对用户而言只是更慢。

### 23.1 复用什么最重要

1. 前缀编码；
2. 检索结果；
3. 已构造的结构化中间状态；
4. 对任务类型和风险的判断结果。

没有复用，级联就很容易从“省成本”变成“双倍算账”。

## 24. 置信度不校准，路由就会系统性出错

置信度驱动升级很常见，但也很脆弱。若小模型置信度过高，就会漏掉该升级的难题；若置信度过低，则会把大量简单请求送进大模型。

### 24.1 置信度校准的几个常见做法

1. 在不同任务桶上单独校准；
2. 对不同语言、模态和长度分桶校准；
3. 用人工评审和线上结果做二次校验；
4. 不把“回答长度”误当成“置信度提升”。

## 25. 多模型服务中的共享组件

许多系统并不是几个模型完全独立运行，而是共享：

1. tokenizer；
2. embedding 模型；
3. 检索索引；
4. reranker；
5. prompt 构造模板；
6. prefix cache。

### 25.1 共享组件的收益与风险

**收益**：

1. 成本更低；
2. 缓存可复用；
3. 运维更简单。

**风险**：

1. 一个共享组件升级会影响整个流量分布；
2. 某个索引或 reranker 退化会被误判成主模型退化；
3. 多模型共用前缀时更难排查路由问题。

## 26. MoE 与多模型系统的可观测性重点不同

### 26.1 MoE 系统要重点看

1. gate 分布；
2. expert token 占比；
3. token drop 或 overflow；
4. all-to-all 通信时长；
5. 热点 expert 队列。

### 26.2 多模型服务要重点看

1. 路由比例；
2. 升级率；
3. 模型命中质量和成本；
4. 升级后收益是否足够；
5. 某模型的局部拥塞是否向全系统传染。

## 27. 一个更现实的企业场景

企业助手往往并非只服务一种任务，而是：

1. FAQ；
2. 合同审阅；
3. 表格抽取；
4. 屏幕操作指导；
5. 工单总结；
6. 数据查询。

**更合理的路径往往是**：

1. FAQ 走便宜对话模型；
2. 合同和表格走文档/VLM 链路；
3. 高风险财税问题直接走强模型 + 检索；
4. 数据查询先走 schema-aware 轻量模型，再决定是否升级。

这类架构比“全部走最强模型”更接近现实可运维系统。

## 28. 设计建议

1. 把多模型路由看成系统级稀疏激活问题，而不是简单 if-else；
2. 路由输入既要看任务特征，也要看系统拥塞状态；
3. 对级联系统优先设计可复用的中间状态，而不是只盯路由阈值；
4. 对 MoE 系统单独建设 gate、expert 和通信三类监控；
5. 把热点 expert 和热点模型视为一等故障源。

## 29. 小结

MoE 和多模型服务看似一个偏模型内部、一个偏系统外部，但本质都在解决“有限资源如何稀疏地服务不同请求”。谁能同时处理好路由精度、负载均衡、可观测性和中间状态复用，谁就更可能把稀疏激活的理论优势兑现成真实服务收益。
