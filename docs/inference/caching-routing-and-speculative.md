# 缓存、路由与投机执行

大模型在线推理一旦进入生产环境，真正拖垮系统的往往不是“模型不够快”，而是：

- KV cache 爆显存
- 不同类型请求互相阻塞
- 大模型被低价值请求占满
- decode 太慢导致长答案体验差

**这一页讲三件相互关联的事**：

1. `缓存`
2. `路由`
3. `投机执行`

## 1. KV cache：推理系统里的第二套显存大户

对 Transformer 来说，每层都会为历史 token 保存 key 和 value。粗略估算：

\[
\text{KV bytes} \approx 2 \times L \times H \times d \times T \times b
\]

其中：

- \(L\) 是层数
- \(H\) 是头数
- \(d\) 是每头维度
- \(T\) 是上下文长度
- \(b\) 是每元素字节数

如果考虑 batch 中有 \(B\) 个并发请求，则：

\[
\text{KV bytes}_{\text{total}} \approx 2 \times B \times L \times H \times d \times T \times b
\]

这就是为什么长上下文一上来，显存压力不是线性“感觉有点大”，而是会非常快地堆满。

### 一个直观例子：32k 上下文企业问答

假设用户上传十几份合同和邮件，RAG 后得到 32k context。  
模型权重量化后也许已经能装下，但一旦同时跑十几个这类请求，KV cache 会迅速成为真正的瓶颈。

## 2. 缓存不是只有 KV cache

完整服务中，常见缓存至少包括：

- 前缀缓存 `prefix cache`
- 检索结果缓存
- 工具结果缓存
- 模板 prompt 缓存
- 多轮会话状态缓存

**可以把命中率写成**：

\[
\text{hit rate} = \frac{\text{cache hits}}{\text{total lookups}}
\]

对一个重复率高的企业内部系统，prefix cache 往往很有价值，因为大量请求共享：

- 固定系统提示词
- 固定工具 schema
- 固定安全模板

若这些前缀能复用，prefill 成本会明显下降。

## 3. Prefix cache 最适合什么场景

### 适合

- 客服系统
- 企业问答
- 固定代理框架
- 模板化工具调用

### 不适合

- 每次 prompt 变化都很大
- 用户上下文高度个性化

### 例子：合同审阅助手

系统 prompt、合规模板、输出 JSON schema 都基本固定。  
真正变化的只有用户传入的合同片段。  
这类任务如果不用 prefix cache，几乎是在反复为同一段前缀付费。

## 4. 缓存的另一个难点：生命周期与碎片

缓存不是“存进去就完了”，还要处理：

- 过期策略
- 回收策略
- 并发抢占
- 内存碎片

如果没有好的管理，系统会出现一种很典型的症状：

- 显存没完全占满
- 但可用块越来越碎
- 新请求进不来

这在长短请求混合场景里尤其明显。

## 5. 路由：不是所有请求都配得上最大模型

设请求特征为 \(\phi(x)\)，模型池为 \(\mathcal{M}\)，则路由器可做如下选择：

\[
m^\star = \arg\min_{m \in \mathcal{M}}
\left(
\lambda_1 \cdot \text{cost}(m)
+ \lambda_2 \cdot \text{latency}(m)
- \lambda_3 \cdot \text{quality}(m, x)
\right)
\]

这里的思想不是数学优化本身，而是非常现实的资源分配：

- 简单问题走小模型
- 复杂问题走大模型
- 高价值请求走高质量链路

## 6. 路由可以按哪些维度做

### 6.1 按请求长度

- 短问题走低延迟链路
- 长文档走大上下文链路

### 6.2 按任务类型

- 闲聊走轻量模型
- 代码生成走代码专用模型
- 表格和 PDF 走文档模型

### 6.3 按置信度

先让小模型试答，若低置信或触发规则，再升级到大模型。

### 6.4 按用户等级或 SLA

不同客户可以走不同质量与成本策略。

### 一个例子：企业问答网关

短问候路由到小模型，复杂表格分析路由到大模型；多轮合规问答再叠加更严格的工具校验链路。  
如果所有请求都直冲同一个大模型，系统的平均成本会非常难看。

## 7. 路由失败的典型情况

## 7.1 误把复杂问题送去小模型

**结果是**：

- 首次响应快
- 但答案错得离谱

后续还得人工重试，整体成本反而更高。

## 7.2 规则过多，维护失控

一开始只想按长度路由，后来不断加规则：

- 是否带附件
- 是否含表格
- 是否多轮
- 是否 VIP

最后路由器自己变成一个难维护系统。

## 7.3 没有在线反馈闭环

若只做离线路由评估，不根据线上成功率和成本回调策略，路由器会越来越脱离真实流量。

## 8. 投机执行：用小模型给大模型打草稿

`speculative decoding` 的核心思想是：

1. 草稿模型先一次提出多个候选 token
2. 大模型并行验证
3. 验证通过的部分直接接受

若一次草稿长度为 \(k\)，接受长度随机变量为 \(A\)，则平均每轮可前进 \(\mathbb{E}[A]\) 个 token。  
加速效果与接受率高度相关。

**可以粗略理解为**：

\[
\text{speedup} \uparrow \quad \text{when} \quad \mathbb{E}[A] \uparrow
\]

## 9. 为什么 speculative decoding 有用

标准 decode 是严格一步一个 token：

\[
y_t \rightarrow y_{t+1} \rightarrow y_{t+2} \rightarrow \cdots
\]

投机执行希望让大模型不必每次都从零想下一个 token，而是验证一个便宜模型提前写好的草稿。  
如果草稿常常对，昂贵模型就能跳步前进。

### 一个直观例子：会议纪要生成

对于模板化较强的输出，例如：

- “会议主题”
- “关键结论”
- “待办事项”

很多 token 序列其实很可预测。  
草稿模型若能先写出大部分骨架，大模型只需重点校验细节，速度就会提升。

## 10. 什么时候 speculative decoding 最划算

### 更适合

- 输出较长
- 模板性较强
- 草稿模型和目标模型风格接近

### 不太适合

- 输出非常短
- 高度开放、分支很多的创作型任务
- 草稿模型质量和目标模型差距太大

因为如果接受率太低，系统等于多跑了一个草稿模型，却没换回多少收益。

## 11. 路由和 speculative 可以联动

一个更现实的生产系统常会把两者结合：

1. 先路由到某个模型族
2. 在该模型族内部决定是否启用 speculative decoding

例如：

- 小模型链路直接普通 decode
- 大模型长答案链路才启用 speculative

这比“对所有请求都无脑 speculative”更合理。

## 12. 三个综合案例

## 12.1 企业知识库问答

**特点**：

- 高频复用系统 prompt
- 文档上下文长
- 结果多为中短回答

**策略**：

- 用 prefix cache 降 prefill
- 用路由区分普通问答和复杂合规问答

## 12.2 客服工单生成

**特点**：

- 输出结构较模板化
- 响应量大

**策略**：

- 用小模型处理常见请求
- 用大模型处理投诉升级和复杂工单
- 对长结构化输出考虑 speculative

## 12.3 Agent 工具调用

**特点**：

- 多轮状态多
- prompt 前缀重复度高
- 输出未必很长

**策略**：

- 更重视会话状态缓存和 prefix cache
- speculative 的收益反而未必是第一优先级

## 13. 常见失败模式

## 13.1 权重量化后以为问题解决了

结果显存还是被 KV cache 吃满。

## 13.2 路由只看离线准确率

忽略真实延迟和线上成功率。

## 13.3 speculative 理论可行，工程上不挣钱

接受率不足，额外复杂度和维护成本抵消了收益。

## 13.4 缓存命中率高，但用户体验没改善

因为真正慢的是长尾请求，缓存优化主要帮到了已经很快的那部分。

## 14. 总结

缓存、路由、投机执行其实是在回答同一个问题：  
如何让“昂贵的大模型推理”尽量少做、做得更晚、或者做得更快。

- 缓存解决的是重复工作
- 路由解决的是资源错配
- speculative 解决的是逐 token 解码过慢

这三者一起设计，推理系统才会从“能跑”走向“跑得起、跑得稳、跑得值”。

## 15. 缓存设计先分“缓存什么”

推理系统里“缓存”是个大词，实际至少包含几类完全不同的对象：

1. **prefix / prompt cache**；
2. **KV cache**；
3. **检索结果或工具结果缓存**；
4. **多模态预处理缓存**；
5. **route / policy 决策缓存**。

### 15.1 为什么必须先分清对象

因为不同缓存的：

1. **命中条件**；
2. **生命周期**；
3. **一致性要求**；
4. **收益指标**；
5. **回收策略**；

都完全不同。把它们混在一起讨论，只会得到含糊结论。

## 16. Prefix Cache 的真正难点是“规范化”

prefix cache 看起来像最直观的优化，但真正让它难用的，常常不是实现，而是规范化。

### 16.1 命中失败的常见原因

1. **prompt 模板有细小差异**；
2. **system prompt 或工具描述里带了动态字段**；
3. **多模态附件顺序不一致**；
4. **路由策略让相似请求去了不同模型池**。

### 16.2 更稳妥的做法

1. **对模板做 canonicalization**；
2. **把动态字段隔离到可变区**；
3. **记录 cache key 组成，便于分析命中率下降原因**；
4. **在路由设计时考虑前缀复用，而不是事后补救**。

## 17. KV Cache 是容量问题，也是调度问题

KV cache 常被看成显存管理问题，但它同时深度影响调度。

### 17.1 为什么

1. **长请求占用 KV 更久**；
2. **cache 生命周期决定 batch 可拼接性**；
3. **淘汰策略会影响尾部请求体验**；
4. **多租户和多模型会让 cache 碎片化更严重**。

### 17.2 实际设计中要同时看

1. **显存占用率**；
2. **碎片率**；
3. **淘汰后带来的重算成本**；
4. **不同租户和长度桶的公平性**。

## 18. 路由策略的收益不能只看模型质量

路由的价值不只是把复杂请求送给更强模型，还包括：

1. **把高成本请求分流到更合适池子**；
2. **让缓存更容易复用**；
3. **让不同 SLO 的任务走不同路径**；
4. **在高峰期优先保护高价值流量**。

### 18.1 一个典型误区

所有高风险请求都送最大模型，结果导致：

1. **高价值短请求也开始排队**；
2. **大模型池被极长请求拖住**；
3. **整体 tail latency 恶化**。

更成熟的路由应该同时看质量、时延、成本和缓存机会。

## 19. 路由器本身也需要被评测和监控

很多团队把路由器当作一段轻逻辑，缺少独立监控。这很危险。

### 19.1 路由器至少要监控

1. **各模型池流量占比**；
2. **不同任务类型的路由分布**；
3. **route 后的质量与成本表现**；
4. **高峰期路由漂移**；
5. **是否出现 route thrashing**，即同类请求在不同版本间来回跳。

### 19.2 为什么 route thrashing 会伤系统

1. **缓存命中下降**；
2. **容量规划失真**；
3. **用户体验不稳定**；
4. **很难解释哪条路径真正有效**。

## 20. Speculative Decoding 最该关注“接受率分布”

投机执行最常见的 KPI 是 acceptance rate，但真正有用的不是一个平均值，而是：

1. **不同任务桶的接受率**；
2. **不同输出长度桶的接受率**；
3. **高峰期和低峰期是否变化**；
4. **接受率与质量退化是否有关联**。

### 20.1 为什么平均接受率会误导

因为投机执行可能在短、简单请求上非常有效，却在长、复杂或高风险请求上几乎无收益。如果不分桶，你会误判它的真实价值。

## 21. Draft Model 选型不应只看便宜

投机执行里，draft model 的角色常被理解成“越便宜越好”，其实更准确的目标是**单位验证成本下的高接受率**。

### 21.1 需要平衡

1. **draft 推理成本**；
2. **接受率**；
3. **对主模型质量边界的偏差**；
4. **对不同任务桶的泛化**。

### 21.2 一个常见错误

选了一个很便宜但行为风格与主模型差异很大的 draft，结果 acceptance 很低，最终既没省钱也没降时延。

## 22. 缓存、路由、speculative 应共同设计

这三者真正高价值的地方，不在于单点最优，而在于组合关系。

### 22.1 一个典型协同

1. **路由让相似请求进入更同质的池子**；
2. **同质池子提高 prefix cache 和 speculative 的收益**；
3. **更高缓存命中又让 KV 压力下降**；
4. **最终尾部体验和单位成本一起改善**。

### 22.2 一个典型冲突

1. **路由频繁抖动**；
2. **缓存命中下降**；
3. **draft model 也难以形成稳定接受率**；
4. **系统表面上开了很多优化，端到端收益却很有限**。

## 23. 一份更工程化的缓存/路由/speculative 清单

1. **缓存对象是否已明确分类**；
2. **prefix cache 是否做了模板规范化**；
3. **KV cache 是否同时纳入容量与调度治理**；
4. **路由器是否有独立观测与评测**；
5. **acceptance rate 是否按任务桶和长度桶分析**；
6. **draft model 是否按单位验证成本选择**；
7. **三者是否作为联动系统而非孤立技巧被设计**。

## 24. 小结

缓存、路由与投机执行的真正价值，不是把几个流行优化名词拼在一起，而是把**重复工作消除、资源错配减少、逐 token 生成加速**放进同一套系统设计里。只有把缓存规范化、路由可观测化、speculative 分桶化，这条链路才会真正从局部技巧升级成稳定的生产收益。

## 快速代码示例

```python
def acceptance_rate(draft_tokens, target_tokens):
    n = min(len(draft_tokens), len(target_tokens))
    if n == 0:
        return 0.0
    hit = sum(int(draft_tokens[i] == target_tokens[i]) for i in range(n))
    return hit / n

def choose_path(prefix_hit, complexity):
    if prefix_hit > 0.8:
        return "cached_route"
    if complexity < 0.3:
        return "small_model_or_speculative"
    return "full_model"
```

这段代码展示了缓存/路由/speculative 联动的最小判断逻辑：先估 prefix 命中，再按任务复杂度选路径，并用接受率指标评估 speculative 是否赚钱。线上通常还要按长度桶和租户桶分开统计，避免平均值掩盖退化。


## 补充：用服务系统视角重写 **缓存、路由与投机执行**

推理主题最容易被写成一串优化名词，但线上系统真正关心的从来不是“有没有这个技巧”，而是 **它是否改变了请求、队列、缓存、成本和高价值流量的命运**。围绕 **缓存、路由与投机执行**，更稳的理解框架是先问：这件事究竟在改变哪一层约束。

1. **prefix 规范化、KV 生命周期、route 一致性、speculative 接受率、fallback**。

只要这几层没有被明确分开，系统就很容易出现一种假象：离线 benchmark 更好了，线上却并没有更稳；或者平均时延下降了，但高价值租户与长尾任务反而更差。对推理系统来说，这类错觉比“完全没有优化”更危险，因为它会把团队引向错误的资源分配方向。

## 补充：更扎实的设计顺序

针对 **缓存、路由与投机执行**，建议把设计顺序固定成下面几步：

1. **先画请求生命周期图**，明确哪些阶段会受这项机制影响；
2. **再画资源账本**，把 GPU、显存、KV、网络、工具链和 CPU 依赖拆开；
3. **然后才讨论策略本身**，例如要不要路由、要不要 cache、要不要 speculative、要不要分层服务；
4. **为高价值流量单独建桶**，不要只看总体平均；
5. **最后定义回退与降级**，确保高峰期和故障期也有可执行动作。

这个顺序的价值在于：它迫使团队先把 **缓存、路由与投机执行** 放进完整服务图里，再讨论局部技巧，否则很容易陷入“看起来懂很多优化，但不知道哪个最值钱”的状态。

## 补充：真实线上最常见的失败模式

围绕 **缓存、路由与投机执行**，线上失败常常并不体现在单次崩溃，而是慢性失真：

1. **缓存命中高但高价值流量仍差**。
2. **route 抖动破坏缓存**。
3. **只看平均 acceptance**。

这些问题之所以难，是因为它们往往混在一起发生。一个路由变化可能同时影响缓存命中、TTFT、单位成本和用户重试率；一个 cache 策略调整可能同时影响长上下文任务、KV 生命周期和多租户公平性。因此，**缓存、路由与投机执行** 相关改动必须天然携带可观测性，否则等于把复杂度埋进黑箱。

## 补充：更像生产系统的验收标准

对 **缓存、路由与投机执行**，更靠谱的验收通常不是一句“线上变快了”，而是至少同时回答：

1. **对命中率分桶**。
2. **route 和 cache 联合观测**。
3. **draft model 按任务桶选型**。
4. **保留降级路径**。

再进一步，建议把这类主题纳入持续治理：

1. **按任务桶和租户桶做长期趋势跟踪**；
2. **保留代表性 replay / shadow case**，用于版本切换对照；
3. **把高价值失败样本回流到训练和系统修复**；
4. **让 oncall 面板能直接看到与它相关的 route、cache、queue 和 fallback 状态**。

只要这套闭环建立起来，**缓存、路由与投机执行** 才真正从“优化点”变成“系统能力”。
