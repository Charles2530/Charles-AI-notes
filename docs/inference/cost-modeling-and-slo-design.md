# 成本建模与 SLO 设计

推理系统上线之后，技术讨论很快会落到两个现实问题上：一是“这套服务到底贵不贵”，二是“多快才算足够快”。前者对应成本建模，后者对应 SLO（Service Level Objective）设计。没有成本模型，团队很难判断量化、缓存、路由、蒸馏是否真带来净收益；没有 SLO，优化会陷入无限追求更低时延或更高吞吐，却不清楚哪些改动值得做。

## 1. 什么是 SLO

SLO 不是一句“越快越好”，而是对服务表现的明确目标，比如：

1. P95 首 token 延迟低于 2 秒；
2. P99 请求成功率高于 99.5%；
3. 普通问答平均成本低于某阈值；
4. 屏幕代理任务成功率高于某水平且平均步数不超过某值。

SLO 的价值，在于把优化目标变成可决策的约束条件，而不是模糊感受。

## 2. 推理成本由哪些部分组成

粗略可把单位时间成本表示为

$$
C_{\text{total}}
=
C_{\text{compute}}
+ C_{\text{memory}}
+ C_{\text{storage}}
+ C_{\text{network}}
+ C_{\text{ops}}.
$$

若换成单请求或单 token 视角，还可进一步分摊固定成本和可变成本。

## 3. 单位 token 成本

假设集群每小时成本为 $C_h$，单位时间吞吐为 $\Theta$ token/s，则单位 token 成本近似为

$$
\text{Cost/token}
=
\frac{C_h}{3600\cdot \Theta}.
$$

但这只是平均数。真实系统中，不同请求长度、模型路由和缓存命中率都会让成本有很大波动。

## 4. 首 token 与后续 token 的不同经济学

Prefill 与 decode 资源画像不同，因此：

$$
C_{\text{req}}
\approx
C_{\text{prefill}}(L_{\text{in}})
+ C_{\text{decode}}(L_{\text{out}})
+ C_{\text{overhead}}.
$$

长上下文系统的主要成本往往在 prefill；实时 agent 系统可能 decode 轮次更多。若你用一个统一“每 token 成本”去概括，容易误导优化方向。

## 5. SLO 必须与业务场景绑定

### 5.1 企业问答

用户可接受 2 到 5 秒内得到有依据的答案，正确率和可引用证据往往比极致速度更重要。

### 5.2 屏幕 agent

首 token 和每步反应都要快，否则用户感觉系统“卡”。这里的 SLO 更像交互控制延迟。

### 5.3 批处理文档分析

单请求延迟没那么关键，吞吐和成本更重要。

因此 SLO 不能抄别人的模板，必须围绕业务路径设计。

## 6. 队列论视角

设请求到达率为 $\lambda$，系统服务率为 $\mu$，则当负载 $\rho = \lambda/\mu$ 接近 1 时，排队延迟会快速恶化。哪怕模型本身推理速度不慢，只要容量规划太紧，尾延迟也会出问题。

这意味着 SLO 设计不能只盯模型推理时间，还要预留足够 headroom，尤其对峰值流量和多租户系统。

## 7. 尾延迟为何关键

用户通常不会因为 P50 很漂亮而满意，如果 P95/P99 经常超时。对 agent 与多步工作流而言，尾延迟还会层层叠加。若单步 P95 较差，整条任务链路的整体时延可能明显恶化。

**因此常见 SLO 应至少覆盖**：

1. P50
2. P95
3. P99
4. 错误率
5. 成功率

## 8. 成本与质量的三角关系

**很多优化都在三角关系中权衡**：

1. 更大模型 -> 质量更好，但更贵、更慢；
2. 更 aggressive batching -> 吞吐更高，但 TTFT 变差；
3. 更低比特量化 -> 更便宜，但可能损失质量；
4. 更强检索 -> 正确率提高，但链路更长。

**可以把决策问题写成**：

$$
\max \; \text{Utility}
=
\alpha Q - \beta C - \gamma L,
$$

其中 $Q$ 是质量，$C$ 是成本，$L$ 是时延。不同场景下 $\alpha,\beta,\gamma$ 完全不同。

## 9. 成本建模需要真实请求分布

不要只用平均输入长度。应统计：

1. 输入长度分布
2. 输出长度分布
3. 模型路由分布
4. 缓存命中率
5. 长上下文占比
6. 工具调用轮次

同一个系统，若 10% 的长上下文请求占据了 60% 的算力，那么优化重点就不该平均铺开。

## 10. 分层 SLO

**成熟系统常有多层 SLO**：

1. **平台层**：可用性、错误率、资源使用率；
2. **模型层**：TTFT、TPOT、上下文上限、质量代理指标；
3. **产品层**：任务完成率、人工接管率、用户满意度；
4. **业务层**：转化率、工单关闭时间、复购率等。

若只设平台 SLO，你可能会把一个“快速但胡说”的系统误判为健康。

## 11. 优先级与降级设计

当资源紧张时，系统不应只有“一起变慢”。更好的做法是按优先级降级：

1. 免费用户切换到小模型；
2. 长上下文请求限制最大长度；
3. 低优先级批处理延迟执行；
4. 关闭昂贵辅助步骤，如二次重排或多样本采样。

降级策略应在成本模型中预先设计，而不是等高峰期临时拍脑袋。

## 12. 例子：企业知识助手

如果知识助手 80% 请求是短问答、15% 是 PDF 文档问答、5% 是长合同比对，那么一条有效策略可能是：

1. 短问答走小模型 + 检索；
2. PDF 走中模型 + VLM；
3. 长合同比对走大模型并允许更高时延。

此时不同请求应有不同 SLO，而不是统一要求“全部 2 秒内返回”。

## 13. 例子：机器人远程操作协助

机器人协助系统可能需要低延迟视觉判断和偶尔的高精规划。低延迟部分 SLO 可能是 300ms 内给出动作建议，而复杂规划允许 3 到 5 秒。把两者混到同一服务目标里，会导致资源配置不合理。

## 14. 常见误区

### 14.1 只看平均成本

忽略热点流量与长上下文，最后总是高估系统承载能力。

### 14.2 只看平均延迟

尾延迟问题上线后会直接被用户感知。

### 14.3 SLO 太理想化

若目标完全脱离硬件预算与业务现实，最终只会逼出大量临时特判。

### 14.4 不把质量纳入 SLO 讨论

一个很快但错误率高的系统，不能算达标。

## 15. 设计建议

1. 用真实请求分布做成本建模，而不是平均值。
2. 至少拆分 prefill 和 decode 成本。
3. 为不同任务路径设计分层 SLO。
4. 同时监控质量、延迟和成本，避免单目标最优化。
5. 提前设计降级策略，并在灰度中验证。

## 16. 小结

成本建模与 SLO 设计的核心，是把推理系统从“技术能跑”转成“业务可运营”。它要求团队明确什么性能最重要、什么成本可以接受、什么质量不能退让，并据此做路由、缓存、量化和容量规划。没有这套框架，再多优化都可能只是局部漂亮、全局亏损。 

## 17. 成本建模先从单位经济学开始

如果一条推理链路连最基本的单位成本都算不清，就很难做出正确的系统决策。

### 17.1 建议至少算清

1. **每请求 GPU 秒数**；
2. **每输入 token 成本**；
3. **每输出 token 成本**；
4. **每种任务路径的总成本**；
5. **工具调用、OCR、检索、网络等外部成本**。

### 17.2 为什么单位经济学重要

它能帮助你判断：

1. **哪个任务最烧钱**；
2. **哪个优化是真正省钱而不只是看起来快**；
3. **哪些请求其实应该异步或降级**。

## 18. Prefill 和 Decode 应该拆成两张账本

很多系统把推理成本做成一个总体均值，这会掩盖关键差异。

### 18.1 Prefill 账本要看

1. **长上下文输入占比**；
2. **prefix cache 命中率**；
3. **多模态附件解析成本**；
4. **TTFT 的预算与峰值压力**。

### 18.2 Decode 账本要看

1. **平均输出长度**；
2. **TPOT 与尾部 token 速度**；
3. **KV cache 占用与显存寿命**；
4. **小 batch / 长尾请求对整体吞吐的影响**。

### 18.3 为什么要拆开

因为不同任务类型对 prefill 与 decode 的依赖差异极大。若不拆开，就很容易把错误的优化放到错误阶段。

## 19. SLO 需要分层，不应只有一个总时延

成熟系统会同时维护多层 SLO：

1. **请求层 SLO**；
2. **阶段层 SLO**；
3. **业务层 SLO**；
4. **成本层 SLO**。

### 19.1 请求层

1. **P50/P95/P99 latency**；
2. **TTFT**；
3. **错误率与拒绝率**。

### 19.2 阶段层

1. **prefill 预算**；
2. **decode 预算**；
3. **tool / OCR / retrieval 预算**；
4. **queue 预算**。

### 19.3 业务层

1. **任务完成率**；
2. **人工接管率**；
3. **关键用户满意度 proxy**。

### 19.4 成本层

1. **单位请求成本**；
2. **高峰期成本上限**；
3. **高价值请求的保底成本配额**。

## 20. 质量、延迟、成本是一个三角

许多系统优化争论之所以反复发生，是因为三方在优化不同目标：

1. **模型团队想保质量**；
2. **平台团队想压时延**；
3. **业务或运维团队想控成本**。

### 20.1 更成熟的做法

把每个优化方案都画成三角中的一条移动：

1. **量化节省了多少成本**；
2. **牺牲了多少质量**；
3. **对 TTFT / TPOT 是改善还是伤害**。

### 20.2 为什么这能减少内耗

因为争论从“我觉得这个方案更好”转成“它沿哪条维度移动、值不值得”。

## 21. 路由策略必须进入成本模型

一条路由规则不仅决定质量，也决定容量和成本。

### 21.1 路由会影响

1. **不同模型池的拥塞程度**；
2. **prefix / cache 的复用可能性**；
3. **高成本请求占比**；
4. **长上下文请求是否集中压垮某个池**。

### 21.2 为什么成本模型里不能忽略路由

因为同样一组请求，若用不同路由策略分配到不同模型、量化版本或异步路径上，单位成本可能差一个明显比例。

## 22. 价格与配额策略会反向塑造流量

系统不是静态的。产品定价、套餐配额和接口限制会反向改变请求结构。

### 22.1 典型影响

1. **提高免费额度会引入更多低价值长尾流量**；
2. **鼓励上传大附件会显著抬高 prefill 与 OCR 成本**；
3. **放宽输出长度上限会直接推高 decode 成本**；
4. **企业客户接入后请求长度和稳定性要求会改变**。

### 22.2 为什么 SLO 设计必须和产品协同

否则系统团队可能在错误的流量假设上做了很多“优化”，一旦业务策略变化就全部失效。

## 23. 降级策略本身也要有成本模型

很多系统设计了 fallback，却没算 fallback 的真实代价。

### 23.1 应计算什么

1. **降级到小模型后节省多少 GPU 成本**；
2. **质量损失是否会引发更多重试或人工接管**；
3. **异步化是否会增加运营复杂度和用户等待成本**；
4. **关闭某些工具后是否会把问题转移到下游流程**。

### 23.2 为什么这很关键

因为某些看似省钱的 fallback，可能最终通过更多重试、更多人工介入把钱又花回去。

## 24. 成本异常也需要告警

团队常给时延和错误率设告警，却不给成本设告警。更成熟的系统应监控：

1. **单位请求成本突然上升**；
2. **某类任务成本异常漂移**；
3. **prefix 命中下降导致 prefill 成本抬升**；
4. **路由变化导致高成本模型占比上升**；
5. **附件大小或输出长度分布异常**。

### 24.1 为什么这很值钱

因为很多成本事故并不会立刻体现为故障，但会在账单和容量压力上长期伤害系统。

## 25. 一份更工程化的 SLO / 成本设计清单

1. **是否建立了请求、阶段、业务、成本四层 SLO**；
2. **是否拆分 prefill 与 decode 账本**；
3. **是否把路由和 fallback 纳入成本模型**；
4. **是否与产品策略联动更新流量假设**；
5. **是否对成本异常建立告警与回滚门槛**。

## 26. 小结

成本建模与 SLO 设计的真正目标，不是做一份静态预算表，而是建立一套**随流量、路由、产品策略和模型版本共同演化的运营控制系统**。只有把成本、质量和时延放在同一套决策框架下，推理平台的优化才不会顾此失彼。


## 补充：用服务系统视角重写 **成本建模与 SLO 设计**

推理主题最容易被写成一串优化名词，但线上系统真正关心的从来不是“有没有这个技巧”，而是 **它是否改变了请求、队列、缓存、成本和高价值流量的命运**。围绕 **成本建模与 SLO 设计**，更稳的理解框架是先问：这件事究竟在改变哪一层约束。

1. **请求价值、prefill/decode 账本、工具成本、峰值 headroom、租户等级**。

只要这几层没有被明确分开，系统就很容易出现一种假象：离线 benchmark 更好了，线上却并没有更稳；或者平均时延下降了，但高价值租户与长尾任务反而更差。对推理系统来说，这类错觉比“完全没有优化”更危险，因为它会把团队引向错误的资源分配方向。

## 补充：更扎实的设计顺序

针对 **成本建模与 SLO 设计**，建议把设计顺序固定成下面几步：

1. **先画请求生命周期图**，明确哪些阶段会受这项机制影响；
2. **再画资源账本**，把 GPU、显存、KV、网络、工具链和 CPU 依赖拆开；
3. **然后才讨论策略本身**，例如要不要路由、要不要 cache、要不要 speculative、要不要分层服务；
4. **为高价值流量单独建桶**，不要只看总体平均；
5. **最后定义回退与降级**，确保高峰期和故障期也有可执行动作。

这个顺序的价值在于：它迫使团队先把 **成本建模与 SLO 设计** 放进完整服务图里，再讨论局部技巧，否则很容易陷入“看起来懂很多优化，但不知道哪个最值钱”的状态。

## 补充：真实线上最常见的失败模式

围绕 **成本建模与 SLO 设计**，线上失败常常并不体现在单次崩溃，而是慢性失真：

1. **只看平均成本**。
2. **只看总时延不看阶段预算**。
3. **忽略高价值流量保护**。

这些问题之所以难，是因为它们往往混在一起发生。一个路由变化可能同时影响缓存命中、TTFT、单位成本和用户重试率；一个 cache 策略调整可能同时影响长上下文任务、KV 生命周期和多租户公平性。因此，**成本建模与 SLO 设计** 相关改动必须天然携带可观测性，否则等于把复杂度埋进黑箱。

## 补充：更像生产系统的验收标准

对 **成本建模与 SLO 设计**，更靠谱的验收通常不是一句“线上变快了”，而是至少同时回答：

1. **拆分账本**。
2. **把路由纳入成本模型**。
3. **为高价值流量保底**。
4. **把成本异常接入告警**。

再进一步，建议把这类主题纳入持续治理：

1. **按任务桶和租户桶做长期趋势跟踪**；
2. **保留代表性 replay / shadow case**，用于版本切换对照；
3. **把高价值失败样本回流到训练和系统修复**；
4. **让 oncall 面板能直接看到与它相关的 route、cache、queue 和 fallback 状态**。

只要这套闭环建立起来，**成本建模与 SLO 设计** 才真正从“优化点”变成“系统能力”。
