# 推理服务系统

训练决定了模型“会不会”，推理系统决定了模型“能不能以可接受的成本稳定地服务用户”。  
这一页重点不是模型原理，而是从系统角度解释：为什么同一个模型，在不同服务栈上，成本和延迟可能差一个数量级。

## 1. 端到端时延分解

对自回归模型，一次请求总时延可粗略写成：

\[
T_{\text{req}} = T_{\text{queue}} + T_{\text{prefill}} + T_{\text{decode}} + T_{\text{post}}
\]

进一步地，若输出长度为 \(n\)，则 decode 阶段可近似展开为：

\[
T_{\text{decode}} \approx \sum_{i=1}^{n} T_{\text{step}}^{(i)}
\]

其中：

- \(T_{\text{queue}}\)：排队等待 GPU 资源
- \(T_{\text{prefill}}\)：把输入上下文一次性编码进 KV cache
- \(T_{\text{decode}}\)：逐 token 生成
- \(T_{\text{post}}\)：反序列化、过滤、工具编排等后处理

### 一个直观例子：长文档问答

用户只问一句“总结这份 80 页 PDF 的关键风险”。  
输出可能只有 200 token，但输入上下文很长。  
此时最贵的往往不是 decode，而是 prefill，因为模型要先把整份文档过一遍并写入 KV cache。

## 2. Prefill 和 Decode 的计算特征完全不同

### Prefill

- 并行度高
- 计算密集
- 对矩阵乘加吞吐更敏感

### Decode

- 一次只生成少量 token
- memory-bound 更明显
- 对 KV cache 读写和调度更敏感

**这导致一个很重要的工程事实**：  
**同一套优化不一定同时优化 prefill 和 decode。**

例如：

- 大 batch prefill 很划算
- 但过大 batch 的 decode 可能拉高首 token 延迟

## 3. 吞吐与 QPS 的基础公式

若单机 token 生成能力为 \(R_{\text{tok}}\) token/s，平均每个请求输出 \(n\) 个 token，则理想吞吐上界近似为：

\[
\text{QPS}_{\max} \lesssim \frac{R_{\text{tok}}}{n}
\]

若把输入长度 \(m\) 也计入，可以写成更现实的资源消耗近似：

\[
C_{\text{req}} \approx \alpha m + \beta n
\]

其中：

- \(\alpha\) 表示 prefill 单 token 成本
- \(\beta\) 表示 decode 单 token 成本

在长上下文场景，\(\alpha m\) 往往主导；在长回答场景，\(\beta n\) 更关键。

## 4. 服务系统最常见的瓶颈不是算力，而是调度

很多团队一开始以为“换更大的 GPU”就能解决问题，结果发现真正拖垮系统的是：

- 请求长度差异过大
- 长请求阻塞短请求
- decode 小步运行导致 GPU 气泡
- KV cache 分配和回收碎片化

### 例子：企业知识库问答

**白天高峰期会混入三类请求**：

- **短闲聊**：输入 50 token，输出 30 token
- **普通问答**：输入 2k token，输出 200 token
- **文档总结**：输入 40k token，输出 500 token

若全部混在一个队列里，长请求会把短请求的 p95 延迟拖得很高。  
这时问题不再是“模型慢”，而是“调度策略错误”。

## 5. Continuous Batching 为什么有效

**传统静态 batch 的问题是**：要等整批请求都准备好再一起跑。  
而连续批处理允许在 decode 过程中动态插入新请求。粗略理解为，每个解码步在处理一个随时间变化的活动集合 \(\mathcal{B}_t\)：

\[
\mathcal{B}_{t+1} = (\mathcal{B}_t \setminus \mathcal{F}_t)\cup \mathcal{A}_t
\]

其中：

- \(\mathcal{F}_t\) 是在时刻 \(t\) 完成的请求
- \(\mathcal{A}_t\) 是在时刻 \(t\) 新加入的请求

**这样做的好处**：

- GPU 更少空转
- 请求能更快进入执行
- 对短请求更友好

### 例子：文档问答服务

若不做 continuous batching，GPU 常会在一批请求 decode 完之前闲等下一批。  
做了之后，新请求可在旧请求之间穿插进入，整体利用率会明显上升。

## 6. 并行策略：TP、PP、DP 各解决什么问题

## 6.1 Tensor Parallel

**把大矩阵切到多卡上**：

\[
Y = XW = \sum_{k=1}^{K} XW_k
\]

**适合**：

- 单模型太大，一张卡放不下
- 希望并行做单层计算

**代价**：

- 每层都要通信
- 小 batch decode 时通信开销可能很显著

## 6.2 Pipeline Parallel

把模型按层切成多段，形成流水线。  
适合超大模型，但会引入 pipeline bubble。若流水线分为 \(S\) 段、微批数为 \(M\)，一个经典近似是：

\[
\text{bubble fraction} \approx \frac{S-1}{M+S-1}
\]

微批越少，bubble 越严重。  
这也是为什么 pipeline 在低并发在线服务里不总是理想。

## 6.3 Data Parallel

**多副本各自处理请求**：

\[
\text{capacity} \propto N_{\text{replica}}
\]

它最容易扩展服务吞吐，但前提是单卡或单节点已经能装下模型。

## 7. KV Cache 才是大模型在线服务的“第二套权重”

在长上下文场景里，KV cache 可能比你想象中更贵。  
**粗略估算**：

\[
\text{KV bytes} \approx 2 \cdot L \cdot T \cdot H \cdot d \cdot b
\]

其中：

- \(L\) 是层数
- \(T\) 是上下文长度
- \(H\) 是头数
- \(d\) 是每头维度
- \(b\) 是每个元素字节数

前面的 2 表示 key 和 value 各一份。

### 一个例子：32k 上下文聊天

模型权重本身可能已经量化到能装进单机，但一旦几十个用户同时跑 32k context，KV cache 会迅速吃满显存。  
这就是为什么很多服务系统表面上看是“推理”，实际上首先是在管理缓存。

## 8. 为什么请求分层和路由很重要

设请求特征为 \(\phi(x)\)，模型池为 \(\mathcal{M}\)。  
**路由器可以做如下选择**：

\[
m^\star = \arg\min_{m\in\mathcal{M}} \lambda_1 \cdot \text{cost}(m) + \lambda_2 \cdot \text{latency}(m) - \lambda_3 \cdot \text{quality}(m, x)
\]

**这意味着**：

- 简单问答走小模型
- 长文档总结走大上下文模型
- 高价值客户请求可以走高质量路由

### 例子：客服网关

“你好，在吗”这种请求没必要占用 70B 模型。  
但“根据这 3 份合同和 2 份邮件，判断违约责任归属”显然不能走同一条链路。

## 9. 首 token 延迟和长尾延迟要分开看

很多系统只盯平均吞吐，忽视了用户真正感受到的是：

- **TTFT**：time to first token
- **TPOT**：time per output token
- P95 / P99 tail latency

如果首 token 很慢，用户会觉得系统“卡住了”；  
如果长尾很高，用户会随机遇到糟糕体验。

## 10. 四个真实部署场景

## 10.1 对话助手

**重点**：

- TTFT 低
- 中短回答稳定
- 高并发排队可控

**策略**：

- 优先 continuous batching
- 控制 prompt 长度
- 对超长上下文做异步或分层处理

## 10.2 RAG 文档问答

**重点**：

- prefill 非常贵
- 检索质量直接影响推理成本

**策略**：

- 先压缩召回片段
- 做 chunk rerank
- 必要时把超长任务拆成多轮总结

## 10.3 Agent 工具调用

**重点**：

- 请求数不大，但单请求链路长
- 模型推理和工具等待交替出现

**策略**：

- 不只是优化模型本身
- 更要优化工具并发、缓存和中间状态复用

## 10.4 多模态服务

**重点**：

- 图像编码开销额外存在
- 输入 token 和视觉 token 混合

**策略**：

- 区分视觉前端和语言后端的资源池
- 对高分辨率图像做分级处理

## 11. 常见失败模式

## 11.1 Head-of-line blocking

一个超长请求堵在前面，后面一串短请求全被拖慢。  
**解决思路通常是**：

- 分队列
- 限流
- 最大上下文长度隔离

## 11.2 显存没爆，但性能突然崩

这常常是 KV cache 碎片、调度抖动、通信瓶颈或 CPU/GPU 数据搬运出了问题。

## 11.3 GPU 利用率很高，但用户仍然觉得慢

因为高 GPU utilization 不代表低用户时延。  
系统可能只是把很多请求堆在一起“忙碌地慢”，而不是“高效地快”。

## 11.4 指标只看平均值

平均时延漂亮，但 P99 巨差，会在生产里持续伤害体验。  
服务系统必须默认关注 tail latency。

## 12. 工程上的最小闭环

一个像样的推理服务系统，至少应该持续观测这些指标：

- 请求数、拒绝率、排队时长
- TTFT、TPOT、P50/P95/P99
- GPU 利用率、显存占用、KV cache 命中与回收
- 各长度区间请求分布
- 错误类型分布

没有这层观测，所有“优化”都很容易变成拍脑袋。

## 13. 总结

推理服务系统的核心，不是让模型跑起来，而是让它在真实流量下表现稳定。  
真正决定体验的，往往不是某个 kernel 再快 8%，而是：

- 是否区分 prefill 和 decode 两种负载
- 是否用好了 continuous batching
- 是否把 KV cache 当成一级资源去管理
- 是否让不同类型请求走了不同服务路径

如果把训练看作“造一台引擎”，那推理系统就是整辆车的变速箱、冷却系统和道路调度。  
引擎再强，没有服务系统，最终也只会在实验室里跑得漂亮。

## 14. 服务系统首先是一个排队系统

在线服务经常被误解成“GPU 跑模型”，但更准确的抽象应是：

\[
\text{arrival} \rightarrow \text{queue} \rightarrow \text{scheduler} \rightarrow \text{prefill/decode workers} \rightarrow \text{post-processing}.
\]

也就是说，服务系统不是单次推理，而是一个持续接收、调度、合并、拆分、回收请求的排队系统。

### 14.1 为什么排队论直觉很重要

很多看似“模型太慢”的问题，实际上是：

1. 瞬时流量波峰导致队列拥塞；
2. 长请求把短请求拖住；
3. 某类用户触发了更昂贵的路由路径；
4. cache 碎片化导致同样流量下有效容量下降。

如果不先把系统理解成排队系统，优化就会过度聚焦单请求 benchmark。

## 15. 服务系统的资源不只有 GPU 算力

实际部署时至少要同时管理四类资源：

1. GPU compute；
2. GPU 显存和 KV cache；
3. CPU 预处理与网络传输；
4. 外部系统配额，如检索、数据库、浏览器、工具 API。

### 15.1 为什么外部工具常成为隐形瓶颈

在 agent 或 RAG 场景，模型本身也许只占关键路径的一部分。若：

1. 检索慢；
2. OCR 慢；
3. 浏览器工具响应不稳定；
4. 排序器部署在另一套集群；

那么就算你把主模型 kernel 再优化一轮，端到端收益也可能有限。

这类瓶颈最难发现的地方在于，它们经常不会出现在模型 profiling 里。GPU 看起来不忙，模型日志也没报错，但用户体感仍然很慢。根因可能是外部依赖把关键路径拉长了，导致模型优化收益被完全淹没。因此服务系统做优化前，必须先把“模型时间”和“模型外时间”拆开看。

## 16. Prefill 和 Decode 应该被视为两种工作负载

成熟系统会把它们当成两类不同 workload，而不是一个统一“推理请求”。

### 16.1 Prefill worker 的特点

1. 高并行度；
2. 更偏 compute-bound；
3. 对 batch 合并和 attention kernel 吞吐更敏感；
4. 更适合大块调度。

因此 prefill worker 更像一次性吃下大段上下文的重计算节点。它通常受益于更大的 batch、更强的 attention kernel、更积极的 prefix 复用和更稳定的显存布局。对这类 worker 来说，关键不是每步调度得多精细，而是能否把大块输入高效吞进去。

### 16.2 Decode worker 的特点

1. 步长小；
2. 更偏 memory-bound；
3. 受 KV cache 布局、scheduler 和 continuous batching 影响更大；
4. 尾延迟更容易被抖动放大。

decode worker 则完全不同。它更像一个高频、小步、持续调度的服务循环。哪怕单步计算量不大，只要 KV 访存、batch 变动、cache 回收或调度器抖动稍有异常，TTFT 和 TPOT 就会被明显放大。所以 decode 系统更依赖稳定的 runtime 和细粒度调度，而不是单次大吞吐。

### 16.3 为什么有些系统会拆池

对长文档、RAG、PDF 问答等场景，prefill 特别重，decode 相对轻。此时常见策略是：

1. 拆 prefill/decode 资源池；
2. 把长文档请求先送到 prefill 池；
3. decode 阶段再进入常规连续批处理池。

这样能避免“巨长 prefill 请求”把短对话 decode 也拖慢。

拆池的本质不是为了“架构更高级”，而是承认 prefill 和 decode 根本不是同一种资源竞争。只要两者混在一起，长输入请求就容易用 prefill 峰值去挤压 decode 的尾延迟。拆池之后，系统至少能分别规划容量、分别做 admission control，也更容易对不同池子采用不同的 kernel、量化和缓存策略。

## 17. Admission control 比想象中更重要

如果系统不控制谁能进入执行队列，就会被极端请求拉爆。常见准入策略包括：

1. 最大上下文长度限制；
2. 最大输出长度限制；
3. 按租户或用户等级限流；
4. 高峰期对昂贵任务转异步；
5. 工具型请求和普通问答分队列。

### 17.1 为什么 admission control 不是“拒绝用户”

没有准入控制时，结果往往是所有人一起变慢。  
合理的 admission control 实际上是在做系统保护：

1. 简单请求及时返回；
2. 极长任务转为异步或批处理；
3. 高价值用户流量优先；
4. 系统在峰值期间不被少数极端请求拖垮。

换句话说，admission control 是把不可控的拥塞变成可控的服务策略。没有它，系统看起来更“开放”，但实际上会让所有请求一起退化；有了它，团队至少能明确哪些请求应实时保证、哪些请求应延后、哪些请求必须限额处理。真正成熟的推理平台，都会把 admission control 视为服务品质的一部分，而不是不得已的限制。

## 18. Continuous batching 也有代价

它当然很有效，但不是无脑越 aggressive 越好。常见代价有：

1. 调度逻辑更复杂；
2. 某些长度分布下尾延迟反而恶化；
3. cache 管理复杂度上升；
4. 对多模型级联和工具调用系统整合更难。

### 18.1 什么时候要限制 batch 异质性

**如果一批请求的**：

1. 输入长度差异极大；
2. 目标输出长度差异极大；
3. 路由模型不一样；
4. 采样参数差异显著；

那么强行放在同一活动批次中，往往会让短请求体验变差。因此很多系统会按：

1. 长度桶；
2. 模型类型；
3. 是否带附件；
4. 是否工具链任务；

先分层，再做批处理。

batch 异质性过高时，最直接的问题是系统开始为“最慢的那类请求”买单。短请求要等长请求，轻任务要和重任务共享相同节奏，不同模型或不同采样参数还会让执行路径更加分裂。所以连续批处理不是一味扩大活动集合，而是要在“更多合批收益”和“更大异质性代价”之间找平衡。

## 19. KV cache 管理决定可持续吞吐

大模型服务常见的一种错觉是“权重放得下就能服务”。实际上更经常把系统压满的是 KV。

### 19.1 KV cache 的管理目标

1. 分配要快；
2. 访问要连续；
3. 回收要低碎片；
4. 跨请求复用尽量多；
5. 不能让冷长请求长期占住热资源。

这几项目标往往彼此拉扯。为了复用更多前缀，你可能想把缓存留久一点；但缓存留得越久，碎片和资源占用问题就越明显。为了让分配足够快，你可能选择更简单的页式管理；但页粒度一旦设计不合理，又会影响访存连续性。因此 KV 管理不是简单的内存问题，而是吞吐、尾延迟和命中率的联合优化问题。

### 19.2 为什么 prefix cache 会改变系统形态

当系统 prompt、模板提示、常用文档片段大量重复时，prefix cache 会显著降低 prefill 成本。但它同时引入新问题：

1. 哪些前缀值得缓存；
2. 缓存命中是否因 prompt 版本细小变化失效；
3. 多租户环境下如何隔离；
4. 缓存失效和回收策略如何设计。

一旦 prefix cache 真正发挥作用，服务系统就不再只是“每个请求独立执行”，而是开始出现显著的请求间共享。这会改变调度决策、容量估计和命中策略，甚至会影响 prompt 模板设计本身。很多团队后来会发现，想让 prefix cache 稳定工作，必须把系统 prompt、检索拼接和模板版本管理一起规范起来。

## 20. 多租户服务需要明确隔离策略

企业环境很少只有一类请求。多租户常见冲突包括：

1. 一个客户长文档请求过多；
2. 一个产品线突然启用推理链；
3. 内部批任务和外部实时请求共享集群；
4. 某租户的系统 prompt 特别长。

### 20.1 隔离方式

1. **物理隔离**：不同 GPU 池；
2. **逻辑隔离**：队列和权重隔离；
3. **优先级隔离**：高价值流量优先；
4. **成本隔离**：昂贵请求单独记账和限额。

如果隔离不存在，问题不会表现成“某个租户差一点”，而是整个系统尾延迟一起抖。

隔离设计真正要回答的是：当某个租户突然放大量长文档、图片附件或工具链请求时，影响能否被限制在局部。没有隔离，系统里的每一层都会被波及，从 queue 到 KV cache，再到外部检索和浏览器工具。因此多租户治理不是商业问题附属品，而是架构问题本身。

## 21. 服务系统里的回退路径必须先设计

**每个推理系统都需要明确回答**：当最优链路失效时，退到哪里。

**常见回退路径包括**：

1. 大模型降级到小模型；
2. 附件过大时转异步任务；
3. 检索失败时返回限定能力回答；
4. 工具系统超时时降级为只读模式；
5. 视觉前端失效时转文本 fallback。

### 21.1 为什么回退设计是质量问题，不只是稳定性问题

如果没有清晰 fallback，系统在异常时会：

1. 直接超时；
2. 返回半截结果；
3. 工具中途失败却不给出合理说明；
4. 明明是检索层故障，却看起来像模型在胡说。

这会极大恶化用户对系统可靠性的感知。

所以 fallback 不能只设计“活下来”的路径，还要设计“解释得清、体验可接受”的路径。对用户而言，降级到小模型但能稳定给出明确答复，往往好过主路径随机超时或输出半截内容。服务质量很多时候不是由最优路径定义，而是由异常路径定义。

## 22. 服务系统的核心指标应该怎么分层

### 22.1 请求层

1. QPS；
2. 拒绝率；
3. 平均和分位响应时长；
4. 请求长度分布；
5. 附件和模态分布。

请求层指标回答的是“今天进来的到底是什么流量”。如果这一层看不清，后面的所有资源和模型指标都很难解释。特别是长度分布、附件比例和工具调用占比，一旦变化，往往意味着系统画像已经变了，不能再用昨天的容量假设解释今天的问题。

### 22.2 Token 层

1. TTFT；
2. TPOT；
3. prefill token/s；
4. decode token/s；
5. 输出长度分布。

token 层指标更接近模型执行语义。它能帮助团队分辨问题到底出在 prefill 还是 decode，也能看出某次优化究竟改善了哪一段链路。很多“整体时延变差”的问题，只有拆到 token 层才看得见根因。

### 22.3 资源层

1. GPU utilization；
2. 显存占用；
3. KV cache 使用率和碎片率；
4. 网络和 CPU 利用率；
5. 外部工具超时率。

资源层指标不能只看平均 utilization，还要看波动和局部饱和。GPU 也许整体利用率不高，但 KV cache 已经严重碎片化；CPU 也许平均负载正常，但某个 OCR 线程池在高峰时完全堵住。这一层的任务是回答：当前系统是被哪一种资源真正卡住。

### 22.4 业务层

1. 任务完成率；
2. 人工接管率；
3. 升级到大模型的比例；
4. 异步任务完成率；
5. 用户满意度 proxy。

业务层指标是推理平台最容易缺、却最关键的一层。因为很多技术优化在请求层和资源层看起来都成立，到了业务层却未必有收益。比如 TTFT 更低了，但工具调用失败率升高，最终任务完成率可能反而下降。只有把技术指标和业务结果串起来，优化才有真实意义。

## 23. 高峰期容量规划要回答什么

1. 峰值流量是多少；
2. 峰值时长度分布是否也变化；
3. 最高价值流量是谁；
4. 最昂贵的请求类型是什么；
5. 哪些任务必须实时，哪些可异步；
6. 故障时哪些模块可优先关闭。

### 23.1 为什么容量规划不只是平均负载

很多系统平时跑得很好，但一旦业务活动上线或客户批量上传长文档，就会因为请求形态变化而崩。  
**所以容量规划必须基于**：

1. 流量峰值；
2. 任务结构峰值；
3. 最坏情况下的尾延迟；
4. 故障演练。

平均负载之所以危险，是因为线上系统常常死于“少量极端但昂贵的请求”而不是死于平均请求。长文档、图片附件、复杂 agent、多轮工具链和大模型升级请求，只要一起出现，就会把 queue、KV 和外部依赖同时推到极限。因此容量规划必须围绕最坏桶来做，而不是围绕均值来做。

## 24. 设计建议

1. 把服务系统看成排队和资源治理系统，而不只是模型调用器；
2. 显式区分 prefill 与 decode 的资源和指标；
3. 为极长任务和高风险任务提供独立路径；
4. 把 admission control、fallback 和多租户隔离当成一等设计对象；
5. 所有优化都要同时看平均值和尾部体验。

## 25. 小结

推理服务系统真正管理的不是“一个模型”，而是一组持续到来的请求、一套分层资源和一条不断变化的关键路径。谁能把排队、批处理、KV cache、前缀复用、工具链和回退路径放进同一张系统图里，谁才真正拥有可扩展的服务能力。

## 26. 一次请求在服务系统里究竟经过了什么

把推理系统看清楚，最好从一次请求的生命周期入手。很多性能和稳定性问题，不在模型内部，而在请求从入口到出口的每一个环节。

### 26.1 一个典型生命周期

1. **入口接入**：鉴权、配额检查、路由标签、租户识别；
2. **请求归类**：估算输入长度、模态种类、工具需求、优先级；
3. **admission control**：判断是否进入实时通道、异步通道或排队；
4. **预处理**：tokenize、图像/文档解析、prompt 组装、缓存查询；
5. **模型路由**：选模型、选量化版本、选 batch 池；
6. **执行阶段**：prefill、decode、工具调用、可能的二次模型调用；
7. **后处理与策略层**：过滤、结构化输出、重试、fallback；
8. **观测记录**：trace、指标、日志、成本与质量标签。

### 26.2 为什么这条链必须被完整画出来

因为系统里很多“模型慢”的抱怨，最后查出来其实是：

1. **tokenizer 慢**；
2. **OCR / 文档解析慢**；
3. **路由策略把请求送错了池子**；
4. **工具调用卡住了**；
5. **回退路径比主路径还慢**。

只要生命周期链画不完整，团队就会在错误位置优化。例如把大量时间花在模型 kernel 上，最后发现瓶颈其实在文档解析；或者一味扩 GPU，却没有发现 admission control 和 fallback 语义本身就不合理。完整链路图的价值，不是为了画得漂亮，而是为了让每一次故障和优化都有准确落点。

## 27. Queueing Model 决定系统会怎样坏掉

服务系统不会只因为平均负载高而坏，更常见的是因为**请求分布和调度策略让尾部先崩**。

### 27.1 需要显式建模的维度

1. **到达率波动**；
2. **输入长度分布**；
3. **输出长度分布**；
4. **多模态附件大小分布**；
5. **工具调用占比**；
6. **不同优先级请求的混部比例**。

### 27.2 为什么平均值会误导

同样平均输入 2000 token 的两组流量，若一组全是均匀 2000，另一组是 90% 为 100、10% 为 19100，它们对 queue 和 KV cache 的打击完全不同。

所以 queueing model 最怕只看均值，不看分布形状。系统真正承受的是极端长度、突发到达、混部比例和工具链波动，而不是一个抽象平均数。容量和调度策略若只按平均设计，就会在尾部样本最重要的时候集体失效。

## 28. 多池化服务往往优于一个大池子

很多团队最开始会把所有请求放进同一个调度池，觉得这样利用率高。但当负载变复杂后，**按请求类型分池**往往能显著提升尾部体验。

### 28.1 常见分池方式

1. **短问答池 / 长上下文池**；
2. **纯文本池 / 多模态池**；
3. **实时池 / 异步池**；
4. **高价值租户池 / 普通租户池**；
5. **实验流量池 / 生产稳定池**。

### 28.2 分池的主要收益

1. **避免长请求拖慢短请求**；
2. **让容量规划更清楚**；
3. **让 fallback 和 canary 更安全**；
4. **便于对不同 workload 用不同 kernel、量化和 batching 策略**。

### 28.3 分池的代价

1. **资源碎片化**；
2. **容量共享效率下降**；
3. **路由策略更复杂**；
4. **跨池迁移与 overflow 逻辑必须设计好**。

## 29. 解耦式角色分工越来越常见

随着上下文变长和工具链变重，服务系统越来越倾向于把不同角色拆开。

### 29.1 常见角色

1. **router / scheduler**；
2. **prefill worker**；
3. **decode worker**；
4. **KV service / cache manager**；
5. **tool executor**；
6. **post-processing / policy layer**。

### 29.2 为什么要拆

1. **prefill 和 decode 的资源画像不同**；
2. **工具调用和模型生成不该相互阻塞**；
3. **KV cache 生命周期管理可以独立优化**；
4. **多模型系统里路由与执行的扩展节奏不同**。

### 29.3 但拆分也有隐患

1. **跨节点传输变多**；
2. **失败域扩大**；
3. **trace 更难串起来**；
4. **一致性与重试语义更复杂**。

## 30. 服务系统要有“失败域地图”

成熟的推理平台会明确画出：哪一层出问题，会影响谁，如何降级。

### 30.1 典型失败域

1. **入口失败域**：鉴权、限流、配置发布错误；
2. **路由失败域**：错把请求送进不兼容模型或不合适的池；
3. **执行失败域**：GPU OOM、kernel 回退、runtime 崩溃；
4. **缓存失败域**：KV 污染、碎片过高、prefix cache 错用；
5. **外部依赖失败域**：向量库、OCR、浏览器、数据库超时。

### 30.2 为什么这很重要

若没有失败域地图，故障发生时团队会本能地“全量扩容”或“回滚整个系统”，既慢又贵，而且未必解决根因。

## 31. 模型路由不是只看质量，也要看系统形状

模型 routing 的决定因素至少包含：

1. **任务质量要求**；
2. **输入长度和附件类型**；
3. **实时性预算**；
4. **当前各池子的拥塞程度**；
5. **prefix 命中与缓存收益**；
6. **租户等级和成本预算**。

### 31.1 一个容易忽略的问题

哪怕大模型质量更好，也不意味着所有请求都该打给它。若某类短请求在小模型上已足够好，而且能显著减少 queue 和 tail latency，那么路由策略就不应只按离线质量排序。

更进一步说，模型 routing 的目标不是单请求最优，而是**整池请求的系统最优**。把所有请求都送去大模型，可能让单条回答上限更高，但也会抬高所有人的等待成本。真正成熟的路由器，必须同时关心质量、成本、拥塞和对其他请求的外部性。

## 32. Fallback 不是最后补丁，而是主设计对象

许多系统把 fallback 当成故障时临时兜底，这会导致平时根本没认真测试。更成熟的做法是把 fallback 设计成**平时就会少量运行的正式路径**。

### 32.1 常见 fallback 形式

1. **降级到更小模型**；
2. **降级到更短上下文**；
3. **改成异步任务**；
4. **先返回草稿，后补全高质量答案**；
5. **关闭某些昂贵工具或多模态解析**。

### 32.2 为什么要日常运行少量流量

否则一旦主路径出故障，fallback 也很可能早就坏了、慢了、或者质量已经不符合业务预期。

fallback 只有在平时被真实流量持续擦亮，才有资格在异常时承担主路径职责。否则它往往只是文档里的一个想法，一到故障时才发现模型版本过期、提示词没同步、监控没接上、延迟也不可接受。把少量真实流量常态化导入 fallback，是检验它是否真能用的唯一可靠方式。

## 33. 多租户治理不是加个配额就结束

真正的多租户服务至少要同时治理：

1. **配额与速率限制**；
2. **长度与附件上限**；
3. **优先级和抢占权**；
4. **缓存占用上限**；
5. **长任务对短任务的干扰**；
6. **成本归属与账单核算**。

### 33.1 为什么多租户特别容易伤到 tail latency

因为少数高成本租户如果没有被隔离好，就会通过 queue、KV cache 和外部依赖把整个系统拖慢。

多租户的危险在于，问题会沿着共享资源传染。一个租户多上传几批长文档，看上去只是它自己的业务行为，但很快就会体现在全局 queue 堵塞、prefix cache 污染、外部 OCR 过载和 decode 尾延迟抬升上。所以 tail latency 治理在多租户环境里，本质上就是共享资源治理。

## 34. 观测系统必须串起“请求-模型-资源-业务”

只看 GPU 利用率，或者只看 API 时延，都不足以管理现代推理系统。

### 34.1 一个更完整的观测链

1. **请求层**：租户、任务类型、长度、附件、工具需求；
2. **模型层**：命中哪个模型、哪个量化版本、哪个路由策略；
3. **资源层**：GPU、显存、KV cache、CPU、网络；
4. **业务层**：任务完成率、人工接管率、用户反馈 proxy。

### 34.2 为什么必须四层联动

因为服务系统的很多问题都是跨层的。例如某个新模型版本可能离线质量更好，但因 TTFT 变差导致用户工具调用超时，最终业务完成率反而下降。

四层联动的意义，就是让团队能回答“一个变化最后影响了谁”。只看一层指标，优化很容易出现局部正确、全局错误。把请求、模型、资源和业务层打通后，系统才真正具备闭环诊断能力，也才知道哪些优化值得继续放大，哪些应立即回滚。

## 35. 一份更实用的推理服务设计清单

1. **请求是否已按长度、模态和价值分层**；
2. **是否显式区分 prefill 与 decode 的资源治理**；
3. **是否存在可演练的 fallback 路径**；
4. **是否有多池和 overflow 设计**；
5. **是否已建立失败域地图**；
6. **是否能把 trace 贯通到业务指标**；
7. **是否能对高峰期和异常流量做有损但可控的降级**。

## 快速代码示例

```python
from fastapi import FastAPI
from vllm import LLM, SamplingParams

app = FastAPI()
llm = LLM(model="Qwen/Qwen2.5-7B-Instruct", max_model_len=8192)

@app.post("/generate")
def generate(prompt: str):
    out = llm.generate([prompt], SamplingParams(max_tokens=128, temperature=0.2))
    return {"text": out[0].outputs[0].text}
```

这段代码展示了一个最小服务化形态：`FastAPI` 负责 HTTP 接口，`vLLM` 负责高吞吐生成，采样参数在请求入口统一配置。后续通常再加队列、鉴权、限流和观测指标接入，形成可运维服务。

## 36. 小结

推理服务系统的核心，不是“让模型跑起来”，而是让**请求、队列、缓存、模型、工具和降级策略**共同组成一套稳定的生产机制。谁能把这些层串成一张有失败域、有回退、有容量边界的系统图，谁才能把模型能力真正稳定地交付给用户。
