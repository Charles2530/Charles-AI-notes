# 上下文压缩、KV 淘汰与内存分层

长上下文系统的真正瓶颈，常常不是模型参数，而是“历史上下文到底如何存、如何压、何时丢”。一旦请求从 `4k` 扩到 `32k`、`128k` 甚至更长，`KV cache` 的显存占用、prefill 开销和检索拼接策略都会迅速成为主问题。上下文压缩、KV 淘汰和内存分层，正是在这个背景下变得越来越重要。

**这页重点讨论一个核心判断**：

**长上下文系统的本质，不是把窗口一味做大，而是把记忆管理做成系统工程。**

## 1. 为什么长上下文会迅速吃掉系统

KV cache 大小近似随序列长度线性增长：

\[
M_{\text{KV}} \propto L \cdot T \cdot d,
\]

其中：

- \(L\) 是层数
- \(T\) 是上下文长度
- \(d\) 是隐藏维度

长上下文一多，即使权重已量化，系统仍可能先被 KV 撑爆。

更具体一点，若每层缓存 key 和 value，两者都需保存，那么显存压力大致会与：

\[
M_{\text{KV}} \sim 2 \cdot L \cdot T \cdot H \cdot d_h \cdot b
\]

成正比，其中：

- \(H\) 是头数
- \(d_h\) 是每头维度
- \(b\) 是每元素字节数

**这说明**：

- 长度翻倍，缓存几乎就翻倍；
- 模型越深、头越多、精度越高，缓存越贵。

## 2. Prefill 与 decode 的问题完全不同

长上下文系统很容易把两个问题混在一起：

### 2.1 Prefill 问题

所有 token 一次性送入模型时，计算和内存带宽压力暴增。  
这更像“整本书一次塞进大脑”。

### 2.2 Decode 问题

虽然每次只生成少量 token，但历史缓存会持续存在，并不断影响每一步注意力计算。  
这更像“边说边记，而且记忆越积越多”。

因此长上下文优化不能只盯一种指标。  
有些方法主要降低 prefill 成本，有些方法主要控制 decode 阶段的缓存压力。

## 3. 上下文不是越长越有用

许多真实请求里，大量历史 token 对当前回答边际价值极低。  
问题不是“能不能塞进去”，而是“哪些信息应该保留在高精记忆里”。

因此上下文管理本质上是一种记忆选择问题。

### 一个很直观的例子

在企业助手长对话里，系统可能保留了：

- 十几轮寒暄
- 多次重复确认
- 旧版本工具日志
- 已被修正的错误中间结论

这些内容虽然都在“上下文”里，但真正对当前任务有价值的，往往只有：

- 最新目标
- 关键约束
- 已确认事实
- 当前工具状态

所以好的长上下文系统不是“什么都记”，而是“记得有层次”。

## 4. 压缩的几种方式

### 4.1 摘要压缩

把旧对话总结成短文本。

**优点**：

- 简单
- 易理解
- 适合纯文本对话

**风险**：

- 容易丢细节
- 摘要本身可能产生幻觉
- 事实和约束容易被模糊化

### 4.2 结构压缩

保留字段、事实、工具结果等关键状态。

**例如把一大段对话压成**：

- 用户目标
- 已知事实
- 禁止条件
- 待办事项
- 当前工具状态

这种方式对企业问答、agent 和流程型任务往往比纯摘要更稳。

### 4.3 向量检索

把旧片段转入外部记忆，需要时再召回。

**优点**：

- 长历史不必一直常驻上下文
- 可扩展到极大知识量

**风险**：

- 召回错误会直接造成“记忆失忆”
- 相关片段被漏检时，模型会像没见过一样回答

### 4.4 Token 级压缩

通过 learned token merging、attention-aware pruning 或动态下采样减少序列。

**优点**：

- 可以直接作用于 token 序列
- 对长文档、多模态场景潜力较大

**风险**：

- 压缩标准不稳时容易丢关键信息
- 工程实现复杂

## 5. 为什么结构压缩常常比纯摘要更适合 agent

对于 agent 系统，许多记忆不是自然语言故事，而是结构化状态，例如：

- 哪个工具已经调用过
- 哪个 API 返回了什么
- 哪个网页表单已经填过
- 哪些约束仍然未满足

如果只做自然语言摘要，很容易把“动作状态”压没。  
所以 agent 更适合把记忆拆成：

- 目标状态
- 世界状态
- 工具状态
- 待执行计划

这本质上更接近程序状态保存，而不是普通对话摘要。

## 6. KV 淘汰策略

如果显存有限，就必须淘汰。常见思路包括：

1. 先进先出
2. 保留最近窗口
3. 按 attention 重要度保留
4. 按角色保留，如系统 prompt、工具返回、用户目标优先

一个简单抽象是：给每段记忆分配价值分数 \(u_i\)，然后在预算内保留最大价值集合。

\[
\max_{S \subseteq \{1,\dots,n\}}
\sum_{i \in S} u_i
\quad
\text{s.t.}
\quad
\sum_{i \in S} c_i \le B.
\]

其中：

- \(u_i\) 是记忆价值
- \(c_i\) 是占用成本
- \(B\) 是预算

这让 KV 管理看起来很像一个背包问题。

## 7. 几类常见淘汰策略的优缺点

### 7.1 先进先出

**优点**：

- 实现简单
- 开销低

**缺点**：

- 很容易删掉仍然重要的早期目标信息

### 7.2 固定最近窗口

**优点**：

- 对聊天和短期局部一致性有帮助

**缺点**：

- 对跨长时任务的核心约束不友好

### 7.3 Attention 重要度驱动

**优点**：

- 更贴近模型真实关注

**缺点**：

- 重要度估计不稳定
- 额外统计成本高

### 7.4 角色或语义驱动保留

**例如始终优先保留**：

- system prompt
- 用户主目标
- 关键工具结果
- 最近日志摘要

**优点**：

- 更符合 agent 和任务执行需求

**缺点**：

- 需要额外结构设计

## 8. 内存分层

成熟系统往往不是“全在 GPU 或全不在 GPU”，而是：

1. 热上下文在 GPU KV
2. 温上下文在 CPU 或主存缓存
3. 冷上下文在向量数据库或对象存储

这和操作系统的多级缓存非常像。  
关键难点是分层转移不能把时延吃光。

### 8.1 热、温、冷记忆的直觉

**可以粗略理解为**：

- **热记忆**：马上就要用，必须低延迟。
- **温记忆**：可能很快会用，允许稍慢恢复。
- **冷记忆**：大概率不会立刻用，但不能丢。

### 8.2 一个类比

**这很像你写作时的工作方式**：

- 当前正在看的段落放在桌面上
- 刚才翻过的资料放在手边
- 更多材料放在书架或电脑里

没有人会把整座图书馆都摊在桌上。  
长上下文系统也一样。

## 9. 一个长文档问答例子

在长文档问答里，真正应该常驻 GPU 的通常不是整份文档，而是：

1. 当前问题相关的检索片段
2. 系统 prompt
3. 当前已生成的局部思路或工具状态

其余内容更适合做结构化缓存或外部召回。

如果用户在问“合同第 4.2 条的违约金规则与附件二是否冲突”，那么真正高价值常驻的可能只有：

- 第 4.2 条附近内容
- 附件二相关片段
- 前面已经确定的法务解释

而不是整份 200 页合同全部塞在显存里。

## 10. 一个 agent 例子

网页代理在执行多步任务时，经常会积累大量低价值 token：

- 冗长 HTML 文本
- 重复截图描述
- 工具链调试日志
- 已过期页面状态

如果这些都被原样塞进长上下文，系统会越来越慢。  
**更合理的做法是**：

- 结构化保存页面状态
- 保留关键操作摘要
- 必要时重新抓取当前页面

这样系统记住的是“状态”，而不是“所有过程噪声”。

## 11. 为什么“无限扩窗”不是最终答案

表面看，上下文窗口越大，模型似乎越聪明。  
但真实系统里至少会遇到三类问题：

### 11.1 成本问题

上下文越长，prefill 和 KV cache 成本越高。

### 11.2 信号稀释问题

相关信息在极长上下文里会被无关 token 稀释，模型并不一定能稳定抓住重点。

### 11.3 记忆治理问题

没有分层和淘汰机制的长窗口，只会把系统变成“昂贵但混乱的记忆堆”。

## 12. 一份实践建议

如果你在做长上下文系统，可按下面顺序推进：

1. 先统计哪些 token 真正被频繁需要。
2. 区分文本记忆、结构记忆和工具状态。
3. 给不同记忆定义热度和优先级。
4. 对冷记忆使用检索或外部存储。
5. 对长对话增加结构摘要而不是纯自然语言摘要。
6. 单独评测压缩和淘汰后的任务成功率，而不是只看能否塞更长窗口。

## 13. 小结

长上下文服务的核心，不是无限扩窗，而是把上下文当成一种分层内存来管理。  
谁留下、谁压缩、谁外置、谁淘汰，决定了系统能否在可接受成本下真正利用长历史。

**更进一步说**：

- 上下文压缩解决的是“如何把历史变成更高信息密度的记忆”；
- KV 淘汰解决的是“在有限预算下谁有资格留下”；
- 内存分层解决的是“不同价值的记忆该放在哪一层系统里”。

只有这三件事一起做好，长上下文能力才会从“实验室窗口长度”变成“真实可用记忆系统”。

## 14. 记忆分层比单次压缩更重要

很多系统在谈长上下文时，只关注“怎么压缩旧内容”。更实用的视角其实是把记忆做分层：

1. `热记忆`：当前 GPU 上常驻，供即时 decode 使用；
2. `温记忆`：较低频访问，可在 CPU、主存或更便宜介质上保存；
3. `冷记忆`：不常访问，放入检索库或结构化存储，需要时再召回。

### 14.1 为什么分层更稳

因为现实任务并不是“所有旧信息价值一样”。更常见的情况是：

1. 当前目标和最近状态最热；
2. 关键事实和历史结论次热；
3. 大量过程噪声和冗余说明很冷。

单纯做一次总结，往往会把这三类信息混在一起，导致真正重要的状态也被压扁。

## 15. 摘要压缩和结构压缩各自适合什么

### 15.1 摘要压缩更适合对话语义

例如：

1. 用户长期目标；
2. 历史偏好；
3. 已确定结论；
4. 待办列表。

### 15.2 结构压缩更适合 agent 和流程系统

**例如把历史压成**：

1. 工具状态；
2. 已完成步骤；
3. 当前环境状态；
4. 待检查约束；
5. 失败恢复点。

### 15.3 为什么两者通常要结合

如果只有摘要，没有结构，agent 会忘记程序状态；  
如果只有结构，没有摘要，系统会失去人类可读的高层语义。  
**因此更稳的方案通常是**：

1. 一份自然语言摘要；
2. 一份结构化状态；
3. 必要时保留关键原始片段索引。

## 16. 不是所有 token 都值得放进 KV

从服务视角看，token 的价值并不均匀。高价值 token 常包括：

1. system prompt；
2. 用户主目标；
3. 关键证据；
4. 工具返回的关键字段；
5. 最近几轮交互中的约束变化。

**低价值 token 常包括**：

1. 礼貌寒暄；
2. 重复解释；
3. 旧版错误中间推理；
4. 已过期日志；
5. 可从外部系统重新获取的信息。

### 16.1 为什么这会影响淘汰策略

如果淘汰策略只看“最新”或“最旧”，它就无法区分重要与不重要。  
**因此更好的做法通常会综合**：

1. recency；
2. role；
3. semantic importance；
4. 是否可重建；
5. 与当前任务的相关性。

## 17. RAG 系统中的上下文管理是双层问题

对 RAG 而言，长上下文不仅是“对话历史怎么保留”，还是“检索证据怎么拼接”。

### 17.1 第一层：历史记忆

**系统需要记住**：

1. 用户当前目标；
2. 已确认事实；
3. 先前引用过的证据；
4. 工具和检索状态。

### 17.2 第二层：证据装配

每次新回答前，还要决定：

1. 召回哪些片段；
2. 片段排序如何做；
3. 是否保留旧证据；
4. 是否需要摘要旧证据。

如果这两层混在一起，系统很容易把“长期记忆”和“本轮证据”都做得不稳定。

## 18. Agent 系统里的记忆更像状态机

对网页 agent、代码 agent、企业流程 agent 来说，历史不只是文本，它更像一个不断更新的状态机：

1. 当前页面或系统所处位置；
2. 哪些步骤已经完成；
3. 哪些约束已满足；
4. 哪个工具返回了什么；
5. 下一步允许做什么。

### 18.1 为什么这意味着记忆应当外化

如果你把这些状态全都塞回自然语言上下文，模型每一轮都要重新解析一次“世界状态”，成本高且不稳定。更合理的做法是：

1. 把状态外化成结构对象；
2. 只把当前最相关部分重新写回 prompt；
3. 允许模型在需要时查询完整状态。

这比单纯“窗口更长”更像真正的系统记忆。

## 19. 上下文压缩要单独评测信息损失

压缩如果只看 token 变少，是不够的。更关键的是：

1. 关键事实是否丢了；
2. 数字、日期、实体是否失真；
3. 工具状态是否还能正确恢复；
4. 长任务成功率是否下降；
5. 用户偏好和禁忌是否仍被保留。

### 19.1 一个简单评测套路

**可以分别测试**：

1. 原始全量上下文；
2. 摘要压缩版本；
3. 结构压缩版本；
4. 摘要+结构混合版本；

**然后比较**：

1. 任务正确率；
2. 事实一致性；
3. TTFT；
4. 显存占用；
5. 长会话持续成功率。

## 20. KV 淘汰策略应和任务类型绑定

### 20.1 对话助手

**更适合保留**：

1. 用户目标；
2. 最近纠错；
3. 长期偏好；
4. 最近几轮高相关内容。

### 20.2 文档问答

**更适合保留**：

1. 当前问题；
2. 核心证据；
3. 已生成的中间摘要；
4. 引用映射关系。

### 20.3 Agent 系统

**更适合保留**：

1. 世界状态；
2. 工具状态；
3. 计划状态；
4. 最近失败点和恢复策略。

同样的淘汰策略，不一定适合这三类系统。

## 21. 何时该外置而不是继续压缩

有些信息与其继续压缩，不如直接外置到检索库或结构化存储。通常满足下列任一条件时，外置更划算：

1. 可被稳定重建；
2. 访问频率低；
3. 容量大但局部查询多；
4. 本身就是结构化数据；
5. 对当前一步 decode 的直接影响较小。

### 21.1 典型适合外置的内容

1. 历史长日志；
2. 完整网页 DOM；
3. 大型附件全文；
4. 工具原始返回体；
5. 旧轮次的完整推理痕迹。

## 22. 一个更贴近系统的记忆策略模板

可以把长上下文系统设计成如下四步：

1. 先识别热信息，放进当前 prompt；
2. 把结构化状态单独保存；
3. 把冷信息送入检索层；
4. 对过期信息做摘要归档或彻底淘汰。

### 22.1 这个模板的优势

1. 更容易控制显存；
2. 更容易解释为什么某些信息被保留；
3. 更适合多轮和长任务；
4. 更容易做回归评测。

## 23. 设计建议

1. 不要把长上下文问题简化成“窗口越大越好”；
2. 优先建设分层记忆，而不是只做一次性摘要；
3. 摘要、结构状态、检索召回三者同时设计；
4. 单独评测压缩后的信息损失和长任务成功率；
5. 根据任务类型定制 KV 淘汰和外置策略。

## 24. 小结

长上下文系统真正要管理的不是 token 数，而是记忆价值。谁是热记忆、谁可外置、谁应被淘汰，决定了系统是在“昂贵地记住一切”，还是“高效地记住真正重要的事”。只有把压缩、淘汰和分层内存当作统一问题来做，长上下文能力才会从模型窗口参数变成可运营的系统能力。


## 补充：用服务系统视角重写 **上下文压缩、KV 淘汰与内存分层**

推理主题最容易被写成一串优化名词，但线上系统真正关心的从来不是“有没有这个技巧”，而是 **它是否改变了请求、队列、缓存、成本和高价值流量的命运**。围绕 **上下文压缩、KV 淘汰与内存分层**，更稳的理解框架是先问：这件事究竟在改变哪一层约束。

1. **压缩损失、KV 淘汰、热上下文识别、分层存储、回退策略**。

只要这几层没有被明确分开，系统就很容易出现一种假象：离线 benchmark 更好了，线上却并没有更稳；或者平均时延下降了，但高价值租户与长尾任务反而更差。对推理系统来说，这类错觉比“完全没有优化”更危险，因为它会把团队引向错误的资源分配方向。

## 补充：更扎实的设计顺序

针对 **上下文压缩、KV 淘汰与内存分层**，建议把设计顺序固定成下面几步：

1. **先画请求生命周期图**，明确哪些阶段会受这项机制影响；
2. **再画资源账本**，把 GPU、显存、KV、网络、工具链和 CPU 依赖拆开；
3. **然后才讨论策略本身**，例如要不要路由、要不要 cache、要不要 speculative、要不要分层服务；
4. **为高价值流量单独建桶**，不要只看总体平均；
5. **最后定义回退与降级**，确保高峰期和故障期也有可执行动作。

这个顺序的价值在于：它迫使团队先把 **上下文压缩、KV 淘汰与内存分层** 放进完整服务图里，再讨论局部技巧，否则很容易陷入“看起来懂很多优化，但不知道哪个最值钱”的状态。

## 补充：真实线上最常见的失败模式

围绕 **上下文压缩、KV 淘汰与内存分层**，线上失败常常并不体现在单次崩溃，而是慢性失真：

1. **只看显存节省**。
2. **忽略压缩对答案质量的局部破坏**。
3. **淘汰策略不可解释**。

这些问题之所以难，是因为它们往往混在一起发生。一个路由变化可能同时影响缓存命中、TTFT、单位成本和用户重试率；一个 cache 策略调整可能同时影响长上下文任务、KV 生命周期和多租户公平性。因此，**上下文压缩、KV 淘汰与内存分层** 相关改动必须天然携带可观测性，否则等于把复杂度埋进黑箱。

## 补充：更像生产系统的验收标准

对 **上下文压缩、KV 淘汰与内存分层**，更靠谱的验收通常不是一句“线上变快了”，而是至少同时回答：

1. **按任务桶评测压缩损失**。
2. **淘汰前后做 replay**。
3. **热上下文单独追踪**。
4. **保留无压缩对照**。

再进一步，建议把这类主题纳入持续治理：

1. **按任务桶和租户桶做长期趋势跟踪**；
2. **保留代表性 replay / shadow case**，用于版本切换对照；
3. **把高价值失败样本回流到训练和系统修复**；
4. **让 oncall 面板能直接看到与它相关的 route、cache、queue 和 fallback 状态**。

只要这套闭环建立起来，**上下文压缩、KV 淘汰与内存分层** 才真正从“优化点”变成“系统能力”。
