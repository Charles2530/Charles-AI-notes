# vLLM、SGLang、TensorRT-LLM 与推理运行时栈

模型推理今天早已不只是“把权重加载到 GPU 然后调用 `generate()`”。真正决定线上成本和时延的，往往是一层专门的运行时系统：它管理 continuous batching、KV cache、prefix cache、paged attention、量化 kernel、图捕获、路由、分布式调度和多模型级联。`vLLM`、`SGLang`、`TensorRT-LLM` 以及相关 serving runtime，正是围绕这些问题形成的主流路线。

这页重点不是做产品宣传，而是给出一个更工程化的理解：

1. 推理运行时到底在比什么；
2. `vLLM`、`SGLang`、`TensorRT-LLM` 各自擅长的工作负载是什么；
3. 为什么 runtime 选型会反过来塑造模型格式、量化策略和服务架构；
4. 如何避免把“框架 benchmark 更快”误解成“真实业务更优”。

## 1. 推理运行时在系统栈中的位置

如果把在线服务栈拆开，可以粗略写成：

1. 网关与流量管理；
2. 路由、检索和工具编排；
3. 推理运行时；
4. 低层 kernel、通信和硬件执行；
5. 监控、日志与回流。

这里的推理运行时，主要负责：

1. 模型加载和权重布局；
2. 请求编队与调度；
3. prefill / decode 协同；
4. KV cache 分配与回收；
5. prefix cache、speculative decoding、量化路径接入；
6. 多 GPU 或多节点执行编排。

它既不是纯模型代码，也不是纯硬件内核，而是把两者连起来的执行层。

## 2. 为什么推理运行时会成为一等问题

同一个模型，在不同运行时上的表现可能差很多，原因通常不在“数学不同”，而在：

1. cache 管理不同；
2. 调度粒度不同；
3. kernel 选择和图优化不同；
4. 长短请求混部策略不同；
5. 多模型和多租户支持能力不同。

因此 runtime 选型不是“包管理器偏好”，而是服务系统设计的一部分。

## 3. `vLLM` 解决了什么典型问题

`vLLM` 最广为人知的贡献，是把推理中的 KV cache 管理和 continuous batching 做成了高可用的通用运行时能力。

### 3.1 为什么它有代表性

因为大模型服务里两个最现实的瓶颈就是：

1. decode 阶段的连续批处理；
2. KV cache 的内存碎片和复用。

`vLLM` 的工程价值，就是围绕这些问题建立了更稳定的运行时抽象。

### 3.2 哪类场景通常受益明显

1. 通用聊天；
2. 中高并发文本服务；
3. 长短请求混合但以解码吞吐为主的场景；
4. 需要 prefix cache 和高效 KV 管理的部署。

### 3.3 什么时候不能只看它的 benchmark

**如果你的系统高度依赖**：

1. 多工具编排；
2. 推理轨迹控制；
3. 复杂结构化生成；
4. 更激进的程序调度和请求编程模型；

那么单看 token/s benchmark 未必足以决定选型。

## 4. `SGLang` 为什么越来越受关注

`SGLang` 的一个重要特征，是它更强调“为复杂推理路径和程序化请求组织提供更灵活的执行框架”。它常被讨论的价值包括：

1. 对复杂生成工作流更友好；
2. 对结构化调用、程序化调度、分阶段生成有较强表达力；
3. 更容易和多轮推理链、工具流程结合。

### 4.1 它的工程直觉

如果说一部分 runtime 更像是在优化“单模型高吞吐文本生成”，那么 `SGLang` 更像在优化“复杂请求程序如何被高效执行”。

### 4.2 哪类系统更可能受益

1. 复杂 agent；
2. 多阶段结构化生成；
3. 同一请求中包含不同生成模式切换；
4. 想把请求逻辑组织成更显式程序流的系统。

## 5. `TensorRT-LLM` 的典型价值

`TensorRT-LLM` 常被放在“更强硬件贴合和推理优化”的上下文里讨论。它的吸引力通常来自：

1. 更深的图优化和 kernel 选择；
2. 更强的硬件特化；
3. 对量化、图捕获、融合路径更积极；
4. 在某些固定部署环境下能取得非常强的吞吐和时延表现。

### 5.1 为什么它常出现在高性能部署讨论里

因为当工作负载较稳定、目标硬件明确、部署链路愿意接受更强工程约束时，这类更深度优化的 runtime 往往能吃到更多硬件红利。

### 5.2 代价是什么

1. 集成复杂度可能更高；
2. 某些动态工作负载适配成本更高；
3. 模型转换、量化和图优化路径需要更谨慎；
4. 团队必须更理解硬件和编译执行细节。

## 6. 运行时选型不能脱离工作负载

**选型前至少应先回答**：

1. 请求以短对话为主，还是长文档/长推理为主；
2. 更在意 TTFT，还是总吞吐；
3. 是否需要复杂 agent / tool calling；
4. 是否多模型共存；
5. 是否有固定硬件和稳定模型格式；
6. 是否计划大量量化和 kernel 特化。

### 6.1 不同负载的典型偏好

#### 高并发文本聊天

**更关心**：

1. continuous batching；
2. decode 吞吐；
3. prefix cache；
4. 多租户稳定性。

#### 长文档问答

**更关心**：

1. prefill 吞吐；
2. KV cache 管理；
3. 长上下文 kernel；
4. 异步任务支持。

#### 复杂 agent

**更关心**：

1. 请求可编程性；
2. 多阶段执行组织；
3. 结构化输出控制；
4. 外部工具整合。

#### 固定场景高吞吐部署

**更关心**：

1. 图优化；
2. 静态路径稳定性；
3. 量化后极致吞吐；
4. 硬件特化。

## 7. KV cache 运行时抽象为什么这么重要

无论选哪种 runtime，KV cache 都是核心资源。运行时需要回答：

1. 如何分配页面或块；
2. 如何回收；
3. 如何避免碎片；
4. 如何支持 prefix 复用；
5. 多租户下如何隔离。

### 7.1 为什么 runtime 之间差异会在这里被放大

因为 decode 阶段本来就对 KV cache 极其敏感。  
只要：

1. 分配策略不稳；
2. 碎片太多；
3. 页面过小或过大；
4. 回收滞后；

用户体验和尾延迟就会明显受影响。

## 8. Prefix cache 与共享前缀优化

在很多真实系统里，请求并非彼此独立。它们常共享：

1. system prompt；
2. 常用工具说明；
3. 常见模版；
4. 相同的文档前缀；
5. 相同的 few-shot 示例。

### 8.1 为什么 prefix cache 能带来巨大收益

因为它直接降低了 prefill 成本，尤其在：

1. 企业助手有固定系统提示；
2. 客服场景有统一规则；
3. 多用户查询同一知识库；
4. 多轮 agent 共用大段上下文；

这些场景中收益最明显。

### 8.2 但 prefix cache 也会带来新问题

1. prompt 版本稍改就失效；
2. 多租户隔离复杂；
3. 不同模型之间是否可共享前缀并不总成立；
4. cache 失效策略会影响系统稳定性。

## 9. 图捕获、静态形状和动态工作负载

某些 runtime 强调图捕获和更深的编译优化，这通常在：

1. 形状较稳定；
2. 模型固定；
3. 工作负载规律；
4. 量化路径明确；

的情况下收益明显。

### 9.1 为什么动态工作负载会带来挑战

**真实在线流量常常意味着**：

1. 输入长度变化大；
2. 输出长度变化大；
3. 工具路径变化大；
4. 模型和租户配置变化大。

这会让某些静态优化难以充分发挥。  
因此运行时选型，必须把动态性也当成一等输入。

## 10. 量化路径和 runtime 是强耦合的

运行时并不只是“接受一个量化模型”。它还决定：

1. 量化权重如何布局；
2. 反量化在何处发生；
3. 混合精度路径如何走；
4. 哪些 kernel 被替换；
5. KV cache 是否也量化。

### 10.1 为什么同样的量化方案在不同 runtime 上表现不同

因为真实系统性能不只看参数更小，还看：

1. 布局变换是否昂贵；
2. 是否引入额外同步；
3. decode kernel 是否真正受益；
4. accuracy 损失是否能接受。

量化路线必须和 runtime 共同设计，而不是后贴上去。

## 11. 多模型共存时，runtime 不只是执行器

企业系统越来越少是“单模型服务”，更多是：

1. 对话模型；
2. 推理模型；
3. VLM；
4. reranker；
5. embedding；
6. 小模型级联。

### 11.1 这时 runtime 要回答什么

1. 多模型是否共享 tokenizer、prefix 或 cache；
2. 路由后切换模型是否能复用中间状态；
3. 是否支持不同模型分池部署；
4. 是否方便接入上层路由和回退机制。

单模型 benchmark 再强，如果多模型场景下集成成本极高，也未必适合真实业务。

## 12. Agent 和工具调用为何会放大 runtime 差异

对 agent 而言，请求不再是“一次输入，一次输出”。它更像：

1. 生成；
2. 调工具；
3. 解析结果；
4. 再生成；
5. 再调用工具。

### 12.1 这对 runtime 的要求

1. 更灵活的请求编程模型；
2. 更好的阶段切换和状态管理；
3. 更细粒度的 tracing；
4. 更稳定的结构化输出支持；
5. 更好的中间状态复用。

这也是为什么很多 agent 团队在 runtime 选型上，不会只看裸 token/s。

## 13. 运行时 benchmark 应该怎么测

更靠谱的 benchmark 通常至少要覆盖：

1. 短输入短输出；
2. 长输入短输出；
3. 中输入长输出；
4. 混合长度并发；
5. 带结构化输出；
6. 带工具链或多阶段请求。

### 13.1 为什么单一 benchmark 很危险

某个 runtime 也许在固定 batch、固定长度、固定量化下非常快，但真实系统的长度和任务分布完全不同。  
如果不做 workload-matched benchmark，选型很容易失真。

## 14. 常见失效模式

### 14.1 只看吞吐，不看 TTFT

对聊天产品非常致命。

### 14.2 只看单模型，不看路由和级联

上线后会发现真正瓶颈根本不在主模型。

### 14.3 只看平均，不看 P95/P99

runtime 在混合负载下的 tail latency 行为才更接近真实部署。

### 14.4 只看文本问答，不看结构化和 agent

最终会在真正复杂任务上线时补很多系统债。

## 15. 运行时选型的实践框架

**可以按以下顺序做**：

1. 先按业务拆 workload 画像；
2. 再定义质量、延迟、吞吐、成本四类目标；
3. 在真实长度和并发分布上压测多种 runtime；
4. 单独测试量化、prefix cache、工具链和多模型级联；
5. 最后再考虑运维复杂度和团队维护能力。

### 15.1 什么时候优先考虑更通用的 runtime

1. 工作负载变化大；
2. 多模型迭代快；
3. 需要快速接入新模型；
4. 团队更重视灵活性而非极致特化。

### 15.2 什么时候优先考虑更深度优化的 runtime

1. 模型和硬件都稳定；
2. 工作负载规律较强；
3. 目标是极致成本和时延；
4. 团队能接受更高系统复杂度。

## 16. 设计建议

1. 不要把 runtime 选型简化成“谁 benchmark 更高”；
2. 先明确你的系统更像聊天、长文档、agent 还是固定高吞吐服务；
3. 单独验证 KV cache、prefix cache、量化和多模型集成路径；
4. 对 TTFT、TPOT、P95/P99 和成本同时做 workload-matched 测试；
5. 把团队维护成本和调试能力纳入选型，而不是只看一时峰值。

## 17. 小结

`vLLM`、`SGLang`、`TensorRT-LLM` 等推理运行时，本质上是在回答同一个问题：如何把模型、cache、调度、量化和请求程序组织成一条高效执行路径。它们的差异，并不只是代码风格不同，而是对不同工作负载、不同集成方式和不同硬件约束做了不同取舍。真正成熟的选型，不是追求抽象上的“最好 runtime”，而是让 runtime 与你的业务负载、模型结构和团队能力一起匹配。

## 18. 运行时选型后的第一批验证

真正落地时，建议先验证：

1. 主 workload 上的 TTFT、TPOT、P95/P99；
2. prefix cache 和 KV 行为是否符合预期；
3. 多模型、LoRA 或结构化输出路径是否稳定；
4. 量化后性能和质量是否一起可接受；
5. 线上 tracing 能否把 runtime 内部决策暴露出来。

### 18.1 为什么 tracing 也属于 runtime 验收

因为没有足够可观测性，runtime 再强，一旦遇到混合负载退化也很难定位。

## 19. 小结

推理运行时的好坏，不该只由离线 benchmark 决定，而应由“在真实工作负载、真实路由和真实缓存压力下，能否持续交付可解释的性能与稳定性”来决定。只有在这个标准下，runtime 选型才真正和业务目标对齐。

## 20. Runtime 选型先看“主工作负载”

讨论 `vLLM`、`SGLang`、`TensorRT-LLM` 时，最容易犯的错误，是脱离工作负载谈优劣。

### 20.1 需要先回答的问题

1. **你的主任务是长上下文、短问答、agent、还是多模态**；
2. **你更看重快速迭代，还是极致吞吐**；
3. **你是否高度依赖动态路由和复杂工具调用**；
4. **你是否需要更强的硬件特化和部署封装**；
5. **团队是否有能力长期维护更底层的 runtime 集成**。

### 20.2 为什么这一步先于 benchmark

因为某个 runtime 在离线 token/s 上更好，不代表它在你的真实混合负载里更优。

## 21. `vLLM` 的价值常在“通用可用 + Paged KV”

很多团队会把 `vLLM` 当成默认起点，是因为它通常在以下几方面组合得比较均衡：

1. **continuous batching 能力**；
2. **paged KV cache 设计**；
3. **较好的生态与模型兼容性**；
4. **相对平衡的易用性与性能**。

### 21.1 但它不是无条件最优

1. **某些极度特化的 deployment 或硬件路径未必最强**；
2. **复杂 tool / agent pipeline 里，还要看上层编排是否合拍**；
3. **当模型结构或量化路径很特殊时，定制成本仍然存在**。

## 22. `SGLang` 更值得从“编排 + runtime 协同”看

`SGLang` 的讨论常不只是 runtime 内核本身，而是它如何服务更复杂的生成与编排工作流。

### 22.1 它更吸引人的地方

1. **对复杂 prompt / program-like generation 更友好**；
2. **更容易与结构化生成、路由和多步执行逻辑结合**；
3. **在 agent 或复杂 serving workflow 中更强调系统级协同**。

### 22.2 这意味着什么

选 `SGLang` 时，不能只看单模型吞吐，还要看：

1. **它是否让你的上层流程更简单**；
2. **是否降低了多步执行的整体系统复杂度**；
3. **是否便于把工具调用和 runtime trace 串起来**。

## 23. `TensorRT-LLM` 更像“硬件特化与部署工程”的代表

讨论 `TensorRT-LLM` 时，重点常常不只是功能，而是：

1. **它对特定 NVIDIA 硬件路径的深度优化**；
2. **其引擎构建与部署流程**；
3. **对量化、kernel 特化和推理图优化的系统性封装**。

### 23.1 更适合什么场景

1. **对吞吐和时延有严格目标**；
2. **部署环境相对可控**；
3. **愿意接受更重的构建与集成流程**；
4. **需要深度吃硬件特化收益**。

### 23.2 代价在哪

1. **灵活性相对下降**；
2. **版本和部署链路更重**；
3. **某些快速试验路径不如通用 runtime 轻**。

## 24. Runtime 的真正差异常在“默认假设”

看起来都支持 batching、KV cache、量化、LoRA，但它们的默认假设并不一样。

### 24.1 常见默认假设维度

1. **请求长度分布**；
2. **模型结构的规则程度**；
3. **是否高频做动态编排**；
4. **是否更偏离线构建还是在线灵活调度**；
5. **目标是研究快迭代还是生产稳运行**。

### 24.2 为什么理解默认假设重要

因为你真正选择的，不只是一个软件包，而是一套对工作负载的隐含建模方式。

## 25. Runtime 选型应结合可观测性一起判断

再强的 runtime，如果内部决策不可观测，线上一旦退化就会很难管理。

### 25.1 建议重点看

1. **batch 组织过程能否追踪**；
2. **cache 命中、淘汰、碎片是否可见**；
3. **route 到不同 kernel / path 的原因是否可解释**；
4. **TTFT / TPOT 是否能拆到 runtime 内部阶段**；
5. **异常形状和回退路径是否有 trace**。

### 25.2 为什么这和性能同样重要

因为生产系统里，不可观测的 5% 性能损失，往往比可观测的 10% 性能损失更贵。

## 26. 多模型、多 LoRA、Agent 流量会放大 runtime 差异

在单模型 benchmark 下接近的两个 runtime，进入复杂流量后差距可能迅速拉开。

### 26.1 复杂流量会暴露什么

1. **batch shaping 能力**；
2. **多租户隔离**；
3. **KV cache 管理成熟度**；
4. **与上层编排框架的协同**；
5. **异常回退和 tracing 能力**。

### 26.2 为什么这很关键

因为真实业务很少永远只有一种干净 workload。谁更能承受混合负载，谁才更接近长期可用。

## 27. 一份更实用的 runtime 选型清单

1. **主工作负载是什么**；
2. **是否优先追求快速迭代还是极致特化**；
3. **对多模型、多 LoRA、agent 编排的需求有多强**；
4. **cache、batch、kernel 路径是否足够可观测**；
5. **部署链路和版本维护成本是否可接受**。

## 28. 小结

`vLLM`、`SGLang`、`TensorRT-LLM` 的差异，真正值得比较的不是“谁绝对更强”，而是**谁的默认假设、部署方式、可观测性和工作负载契合度更适合你的系统**。只有把 runtime 放回真实流量、真实路由和真实运维约束里，选型才不会停留在 benchmark 层面。

## 快速代码示例

```python
def choose_runtime(req):
    if req.need_static_graph and req.shape_stable:
        return "tensorrt-llm"
    if req.multi_lora or req.long_context:
        return "vllm"
    return "sglang"  # 适合更强的编排和工具链协同
```

这段代码示意了一个基于请求特征的 runtime 选择器：先看是否适合静态图特化，再看长上下文和多 LoRA 需求，最后回落到通用编排路径。真实系统里通常会把这个策略和线上观测指标绑定成可回滚规则。


## 补充：用服务系统视角重写 **vLLM、SGLang、TensorRT-LLM 与推理运行时栈**

推理主题最容易被写成一串优化名词，但线上系统真正关心的从来不是“有没有这个技巧”，而是 **它是否改变了请求、队列、缓存、成本和高价值流量的命运**。围绕 **vLLM、SGLang、TensorRT-LLM 与推理运行时栈**，更稳的理解框架是先问：这件事究竟在改变哪一层约束。

1. **runtime 假设、cache 组织、可观测性、集成成本、混合负载弹性**。

只要这几层没有被明确分开，系统就很容易出现一种假象：离线 benchmark 更好了，线上却并没有更稳；或者平均时延下降了，但高价值租户与长尾任务反而更差。对推理系统来说，这类错觉比“完全没有优化”更危险，因为它会把团队引向错误的资源分配方向。

## 补充：更扎实的设计顺序

针对 **vLLM、SGLang、TensorRT-LLM 与推理运行时栈**，建议把设计顺序固定成下面几步：

1. **先画请求生命周期图**，明确哪些阶段会受这项机制影响；
2. **再画资源账本**，把 GPU、显存、KV、网络、工具链和 CPU 依赖拆开；
3. **然后才讨论策略本身**，例如要不要路由、要不要 cache、要不要 speculative、要不要分层服务；
4. **为高价值流量单独建桶**，不要只看总体平均；
5. **最后定义回退与降级**，确保高峰期和故障期也有可执行动作。

这个顺序的价值在于：它迫使团队先把 **vLLM、SGLang、TensorRT-LLM 与推理运行时栈** 放进完整服务图里，再讨论局部技巧，否则很容易陷入“看起来懂很多优化，但不知道哪个最值钱”的状态。

## 补充：真实线上最常见的失败模式

围绕 **vLLM、SGLang、TensorRT-LLM 与推理运行时栈**，线上失败常常并不体现在单次崩溃，而是慢性失真：

1. **只看单模型 benchmark**。
2. **忽略编排成本**。
3. **上线后缺乏路径级追踪**。

这些问题之所以难，是因为它们往往混在一起发生。一个路由变化可能同时影响缓存命中、TTFT、单位成本和用户重试率；一个 cache 策略调整可能同时影响长上下文任务、KV 生命周期和多租户公平性。因此，**vLLM、SGLang、TensorRT-LLM 与推理运行时栈** 相关改动必须天然携带可观测性，否则等于把复杂度埋进黑箱。

## 补充：更像生产系统的验收标准

对 **vLLM、SGLang、TensorRT-LLM 与推理运行时栈**，更靠谱的验收通常不是一句“线上变快了”，而是至少同时回答：

1. **用真实工作负载选型**。
2. **保留 shadow 跑法**。
3. **统计异常 shape**。
4. **把 runtime 决策暴露到 trace**。

再进一步，建议把这类主题纳入持续治理：

1. **按任务桶和租户桶做长期趋势跟踪**；
2. **保留代表性 replay / shadow case**，用于版本切换对照；
3. **把高价值失败样本回流到训练和系统修复**；
4. **让 oncall 面板能直接看到与它相关的 route、cache、queue 和 fallback 状态**。

只要这套闭环建立起来，**vLLM、SGLang、TensorRT-LLM 与推理运行时栈** 才真正从“优化点”变成“系统能力”。
