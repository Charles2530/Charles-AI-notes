# 推理优化清单与容量规划

推理系统一旦上线，团队很快会遇到两个问题：

1. 现在还能撑多久？
2. 下一步该先优化哪里？

这两个问题表面上像是“算资源”，本质上却是在做系统建模。  
因为大模型推理的容量，不是一个固定常数，而是下面这些因素共同作用的结果：

- 请求到达分布
- 输入和输出 token 分布
- 模型结构与量化方式
- batch 策略
- prefix / KV cache 命中率
- 工具调用与外部依赖
- 用户可接受的延迟上限

所以这页把推理优化从“零散技巧”整理成一套更接近容量规划和瓶颈归因的框架。

## 1. 容量规划先从请求画像开始

设系统接收请求到达率为 \(\lambda\)，平均服务时间为 \(S\)，则最基本的负载强度概念是：

\[
\rho = \lambda S.
\]

虽然真实系统远比单服务器队列复杂，但这个直觉非常重要：

- 请求到达更快，会推高 \(\rho\)
- 平均服务时间更长，也会推高 \(\rho\)

一旦 \(\rho\) 接近甚至超过系统的有效处理能力，排队时间会急剧上升，尾延迟会先于平均延迟失控。

### 一个容易被忽略的点

在 LLM/VLM 系统里，\(S\) 不是固定值，而常常近似满足：

\[
S \approx T_{\text{prefill}}(L_{\text{in}})
 + T_{\text{decode}}(L_{\text{out}})
 + T_{\text{tool}}
 + T_{\text{network}},
\]

其中：

- \(L_{\text{in}}\) 是输入长度
- \(L_{\text{out}}\) 是输出长度
- \(T_{\text{tool}}\) 是检索、OCR、浏览器、数据库等外部调用耗时
- \(T_{\text{network}}\) 是服务编排和网络开销

**这意味着**：  
你以为自己在做“模型容量规划”，实际上常常是在给整个多段链路做容量规划。

## 2. 先分清三类请求，而不是看一个平均值

容量规划时，建议至少把请求分成三类：

- 短请求
- 中请求
- 长请求

如果把它们混成一个平均值，很多系统判断会失真。

### 一个直观例子

100 个短问答和 5 个超长 PDF 分析，平均 token 数也许看起来还好，但系统真实负担完全不同。

对一个大模型服务来说，下面两组流量即使平均输入长度相同，容量压力也可能差很多：

- **组 A**：全部是 400 token 的普通问答
- **组 B**：90% 是 80 token，10% 是 8000 token 的长文档分析

**组 B 的问题在于**：

- 长请求占用显存和 queue 时间更久
- continuous batching 更容易被大请求拖慢
- cache 生命周期更长
- tail latency 更容易被少量长请求拉高

## 3. 先把推理时间拆成 prefill 和 decode 两段

这是容量规划里最重要但最常被略过的一步。

### 3.1 Prefill 阶段

Prefill 需要一次性处理全部输入 token。  
通常更偏计算密集和显存带宽敏感，尤其在长上下文时成本很高。

### 3.2 Decode 阶段

Decode 每次只生成一个或少量 token，但要重复很多轮。  
它往往更偏 memory-bound，并深受：

- batch 动态变化
- KV cache 访问
- 并发请求数量

影响。

**因此一个常见的近似分解是**：

\[
T_{\text{total}}
=
T_{\text{queue}}
 + T_{\text{prefill}}
 + T_{\text{decode}}.
\]

很多团队会把“延迟高”笼统归因于模型太大，但更准确的问法应该是：

- 是 queue 太长？
- 是 prefill 太重？
- 是 decode 太慢？

## 4. 优化清单应该按层来排

推理优化最怕“想到什么做什么”。  
更稳的做法是分四层。

### 第 1 层：请求层

**关注**：

- 输入长度
- 输出长度
- 检索拼接长度
- 多图或多模态输入大小
- 并发模式

如果请求层就失控，后面的优化常常只是延缓问题。

### 第 2 层：调度层

**关注**：

- continuous batching
- queue 策略
- priority scheduling
- route 策略
- prefix cache 策略

这层决定系统能否把 GPU 时间切得足够“满”。

### 第 3 层：模型层

**关注**：

- 量化
- speculative decoding
- early exit / draft model
- 更小模型替换
- prompt template 收缩

这层决定单次请求本身要花多少模型计算。

### 第 4 层：系统层

**关注**：

- cache
- kernel
- 内存管理
- 异构加速
- 横向扩容

这层通常最重，但并不总是最先该做。

## 5. 为什么“先优化哪里”比“会不会优化”更关键

很多团队一上来就做最酷的 kernel 或 speculative，但真正瓶颈其实可能只是：

- prompt 太长
- route 不合理
- cache 命中低
- RAG 拼接过量
- 文本模板过于冗长

**这类问题的特点是**：

- 不需要重写底层系统
- 往往一周内就能显著见效
- 解决后还会让后续底层优化更有效

所以更稳的做法是先做瓶颈归因，再决定优化顺序。

## 6. 一份更完整的推理优化 checklist

建议按下面顺序逐项检查。

### 6.1 请求设计

1. 输入是不是过长？
2. 检索片段是不是过多？
3. system prompt 是否在重复堆砌规范？
4. 是否把不必要的工具日志也送回模型？
5. 多模态输入分辨率是否过高？

### 6.2 调度与队列

1. batch 和 queue 是否合理？
2. 长请求是否和短请求混在同一队列？
3. 是否需要优先级队列？
4. prefix 相同的请求是否被重算？
5. 是否能根据请求画像做路由分桶？

### 6.3 模型与解码

1. KV cache 是否爆显存？
2. prefix cache 是否可用？
3. speculative 是否真的带来收益？
4. 量化是否在真实业务下仍稳？
5. 采样参数是否导致输出过长？
6. 是否可以用更小模型处理简单请求？

### 6.4 系统与基础设施

1. kernel 是否和当前量化方式匹配？
2. GPU 利用率低的根因是计算不足还是调度不足？
3. NCCL / 多卡通信是否拖慢整体吞吐？
4. 外部依赖是否成为整个链路瓶颈？
5. 是否需要做冷热流量拆分？

## 7. 用一个简化模型做容量估算

若系统共有 \(M\) 张等效 GPU，每张卡在当前模型和配置下的稳定吞吐为 \(r\) tokens/s，则理想情况下的总 decode 能力约为：

\[
C_{\text{decode}} \approx M r.
\]

若平均每个请求输出 \(\mathbb{E}[L_{\text{out}}]\) 个 token，则在 decode 主导的场景下，粗略请求吞吐可近似为：

\[
QPS \approx \frac{M r}{\mathbb{E}[L_{\text{out}}]}.
\]

但这只是乐观上界，因为真实系统还会被以下因素折损：

\[
QPS_{\text{real}}
 \approx
 \eta_{\text{sched}}
 \cdot
 \eta_{\text{cache}}
 \cdot
 \eta_{\text{network}}
 \cdot
 \eta_{\text{tool}}
 \cdot
 \frac{M r}{\mathbb{E}[L_{\text{out}}]},
\]

其中 \(\eta\) 都是小于 1 的效率系数。  
真实容量规划的难点，恰恰在于这些效率系数经常波动很大。

## 8. 一个更贴近业务的例子

假设你有 4 张 GPU，某模型在当前 batch 策略下 decode 吞吐约为每卡 140 token/s。  
平均每个请求输出 280 token。  
不考虑 prefill 和外部工具时，理想上界约为：

\[
QPS \approx \frac{4 \times 140}{280} = 2.
\]

但上线后你发现实际只有 1.1 QPS。  
**排查后可能发现**：

- 20% 请求带 6k token 的检索上下文，prefill 很重
- prefix cache 命中率只有 8%
- 30% 请求会调用外部检索和 OCR
- 长请求和短请求混队，导致尾部拖慢

这说明“算力不足”只是表象，根因在于请求分布和系统分层设计。

## 9. 先抓哪类优化，通常有一个经验顺序

### 第一优先级：先压无效 token

这是性价比最高的一层。  
**常见动作包括**：

- 缩短 system prompt
- 精简 tool schema
- 减少无效历史消息
- 控制 RAG 拼接数量
- 只保留必要 citation 片段

如果单次请求的 token 数下降 30%，很多时候收益比复杂 speculative 还直接。

### 第二优先级：把请求分层

把请求按长度、业务类型、模型需求拆开后，很多瓶颈会立刻清晰：

- FAQ 走小模型
- 复杂写作走大模型
- OCR 请求走专门的文档模型
- 超长上下文请求单独排队

### 第三优先级：优化缓存和 batching

这层往往是系统真正的吞吐增益来源，包括：

- prefix cache
- KV cache 管理
- continuous batching
- admission control

### 第四优先级：再上更重的模型优化

例如：

- speculative decoding
- 激进量化
- draft model
- 并行解码新策略

因为这些优化更依赖基础设施、兼容性和回归验证。

## 10. 容量规划里最容易漏掉的东西

### 10.1 尾延迟

平均值好看，不代表系统稳。  
用户真正感知到的卡顿，往往是 P95 或 P99。

### 10.2 版本升级后的 embedding / cache 漂移

模型一换，上游和下游都可能受影响，例如：

- embedding 相似度分布变了
- prefix cache 不再命中
- 检索召回更长或更短
- 工具调用格式变化

### 10.3 工具链外部依赖

Agent 系统里，模型只是链路中的一段，不是全部。  
如果浏览器执行、OCR 服务、数据库或向量库变慢，整条链路就会被拖住。

### 10.4 资源碎片化

即使平均显存看起来够，也可能因为：

- batch 动态变化
- cache 生命周期不同
- 多租户模型混跑

导致实际可用容量明显低于理论容量。

## 11. 一份故障导向的排查表

| 现象 | 更可能的根因 | 第一排查动作 |
| --- | --- | --- |
| 平均延迟正常，P99 爆炸 | 长请求混队、外部依赖慢 | 看队列分布与 trace |
| GPU 利用率低 | 调度不足、batch 太碎 | 看 token/s 与 batch 演化 |
| 显存频繁爆 | KV cache 或多模态输入过大 | 统计 cache 占用和长请求占比 |
| 吞吐下降但质量没变 | prompt 变长或 route 变差 | 对比 token 分布和模型选择 |
| speculative 无收益 | draft 质量低或接受率低 | 看 acceptance rate |
| 量化后更慢 | kernel 不匹配、反量化开销大 | 重新做端到端 profile |

## 12. 一个更完整的监控面板建议

如果你要长期做容量治理，建议监控至少覆盖：

- `QPS`
- `TTFT`
- `TPOT`
- `P50 / P95 / P99 latency`
- 输入 token 分布
- 输出 token 分布
- prefix cache 命中率
- KV cache 占用率
- GPU 利用率
- 外部工具调用耗时
- route 到各模型的流量比例

没有这些指标，很多所谓“优化效果”其实无法验证。

## 13. 这页的一个总判断

推理容量规划的核心不是“算现在还能扛多少 QPS”，而是建立一种分层意识：  
到底是请求、调度、模型、缓存，还是外部依赖在限制系统。  
一旦层次分清，优化顺序才会自然。

真正成熟的团队不会把推理优化理解成“堆几个炫技技巧”，而是把它当成一套长期的容量治理过程：

- 先理解流量和请求结构
- 再识别瓶颈所在层
- 再按投入产出比排优化顺序
- 最后用持续监控确认收益是否真实存在

只有这样，推理系统才会从“临时能跑”走向“长期稳定”。
