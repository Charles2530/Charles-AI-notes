# 推理优化清单与容量规划

推理系统一旦上线，团队很快会遇到两个问题：

1. 现在还能撑多久？
2. 下一步该先优化哪里？

这两个问题表面上像是“算资源”，本质上却是在做系统建模。  
因为大模型推理的容量，不是一个固定常数，而是下面这些因素共同作用的结果：

- 请求到达分布
- 输入和输出 token 分布
- 模型结构与量化方式
- batch 策略
- prefix / KV cache 命中率
- 工具调用与外部依赖
- 用户可接受的延迟上限

所以这页把推理优化从“零散技巧”整理成一套更接近容量规划和瓶颈归因的框架。

## 1. 容量规划先从请求画像开始

设系统接收请求到达率为 \(\lambda\)，平均服务时间为 \(S\)，则最基本的负载强度概念是：

\[
\rho = \lambda S.
\]

虽然真实系统远比单服务器队列复杂，但这个直觉非常重要：

- 请求到达更快，会推高 \(\rho\)
- 平均服务时间更长，也会推高 \(\rho\)

一旦 \(\rho\) 接近甚至超过系统的有效处理能力，排队时间会急剧上升，尾延迟会先于平均延迟失控。

### 一个容易被忽略的点

在 LLM/VLM 系统里，\(S\) 不是固定值，而常常近似满足：

\[
S \approx T_{\text{prefill}}(L_{\text{in}})
 + T_{\text{decode}}(L_{\text{out}})
 + T_{\text{tool}}
 + T_{\text{network}},
\]

其中：

- \(L_{\text{in}}\) 是输入长度
- \(L_{\text{out}}\) 是输出长度
- \(T_{\text{tool}}\) 是检索、OCR、浏览器、数据库等外部调用耗时
- \(T_{\text{network}}\) 是服务编排和网络开销

**这意味着**：  
你以为自己在做“模型容量规划”，实际上常常是在给整个多段链路做容量规划。

## 2. 先分清三类请求，而不是看一个平均值

容量规划时，建议至少把请求分成三类：

- 短请求
- 中请求
- 长请求

如果把它们混成一个平均值，很多系统判断会失真。

### 一个直观例子

100 个短问答和 5 个超长 PDF 分析，平均 token 数也许看起来还好，但系统真实负担完全不同。

对一个大模型服务来说，下面两组流量即使平均输入长度相同，容量压力也可能差很多：

- **组 A**：全部是 400 token 的普通问答
- **组 B**：90% 是 80 token，10% 是 8000 token 的长文档分析

**组 B 的问题在于**：

- 长请求占用显存和 queue 时间更久
- continuous batching 更容易被大请求拖慢
- cache 生命周期更长
- tail latency 更容易被少量长请求拉高

## 3. 先把推理时间拆成 prefill 和 decode 两段

这是容量规划里最重要但最常被略过的一步。

### 3.1 Prefill 阶段

Prefill 需要一次性处理全部输入 token。  
通常更偏计算密集和显存带宽敏感，尤其在长上下文时成本很高。

### 3.2 Decode 阶段

Decode 每次只生成一个或少量 token，但要重复很多轮。  
它往往更偏 memory-bound，并深受：

- batch 动态变化
- KV cache 访问
- 并发请求数量

影响。

**因此一个常见的近似分解是**：

\[
T_{\text{total}}
=
T_{\text{queue}}
 + T_{\text{prefill}}
 + T_{\text{decode}}.
\]

很多团队会把“延迟高”笼统归因于模型太大，但更准确的问法应该是：

- 是 queue 太长？
- 是 prefill 太重？
- 是 decode 太慢？

## 4. 优化清单应该按层来排

推理优化最怕“想到什么做什么”。  
更稳的做法是分四层。

### 第 1 层：请求层

**关注**：

- 输入长度
- 输出长度
- 检索拼接长度
- 多图或多模态输入大小
- 并发模式

如果请求层就失控，后面的优化常常只是延缓问题。

### 第 2 层：调度层

**关注**：

- continuous batching
- queue 策略
- priority scheduling
- route 策略
- prefix cache 策略

这层决定系统能否把 GPU 时间切得足够“满”。

### 第 3 层：模型层

**关注**：

- 量化
- speculative decoding
- early exit / draft model
- 更小模型替换
- prompt template 收缩

这层决定单次请求本身要花多少模型计算。

### 第 4 层：系统层

**关注**：

- cache
- kernel
- 内存管理
- 异构加速
- 横向扩容

这层通常最重，但并不总是最先该做。

## 5. 为什么“先优化哪里”比“会不会优化”更关键

很多团队一上来就做最酷的 kernel 或 speculative，但真正瓶颈其实可能只是：

- prompt 太长
- route 不合理
- cache 命中低
- RAG 拼接过量
- 文本模板过于冗长

**这类问题的特点是**：

- 不需要重写底层系统
- 往往一周内就能显著见效
- 解决后还会让后续底层优化更有效

所以更稳的做法是先做瓶颈归因，再决定优化顺序。

## 6. 一份更完整的推理优化 checklist

建议按下面顺序逐项检查。

### 6.1 请求设计

1. 输入是不是过长？
2. 检索片段是不是过多？
3. system prompt 是否在重复堆砌规范？
4. 是否把不必要的工具日志也送回模型？
5. 多模态输入分辨率是否过高？

### 6.2 调度与队列

1. batch 和 queue 是否合理？
2. 长请求是否和短请求混在同一队列？
3. 是否需要优先级队列？
4. prefix 相同的请求是否被重算？
5. 是否能根据请求画像做路由分桶？

### 6.3 模型与解码

1. KV cache 是否爆显存？
2. prefix cache 是否可用？
3. speculative 是否真的带来收益？
4. 量化是否在真实业务下仍稳？
5. 采样参数是否导致输出过长？
6. 是否可以用更小模型处理简单请求？

### 6.4 系统与基础设施

1. kernel 是否和当前量化方式匹配？
2. GPU 利用率低的根因是计算不足还是调度不足？
3. NCCL / 多卡通信是否拖慢整体吞吐？
4. 外部依赖是否成为整个链路瓶颈？
5. 是否需要做冷热流量拆分？

## 7. 用一个简化模型做容量估算

若系统共有 \(M\) 张等效 GPU，每张卡在当前模型和配置下的稳定吞吐为 \(r\) tokens/s，则理想情况下的总 decode 能力约为：

\[
C_{\text{decode}} \approx M r.
\]

若平均每个请求输出 \(\mathbb{E}[L_{\text{out}}]\) 个 token，则在 decode 主导的场景下，粗略请求吞吐可近似为：

\[
QPS \approx \frac{M r}{\mathbb{E}[L_{\text{out}}]}.
\]

但这只是乐观上界，因为真实系统还会被以下因素折损：

\[
QPS_{\text{real}}
 \approx
 \eta_{\text{sched}}
 \cdot
 \eta_{\text{cache}}
 \cdot
 \eta_{\text{network}}
 \cdot
 \eta_{\text{tool}}
 \cdot
 \frac{M r}{\mathbb{E}[L_{\text{out}}]},
\]

其中 \(\eta\) 都是小于 1 的效率系数。  
真实容量规划的难点，恰恰在于这些效率系数经常波动很大。

## 8. 一个更贴近业务的例子

假设你有 4 张 GPU，某模型在当前 batch 策略下 decode 吞吐约为每卡 140 token/s。  
平均每个请求输出 280 token。  
不考虑 prefill 和外部工具时，理想上界约为：

\[
QPS \approx \frac{4 \times 140}{280} = 2.
\]

但上线后你发现实际只有 1.1 QPS。  
**排查后可能发现**：

- 20% 请求带 6k token 的检索上下文，prefill 很重
- prefix cache 命中率只有 8%
- 30% 请求会调用外部检索和 OCR
- 长请求和短请求混队，导致尾部拖慢

这说明“算力不足”只是表象，根因在于请求分布和系统分层设计。

## 9. 先抓哪类优化，通常有一个经验顺序

### 第一优先级：先压无效 token

这是性价比最高的一层。  
**常见动作包括**：

- 缩短 system prompt
- 精简 tool schema
- 减少无效历史消息
- 控制 RAG 拼接数量
- 只保留必要 citation 片段

如果单次请求的 token 数下降 30%，很多时候收益比复杂 speculative 还直接。

### 第二优先级：把请求分层

把请求按长度、业务类型、模型需求拆开后，很多瓶颈会立刻清晰：

- FAQ 走小模型
- 复杂写作走大模型
- OCR 请求走专门的文档模型
- 超长上下文请求单独排队

### 第三优先级：优化缓存和 batching

这层往往是系统真正的吞吐增益来源，包括：

- prefix cache
- KV cache 管理
- continuous batching
- admission control

### 第四优先级：再上更重的模型优化

例如：

- speculative decoding
- 激进量化
- draft model
- 并行解码新策略

因为这些优化更依赖基础设施、兼容性和回归验证。

## 10. 容量规划里最容易漏掉的东西

### 10.1 尾延迟

平均值好看，不代表系统稳。  
用户真正感知到的卡顿，往往是 P95 或 P99。

### 10.2 版本升级后的 embedding / cache 漂移

模型一换，上游和下游都可能受影响，例如：

- embedding 相似度分布变了
- prefix cache 不再命中
- 检索召回更长或更短
- 工具调用格式变化

### 10.3 工具链外部依赖

Agent 系统里，模型只是链路中的一段，不是全部。  
如果浏览器执行、OCR 服务、数据库或向量库变慢，整条链路就会被拖住。

### 10.4 资源碎片化

即使平均显存看起来够，也可能因为：

- batch 动态变化
- cache 生命周期不同
- 多租户模型混跑

导致实际可用容量明显低于理论容量。

## 11. 一份故障导向的排查表

| 现象 | 更可能的根因 | 第一排查动作 |
| --- | --- | --- |
| 平均延迟正常，P99 爆炸 | 长请求混队、外部依赖慢 | 看队列分布与 trace |
| GPU 利用率低 | 调度不足、batch 太碎 | 看 token/s 与 batch 演化 |
| 显存频繁爆 | KV cache 或多模态输入过大 | 统计 cache 占用和长请求占比 |
| 吞吐下降但质量没变 | prompt 变长或 route 变差 | 对比 token 分布和模型选择 |
| speculative 无收益 | draft 质量低或接受率低 | 看 acceptance rate |
| 量化后更慢 | kernel 不匹配、反量化开销大 | 重新做端到端 profile |

## 12. 一个更完整的监控面板建议

如果你要长期做容量治理，建议监控至少覆盖：

- `QPS`
- `TTFT`
- `TPOT`
- `P50 / P95 / P99 latency`
- 输入 token 分布
- 输出 token 分布
- prefix cache 命中率
- KV cache 占用率
- GPU 利用率
- 外部工具调用耗时
- route 到各模型的流量比例

没有这些指标，很多所谓“优化效果”其实无法验证。

## 13. 这页的一个总判断

推理容量规划的核心不是“算现在还能扛多少 QPS”，而是建立一种分层意识：  
到底是请求、调度、模型、缓存，还是外部依赖在限制系统。  
一旦层次分清，优化顺序才会自然。

真正成熟的团队不会把推理优化理解成“堆几个炫技技巧”，而是把它当成一套长期的容量治理过程：

- 先理解流量和请求结构
- 再识别瓶颈所在层
- 再按投入产出比排优化顺序
- 最后用持续监控确认收益是否真实存在

只有这样，推理系统才会从“临时能跑”走向“长期稳定”。

## 14. 容量规划必须显式保留 Headroom

很多系统在平均负载下能跑得很好，却在波峰时突然崩，根因往往不是规划没算，而是**默认把系统按接近满载来设计**。

### 14.1 为什么推理系统更需要 headroom

1. **请求长度分布高度波动**；
2. **KV cache 占用有长尾**；
3. **工具调用和外部依赖不稳定**；
4. **模型路由和 batch 结构会随业务形态变化**；
5. **故障时 fallback 流量会额外涌入其他池**。

### 14.2 一个更实用的 headroom 观念

不要只问“理论最大 QPS”，而要问：

1. **在 P95 长度分布下还能撑多少**；
2. **当 prefix 命中下降时还能撑多少**；
3. **当某个池故障需要溢出到邻池时还能撑多少**；
4. **当工具链或 OCR 变慢时还能撑多少**。

## 15. 队列预算应该按阶段拆，不应只盯总延迟

若只给总延迟设预算，团队很难知道该从哪层挤时间。更稳妥的做法是把预算拆给各阶段。

### 15.1 一个常见拆法

1. **入口与预处理预算**；
2. **queue 预算**；
3. **prefill 预算**；
4. **decode 预算**；
5. **tool/post-processing 预算**。

### 15.2 为什么这很值钱

如果发现：

1. **prefill 预算经常超**，应看长上下文和 prefix cache；
2. **queue 预算经常超**，应看 admission control、分池和 batch 策略；
3. **tool 预算经常超**，应看外部依赖和异步化，而不是继续优化 GPU。

## 16. Token Mix Planning 比平均长度更重要

许多团队会统计平均输入长度和平均输出长度，但真正决定容量的通常是**token mix**。

### 16.1 什么是 token mix

可以粗略理解为不同请求类型在资源上的组合：

1. **短输入短输出**；
2. **长输入短输出**；
3. **短输入长输出**；
4. **长输入长输出**；
5. **多模态大附件 + 短文本输出**；
6. **tool-heavy agent 任务**。

### 16.2 为什么 token mix 比平均值重要

因为：

1. **长输入短输出主要吃 prefill**；
2. **短输入长输出主要吃 decode**；
3. **长输入长输出同时吃 queue、显存和 wall time**；
4. **多模态请求会把 CPU、网络和 GPU 一起拉进瓶颈**。

## 17. 容量规划要回答“峰值时是谁最值钱”

一旦系统进入高峰期，不是所有流量都该被同等对待。

### 17.1 需要事先定义的优先级问题

1. **哪些请求必须实时返回**；
2. **哪些可以异步或降级**；
3. **哪些租户和场景价值最高**；
4. **哪些请求最贵但收益最低**；
5. **哪些能力可以在高峰时临时关闭**。

### 17.2 为什么这不是产品问题，而是系统问题

如果没有优先级策略，系统就只能用一种最笨的方法处理高峰：所有请求一起排队一起慢，最后伤害最高价值用户。

## 18. Prefix 命中率需要进入容量预算

很多系统在规划容量时默认 prefix cache 命中率稳定，但实际业务里它会随着 prompt 变化、用户分布和模板升级而显著波动。

### 18.1 这会带来什么

1. **prefill 成本突然上升**；
2. **TTFT 变差**；
3. **显存与 cache 生命周期变化**；
4. **某些原本可复用的工作重新变成重复计算**。

### 18.2 一个更稳妥的做法

容量预算至少要有三档：

1. **高命中场景**；
2. **中等命中场景**；
3. **低命中或几乎无命中场景**。

只按第一档做规划，系统会在真实业务变动时变得很脆。

## 19. Canary 和 Rollback 也属于容量治理

模型或 runtime 升级时，很多事故不是“结果错”，而是**容量画像变了**。

### 19.1 Canary 时至少应观察

1. **TTFT / TPOT 是否变化**；
2. **KV cache 占用和碎片率是否变化**；
3. **route 到新版本后的队列结构是否变化**；
4. **工具调用链路是否出现更多超时**；
5. **不同长度桶里的 P95 / P99 是否恶化**。

### 19.2 Rollback 条件不能只看平均延迟

很多回滚太慢，是因为团队等到平均值也崩了才行动。更好的做法是：

1. **给关键租户和关键长度桶设独立门槛**；
2. **给尾延迟和拒绝率设硬阈值**；
3. **把 cache 命中和显存异常也纳入回滚条件**。

## 20. 容量规划最终要落到成本账本

如果没有成本账本，团队就很难判断某次优化是不是值得。

### 20.1 建议至少记录

1. **每请求平均 GPU 秒数**；
2. **每输出 token 成本**；
3. **每种任务类型的单位成本**；
4. **高峰期额外扩容成本**；
5. **fallback、异步化或小模型路由带来的成本节省**。

### 20.2 为什么这能改变优化顺序

有些优化让 benchmark 很漂亮，但对整体成本几乎没影响；有些策略虽然不炫，比如把一类长尾请求异步化，却能立刻把高峰成本打下来。

## 21. 一份故障期应急剧本

容量治理不只是平时算预算，还要准备系统进入拥塞时怎么活下来。

### 21.1 常见应急动作

1. **提高 admission 阈值，拒绝最低价值流量**；
2. **把超长请求改成异步**；
3. **关闭最昂贵的工具链或附件解析**；
4. **把部分请求路由到更小模型**；
5. **缩短最大输出长度**；
6. **临时冻结实验和灰度流量**。

### 21.2 为什么要提前演练

故障发生时没有人有空现场发明策略。没有提前演练的降级路径，大概率会在真正需要时露出新的 bug。

## 22. 一份更工程化的容量治理顺序

1. **先按请求类型画出 token mix**；
2. **再按阶段拆延迟和成本**；
3. **确认是 queue、prefill、decode、cache 还是外部依赖在主导**；
4. **为不同池建立 headroom 和 overflow 规则**；
5. **把 canary、rollback 和故障剧本并入容量治理**；
6. **最后才去讨论更深的 kernel 或模型优化**。

### 22.1 为什么这是更稳的顺序

因为很多团队一上来就卷 kernel，却没发现真实问题是流量结构已经变了，或者请求分层做得太差。

## 23. 小结

推理容量规划的核心，不是算出一个漂亮的“最大 QPS”，而是建立一套**能承受流量变化、命中率变化、模型升级和故障溢出的长期治理机制**。谁能把 headroom、token mix、回滚门槛和成本账本一起纳入设计，谁的服务系统就更不容易在真实业务里失真。
