# QAT、Kernel 与 KV Cache

很多量化讨论只停留在“精度掉不掉”，但真实部署里，量化是否成功还取决于三件紧密耦合的事：

1. `QAT`
2. `Kernel`
3. `KV Cache`

如果只看前两者而不看第三者，很容易出现“权重量化做完了，系统却还是不够快也不够省”的情况。这一页要回答的核心问题是：

1. `QAT` 到底在优化什么；
2. 为什么很多论文里的低比特方案上线后并没有兑现吞吐；
3. 为什么长上下文服务最终常常被 `KV cache` 重新拉回系统瓶颈。

## 1. 量化问题的统一写法

**最基础的均匀量化可写成**：

$$
q = \mathrm{clip}\left(\left\lfloor \frac{x}{s} \right\rceil + z, q_{\min}, q_{\max}\right),
\qquad
\hat{x} = s(q-z),
$$

其中：

- \(s\) 是 scale；
- \(z\) 是 zero-point；
- \(q\) 是量化后的整数表示；
- \(\hat{x}\) 是反量化后的近似值。

真正部署时，不同对象会有不同粒度：

- per-tensor；
- per-channel；
- per-group；
- per-token；
- per-head。

后续所有 `QAT`、kernel 和 cache 讨论，本质上都绕不开这个近似误差如何传播的问题。

## 2. 为什么需要 QAT

后训练量化 `PTQ` 足够便宜，但在极低比特或激活量化场景中，往往不够稳。  
`QAT` 的基本思路是在训练时显式把量化误差模拟进来：

$$
\hat{x} = \mathrm{Dequant}(\mathrm{Quant}(x))
$$

然后让前向过程始终在这种近似低比特环境里运行。

直觉上，模型不再是“训练完再被突然压缩”，而是在训练期间就学会适应量化噪声。

## 3. QAT 真正做的不是“重新训一遍”

很多人把 `QAT` 理解成“拿原模型再训练一下”，但更准确地说，它是在重新优化以下几件事的兼容性：

1. 权重分布与量化区间是否匹配；
2. 激活离群值是否可控；
3. 各层误差传播是否可被后续层吸收；
4. 低比特环境下 attention、MLP 和输出头是否还能保持数值稳定。

也就是说，`QAT` 不是单纯把精度损失补回来，而是在训练阶段让模型参数主动适应低比特约束。

## 4. Fake quant 与 STE

训练时常见做法是 fake quant：前向按量化后的 \(\hat{x}\) 走，反向则使用近似梯度。  
最常见近似是 `STE`（Straight-Through Estimator），可粗略理解为：

$$
\frac{\partial \hat{x}}{\partial x} \approx 1
$$

在未被 clip 的区间内直接把梯度传回去。

### 4.1 为什么需要 STE

因为真正的取整和截断操作不可导。  
若没有 `STE` 或其他近似梯度，模型几乎没法在量化算子存在时稳定训练。

### 4.2 STE 不是严格正确，但很实用

`STE` 并不意味着量化问题被理论上完美解决，而是在工程上提供了一种可训练近似。  
这也解释了为什么 `QAT` 很依赖：

- 学习率；
- scale 初始化；
- clipping 范围；
- 哪些层参与量化、哪些层保留高精。

## 5. QAT 主要在哪些场景更值得

### 5.1 极低比特部署

如 `4bit` 甚至更低。

### 5.2 激活量化

因为激活分布更动态，`PTQ` 容易失稳。

### 5.3 高精度损失代价很大的业务

例如结构化抽取、长推理、专业问答、多轮工具调用。

### 5.4 想同时处理权重、激活和 cache

单独压权重常不够，而 `QAT` 更适合在训练期一起适配多种低精度对象。

## 6. QAT 的现实代价

`QAT` 的问题也非常现实：

- 训练更贵；
- 调参更难；
- 数值稳定性更敏感；
- 不同 kernel 约束会反过来影响训练目标。

所以它不总是第一选择，而更像在“`PTQ` 不够稳时的下一层保险”。  
真正值得做 `QAT` 的前提通常是：

1. 目标量化格式已经基本确定；
2. kernel 栈基本可用；
3. 业务确实需要再压一层成本。

## 7. Kernel 为什么会决定量化是否值钱

一个量化方案理论上很优，不代表实际部署一定快。  
真正速度和吞吐的落点，取决于内核实现：

- 是否支持该 bit 宽；
- 是否支持 group size / packing 方式；
- 是否支持 fused kernel；
- 是否与现有 serving 框架兼容；
- 是否能避免频繁 dequant-requant；
- 是否能高效利用显存带宽和寄存器。

因此很多时候，工程选择不是“最优量化论文方案”，而是“在当前硬件和 kernel 上最值得跑的方案”。

### 一个直观例子

某种分组 `4bit` 方法在论文里质量最好，但如果现有 kernel 对它支持很差，那么线上吞吐可能还不如一个稍逊但实现成熟的方案。  
原因不在数学，而在执行链：

1. 数据布局不匹配；
2. 需要额外 unpack；
3. 无法和 `GEMM`、bias、激活函数做融合；
4. dequant 成本把理论收益吃掉。

## 8. 为什么理论压缩比经常兑现不了

很多团队看到“权重从 `16bit` 到 `4bit`，理论缩小 `4x`”，就自然期待显存和时延都接近 `4x` 改善。现实里常不会这样，原因包括：

1. 并非所有层都能压到同一比特；
2. metadata、scale、zero-point 也占空间；
3. 中间激活和 `KV cache` 仍可能是大头；
4. kernel 若不高效，运算和访存仍是瓶颈；
5. 调度、batching、分页缓存会带来额外系统开销。

因此量化收益应始终在真实 serving 链路里测，而不是只看离线模型文件大小。

## 9. QAT 与 kernel 不是两张表，而是一条链

真正的部署优化通常不是单点，而是：

$$
\text{weights} + \text{activations} + \text{cache} + \text{kernel} + \text{scheduling}
$$

只要其中某一环没跟上，整体收益就可能被吃掉。  
比如：

- 训练时假设 per-channel `INT4`；
- 线上 kernel 只能高效跑另一种 packing；
- 那么最终部署格式可能和训练假设不一致。

这也是为什么成熟团队会在量化设计早期就同时拉上训练、runtime 和服务工程一起看。

## 10. KV Cache 为什么会把量化问题重新拉回系统层

即使权重已经压到 `4bit`，长上下文服务中显存大头仍可能是：

$$
\text{KV bytes} \approx 2 \times B \times L \times H \times d \times T \times b,
$$

这里：

- \(B\) 是并发请求数；
- \(L\) 是层数；
- \(H\) 是头数；
- \(d\) 是每头维度；
- \(T\) 是上下文长度；
- \(b\) 是每元素字节数。

**因此你可能遇到一种常见现象**：

- 权重量化做得很好；
- 但长文档服务一跑并发，还是爆显存。

这时候真正该考虑的，往往不是再压权重，而是：

- `KV cache` 精度；
- cache 管理策略；
- prefix reuse；
- paged attention；
- 长上下文路由。

## 11. 权重量化与 KV cache 量化不是同一个问题

### 11.1 权重量化

静态、离线、好校准。

### 11.2 KV cache 量化

动态、在线、对生成稳定性影响更直接。

尤其在长上下文和多轮对话里，`KV cache` 的量化若过激，可能会：

- 降低长期一致性；
- 让长链推理更不稳；
- 影响早期 token 的长期记忆；
- 让 attention 分布更尖锐或更散。

所以它常需要单独验证，而不能想当然地“权重能 `4bit`，cache 也该 `4bit`”。

## 12. KV cache 量化里最容易被忽略的细节

### 12.1 不同头和不同 token 的动态范围不同

若只用粗粒度 scale，局部高能量 token 容易被截断。

### 12.2 旧 token 的误差会长期留存

权重误差每次前向都会重新参与计算；  
而 cache 误差会在后续大量步里反复被读取，因此影响可能被放大。

### 12.3 长上下文任务更敏感

例如：

- 文档问答；
- 代码补全；
- 长链工具调用；
- 多轮代理状态跟踪。

这类任务往往比普通短对话更容易暴露 `KV cache` 量化问题。

## 13. 一个更现实的部署视角

长上下文系统里，真正值得问的问题通常是：

1. 权重量化后模型能否塞进目标硬件；
2. cache 是否成为新瓶颈；
3. kernel 是否支持目标精度与缓存格式；
4. page/block 管理是否和量化格式兼容；
5. 在真实并发、真实上下文长度和真实 batch 下是否仍稳。

只回答第 1 个问题，通常远远不够。

## 14. 三种典型场景

### 场景 1：单卡小模型部署

重点先看权重量化和 kernel 兼容。

### 场景 2：长上下文企业问答

重点看 `KV cache`、prefix cache 和 route 策略。

### 场景 3：低资源微调后上线

重点是 `QLoRA` 产物如何与实际量化格式、kernel 和 serving 栈接起来。

## 15. 常见失效模式

### 15.1 离线基准很好，线上吞吐没起来

多半是 kernel、packing 或调度没有兑现理论收益。

### 15.2 短任务没问题，长任务明显漂

多半要检查 `KV cache` 量化和缓存管理。

### 15.3 QAT 训练稳定，但导出格式后质量掉

说明训练期假设和部署期实际算子不一致。

### 15.4 压权重有效，整体显存却没降多少

说明真正的大头已经转移到激活或 cache。

## 16. 工程建议

如果你在做量化部署，建议按顺序排查：

1. 权重量化是否稳定；
2. 激活和关键层是否需要 `QAT`；
3. kernel 是否真正带来吞吐收益；
4. `KV cache` 是否成为新瓶颈；
5. 长上下文和高并发下是否仍稳；
6. 训练假设、导出格式和线上 kernel 是否完全一致。

这几步比单看一个离线精度表更接近真实上线问题。

## 17. 小结

`QAT`、kernel 与 `KV cache` 把量化从一个“数值压缩问题”变成了一个“训练、运行时和服务系统的联动问题”。真正成熟的量化方案，不是论文表里某一列最低 bit，而是：

- 训练时能适应；
- kernel 上能跑快；
- cache 上能省住；
- 线上长任务和高并发下仍可控。

只有这四件事同时成立，量化才真正从方法变成能力。
