# 量化总览

量化的目标，是用更少的比特表示权重和激活，从而降低显存、带宽和推理成本。

## 1. 一个统一数学骨架

把浮点数 \(x\) 映射到整数 \(q\)：

\[
q = \text{round}\left(\frac{x}{s}\right) + z
\]

**反量化为**：

\[
\hat{x} = s(q-z)
\]

其中：

- \(s\) 是 scale
- \(z\) 是 zero-point

这看起来只是数值压缩，但真正要保住的是模型输出：

\[
Wx \approx \hat{W}\hat{x}
\]

也就是说，量化真正关心的不是单个参数值，而是最终算子的误差传播。

## 2. 为什么量化在大模型时代变成核心工程问题

因为大模型部署的成本主要集中在：

- 权重显存
- KV cache
- 带宽
- 推理吞吐

如果不做量化，一个 7B 或 13B 模型很快就会撞上部署门槛。

### 一个直观例子：7B 模型显存

若 7B 参数模型用 FP16 存储，权重约占：

\[
7 \times 10^9 \times 2 \text{ bytes} \approx 14 \text{ GB}
\]

若改成 INT4，则理论上只需约 3.5GB 左右，再加上量化元数据，整体仍会显著下降。这就是为什么“能不能量到 4bit 且还能用”在工程上这么重要。

## 3. 量化真正解决的不是压缩，而是部署边界

从研究视角看，量化像在研究数值逼近；从工程视角看，它更像在回答：

1. 模型能不能塞进目标硬件；
2. 并发能不能提高；
3. 带宽和显存是否不再是第一瓶颈；
4. 能否在不牺牲关键业务质量的前提下降本。

因此量化从来不是“压得越低越先进”，而是“在目标场景里压到多低仍然划算”。

## 4. 量化的三条主线

### 后训练量化 PTQ

模型训练完后再量化。优点是便宜、快、适合现成模型。

### 量化感知训练 QAT

在训练中显式模拟量化误差。优点是效果通常更稳，但训练成本更高。

### 量化底座上的适配训练

如 `QLoRA`。目标不是只部署，而是让量化模型还能继续做高效微调。

### 4.1 还有一条现实系统主线：混合精度

很多上线系统并不会全模型统一到同一比特，而是：

1. 权重量化；
2. 某些敏感层保留高精；
3. 激活维持更保守精度；
4. KV cache 单独做折中。

这条混合精度路线往往比“最小比特挑战”更接近真实工程。

## 5. 权重和激活是两个不同问题

### 权重量化

静态、离线、相对容易控制。

### 激活量化

动态、分布变化大、易受离群值影响。

很多系统先做权重量化，再视硬件和任务决定是否推进激活量化。

### 5.1 为什么激活更难

**因为激活分布会随**：

1. 输入长度；
2. prompt 类型；
3. 多模态输入结构；
4. 工具返回结果；
5. 长上下文记忆状态

持续变化。所以很多离线看起来稳定的量化方案，一到线上复杂请求就暴露问题。

## 6. 多模态和长上下文为什么让量化更复杂

**一旦系统进入**：

- 文档理解
- 图表与表格
- 屏幕代理
- 视频理解
- 机器人动作

量化的挑战就不再只是“平均精度掉多少”，而是：

1. 细粒度定位是否还准；
2. 数字一致性是否还在；
3. 长时动作是否更抖；
4. 长上下文和 KV cache 是否成为新瓶颈。

这也是为什么量化在多模态和具身系统里需要更谨慎的混合精度与专项评测。

## 7. 一个够用的理解方式

**量化本质上是在做一件事**：把不可避免的数值误差，尽量安排到模型最不敏感的地方去。

**这也是为什么后面会出现**：

- GPTQ
- AWQ
- SmoothQuant
- FP8
- 混合精度服务

这些方法其实都在回答同一个问题：

**误差到底该落在哪，才最不伤输出。**

## 8. 量化在系统中的位置

量化不是孤立模块，它会和下面几件事强耦合：

1. kernel 实现；
2. serving runtime；
3. KV cache 管理；
4. batching 和路由；
5. 目标硬件；
6. 线上评测与回退。

因此看量化方案时，不能只看论文压到几 bit，还要问：

1. 运行栈是否支持；
2. 系统是否真的更快；
3. 线上是否有可回退路径；
4. 高风险任务是否被保护。

## 10. 一个总判断

量化的真正价值，不在于把模型变成一个更小的文件，而在于把“原本难以部署、难以扩并发、难以降成本的系统”推回可运营区间。它既是数值近似问题，也是服务架构问题，更是业务取舍问题。理解量化，不能只盯住 bit 数，而要同时盯住误差传播、系统瓶颈和最终可交付收益。 

## 快速代码示例

```python
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig

model_id = "Qwen/Qwen2.5-7B-Instruct"
quant_cfg = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.bfloat16,
)

tokenizer = AutoTokenizer.from_pretrained(model_id)
model = AutoModelForCausalLM.from_pretrained(
    model_id, quantization_config=quant_cfg, device_map="auto"
)
print("model footprint (GB):", model.get_memory_footprint() / 1024**3)
```

这段代码演示了 `bitsandbytes` 的 4bit 加载流程：`nf4` 用于权重量化，`bf16` 用于计算类型，最后用 `get_memory_footprint` 快速估算显存占用。部署前还应补充任务精度回归，确认压缩收益可接受。


## 补充：把 **量化总览** 写成一套系统取舍

量化主题最容易被误写成“精度换速度”的线性故事，但真正进入训练和服务系统后，事情远比这复杂。围绕 **量化总览**，更扎实的写法应该明确指出：量化从来不是单独的数学压缩问题，而是同时涉及 **权重布局、激活分布、校准制度、kernel 支持、缓存路径、部署链路和回退治理** 的系统设计问题。

从这个角度看，把量化专题从方法表扩展成选型、评测、部署和回退的完整路线。也就是说，量化收益从来不是白来的。你在一处省下来的显存和带宽，往往会在另一处以更复杂的 scale 管理、更多的运行时判断、更多的回归验证或更窄的适用边界重新付出去。页面如果不把这些代价写透，读者很容易高估某条量化路线的泛化能力。

## 补充：量化决策真正要回答的四个问题

围绕 **量化总览**，一个更稳妥的决策框架通常是：

1. **量化对象是什么**：仅权重，还是权重加激活，还是连 KV cache、优化器状态、LoRA 适配器也一起纳入；
2. **服务目标是什么**：单机吞吐、在线时延、长上下文成本、多模型混部，还是端侧/边缘部署；
3. **硬件路径是什么**：不同 GPU、不同 runtime、不同 kernel 栈对量化格式的支持差异非常大；
4. **回退与验证如何做**：没有全精度对照、没有回滚门槛、没有尾部任务桶评测，量化方案就很难称为可上线方案。

只要这四个问题被提前写清楚，很多常见争论会自然化解。因为多数争论并不是“方法谁更先进”，而是“目标场景到底是不是同一个”。

## 补充：最容易被平均值掩盖的风险

量化路线最危险的地方，往往不在平均基准，而在尾部。比如长上下文、少数高价值任务、复杂多模态输入、工具调用路径、动作头、控制头或世界模型里的风险预测头，这些位置常常对精度更敏感，却最容易被总体平均值稀释。再比如 calibration 集过于干净、只覆盖短文本或静态图像，也会让量化方案在真实流量里突然失真。

因此，更扎实的页面内容必须明确指出：**量化不是只要“整体差不多”就够了**。真正成熟的量化验收，要能说清哪些层最敏感、哪些任务桶最脆弱、哪些硬件路径收益能兑现、哪些场景应直接保留高精度回退。

## 补充：更像部署文档的验收清单

对 **量化总览**，建议至少把以下几类验证写进页面：

1. **数值验证**：和高精度参考路径逐层或逐任务桶对照；
2. **系统验证**：不仅看显存，还看 TTFT、TPOT、cache 占用、编译和启动成本；
3. **场景验证**：长文本、多模态、agent、动作控制等高风险路径要单独建桶；
4. **运维验证**：明确什么时候回退、如何 shadow、新旧版本如何并行观察。

只有当页面把这些内容写扎实，**量化总览** 才不会停留在“量化方法列表”，而会真正成为能指导选型与上线的系统知识。


## 补充：和相邻页面的接口要怎么看

围绕 **量化总览**，页面真正该帮读者建立的判断，不是“它是不是一种量化方法”，而是 方法谱系、部署路径和评测框架的接口。只有把接口讲清楚，读者才能知道为什么同一条量化路线在训练、推理、KV cache、多模态或不同硬件上表现会差这么多。

## 补充：一条更实用的落地顺序

更稳的量化推进顺序通常是：先确定目标场景和硬件边界，再选量化对象，再做校准与 kernel 路径验证，最后再看是否值得把它推到主线。顺序如果反过来，团队很容易先得到一个好看的压缩率，再花很久解释为什么端到端收益并没有兑现。

## 补充：还值得继续深挖的问题

围绕 **量化总览**，后续最值得加厚的内容通常包括：不同任务桶上的量化敏感性；校准集构造和激活尾部分布；多机服务与多模型混部时的真实收益；以及一旦线上出现局部退化，应该如何快速回到更高精度路径。把这些写透，量化页才真正具备上线指导价值。


## 补充：如果要把 **量化总览** 真正推进成主线工作

一个很现实的问题是：团队即使认同 **量化总览** 很重要，也未必知道应该先投入哪一层。更稳的推进方式通常不是一上来就把所有可能相关的方法和组件都拉满，而是先判断 **它当前最可能成为主线收益点，还是主线阻塞点**。若它更像收益点，重点是尽快做小规模验证、建立对照组和收益归因；若它更像阻塞点，重点则应放在接口澄清、失败样本回放、基础设施补齐和回退路径准备上。把这一步判断写进页面，读者会更容易把“知道这个方向”转成“知道现在该做什么”。

## 补充：比继续堆内容更重要的，是明确暂时不做什么

很多方向越写越散，根因不是信息不够，而是没有明确边界。对 **量化总览** 来说，一份更成熟的文档通常还要告诉读者：在当前阶段，哪些复杂方法可以先不碰，哪些收益看起来诱人但短期不值得，哪些局部最优会把系统拖进更高维护成本。只要把“暂时不做什么”写明白，页面的决策价值会大幅提高，因为真实项目里最贵的往往不是少做了一件事，而是过早做了十件还没有基础条件支撑的事。
