# 自定义算子与框架集成

很多人第一次写自定义 kernel 时，会把注意力放在“这个 kernel 跑得够不够快”。但在真实工程里，性能只是其中一部分。一个自定义算子想真正进入训练或推理系统，还必须解决：

1. 怎样被框架调用；
2. 怎样与 autograd、图编译和 profiler 兼容；
3. 怎样处理 shape、dtype、device dispatch；
4. 怎样做 fallback、测试和版本演进。

因此“自定义算子开发”其实更像一条完整集成链，而不是孤立的 kernel 编写。

!!! note "初学者先抓住"

    一个 custom op 的工程价值不只在快，还在能不能被框架稳定调用、能不能反向传播、能不能 profile、能不能 fallback、能不能覆盖真实 shape。只写出一个 microbenchmark 很快的 CUDA kernel，还不算真正接入系统。

!!! example "有趣例子：赛车发动机装不上车"

    发动机单独测试很强，但如果接口、散热、变速箱和维护都不匹配，整车上路仍然失败。自定义算子也是：kernel 快只是零件好，框架集成才决定能不能上生产线。

## 1. 为什么很多高性能 kernel 最终没被用起来

常见原因不是它不够快，而是：

1. 集成太难；
2. 只支持少量 shape；
3. autograd 路径缺失；
4. 无法进入 graph compile；
5. fallback 和错误处理太弱；
6. 部署、打包和兼容成本太高。

也就是说，真正让一个 kernel 有工程价值的，不只是性能，而是可接入性。

## 2. 自定义算子的典型组成

一个较完整的 custom op 通常包含：

1. 前端 API；
2. shape / dtype 检查；
3. forward kernel；
4. backward kernel 或 autograd 定义；
5. dispatch 注册；
6. 测试与 benchmark；
7. 可能还要有图编译兼容和 profiler 标记。

### 2.1 为什么这和“写个 CUDA 文件”不是一回事

因为一旦进入真实系统，你还要考虑：

1. contiguous 和 non-contiguous 输入；
2. CPU fallback 或 error path；
3. 多卡与 stream 语义；
4. mixed precision；
5. 编译缓存和运行时加载。

## 3. 与 PyTorch 集成时要想清楚的几件事

### 3.1 接口边界

**这个 op 是**：

1. 完全替换现有算子；
2. 作为 fused op；
3. 只服务某些 shape；
4. 只服务训练或只服务推理。

**不同定位会决定**：

1. API 如何设计；
2. fallback 如何写；
3. 是否需要多个后端实现。

### 3.2 Autograd 边界

如果是训练路径，还必须考虑：

1. backward 是否自己写；
2. 是否可由已有算子组合求导；
3. 中间缓存如何保存；
4. backward 是否也要高性能。

很多 kernel 的集成难点其实不在 forward，而在 backward。

## 4. 与 `torch.compile` / 图编译的关系

自定义算子如果完全绕开图编译器，有时会导致：

1. 断图；
2. 融合机会丢失；
3. 上层优化无法继续穿透；
4. profiler 看到一大片黑盒。

这并不一定意味着不能写 custom op，但意味着你需要清楚：

1. 它是否值得成为黑盒边界；
2. 它是否会阻断更大的图优化；
3. 它是否只在真正热点处使用。

## 5. Triton 自定义算子与 CUDA 自定义算子的差异

### 5.1 Triton

**优点**：

1. 开发快；
2. 更适合快速试 fusion；
3. 与 Python / PyTorch 路径更近。

**缺点**：

1. 某些极致优化场景不如底层实现；
2. 某些复杂同步和特化路径不够灵活。

### 5.2 CUDA

**优点**：

1. 底层控制更强；
2. 可做到更极致特化；
3. 对复杂同步和特定硬件路径支持更充分。

**缺点**：

1. 集成与维护成本更高；
2. 开发调试更慢；
3. 对团队要求更高。

## 6. 什么时候值得做 custom op

**通常需要满足以下几条中的多条**：

1. 热点明显；
2. 默认实现慢；
3. 可融合收益明确；
4. 形状相对稳定；
5. 业务收益足以覆盖维护成本。

如果只是“理论上也许能快一点”，往往不值得。

## 7. 测试与验收不能省

一个可上线的 custom op 至少要经过：

1. 正确性对比；
2. 多 shape 测试；
3. dtype 测试；
4. contiguous / stride 测试；
5. 训练或推理端到端回归；
6. profiler 对照；
7. fallback 验证。

否则一个快 kernel 很容易变成线上隐患。

## 8. 版本演进与兼容

自定义算子不是“一次写完永远不动”的资产。  
**它会持续受以下因素影响**：

1. 新 GPU 架构；
2. 新 PyTorch 版本；
3. 编译器后端变化；
4. shape 分布变化；
5. 模型结构变化。

因此一个工程上成熟的 custom op，还必须有：

1. 版本管理；
2. 持续 benchmark；
3. fallback；
4. 清晰的适用边界。

## 9. 一个形象比喻

自定义算子就像为工厂定制一台专用设备。设备本身可能很快，但如果它接不上现有流水线、维护手册不全、只能处理一种规格的零件、坏了又没有备用流程，那么它的工程价值就会大打折扣。真正有价值的 custom op，不只是“局部最快”，而是能顺滑进入整条生产线。

## 10. 小结

自定义算子开发的关键，不只是 kernel 写得快，而是把性能、集成、autograd、图编译、测试、fallback 和版本演进放在同一张设计图里。只有这样，一个自定义 kernel 才会从 demo 变成真正可维护、可部署、可复用的系统组件。


## 实践补充与检查

### 把 **自定义算子与框架集成** 放回真实系统里理解

很多页面在介绍算子或排查方法时，读者最容易获得的是一组术语，但真正落地时更关键的是：**这个主题究竟在训练、推理、编译器、运行时和硬件拓扑之间扮演什么角色**。围绕 **自定义算子与框架集成**，更实用的切入方式不是先记定义，而是先把问题拆成几个稳定坐标：

1. **API 语义、张量布局、autograd、打包发布、兼容矩阵**。

之所以要先做这一步，是因为大量“优化失败”并不是实现不够努力，而是一开始就把问题建模错了。比如明明是 **shape family** 失控，却去盯某个单一 kernel；明明是 **拓扑或缓存生命周期** 在拖后腿，却把矛头指向单步 FLOPs；明明应该在 **运行时** 解决，却试图把所有复杂度压进一个更重的 fused kernel。只要坐标系没建立，后续 benchmark、profile、灰度和上线都容易失真。

### 更像工程手册的排查顺序

针对 **自定义算子与框架集成**，一个更稳妥的工程排查顺序通常是：

1. **先确认真实工作负载**。不要只拿一个热 shape 或论文默认配置来代表全系统；
2. **再确认瓶颈层级**。区分是算子本身、调度路径、拓扑链路、还是缓存/内存生命周期在主导；
3. **然后才进入 microbenchmark**。只有当端到端关键路径已被确认，单算子基准才有解释力；
4. **把热路径和尾路径分开**。很多主题在平均值上看起来很好，但在尾部 shape、极端长度或混合流量上完全是另一回事；
5. **最后写清回退条件**。没有回退条件的优化，就不算真正可上线的优化。

如果团队把这套顺序固化下来，**自定义算子与框架集成** 就不再只是“某个高手会调的点”，而会逐渐变成组织可继承的能力。

### 最常见的误判与失败模式

围绕 **自定义算子与框架集成**，最常见的误判往往不是“完全不知道问题”，而是知道一半：

1. **原型能跑但集成成本高**。
2. **忽略错误处理**。
3. **只测 eager 不测 compile/runtime**。

这些误判之所以反复出现，是因为算子工程天然跨层。一个现象可能同时带着 **kernel**、**runtime**、**拓扑**、**数据布局** 和 **服务负载** 五层因素。若团队只盯其中一层，很容易做出“局部更优、全局更差”的决策。因此，这类主题最需要的往往不是更多零碎技巧，而是**把失败模式写成制度化案例**：什么情况下该怀疑布局，什么情况下该怀疑 cache，什么情况下应先改 dispatch，什么情况下必须回到硬件日志与链路测量。

### 验收、回归与长期维护

真正让 **自定义算子与框架集成** 站得住，不是一次 benchmark 跑得漂亮，而是它能否进入长期维护循环。更成熟的验收方式通常包括：

1. **先定义接口契约**。
2. **提供参考实现**。
3. **把构建与测试自动化**。
4. **记录支持限制**。

除此之外，还建议把 **自定义算子与框架集成** 相关改动长期绑定四类材料：

1. **代表性 probe**，用于快速复验是否退化；
2. **端到端案例**，用于防止只赢 microbenchmark；
3. **已知黑名单 shape / 路径**，避免后来人重复踩坑；
4. **升级对照记录**，包括驱动、编译器、框架、运行时变化前后的差异。

做到这一步后，**自定义算子与框架集成** 才会真正从“概念页”变成“方法页”，再从“方法页”变成“平台资产页”。
