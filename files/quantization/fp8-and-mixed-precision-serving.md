# FP8 与混合精度服务

在量化讨论里，`INT8`、`INT4` 很吸引眼球，但在很多工业服务栈中，真正迅速落地的是 `FP8` 和各种混合精度路径。  
**原因很简单**：它们在硬件支持、数值稳定和 kernel 实现之间往往处在一个更实际的平衡点。尤其是大模型服务、训练后部署和多模态系统中，FP8 往往不是最激进的选择，却可能是最先产生稳定价值的选择。

**这页重点回答四个问题**：

1. 为什么 FP8 在很多系统里比 INT4 更容易先落地？
2. 为什么真实部署几乎从来不是“全模型统一一种精度”？
3. FP8 的收益究竟来自显存、吞吐，还是系统兼容性？
4. 在服务系统里，哪些层值得保高精度，哪些层更适合压低精度？

!!! note "初学者先抓住"
    FP8 不是“越低越好”的竞赛，而是一个更容易在硬件、kernel 和数值稳定之间兑现收益的折中。真实系统往往是混合精度：敏感层保守，热路径降精度。

!!! example "有趣例子：不同路段限速"
    高速公路可以开快，学校门口必须慢。混合精度也是这样：GEMM 热路径可以尝试 FP8，某些 norm、输出头、风险相关模块可能要保留更高精度。

## 1. FP8 在解决什么问题

与 `BF16/FP16` 相比，FP8 主要希望：

1. 降低显存占用；
2. 降低带宽压力；
3. 利用新硬件上的专门张量核心；
4. 保留比纯整数格式更自然的动态范围表示。

因此 FP8 常被视为一种“较温和的低精度路径”。

如果把单个张量元素存储位数从 16 bit 降到 8 bit，理论上存储和带宽需求都能大幅下降。  
但 FP8 的真正意义不只是“省一半空间”，而是：

- 仍保留浮点格式的指数和尾数结构；
- 对动态范围更友好；
- 与现有浮点训练和推理栈更容易衔接。

## 2. 为什么 FP8 比 INT4 更容易先进入生产

### 2.1 浮点格式对动态范围更宽容

整数格式往往要求你非常谨慎地选 scale。  
而 FP8 由于保留指数位，面对范围跨度很大的张量时更自然。

很多激活张量会表现出这样的特性：

\[
\max |x| \gg \mathrm{median}(|x|).
\]

这意味着张量里同时存在大量中小值和少量极端值。  
如果用低比特整数硬压，往往要么：

- 小值被量化成 0；
- 要么大值被截断。

FP8 在这里更有缓冲空间。

### 2.2 硬件和编译链支持更成熟

很多现代加速卡和编译栈已经逐步把 FP8 作为一类原生优化目标。  
**这意味着**：

- kernel 更成熟；
- 张量核心更容易利用；
- 编译器更愿意做自动图优化；
- 服务框架更容易接入。

而非常激进的低比特整数方案，虽然理论压缩率更高，但往往受限于：

- 特殊 kernel 支持不足；
- 模型结构不兼容；
- 某些操作必须回退高精度。

### 2.3 混搭成本更低

很多团队已有大规模 `BF16/FP16` 训练和推理栈。  
FP8 进入系统时更像在原有浮点链上降一级，而不是彻底换世界观。

**这意味着**：

- 调试成本更低；
- 数值异常更容易定位；
- 和现有模块混用更自然。

## 3. 先建立一个概念：混合精度不是妥协，而是资源分配

在线服务里最常见的并不是“全模型 FP8”，而是：

1. 权重某些层 FP8；
2. 激活保留 BF16；
3. softmax、norm、输出头保留高精度；
4. 某些跨模态投影或动作头单独保护。

**本质上这是在做精度预算分配**：把高精度留给最敏感的环节。

更准确地说，混合精度是在回答下面这个问题：

\[
\text{在给定误差预算下，哪些张量最值得保高精度？}
\]

**这和资金预算分配很像**：

- 不是所有层都同样重要；
- 不是所有操作对数值误差都一样敏感；
- 合理分配精度，往往比追求“全局最低 bit”更稳。

## 4. FP8 的数值直觉

若张量 \(x\) 映射到 FP8，实际关注的是动态范围和舍入误差。  
与整数不同，FP8 可以在指数位上保留更大的范围变化，因此对范围跨度大的场景通常更稳。  
**但它仍然会在以下位置遇到问题**：

- 极小值累积
- 归一化
- 残差叠加
- 长序列激活
- 极端 outlier

### 4.1 误差粗略写法

若量化映射为 \(\hat{x} = Q_{\text{FP8}}(x)\)，则误差可写成：

\[
e = x - \hat{x}.
\]

在矩阵乘法 \(y = Wx\) 中，权重和激活误差共同传播：

\[
\hat{y} = (\hat{W})(\hat{x}) = (W + \delta_W)(x + \delta_x).
\]

**展开可得**：

\[
\hat{y} - y
=
W\delta_x + \delta_W x + \delta_W \delta_x.
\]

这说明数值退化不是单点发生，而是权重误差和激活误差一起作用。

### 4.2 为什么某些操作常常要保高精度

例如：

- `LayerNorm`
- `Softmax`
- 输出 logits
- 某些 attention score 计算

**这些操作对**：

- 小量差异
- 指数放大
- 归一化稳定性

非常敏感。  
因此真实系统里，哪怕大部分 matmul 都压到 FP8，上述环节也经常保留更高精度。

## 5. 一个直观例子：为什么多模态系统更偏爱稳妥的混合精度

设想一个文档 VLM 系统，要同时处理：

- OCR 小字
- 布局坐标
- 图像 patch
- 长文本上下文

这类系统往往比纯文本 LLM 更容易受到数值精度变化影响。  
**原因在于**：

- 不同模态张量的尺度差异更大；
- 视觉投影层和文本层的数值分布不同；
- 局部视觉细节对小数值误差更敏感。

如果直接上激进的低比特整数方案，可能会出现：

- OCR 细节丢失
- 表格边界理解退化
- grounding 精度下降

这时 FP8 或 FP8 + BF16 混合精度就成为更现实的折中。

## 6. 服务时应如何看 FP8 价值

真正要看的不是“文件小了多少”，而是：

1. `TTFT` 是否下降；
2. `TPOT` 是否下降；
3. 显存释放后并发是否增加；
4. 长上下文下是否仍稳；
5. 多模态或长尾任务是否有不可接受退化；
6. 工程链是否更稳定。

很多时候，FP8 最大收益是让系统更容易塞进目标硬件，而不一定是理论吞吐数字最亮眼。

### 6.1 一个更实用的判断式

**可以把上线价值粗略看成**：

\[
\text{Value}_{\text{FP8}}
=
\Delta \text{capacity}
 + \Delta \text{throughput}
 - \Delta \text{quality risk}
 - \Delta \text{engineering complexity}.
\]

**这提醒我们**：

- 性能提升要看端到端；
- 风险和复杂度也必须算进去。

## 7. 哪些层适合优先尝试 FP8

虽然不同模型差异很大，但经验上更适合优先尝试 FP8 的常见位置包括：

- 大型线性层权重
- attention / MLP 的主要 matmul
- 某些激活张量

**而更常需要保高精度的部分包括**：

- embedding / 输出头
- norm 相关计算
- softmax / logit 相关部分
- 某些视觉投影或多模态融合层

## 8. 三种典型部署场景

### 场景 A：70B 级文本模型服务

**需求**：

- 模型装进有限 GPU
- 延迟可控
- 不希望质量掉太多

这时 FP8 常常是一个好起点，因为：

- 比 BF16 更省资源；
- 比 INT4 风险更低；
- 和成熟服务栈更容易兼容。

### 场景 B：企业文档 VLM

**需求**：

- 文本和图像都要保住细节
- OCR 错误容忍度低
- 需要稳定上线

**这里更常见的路线是**：

- 主干大部分 matmul 尝试 FP8
- OCR 或关键结构层保 BF16
- 输出侧和规则判定部分保高精度

### 场景 C：持续演化中的线上系统

如果团队仍在频繁更换模型版本、prompt 模板和工具链，那么采用一种过于激进的量化方案会显著加大回归成本。  
**FP8 的优势在于**：

- 过渡平滑
- 异常更容易定位
- 便于逐步扩展到更复杂混合精度设计

## 9. FP8 也有明确短板

### 9.1 它不是最极致的压缩路线

**若你目标是**：

- 单卡塞超大模型
- 极限压缩
- 边缘设备部署

那么 FP8 往往不如 `INT4/W4A8` 这类方案激进。

### 9.2 不是所有收益都会自动兑现

如果 kernel、框架或算子路径不成熟，理论上的 FP8 优势可能无法变成端到端收益。  
例如：

- 某些算子回退高精度；
- 频繁数据格式转换；
- 跨设备通信仍是瓶颈。

### 9.3 层选择不当仍会伤质量

“FP8 比 INT4 稳”不代表“随便压都没事”。  
**尤其在**：

- 长上下文
- 复杂推理
- 文档细粒度理解
- agent 多工具任务

中，局部敏感层被压坏后，问题依然明显。

## 10. 一份实践化落地顺序

若你准备把 FP8 引入服务系统，可按下面顺序推进：

1. 先做端到端 profile，确认瓶颈确实受带宽和显存影响。
2. 先从权重侧或主干 matmul 开始尝试，不要一次性压全部。
3. 关键模块保留 BF16，例如 norm、softmax、输出头。
4. 对长上下文、多模态和长尾样本单独做回归。
5. 统计实际 `TTFT`、`TPOT`、显存和并发收益。
6. 在收益真实可见后，再考虑是否继续向更低比特推进。

## 11. 小结

FP8 与混合精度服务的价值，在于它提供了一条更稳的压缩与加速路径。  
对很多真实系统来说，它不是“最极致”的方案，却是最容易从实验室走向生产的方案。

更准确地说，它的核心价值不只是“8 bit”，而是：

- 保留浮点动态范围；
- 便于与现有栈混合；
- 把低精度部署从一次性赌博变成可分阶段推进的工程设计。

## 快速代码示例

```python
import transformer_engine.pytorch as te

def forward_step(model, x):
    with te.fp8_autocast(enabled=True):
        h = model.backbone(x)   # 主干矩阵计算走 FP8
    logits = model.lm_head(h.float())  # 输出头保留更高精度
    return logits
```

这段代码演示了一个常见混合精度策略：主干高算量模块用 FP8 提升吞吐，敏感输出头保留更高精度以稳住质量。实际部署时还会配合分桶回归，确认长上下文和长尾任务不退化。


## 实践补充与检查

### 把 **FP8 与混合精度服务** 写成一套系统取舍

量化主题最容易被误写成“精度换速度”的线性故事，但真正进入训练和服务系统后，事情远比这复杂。围绕 **FP8 与混合精度服务**，更扎实的写法应该明确指出：量化从来不是单独的数学压缩问题，而是同时涉及 **权重布局、激活分布、校准制度、kernel 支持、缓存路径、部署链路和回退治理** 的系统设计问题。

从这个角度看，围绕精度、显存、kernel 支持、scale 管理和端到端收益讨论 FP8。也就是说，量化收益从来不是白来的。你在一处省下来的显存和带宽，往往会在另一处以更复杂的 scale 管理、更多的运行时判断、更多的回归验证或更窄的适用边界重新付出去。页面如果不把这些代价写透，读者很容易高估某条量化路线的泛化能力。

### 量化决策真正要回答的四个问题

围绕 **FP8 与混合精度服务**，一个更稳妥的决策框架通常是：

1. **量化对象是什么**：仅权重，还是权重加激活，还是连 KV cache、优化器状态、LoRA 适配器也一起纳入；
2. **服务目标是什么**：单机吞吐、在线时延、长上下文成本、多模型混部，还是端侧/边缘部署；
3. **硬件路径是什么**：不同 GPU、不同 runtime、不同 kernel 栈对量化格式的支持差异非常大；
4. **回退与验证如何做**：没有全精度对照、没有回滚门槛、没有尾部任务桶评测，量化方案就很难称为可上线方案。

只要这四个问题被提前写清楚，很多常见争论会自然化解。因为多数争论并不是“方法谁更先进”，而是“目标场景到底是不是同一个”。

### 最容易被平均值掩盖的风险

量化路线最危险的地方，往往不在平均基准，而在尾部。比如长上下文、少数高价值任务、复杂多模态输入、工具调用路径、动作头、控制头或世界模型里的风险预测头，这些位置常常对精度更敏感，却最容易被总体平均值稀释。再比如 calibration 集过于干净、只覆盖短文本或静态图像，也会让量化方案在真实流量里突然失真。

因此，更扎实的页面内容必须明确指出：**量化不是只要“整体差不多”就够了**。真正成熟的量化验收，要能说清哪些层最敏感、哪些任务桶最脆弱、哪些硬件路径收益能兑现、哪些场景应直接保留高精度回退。

### 更像部署文档的验收清单

对 **FP8 与混合精度服务**，建议至少把以下几类验证写进页面：

1. **数值验证**：和高精度参考路径逐层或逐任务桶对照；
2. **系统验证**：不仅看显存，还看 TTFT、TPOT、cache 占用、编译和启动成本；
3. **场景验证**：长文本、多模态、agent、动作控制等高风险路径要单独建桶；
4. **运维验证**：明确什么时候回退、如何 shadow、新旧版本如何并行观察。

只有当页面把这些内容写扎实，**FP8 与混合精度服务** 才不会停留在“量化方法列表”，而会真正成为能指导选型与上线的系统知识。


### 和相邻页面的接口要怎么看

围绕 **FP8 与混合精度服务**，页面真正该帮读者建立的判断，不是“它是不是一种量化方法”，而是 精度管理、kernel 路径、cache 成本和回退策略的接口。只有把接口讲清楚，读者才能知道为什么同一条量化路线在训练、推理、KV cache、多模态或不同硬件上表现会差这么多。

### 一条更实用的落地顺序

更稳的量化推进顺序通常是：先确定目标场景和硬件边界，再选量化对象，再做校准与 kernel 路径验证，最后再看是否值得把它推到主线。顺序如果反过来，团队很容易先得到一个好看的压缩率，再花很久解释为什么端到端收益并没有兑现。

### 还值得继续深挖的问题

围绕 **FP8 与混合精度服务**，后续最值得加厚的内容通常包括：不同任务桶上的量化敏感性；校准集构造和激活尾部分布；多机服务与多模型混部时的真实收益；以及一旦线上出现局部退化，应该如何快速回到更高精度路径。把这些写透，量化页才真正具备上线指导价值。
