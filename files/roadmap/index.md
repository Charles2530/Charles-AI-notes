# 路线图

这套文档站的目标，不是做百科式罗列，而是建立一个可持续扩写、可反复复用、可长期维护的知识图谱。  
当前站点已经具备完整主干，但如果要继续扩到 `400 万字` 规模，最重要的原则不是不停增加新专题，而是把现有专题真正写厚、写透、写出层次。

**这份路线图因此承担三件事**：

1. 说明全站应该怎么读；
2. 说明现有大专题之间如何连接；
3. 说明在**不增加新大专题**的前提下，如何把全站稳步扩写到 `400 万字`。

## 1. 这套站点适合怎样阅读

如果你是第一次接触这些主题，建议优先按“问题链”来读，而不是按目录名字跳着读。  
**目前全站大致有三条主线**：

- `生成与多模态`
- `训练与系统`
- `规划与具身`

这样组织的目的，是让读者既可以：

- 从头系统读一条主线；
- 也可以在某个主题上快速下钻；
- 还可以围绕真实问题跨主题回跳，而不是被目录困住。

## 6. 当前站点的组织原则

为了后续持续扩写，每个主题都应保持稳定结构。  
现有大专题不变，但每个专题内部可以逐渐形成四层：

### 第一层：总览

**回答**：

- 这个主题在解决什么问题；
- 关键词之间的关系是什么；
- 读者第一次进入时应该先抓哪条主线。

### 第二层：核心方法

**回答**：

- 代表方法的公式和直觉；
- 方法之间的继承关系和差异；
- 哪些分支只是写法变化，哪些分支是真正范式变化。

### 第三层：工程落地

**回答**：

- 实际系统怎么选型；
- 常见失败模式是什么；
- 部署和成本怎么平衡；
- 为什么论文里的收益在真实系统里可能打折。

### 第四层：评测、案例与复盘

**回答**：

- 应该怎么验收；
- 线上或实机会在哪里失效；
- 有哪些真实工作流、案例、反例和决策记录可以复用。

## 7. 扩写到 400 万字时的基本约束

如果没有约束，长文档很容易膨胀成重复叠加。  
要把全站扩到 `400 万字`，必须先把边界定清楚。

### 7.1 不增加新大专题

**现有一级主题已经足够完整**：

- 路线图
- 术语表
- 参考文献
- 论文导读
- 扩散模型
- `VLM`
- `VLA`
- 量化
- 训练
- 推理
- 对比学习
- 世界模型
- 具身智能

**后续扩写默认只做三类动作**：

1. 给现有页面增加更多小标题；
2. 把已有段落写得更细、更可操作；
3. 只在原专题内部补充同主题子页，而不新开一级大类。

### 7.2 优先扩现有页面，再考虑拆页

**建议的默认顺序是**：

1. 先把总览页扩到 `1.5 万` 到 `2.5 万字`；
2. 再把方法深度页扩到 `2.5 万` 到 `4 万字`；
3. 再把工程页、评测页、对照页扩到 `2 万` 到 `3.5 万字`；
4. 单页确实过长时，再在同专题内部拆成子页。

这能避免过早拆得太碎，导致目录结构先失控、内容却还不够厚。

### 7.3 字数统计口径要统一

这里的 `400 万字`，按当前 `docs/` 下 `Markdown` 正文的字符规模粗略统计。  
公式、表格、英文 token、链接文本都会计入一部分字符，因此它不是出版业那种极严格的版面字数，而是更适合仓库维护的工程口径。

### 7.4 每次扩写都要回答同一组问题

无论扩的是哪一页，新增内容最好都能回答下面至少三到五个问题：

1. 它在解决什么问题；
2. 为什么旧方法不够；
3. 关键公式、结构或工作流是什么；
4. 它在工程上最常卡在哪里；
5. 它和相邻主题的接口是什么；
6. 它最常见的失败模式是什么；
7. 读者看完之后下一步该跳到哪里。

只要持续这样扩，字数增长才会真正转化为密度增长。

## 8. 从当前体量到 400 万字的总盘子

按当前仓库统计，`docs/` 下 `Markdown` 总体量约为：

- 当前规模：`396310`
- 目标规模：`4000000`
- 尚需扩写：`3603690`

这意味着后续不是“再补一点内容”就够了，而是要把现有站点从“已成型的知识骨架”扩成“可反复查阅的大型知识底稿”。

**一个更直观的理解方式是**：

$$
\text{目标增量} \approx 10 \times \text{当前主干厚度}
$$

所以规划重点不该是发散，而应是：

- 把每个主题都写出真正的内部层级；
- 把每种方法都写出前因后果；
- 把每个工程问题都写到能支持选型与复盘。

## 10. 每类页面应该怎样变厚

如果要在不新增大专题的情况下把字数做大，最有效的方法不是堆更多页名，而是给每一类页面建立统一扩写结构。

### 10.1 总览页

总览页不该只写“这是什么”。  
**后续建议统一补齐这些小标题**：

1. 主题在解决什么问题；
2. 为什么这个问题重要；
3. 主要方法分几条线；
4. 每条线分别解决什么瓶颈；
5. 训练、推理、部署分别卡在哪里；
6. 这一主题与相邻主题如何连接；
7. 初学者最容易误解什么；
8. 推荐阅读顺序和跳转路径。

### 10.2 方法深度页

这类页面是未来字数增长的主力。  
**建议统一补齐**：

1. 问题定义；
2. 历史动机；
3. 统一记号；
4. 核心公式与推导；
5. 结构图或流程拆解；
6. 代表方法对照；
7. 训练细节；
8. 推理细节；
9. 工程坑点；
10. 失效模式；
11. 小案例；
12. 开放问题。

### 10.3 对照表页面

对照表不应只是“短表格”。  
真正有价值的扩法是把每张表后面再补三层内容：

1. 这张表该怎么读；
2. 哪些列最关键；
3. 哪些比较只在某些约束下成立；
4. 每一组方法各自适合什么场景；
5. 最常见的误读是什么。

### 10.4 导读页

导读页未来要承担“把厚内容重新组织成路线”的作用。  
**可以统一增加**：

1. 为什么先读这篇、后读那篇；
2. 每篇论文解决了上一代什么问题；
3. 这条路线在哪些地方分叉；
4. 论文之间哪些是正统延续，哪些只是相邻探索；
5. 读完之后应该如何做复现或实验。

### 10.5 术语表与参考文献

这两类页面很容易被低估，但它们是大型知识库最重要的稳定器。

**术语表可继续补**：

1. 术语定义；
2. 它和哪些词最容易混淆；
3. 在本站语境里它更偏哪层含义；
4. 常见误用；
5. 关联页面。

**参考文献可继续补**：

1. 论文的主题归类；
2. 一句话定位；
3. 推荐阅读顺序；
4. 与站内页面的反向链接；
5. 哪些是入门必读，哪些是专题深入。

## 11. 各专题扩写抓手

下面这部分是后续把字数落到具体专题时的主抓手。  
仍然坚持一个原则：**不新增新大专题，只把现有主题写厚。**

### 11.1 首页

首页不需要变成百科，但需要更像一个真正的总站入口。  
**后续可补**：

- 站点定位与使用场景；
- 三类用户画像；
- 五条推荐阅读线；
- “研究问题如何映射到章节”的导航表；
- 站内常见跳转组合；
- 如何把导读页和深度页配合使用。

### 11.2 路线图

路线图未来要成为全站的总控页，而不只是阅读索引。  
**可以继续扩写**：

- 各主题之间的接口关系；
- 不同读者类型的详细路线；
- 不同项目目标下的章节组合；
- 站点扩写节奏与字数盘子；
- 单页扩写规则；
- 同主题内部拆页规则；
- “从知识地图到工程工作流”的使用方式。

### 11.3 术语表

术语表不需要引入新专题，但可以把现有全站高频概念全部补齐。  
**扩法重点是**：

- 每个术语给出“本站语境定义”；
- 补充最易混淆概念；
- 增加反向链接；
- 给出最短例子；
- 把公式型术语、系统型术语、机器人型术语统一起来。

### 11.4 参考文献

参考文献不只是列论文名。  
**后续可增加**：

- 按主题、年代、问题线重排；
- 每篇一行定位；
- “这篇应该在什么阶段读”；
- 代表性论文之间的继承关系；
- 与站内页面的互链。

### 11.5 论文导读

导读页应当承担“厚内容的导航器”角色。  
**主要扩法包括**：

- 每条研究线补“前史 -> 主线 -> 分叉 -> 工程化”；
- 增加关键论文之间的关系说明；
- 增加读论文时该重点盯什么；
- 增加复现难点；
- 增加常见误读与跳读策略。

### 11.7 扩散模型

扩散专题已经有不错骨架，后续重点不是再开新大类，而是把现有页面写出真正的厚度。

**建议的扩写方向**：

- `index.md` 增加“问题地图、方法谱系、常见误区、阅读入口、部署入口”；
- `evolution.md` 从时间线扩成“每代方法为何出现、解决了什么、引入了什么代价”；
- `training.md`、`score-matching-sde-and-probability-flow.md`、`noise-schedules-and-parameterization.md` 补齐推导、参数化差异、训练稳定性和实验设计；
- `guidance-and-conditioning.md` 深写控制接口、条件冲突、编辑任务和失败案例；
- `inference.md`、`distillation.md`、`consistency-models-and-rectified-flow.md` 增加采样器家族比较、速度质量折中、真实部署工作流；

**建议的扩写方向**：

- `document-understanding-and-ocr.md` 增加版式、表格、字段抽取、长页错误模式；
- `charts-tables-and-structured-reasoning.md` 增加结构化推理链、图表误读、单位与比例陷阱；

### 11.10 量化

## 12. 四轮扩写节奏

为了把 `400 万字` 做得稳，而不是一口气写散，建议按四轮推进。

### 12.1 第一轮：把每页写到“完整”

目标不是极厚，而是先补齐所有页的基础骨架：

- 问题定义；
- 方法主线；
- 工程接口；
- 常见失败；
- 跳转关系。

这一轮完成后，全站会从“骨架齐”变成“每页都能独立阅读”。

### 12.2 第二轮：把每页写到“可比较”

**这一轮重点补**：

- 与相邻方法的比较；
- 为什么旧方法不够；
- 公式之间的关系；
- 场景约束下的取舍。

这一轮完成后，全站会从“能读”变成“能判断”。

### 12.3 第三轮：把每页写到“可落地”

**这一轮重点补**：

- 训练与部署 recipe；
- 数据、显存、时延与成本；
- 评测方案；
- 排障与回滚。

这一轮完成后，全站会从“能判断”变成“能拿去做事”。

### 12.4 第四轮：把每页写到“可复用”

**这一轮重点补**：

- 案例；
- 反例；
- 步骤清单；
- 检查清单；
- 跨页索引；
- 反向引用。

这一轮完成后，全站才真正接近“大型知识底稿”。

## 13. 防止 400 万字变成“水字数”

体量一大，最大的风险不是写不满，而是写得松。  
因此后续扩写最好持续守住以下规则：

1. 不重复定义同一个概念，尽量反向链接到术语表；
2. 不把同一句空泛判断换几个说法反复写；
3. 每一大段都尽量回答一个明确问题；
4. 表格后必须补解释，不让表格变成孤立堆砌；
5. 深度页至少包含一个“小例子”或“失败案例”；
6. 工程页至少包含一个“如何排查”或“如何验收”部分；
7. 每次补内容时，同时补内部链接。

如果这几条守住了，字数越大，反而越容易形成真正的系统性。

## 14. 一个实际可执行的页级目标

**后续可把页级目标大致定成这样**：

| 页面类型 | 建议目标 |
| --- | ---: |
| 首页 / 路线图页 | 15000 - 30000 |
| 总览页 | 15000 - 25000 |
| 方法深度页 | 25000 - 45000 |
| 工程页 / 评测页 | 20000 - 35000 |
| 对照表页 | 18000 - 28000 |
| 导读页 | 12000 - 25000 |
| 术语表 / 参考页 | 持续累积，不设硬上限 |

这意味着后续很多页面完全可以只靠：

- 增加小标题；
- 增加例子；
- 增加失败模式；
- 增加比较段；
- 增加工程细节；

就自然长到合理厚度，而不需要到处新开大专题。

## 15. 小结

把这套站点扩到 `400 万字`，核心不是继续横向发散，而是纵向加厚。  
现有大专题已经足够，下一阶段真正需要做的是：

1. 让每个专题从“有主干”变成“有层级”；
2. 让每个页面从“能介绍”变成“能判断、能落地、能复盘”；
3. 让导读、术语、参考把整个站点重新织紧；
4. 用统一的页级结构和字数盘子，把增量控制在同一套秩序里。

后续只要继续沿这个规划推进，`400 万字` 不是靠新增更多大类完成，而是靠把现有内容真正写深、写透、写成体系。
