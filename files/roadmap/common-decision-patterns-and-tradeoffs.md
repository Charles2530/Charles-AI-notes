# 常见技术决策与取舍

真实的模型系统建设，很少是“看到一个 SOTA 方法就直接上”。更多时候，团队面对的是一连串不完美选择：要不要换更大模型，还是先做检索？要不要做量化，还是先蒸馏？要不要追求一步生成，还是接受十几步但更稳？要不要端到端 VLA，还是保留分层控制？这些问题没有脱离场景的标准答案，但它们确实存在一组反复出现的决策模式。本文试图把这些模式系统化，帮助你在面对复杂路线选择时少走弯路。

## 1. 决策的起点：先分清问题类型

几乎所有技术取舍，都可以先归因到四类问题之一：

1. **能力问题**：系统不会做或做不好。
2. **成本问题**：系统会做，但太贵。
3. **时延问题**：系统会做，但太慢。
4. **可靠性问题**：系统平均可用，但长尾不稳、回归多、难部署。

很多讨论之所以低效，是因为参与者没有先统一问题类型。比如有人主张上更大模型，另一个人主张先做缓存；如果真实问题是时延，两人就不在同一层面争论。

## 2. 决策模式一：扩大模型，还是先补数据

这是最常见的路线分歧。粗略地讲：

1. 若模型在多个任务上普遍能力不足，且已有数据质量较高，扩大模型可能有效。
2. 若模型在少数高价值长尾场景上退化明显，优先补专项数据通常更值。
3. 若训练集存在明显脏噪、重复和错配，先做数据治理往往比扩模更划算。

**可把收益粗略写成**：

$$
\Delta U \approx \Delta U_{\text{model}} + \Delta U_{\text{data}} - \Delta C,
$$

关键是判断当前边际收益主要来自哪一项，而不是机械地“参数越大越好”。

### 2.1 什么时候优先扩模

1. 多项基础 benchmark 都显示能力边界明显；
2. 错误广泛而非集中于少数长尾；
3. 训练数据已较丰富，新增高质量数据难获取；
4. 推理成本还能接受或可通过蒸馏、量化回收。

### 2.2 什么时候优先补数据

1. 问题集中在文档、图表、UI、代码、特定行业术语等垂直领域；
2. 失败模式高度集中；
3. 小模型通过更好数据也能明显修复；
4. 部署预算不足以支持更大模型上线。

## 3. 决策模式二：做检索增强，还是继续追求纯模型能力

当系统需要利用大量外部知识、长文档或频繁变化信息时，常见分歧是：继续把模型做大、上下文做长，还是把知识外置到检索系统。

### 3.1 检索优先的信号

1. 知识频繁更新；
2. 需要权限、时效或版本过滤；
3. 查询常只需要知识库的一小部分；
4. 用户期望看到可引用证据。

### 3.2 纯模型优先的信号

1. 任务更依赖推理结构而非外部知识；
2. 上下文构造成本太高；
3. 检索质量难以保证；
4. 目标设备无法承受复杂外部链路。

### 3.3 现实中的中间路线

大多数成熟系统都会采用“模型能力 + 检索增强”的组合。问题不在二选一，而在于把检索边界放在哪里：是仅用于召回证据，还是让模型大量依赖外部上下文；是作为默认路径，还是只在高不确定任务中触发。

## 4. 决策模式三：量化、蒸馏还是换硬件

当服务成本或显存压力上升时，团队常在三条路线间犹豫：

1. **量化**：压低数值精度；
2. **蒸馏**：训练更小学生模型；
3. 换更高规格硬件或更多副本。

### 4.1 量化适合什么

1. 现有模型能力已经满意；
2. 主要痛点是显存和带宽；
3. runtime 对目标低比特支持较成熟；
4. 可以接受少量精度折损或采用混合精度。

### 4.2 蒸馏适合什么

1. 需要更小、更快、结构更简单的学生；
2. 希望在边缘端、本地部署或多模型并发场景中使用；
3. 可接受一次额外训练成本，换长期推理收益。

### 4.3 换硬件何时反而更划算

1. 工程链路对新精度支持很差；
2. 推理负载不稳定，量化收益兑现困难；
3. 项目时间紧，无法承担新模型训练和验证。

很多时候，最优方案不是单选，而是“小步量化 + 局部蒸馏 + 分层路由”。

## 5. 决策模式四：追求一步生成，还是保留多步但更稳

扩散模型快速发展后，团队经常在“一步 / 少步生成”和“十几步稳健采样”之间取舍。关键不是步数越少越先进，而是目标场景是什么。

### 5.1 极少步优先的场景

1. 实时交互、移动端、本地生成；
2. 需要快速预览与大量迭代；
3. 用户对微小质量损失不敏感。

### 5.2 多步优先的场景

1. 高质量海报、商业素材；
2. 结构和细节容错率低；
3. 多条件控制复杂，如文档、布局、视频编辑。

### 5.3 现实中的折中

**常见做法是**：

1. 预览模式用少步；
2. 最终导出用更多步；
3. 用蒸馏或一致性模型做首轮提案，再高步数精修。

这是一种典型“体验与质量分层”思路。

## 6. 决策模式五：端到端 VLA，还是保留分层控制

VLA 很诱人，因为它似乎能把视觉、语言和动作统一成一个模型。但真实机器人系统常需要在端到端与分层控制之间权衡。

### 6.1 端到端路线的优点

1. 能直接从大规模数据学跨模态关联；
2. 接口简洁；
3. 对复杂语义任务更自然。

### 6.2 分层路线的优点

1. 控制稳定性更强；
2. 更易插入安全规则；
3. 更便于局部调试和实机部署。

### 6.3 判断依据

如果任务主要受高层语义限制，且低层控制成熟，端到端收益更可能体现在高层规划和技能选择；如果任务对接触和精细控制极敏感，保留低层专用控制器往往更稳。

## 7. 决策模式六：世界模型先上，还是先把 reactive policy 做稳

世界模型很有吸引力，但并不是所有场景都适合优先投入。判断时可以问：

1. 任务是否真的需要长时规划？
2. 真实试错是否昂贵到值得做内部想象？
3. 是否已经有足够数据训练稳定世界模型？
4. 当前最主要问题是前瞻不足，还是感知 / 控制基础还没稳？

如果系统连基本反应式策略都不稳定，直接上世界模型往往会把问题复杂化。

## 8. 决策模式七：先做泛化，还是先做可靠性

研究常倾向强调“能做更多任务”，工程则更关心“把核心任务做到非常稳”。这不是谁对谁错，而是阶段不同。

### 8.1 泛化优先的阶段

1. 方向探索初期；
2. 需要验证统一架构是否有潜力；
3. 目标是研究突破。

### 8.2 可靠性优先的阶段

1. 接近上线；
2. 任务范围已明确；
3. 长尾和回归成本高。

技术负责人最大的误区之一，就是在产品阶段仍然用研究阶段的评价方式做决策。

## 9. 决策模式八：先修算法，还是先修评测

很多团队一看到模型表现不佳，就立刻改模型。但真实情况常常是：

1. 评测集不代表线上问题；
2. 指标与业务目标错位；
3. 错误类型没有被分桶；
4. 回归只是某个链路版本漂移。

若评测体系本身不完整，再多改模型也可能是在错误目标上优化。特别在 VLM、VLA、世界模型和量化系统里，“先修评测”常常是更高 ROI 的决策。

## 10. 决策模式九：统一平台，还是专项系统

随着模型越来越通用，团队常想做一个大一统平台：统一训练、统一 serving、统一向量系统、统一评测。但现实中，不同任务的负载画像和质量要求差异很大。

### 10.1 统一平台的优点

1. 复用组件；
2. 降低维护成本；
3. 数据与监控更集中。

### 10.2 专项系统的优点

1. 针对性优化；
2. 更容易满足极端性能约束；
3. 不被通用抽象拖慢。

**一个成熟路线往往是**：基础设施尽量统一，真正影响业务差异的部分保留专项优化空间。

## 11. 决策模式十：立即上线灰度，还是继续离线打磨

这是工程现实中的高频问题。一般而言，满足以下条件更适合灰度：

1. 风险可回退；
2. 已有明确线上监控；
3. 离线收益虽有限，但方向正确；
4. 需要真实流量验证长尾问题。

反之，若安全风险高、回滚困难、离线评测尚不稳定，就不应急着上线。

## 12. 如何建立一套团队决策框架

建议每次重大路线选择都强制回答以下问题：

1. **当前问题类型是什么**：能力、成本、时延还是可靠性？
2. **约束是什么**：数据、硬件、时间、人力、合规？
3. 备选路线有哪些？
4. 每条路线的短期收益、长期收益和失败成本是什么？
5. 若路线失败，有哪些中间资产可复用？

把这些写清楚，会显著减少“拍脑袋式技术路线”。

## 13. 一个综合例子：企业文档智能系统

系统当前问题是复杂合同问答准确率不稳，且长文档时延偏高。面对这个情况，常见路线包括：

1. 更大 VLM；
2. 文档检索与重排优化；
3. 文档长尾数据回流；
4. 模型量化与缓存；
5. 评测体系重建。

如果评测显示问题主要集中于长页表格和 OCR 错误页，那么优先级更可能是：

1. 长尾数据 + 评测修正；
2. 检索与结构化重排；
3. 再考虑扩模；
4. 量化属于后续成本优化。

这比简单“换更大模型”更有把握。

## 14. 再看一个综合例子：仓库机器人 VLA

系统在简单抓取上很好，但复杂遮挡和失败恢复差。备选路线可能是：

1. 扩大 VLA 模型；
2. 引入动作分块和层级技能；
3. 采集失败恢复数据；
4. 引入世界模型做前瞻；
5. 增强实机安全与接管机制。

若当前最痛点是恢复失败而不是高层语义理解，那么更合理的顺序通常是：

1. 失败检测与恢复评测；
2. 失败数据回流；
3. 技能结构改造；
4. 再评估世界模型是否值得引入。

## 15. 一个形象比喻

技术路线选择有点像登山。你不是每次都需要最陡最快的路线，也不是每次都该带最重的装备。你要先看天气、体力、补给、队伍经验和时间窗口。某条路线理论上最短，但若天气不稳、队伍没训练过，可能反而风险最大。模型系统建设也是如此，路线正确与否，从来不是看它听起来多先进，而是看它是否与当前地形真正匹配。

## 16. 小结

常见技术决策的本质，不在于背诵“什么时候用 A、什么时候用 B”，而在于先把问题类型、约束条件和收益结构看清，再做取舍。真正成熟的团队不一定总能选对，但它们通常能把选错的成本降到最低，并让每次路线判断都留下可复用的经验。
