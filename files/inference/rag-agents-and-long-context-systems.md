# RAG、Agent 与长上下文系统

现代推理系统已经很少只是“给模型一个 prompt 然后生成答案”。很多真实系统更接近下面这种结构：

\[
\text{retrieval} \rightarrow \text{context assembly} \rightarrow \text{model} \rightarrow \text{tools/agents}
\]

**这意味着推理系统必须同时处理**：

- 检索：从外部文档、数据库或索引中找可用证据。
- 长上下文：把多份证据、历史和用户问题装进有限 token 预算。
- 工具调用：让模型在需要时调用搜索、数据库、OCR、浏览器或业务 API。
- 多轮状态：持续追踪已经查过什么、得到什么、下一步该做什么。

而这四件事一旦组合起来，系统的核心问题就会从“单次推理质量”转向“信息如何被组织、更新、裁剪和追踪”。这也是为什么很多看似是模型问题的线上故障，最后会落到检索、上下文装配或状态管理上。

!!! note "初学者先抓住"
    RAG、Agent 和长上下文不是简单把更多信息塞进 prompt。真正难点是证据怎么选、上下文怎么装、工具状态怎么追踪、旧信息什么时候丢掉。

!!! example "有趣例子：带资料参加会议"
    你不能把整个档案柜搬进会议室，只能带最相关的几页。RAG 负责找资料，context assembly 负责排版和取舍，Agent 则像会议中不断查资料、执行任务、更新记录的人。

## 1. 为什么 RAG 不只是“加个检索器”

RAG 的核心不只是把文档拼进 prompt，而是：

- 检索对象：决定查文档、表格、数据库、网页还是历史记录。
- 拼接预算：决定放多少证据进上下文，避免 token 被低价值材料占满。
- 证据排序：决定关键证据是否出现在模型更容易使用的位置。
- 噪声控制：避免无关信息塞爆上下文并误导模型。

如果检索链不稳，后面的模型常常会：

- 错误证据误导：模型会基于看似权威但不相关或过期的材料回答。
- 上下文变长但质量变差：token 变多不等于有效信息变多。
- prefill 成本上升：无关材料也要被编码进 KV cache，直接增加延迟和显存。

### 1.1 RAG 的真实目标是证据组织，不是文本召回

很多系统刚做 RAG 时，目标只是“把相关文本找回来”。但更成熟的目标应该是：

1. 找回的证据能否直接支撑回答；
2. 证据是否足够新；
3. 多份证据是否相互一致；
4. 在有限上下文预算内，这些证据是否值得保留。

换句话说，RAG 真正优化的是 evidence orchestration，而不只是 retrieval recall。

## 2. 上下文装配本身就是系统设计

设检索到的候选片段为 \(\{d_1,\dots,d_k\}\)，真正送入模型的上下文不等于全部片段简单拼接。实际需要解决：

\[
\mathcal{C}^\star = \text{Assemble}(d_1,\dots,d_k, q, B)
\]

其中 \(B\) 是上下文预算。这一步包括：

- 去重：移除重复片段，避免同一证据浪费上下文。
- 排序：把最关键、最新或最可信证据放在更合适位置。
- 裁剪：删掉和问题无关的段落、表格行或历史消息。
- 模板包装：保留来源、时间、权限和引用格式，方便模型正确使用证据。

### 2.1 为什么上下文装配经常比模型更影响效果

**因为它决定了**：

1. 哪些信息先被模型看到；
2. 哪些证据被截断；
3. 哪些元数据被保留；
4. 哪些证据会相邻出现，从而形成局部关联。

在长文档问答、法规审查和多模态文档系统里，这一步经常比“换更大模型”更直接地改变答案质量。

## 3. 长上下文不等于信息利用率高

很多系统把上下文长度加大后，确实“放得下更多文本”，但未必“用得好更多文本”。现实中的问题常是：

- 重要证据被埋没：关键句子被大量无关材料挤到不显眼的位置。
- prompt 冗长推高 prefill：模型必须先编码大量低价值 token。
- cache 压力增大：长上下文会线性增加 KV cache 占用。

所以长上下文系统的关键是有效信息密度，而不只是 token 上限。

### 3.1 一个简单的信息密度视角

如果一个 100k 上下文里只有 5k token 与当前问题真正相关，那么剩余 95k token 并不是无害背景，它们还会：

1. 增加 prefill 成本；
2. 稀释模型注意力；
3. 放大无关证据混入带来的风险；
4. 占据缓存预算。

因此长上下文能力若没有配套压缩、重排和淘汰策略，常常只会把系统拖慢，而不会让系统更聪明。

## 4. Agent 系统为什么更复杂

当系统不只回答，而是还会：

- 调搜索：获取公开网页、知识库或实时信息。
- 调数据库：查询结构化业务数据、订单、用户或指标。
- 调 OCR：从图片、票据、截图或扫描件里读取文字。
- 调浏览器：执行网页导航、点击、填写和状态检查。

它就不再是单步生成，而是一个多步控制系统：

\[
a_t \sim \pi_\theta(a_t \mid c_t),\qquad c_{t+1}=U(c_t,a_t,o_t)
\]

其中 \(o_t\) 是工具返回结果。

### 4.1 Agent 为什么更像控制而不是聊天

每一步动作都会改变未来可见信息。模型选择了哪个工具、取回了什么结果、是否继续检索，都会直接影响下一轮决策。因此 agent 更像闭环系统，而不是多轮版 prompt。

这意味着评测 agent 不能只看最终回答，还必须看：

1. 是否会进入低价值工具循环；
2. 是否会在错误证据上越走越偏；
3. 是否会在必要时终止探索；
4. 是否能管理中间状态而不把上下文越堆越乱。

## 5. RAG 与 Agent 的交叉点

**很多真实系统两者会合并**：

- 先检索材料：用 RAG 获取第一批证据和候选方向。
- 再决定工具：根据证据缺口调用数据库、OCR、搜索或浏览器。
- 最后综合证据：把多步工具结果和检索材料合成可引用答案。

这意味着推理系统的真正难点，已经不只是 decode，而是：

- 状态管理：记录已检索证据、工具结果、用户约束和当前任务阶段。
- 上下文复用：避免每轮都重复塞入相同材料，降低 prefill 成本。
- 工具链容错：工具失败、超时或返回冲突结果时要能降级和重试。

### 5.1 一个典型工作流

在企业知识助手中，系统经常会走：

1. 先检索制度文档；
2. 再从检索结果判断是否需要调用数据库；
3. 若数据库结果与文档冲突，再次召回更多证据；
4. 最终输出答案并给出引用。

在这个过程中，RAG 和 agent 早已不是两条线，而是一个统一的多阶段信息流。

## 6. 长上下文系统最常见的瓶颈

### Prefill 成本

上下文越长，prefill 越贵。

### Cache 压力

KV cache 很快成为显存大头。

### 证据污染

无关检索片段反而降低答案质量。

### 多轮漂移

Agent 在多步工具调用后，上下文容易越来越乱。

### 证据版本冲突

检索到的不同文档可能来自不同时间、不同权限层或不同版本。若系统不处理这些冲突，模型会在相互不兼容的证据上推理。

### 状态膨胀

随着多轮交互进行，系统会不断追加：

1. 工具返回结果；
2. 中间推理文本；
3. 历史检索片段；
4. 用户追问。

没有压缩与淘汰机制时，系统很快会变成“上下文很长，但自己也说不清当前状态是什么”。

## 7. 一个直观例子：企业合规问答代理

**系统可能要同时处理**：

- 合同：提供具体义务、违约条款和责任边界。
- 邮件：补充协商过程、时间线和双方承诺。
- 内部流程文档：说明公司内部审批和执行规则。
- 法规片段：提供外部法律或合规约束。

用户问的是一句话，但系统背后可能经历：

1. 检索相关条款；
2. 再检索内部规范；
3. 调表格解析工具；
4. 汇总结果。

这已经不是普通 chat inference，而是完整的多阶段信息流。

### 7.1 这个例子里最容易出错的地方

1. 初始检索召回了过期法规；
2. 内部规范与公开法规版本不一致；
3. 表格工具把金额字段识别错；
4. 最终答案只引用了其中一条证据，却忽略了另一条冲突证据。

用户最终只会看到“答案错了”，但系统内部其实可能是多个环节一起导致的失败。

## 8. 工程建议

做这类系统时，建议单独评估：

1. 检索质量
2. 上下文装配质量
3. 模型推理质量
4. 工具调用成功率

不要只看最后答案。否则出了问题，你会很难知道是检索错了、拼接错了，还是模型没用好证据。

### 8.1 建议增加的专项评测

**更成熟的系统还应增加**：

1. 证据一致性评测；
2. 上下文压缩前后效果对比；
3. 多轮工具链稳定性评测；
4. 长上下文成本与质量联合评测；
5. 版本更新后的回归集。

## 9. 一个实用抽象

**可以把这类系统拆成三层**：

1. `retrieval layer`：决定找什么；
2. `assembly layer`：决定怎么组织证据；
3. `reasoning layer`：决定怎么用这些证据和工具完成任务。

分层优化往往比端到端黑箱调 prompt 更有效，也更便于定位问题。

## 10. 一个总判断

RAG、Agent 和长上下文系统的结合，代表着推理系统已经从“单次生成”走向“受约束的信息编排与决策系统”。因此它们更像系统工程问题，而不是单纯模型推理问题。真正强的系统，不只是能读很多 token，而是知道何时检索、何时压缩、何时调用工具、何时停止继续扩张上下文。 


## 实践补充与检查

### 用服务系统视角重写 **RAG、Agent 与长上下文系统**

推理主题最容易被写成一串优化名词，但线上系统真正关心的从来不是“有没有这个技巧”，而是 **它是否改变了请求、队列、缓存、成本和高价值流量的命运**。围绕 **RAG、Agent 与长上下文系统**，更稳的理解框架是先问：这件事究竟在改变哪一层约束。

1. **检索质量、上下文压缩、工具链时延、长文档 prefill、会话级反馈**。

只要这几层没有被明确分开，系统就很容易出现一种假象：离线 benchmark 更好了，线上却并没有更稳；或者平均时延下降了，但高价值租户与长尾任务反而更差。对推理系统来说，这类错觉比“完全没有优化”更危险，因为它会把团队引向错误的资源分配方向。

### 更扎实的设计顺序

针对 **RAG、Agent 与长上下文系统**，建议把设计顺序固定成下面几步：

1. **先画请求生命周期图**，明确哪些阶段会受这项机制影响；
2. **再画资源账本**，把 GPU、显存、KV、网络、工具链和 CPU 依赖拆开；
3. **然后才讨论策略本身**，例如要不要路由、要不要 cache、要不要 speculative、要不要分层服务；
4. **为高价值流量单独建桶**，不要只看总体平均；
5. **最后定义回退与降级**，确保高峰期和故障期也有可执行动作。

这个顺序的价值在于：它迫使团队先把 **RAG、Agent 与长上下文系统** 放进完整服务图里，再讨论局部技巧，否则很容易陷入“看起来懂很多优化，但不知道哪个最值钱”的状态。

### 真实线上最常见的失败模式

围绕 **RAG、Agent 与长上下文系统**，线上失败常常并不体现在单次崩溃，而是慢性失真：

1. **只靠塞更多上下文**。
2. **agent 链路太长无人观测**。
3. **离线检索分数无法解释线上效果**。

这些问题之所以难，是因为它们往往混在一起发生。一个路由变化可能同时影响缓存命中、TTFT、单位成本和用户重试率；一个 cache 策略调整可能同时影响长上下文任务、KV 生命周期和多租户公平性。因此，**RAG、Agent 与长上下文系统** 相关改动必须天然携带可观测性，否则等于把复杂度埋进黑箱。

### 更像生产系统的验收标准

对 **RAG、Agent 与长上下文系统**，更靠谱的验收通常不是一句“线上变快了”，而是至少同时回答：

1. **区分检索失败与推理失败**。
2. **记录每轮工具收益**。
3. **长上下文走专门通路**。
4. **维护 session 回放**。

再进一步，建议把这类主题纳入持续治理：

1. **按任务桶和租户桶做长期趋势跟踪**；
2. **保留代表性 replay / shadow case**，用于版本切换对照；
3. **把高价值失败样本回流到训练和系统修复**；
4. **让 oncall 面板能直接看到与它相关的 route、cache、queue 和 fallback 状态**。

只要这套闭环建立起来，**RAG、Agent 与长上下文系统** 才真正从“优化点”变成“系统能力”。
