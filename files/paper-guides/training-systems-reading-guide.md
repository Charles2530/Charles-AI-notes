# 训练与系统论文导读

这条导读的目标，是帮助你把“模型是怎么训出来的”和“模型为什么能稳定跑起来”放到同一条主线上看。很多人会把“训练论文”和“系统论文”分开读，结果形成两个割裂的世界：

1. 一边谈 loss、目标函数和 benchmark；
2. 一边谈吞吐、缓存和延迟；
3. 另一边又谈 kernel、Triton、CUDA、Megatron、DeepSpeed。

但真实项目里，这几条线是连着的。  
训练阶段塑造什么能力和行为，最终会直接影响：

1. 推理长度；
2. 工具使用模式；
3. 上下文需求；
4. 容量规划；
5. 安全边界；
6. 算子和 kernel 热点。

因此这条导读的重点，就是把它们重新接回一条因果链。

!!! note "初学者先抓住"

    训练论文和系统论文不能分开读。训练目标和数据决定模型行为，系统栈决定它能否稳定、便宜、可复现地训出来，推理系统又反过来约束可接受的模型结构和输出方式。

!!! example "有趣例子：造车不是只研究发动机"

    只懂模型 loss 像只懂发动机，能解释动力来源；但真正上路还要底盘、油耗、维修、道路和安全。训练系统论文就是把这些工程条件补齐。

## 1. 先建立一条总主线

建议你把训练与系统研究先压缩成下面这条主线：

```text
预训练
 -> 指令微调
 -> 偏好对齐
 -> 数据系统
 -> 分布式训练
 -> Kernel / 编译栈
 -> 推理服务与部署
```

如果缺少其中任何一段，你对模型系统的理解都会偏：

1. 只懂预训练，不懂行为塑形；
2. 只懂对齐，不懂底座能力边界；
3. 只懂 loss，不懂数据和吞吐；
4. 只懂系统，不懂这些系统到底在服务什么训练目标。

## 2. 第一阶段：先理解预训练在塑造什么

读预训练类工作时，最该回答的问题不是“用了多少参数”，而是：

1. 它在塑造哪种能力边界；
2. 数据规模和模型规模怎样互动；
3. 为什么简单的 next-token prediction 会学出复杂能力；
4. 为什么有些能力在数据扩张后是平滑出现，有些则像阈值现象。

如果这个底层认识没有建立，后面的 `SFT` 和 `DPO` 会显得像在“凭空变聪明”。

### 2.1 读预训练时不要只盯损失曲线

一个常见误区是把预训练理解成“loss 降低过程”。  
**更实用的读法是一直问**：

1. 这个目标函数究竟在塑造哪类可迁移能力？
2. 数据分布决定了模型更擅长什么，不擅长什么？
3. token 预测为何能隐式学到推理、规划、代码模式和世界知识？
4. 预训练的上限究竟受模型规模、数据规模还是数据质量主导？

### 2.2 Scaling law 的价值是什么

缩放律工作的价值，不只是告诉你“更大更强”，而是帮助你建立资源分配直觉。  
**例如粗略形式**：

\[
\mathcal{L}(N, D, C)
\approx
\mathcal{L}_\infty
 + aN^{-\alpha}
 + bD^{-\beta}
 + cC^{-\gamma},
\]

其中 \(N\)、\(D\)、\(C\) 分别对应参数、数据和算力规模。  
**这类工作帮你理解**：

1. 为什么有时该补数据，而不是盲目加参数；
2. 为什么 compute-optimal training 重要；
3. 为什么预训练阶段的资源决策会决定下游全部天花板；
4. 为什么“同样算力预算”下，不同分配策略差异可能很大。

## 3. 第二阶段：从底座能力到任务行为

指令微调类工作值得读，不是因为它们总有最新 benchmark，而是因为它们让你看清：

1. 同一个底座，为什么会因为数据组织不同而表现出完全不同的行为；
2. 为什么 `SFT` 更像行为克隆，而不是重建世界知识；
3. 为什么输出格式、回答风格和工具调用倾向会被强烈塑形；
4. 为什么多轮、结构化、工具型 SFT 会直接影响服务负载。

这一步非常适合理解“能力”和“行为”的分工。

### 3.1 一个简单的直觉

**可以把模型输出理解成**：

\[
p(y|x) = \text{capability} \times \text{behavior policy}.
\]

预训练更像在塑造“能不能”，SFT 更像在塑造“平时倾向怎么做”。  
它不会凭空创造所有能力，但会强烈改变：

1. 回答结构；
2. 风格；
3. 是否愿意执行任务；
4. 是否会优先调用工具；
5. 是否习惯给冗长解释。

### 3.2 读 SFT 论文时最该看的不是“用了多少指令”

**而是**：

1. 指令数据来自哪里；
2. 是否覆盖关键任务分布；
3. 输出格式约束是否清楚；
4. 是否有多轮上下文；
5. 是否存在高质量拒答或澄清样本；
6. 是否引入了行为模板化副作用。

这些因素对最终行为影响常常比“多几个点 benchmark”更有现实意义。

## 4. 第三阶段：偏好对齐不是额外装饰，而是分布重塑

`RLHF` / `DPO` 一类工作要重点看三件事：

1. 它们在优化什么偏好；
2. 它们如何改变输出风格和风险边界；
3. 为什么这种变化有时会压制能力，有时会提升体验。

**如果读完后你能区分**：

1. 预训练问题；
2. SFT 问题；
3. 对齐问题；

那这条线就已经建立起来了。

### 4.1 对齐读法的关键

读对齐论文时，不要只看公式推导。  
**更重要的是问**：

1. 偏好标签究竟从哪里来？
2. 奖励模型或隐式奖励在鼓励什么行为？
3. 哪些行为在离线评测里看起来更优，但在线上反而更糟？
4. reference model 和 KL 约束起了多大作用？

### 4.2 为什么对齐和系统相关

对齐策略会直接影响系统层的很多问题，例如：

1. 回复更长还是更短；
2. 是否频繁拒答；
3. 是否更爱调用工具；
4. 是否喜欢给结构化输出；
5. 是否倾向多步思考。

**这些都会改变**：

1. 平均输出 token；
2. 工具链调用频率；
3. 服务延迟；
4. 容量规划；
5. cache 行为。

所以训练和系统根本不是断开的。

## 5. 第四阶段：训练论文必须和数据系统一起看

现实里，一个模型训得再漂亮，也会马上遇到：

1. 数据去重；
2. packing；
3. curriculum；
4. batch 组织；
5. 数据质量漂移；
6. 长尾样本回流；
7. 数据版本化。

很多训练论文如果只看优化器和目标函数，会漏掉最决定成败的一层：  
**数据生产线本身。**

### 5.1 为什么数据系统是“隐形模型”

同一个训练算法，不同数据系统会产生非常不同的模型行为。  
**因为数据系统决定**：

1. 看到了什么；
2. 没看到什么；
3. 某类行为被重复强化多少次；
4. 长尾失败模式是否能被回流；
5. 哪些 token 真正变成了有效监督。

因此读训练系统论文时，建议特别关注：

1. 去重策略；
2. 数据过滤标准；
3. mixture 比例；
4. 长短样本混合方式；
5. packing 对 token 利用率的影响；
6. 数据与系统指标如何被联动监控。

## 6. 第五阶段：分布式训练是主问题，不是附录

到了几十亿参数以上，训练论文必须继续往下读到：

1. 数据并行；
2. 张量并行；
3. 流水线并行；
4. ZeRO / FSDP；
5. activation checkpointing；
6. checkpoint 与恢复。

这条线的关键，不是记住术语，而是看清楚：

1. 为什么单卡思维会失效；
2. 通信与显存如何共同约束训练；
3. 同样的模型结构，为什么系统栈不同会产生完全不同吞吐和成本。

### 6.1 读这一层时建议重点追三类名字

1. `Megatron-LM / Megatron Core`：理解多维并行和大模型训练参考实现；
2. `DeepSpeed / ZeRO`：理解状态分片、offload 和 pipeline runtime；
3. 原生 `FSDP` 路线：理解全分片训练如何融入主流框架生态。

这些名字之所以重要，不是因为它们必须被全部使用，而是因为它们分别代表了训练系统的不同工程思路。

## 7. 第六阶段：Kernel 与编译栈必须补上

如果读到这里就停，你仍然会漏掉一层：为什么这些训练和推理系统最后都会回到 kernel 问题。

**建议在这一步开始补下面几条线**：

1. FlashAttention；
2. quantized GEMM；
3. Triton kernel；
4. CUDA / CUTLASS / CuTe；
5. 编译器如 Inductor / TensorRT-LLM / TVM / XLA 等的角色。

### 7.1 为什么这一步不能省

因为：

1. 推理慢不一定是调度问题，可能是 decode kernel 问题；
2. 训练吞吐低不一定是并行设计问题，可能是数据 layout 或 kernel 问题；
3. 量化没提速不一定是量化算法问题，可能是 dequant kernel 问题。

你不一定要手写 CUDA，但必须能读懂系统 bottleneck 为什么会落到这一层。

### 7.2 这也是为什么你给的参考源很互补

1. `AI System Docs` 提供从芯片、编译器、框架到推理系统的全栈纵深；
2. `how-to-optim-algorithm-in-cuda` 提供 GEMM、Softmax、LayerNorm、FlashAttention、Triton、Cutlass 等算子实践入口；
3. 它们一起能把训练系统阅读从“会看论文”推进到“会看实现”。

## 8. 第七阶段：训练论文最终要落到推理和部署

很多人读到 DPO、SFT 或 ZeRO 就停了。  
但真实系统里，一个模型训好后还要面临：

1. continuous batching；
2. cache；
3. prefix reuse；
4. tail latency；
5. route to model；
6. 量化和部署；
7. observability 与 online evaluation。

所以建议这条导读不要停在对齐或训练本身，而要继续补服务与系统层的代表性工作。

### 8.1 一个很现实的例子

假设某次对齐让模型更乐于详细解释。  
离线用户评分上升了。  
**但上线后你发现**：

1. 平均输出 token 增加 `35%`；
2. decode 时间显著拉长；
3. P95 延迟恶化；
4. 工具调用次数上升。

这就是训练决策直接改变系统负担的例子。

这个例子真正想说明的是：训练和系统不是先后关系，而是因果关系。一次看似只影响“回答风格”的对齐更新，可能立刻改写服务的 token 预算、缓存行为和工具链负载。若你在读论文时没有把这条因果链记住，就会很容易把系统问题误判成服务实现问题，而忽略了它其实起源于训练目标。

## 9. 如果你只想建立一个“够用的阅读骨架”

**建议用下面六层骨架去组织阅读**：

1. **能力上限**：预训练与 scaling；
2. **行为塑形**：SFT；
3. **偏好重塑**：RLHF / DPO；
4. **数据生产线**：去重、mixture、packing、回流；
5. **训练底座**：并行、checkpoint、Megatron、DeepSpeed、FSDP；
6. **推理与算子**：FlashAttention、continuous batching、KV cache、量化 kernel、编译栈。

只要这六层串起来，你对训练与系统的理解就已经不再碎片化。

这六层骨架的真正用途，不是帮你背目录，而是帮你在读任何一篇论文时迅速定位它到底改了哪一层、收益转移到了哪一层、代价又落到了哪一层。只要这件事做得到，阅读就会从“记住结论”变成“建立系统地图”。

## 10. 建议阅读顺序

### 10.1 第一层：能力上限

**建议至少读**：

1. 一篇预训练或 scaling law 代表工作。

**目标**：理解底座能力如何随数据和规模变化。

这一层建议精读，因为它决定了后面所有阶段的能力天花板。若你没理解预训练和 scaling 在塑造什么，就很容易把后训练收益看得过大，或者把系统问题错误归因到模型“还不够大”。

### 10.2 第二层：行为塑形

**建议读**：

1. 一篇 instruct / SFT 代表工作；
2. 一篇多轮或工具型 SFT 代表工作。

**目标**：理解行为和格式是如何被训练数据塑形的。

这一层最重要的收获，是把“会不会”和“平时怎么做”分开。很多真实系统的问题，其实不是底座没有能力，而是行为分布没有被正确塑形。

### 10.3 第三层：偏好与对齐

**建议读**：

1. 一篇 `RLHF` 或 InstructGPT；
2. 一篇 `DPO`。

**目标**：理解为什么“更像助手”既可能提升体验，也可能压制能力。

建议在这一层特别关注 reward 信号到底在鼓励什么。对齐论文最容易被误读成“让模型更安全、更好用”，但它们经常也会同时改变回复长度、保守性、拒答阈值和工具使用倾向，这些都会直接进入系统层。

### 10.4 第四层：数据系统与实验经济学

**建议读**：

1. 一篇数据去重 / mixture / curriculum / data quality 工作；
2. 一篇 scaling 和 compute allocation 工作。

**目标**：理解训练不是只拼模型，也是在拼数据生产线。

这层若看懂了，你会开始自然追问：某个结论到底是模型改动带来的，还是数据过滤、mixture、packing 和回流策略带来的。很多训练提升如果不把这一层看进去，其实是无法正确归因的。

### 10.5 第五层：训练系统

**建议读**：

1. `Megatron-LM` / `Megatron Core` 相关资料；
2. `DeepSpeed ZeRO` 和 pipeline 相关资料；
3. 一篇 FSDP 或 checkpoint / recovery 路线资料。

**目标**：理解训练为什么会被并行、显存、通信和容错塑形。

这一层最值得建立的直觉是：很多“模型训练设计”本质上是系统设计。上下文能开多长、global batch 能开多大、恢复是否稳定、低精度是否可用，最终都会改写研究能做什么实验、能不能复现以及能否真正迭代。

### 10.6 第六层：推理与算子

**建议读**：

1. 一篇 LLM serving / continuous batching 工作；
2. 一篇 attention / cache / long context 系统工作；
3. 一篇 FlashAttention 或 Triton / CUDA 算子实现资料。

**目标**：把训练端的能力和行为变化接到服务端成本与硬件实现上。

把这一层补上之后，前面五层才真正闭环。否则你会知道模型怎么训，却不知道为什么上线后成本突然变高、tail latency 变差、量化提速没兑现，或者某个行为变化为什么会变成 decode 热点。

## 11. 阅读时最值得记的十个问题

阅读这条线时，建议每篇都记下面十个问题：

1. 这篇工作主要改的是能力、行为，还是系统？
2. 它默认底座已经具备什么？
3. 它的收益更像“提高上限”还是“改善体验”？
4. 它对数据系统提出了什么要求？
5. 它对推理长度、工具调用或输出格式有什么影响？
6. 它留给工程落地的最大难点是什么？
7. 它最可能在哪个环节把成本转移出去？
8. 它是否依赖特定硬件或训练栈？
9. 它的评测指标和真实部署指标是否一致？
10. 如果你要复现，最先会卡在哪一层？

这十个问题会强迫你把论文从“结论摘要”变成“系统理解”。

## 12. 三种常见误区

### 12.1 只读模型论文，不读系统论文

**结果通常是**：

1. 理解了 benchmark；
2. 却不知道为什么复现不出来；
3. 更不知道为什么上线以后成本完全不同。

这种误区的本质，是把论文里的“能力提升”误以为能自动转化为工程收益。现实里，很多收益会在并行策略、数据路径、内存系统、KV cache 或工具链里被重新折算。

### 12.2 只读系统论文，不读训练目标

**结果通常是**：

1. 知道怎么把吞吐做高；
2. 却不知道模型到底被训练成了什么样；
3. 不知道行为变化会不会破坏产品体验。

这类误区常见于只盯 serving、量化和 kernel 的阅读方式。系统可以把一件事做得很快，但如果你不知道模型被优化成了什么行为分布，也很难判断“更快”是否还在为正确目标服务。

### 12.3 只读摘要，不读限制条件

**很多工作的真正信息量在于**：

1. 数据前提；
2. 评测口径；
3. 硬件环境；
4. shape 分布；
5. 哪些场景不成立。

如果忽略这些，只会积累很多没有边界感的“方法印象”。

而没有边界感，正是技术判断最危险的状态。你会记得某个方法“很强”“很省”“很稳”，却不知道这些结论是在哪种数据、哪种卡型、哪种流量画像和哪种任务约束下才成立。

## 13. 如何把论文阅读和工程文档结合起来

比较好的做法是每读完一类论文，就回看本知识库对应专题：

1. 读完预训练 / SFT / DPO，看训练章节；
2. 读完 ZeRO / Megatron / checkpoint，看分布式训练章节；
3. 读完 FlashAttention / Triton / CUDA，看算子与编译器章节；
4. 读完 serving 论文，看推理章节；
5. 读完世界模型工作，看世界模型章节。

这样能把“论文结论”变成“系统知识结构”，而不是停留在 paper list。

更实用一点的做法，是每读完一个阶段就顺手写一张两列表：左边记论文结论，右边记它在系统里的接口位置。久而久之，你得到的就不是一堆摘要，而是一张能指导训练、推理和部署决策的知识图。

## 14. 小结

训练与系统论文真正难的地方，不在于单篇有多复杂，而在于它们跨越了能力、行为、数据、并行、kernel、编译器和服务部署多个层次。只要把这些层次拆开又重新连起来，你就会逐渐建立一种更成熟的判断力：

1. 某个提升到底来自哪里；
2. 某个代价被转移到了哪里；
3. 某个方法适合研究原型，还是适合真实系统。

也只有在这种视角下，“训练论文”和“系统论文”才不再是两个世界，而是一条从目标函数一直延伸到 GPU kernel 与线上容量规划的完整因果链。


## 实践补充与检查

### 把 **训练与系统论文导读** 写成真正可执行的阅读路径

导读页最容易的问题，是“列得很全，但读者读完仍然不知道该先看什么、为什么看、读到哪一步算过关”。围绕 **训练与系统论文导读**，更有价值的写法是把导读变成一种执行框架，而不只是导航页。

更具体地说，把训练系统阅读顺序、关键问题和复现难点补齐。这意味着导读页不应只给论文名，还应告诉读者：应该先建立哪几个核心问题意识，哪些论文负责打开视野，哪些论文负责建立方法主线，哪些系统论文负责让抽象概念落到真实工程里，哪些内容适合快速跳读，哪些地方最值得做复现和做笔记。

### 更像学习计划的组织方式

更成熟的导读页可以按三条线并行组织：

1. **问题线**：这个方向最本质的问题是什么；
2. **方法线**：主流方法是如何一步步逼近这些问题的；
3. **工程线**：哪些结论一旦落到系统、数据、评测和部署上会发生变化。

一旦这三条线写清楚，读者就不容易陷入“知道很多论文标题，但仍然没有判断力”的状态。尤其对系统、世界模型、VLM/VLA 这类主题，真正值钱的往往不是读过多少论文，而是能不能把论文放回完整问题空间里判断。

### 导读页也应明确复现与跳读策略

很多读者最需要的其实不是再多十篇论文，而是知道：

1. 哪几篇适合精读；
2. 哪几篇只需抓核心图表和结论；
3. 哪几篇如果不看源码或系统实现就很难真正理解；
4. 哪些部分最适合做复现、做对照实验或做总结表。

只要把这些策略写进导读页，**训练与系统论文导读** 就会从“目录”升级成“真正能帮人建立学习路径的工具页”。


### 导读页最该补的是接口说明

围绕 **训练与系统论文导读**，真正高价值的扩写不是继续堆论文，而是把 并行、checkpoint、数据系统和稳定性页之间的接口 讲清楚。这样读者才知道：看完这页后，应该跳到哪一类方法页、系统页或评测页，才能把阅读路径真正闭环起来。

### 一条更实用的执行顺序

更好的导读页应明确告诉读者：先看哪些奠基问题，再看哪些代表性方法，再看哪些工程化页面，最后在哪些位置做复现或做总结表。只要这条顺序写清楚，导读页就不再只是入口，而会真正变成学习计划。

### 后续最值得继续补强的内容

导读页后续最值得加厚的，通常不是再加一串参考文献，而是补：跳读策略、复现优先级、经典误读、系统化做笔记的方法，以及不同背景读者各自最短的入门路径。
