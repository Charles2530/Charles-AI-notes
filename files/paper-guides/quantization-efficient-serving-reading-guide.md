# 量化与高效推理论文导读

这条导读适合你已经有一个可用模型，现在关心：

- 显存够不够
- 延迟稳不稳
- 吞吐高不高
- 成本能不能降
- 量化之后模型还能不能继续训练、继续部署、继续演进

它要帮你建立的主线不是“背几篇压缩论文”，而是形成下面这条完整理解链：

```text
后训练量化
 -> 激活与权重误差控制
 -> 量化底座上的训练
 -> cache / routing / speculative
 -> serving 系统与容量治理
```

如果你读完后仍然只记得“GPTQ 很火、AWQ 很常用、QLoRA 很省显存”，那说明这条主线还没有真正建立起来。  
**你最终应该能回答的是**：

- 为什么某些量化论文在线下分数很好，线上却未必提速？
- 为什么有时真正的瓶颈不在权重，而在 KV cache？
- 为什么“模型压缩”和“系统优化”必须放在一起看？

!!! note "初学者先抓住"

    量化与高效推理论文要按瓶颈迁移来读：先看权重和激活误差，再看 KV cache、batching、routing、speculative 和 runtime。线上收益来自完整服务链路，不只是某个低比特方法。

!!! note "难点解释：为什么线下分数和线上速度会脱节"

    线下量化评测通常关注质量掉点，线上服务还要处理 dequant、kernel layout、cache、batch、tail latency 和 fallback。质量不掉只能说明“可能可用”，不能说明“端到端更快”。

## 1. 先明确：这条导读读的不是单点技巧，而是系统演化

量化与高效推理这条线最容易被读散。  
表面上看，相关论文分别在讨论：

- 低比特表示
- 精度损失
- 解码优化
- cache 管理
- batch 调度
- serving 框架

但如果从工程视角重排，它们其实都围绕一个总目标：

\[
\min \text{cost},\qquad
\min \text{latency},\qquad
\max \text{quality}.
\]

所以这条导读不是按术语堆叠，而是按“瓶颈迁移”来读。  
**也就是**：

1. 先解决模型太大、装不下。
2. 再解决虽然装下了，但不够快。
3. 再解决虽然快了，但长上下文或高并发时仍不稳。
4. 最后把这些点拼成完整服务系统。

## 2. 第一阶段：先建立量化误差直觉

建议先读 `GPTQ`、`AWQ`、`SmoothQuant` 这一组工作。  
阅读重点不是代码实现，而是先建立误差直觉：

- 量化误差为什么不能只看权重本身
- 为什么激活分布会决定量化成败
- 为什么“可部署”和“理论压缩率高”不是一回事
- 为什么同样是 4bit，不同方法实际效果能差很多

### 2.1 阅读时先抓公式含义，不急着记所有细节

**一个最基本的量化近似写法是**：

\[
q = \mathrm{clip}\left(\left\lfloor \frac{x}{s} \right\rceil + z \right),\qquad
\hat{x} = s(q-z).
\]

读论文时，先抓住这里的三个问题：

1. `scale` 和 `zero-point` 是怎么选的？
2. 误差主要来自截断、饱和，还是传播？
3. 论文是在保护权重、激活，还是某些敏感通道？

### 2.2 GPTQ 适合建立“二阶误差补偿”直觉

阅读 GPTQ 时，不必一开始陷在实现细节里。  
**先记住一个基本近似**：

\[
\Delta \mathcal{L}
\approx
\frac{1}{2}(w-\hat{w})^\top H (w-\hat{w}).
\]

它表达的不是“要精确求 Hessian”，而是提醒你：  
不同方向上的量化误差影响不一样大。  
**这会帮助你形成一个重要判断**：

**量化不是平均误差最小化，而是要保护真正敏感的方向。**

### 2.3 AWQ 适合建立“关键通道保护”直觉

**AWQ 很适合让你理解**：

- 权重是否重要，不只由权重自身大小决定
- 激活分布会把某些通道的重要性放大很多倍

读到这里时，建议你反复问自己：

**如果一个通道平时很少被激活，但一旦激活就至关重要，量化时该如何保护它？**

### 2.4 SmoothQuant 适合建立“激活 outlier 是第一麻烦”直觉

很多人一开始把量化难点理解成“权重不够精确”。  
但大量部署实践表明，激活尖峰往往更棘手。  
SmoothQuant 的重参数化思想：

\[
W x = (W D^{-1})(D x)
\]

**可以帮你意识到**：  
有时最好的量化办法不是硬扛 outlier，而是把问题换个位置处理。

## 3. 第二阶段：理解量化不只用于部署

`QLoRA` 这类工作非常适合放在第二阶段读，因为它回答了一个很现实的问题：

**当底座已经量化后，模型还能不能继续被高效适配？**

这能帮助你把“模型压缩”和“训练适配”两条线接起来。

### 3.1 为什么这一步重要

**很多人把量化理解成部署的终点**：模型训完了，最后再压一下。  
但现实里，模型上线后还可能需要：

- 领域微调
- 指令微调
- 适配企业数据
- 快速增量更新

这时如果量化底座完全不能训练，工程价值就会打折。

### 3.2 读 QLoRA 时重点看什么

不要只记住“4bit + LoRA 很省显存”。  
**更重要的是理解**：

- 为什么底座可以量化，而适配参数仍保持较高精度
- 为什么这种做法能在资源有限时保留可训练性
- 它在哪些任务上足够强，在哪些任务上容量可能不够

## 4. 第三阶段：从模型压缩走向服务系统

很多人读完量化论文会误以为部署问题快结束了。  
**现实里瓶颈会迅速转移到**：

- KV cache
- prefill
- scheduling
- tail latency

**所以建议接着补**：

- continuous batching
- prefix cache
- speculative decoding
- route-to-model 的系统工作

**这里最关键的认识是**：

**模型文件变小，不等于系统一定更快。**

### 4.1 为什么量化后不一定提速

真实系统里，速度还会受到：

- 低比特 kernel 是否成熟
- 反量化开销
- 显存访问模式
- batch 大小时的吞吐曲线
- cache 读写成本

影响。

因此阅读 serving 论文时，要有意识地把“算法压缩率”与“端到端收益”区分开。

### 4.2 KV cache 是很多团队第二次踩坑的地方

权重量化做好后，模型似乎已经装下了。  
可一旦进入长上下文、多轮对话或 agent 任务，KV cache 可能迅速成为真正瓶颈。

读这部分工作时，建议重点追问：

- cache 占显存的比例有多高？
- cache 的精度下降会如何影响长程依赖？
- cache 压缩和 prefix reuse 能否结合？

## 5. 第四阶段：真正的工程问题是综合优化

**到这一步你应该逐渐明白**：  
部署优化不是单点问题，而是一个多目标系统问题。

\[
\min \text{cost},\qquad
\min \text{latency},\qquad
\max \text{quality},\qquad
\max \text{stability}.
\]

量化、cache、routing、speculative 都只是不同抓手。  
这条导读的价值，就是把它们放到同一个框架下看。

### 一个生动例子

**设想你要上线一个企业知识助手**：

- 权重量化后模型终于能塞进 2 张卡
- 但用户常问长文档，prefill 很慢
- 你加了 prefix cache，短请求变快
- 上线后又发现少量超长请求把 P99 拖爆
- 于是又要做路由分层和长请求隔离

这说明高效推理不是某一篇论文的功能，而是一整串相互作用的设计。

## 6. 建议阅读顺序

下面给出一个更实用的阅读顺序，而不是按年份硬排。

### 6.1 起步包：先建立误差与部署直觉

建议至少从下面几类中各读一篇代表工作：

1. GPTQ
2. AWQ
3. SmoothQuant

这一步目标不是背方法细节，而是建立下面的判断：

- 权重误差和激活误差不是一回事
- offline 精度和 online 吞吐也不是一回事

### 6.2 第二包：把训练适配接上

**建议接着读**：

1. QLoRA
2. 一篇 QAT 或量化训练代表工作

**这一步要建立的认识是**：

- 量化不是部署末端操作，也可以是训练系统的一部分
- 底座压缩后，模型仍有可能继续学习

### 6.3 第三包：从模型走向服务

**建议读**：

1. 一篇 LLM serving / continuous batching 代表工作
2. 一篇 speculative decoding 代表工作
3. 一篇 KV cache 优化工作

**这一步的重点是**：  
把你对“模型压缩”的理解升级为对“推理系统”的理解。

## 7. 阅读时最值得记的七件事

读这条线的论文时，建议每篇都按下面七个问题记笔记：

1. 这篇工作优化的是权重、激活、KV cache，还是服务调度？
2. 它改善的是显存、吞吐，还是延迟？
3. 它的收益最主要体现在离线 benchmark 还是线上服务？
4. 它的副作用主要体现在质量、复杂度还是兼容性？
5. 它需要什么硬件或 kernel 支持？
6. 它是否只是把瓶颈转移到了别的地方？
7. 它最适合单卡研究、批量离线，还是生产服务？

如果你持续这样记录，阅读会越来越系统，而不是变成“又记了一个新缩写”。

## 8. 最小阅读包

如果时间非常有限，建议至少建立如下阅读包：

1. GPTQ
2. AWQ
3. SmoothQuant
4. QLoRA
5. 一篇 LLM serving / continuous batching 代表工作
6. 一篇 speculative decoding 或 cache 优化代表工作

如果有额外时间，再补：

7. 一篇 QAT 代表工作
8. 一篇 FP8 / mixed precision 工程报告
9. 一篇 KV cache 量化或 eviction 工作

## 9. 读完后你应该能形成的最终判断

真正成熟的“量化与高效推理”理解，不会停留在“模型压到几 bit”。  
你应该能用一句完整的话描述一个方案：

- 它在什么层面压缩了什么对象；
- 它把误差控制在什么位置；
- 它兑现收益依赖哪些 kernel 或系统假设；
- 它上线后最可能把瓶颈转移到哪里。

只要形成了这套判断，你看后面的量化、serving、cache、speculative 论文时就会越来越像系统工程师，而不只是读单篇算法的人。
