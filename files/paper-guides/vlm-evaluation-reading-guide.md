# VLM 评测论文导读

这条导读的目标，是帮助你区分三类经常被混在一起的事情：

1. 模型语言上答得像不像
2. 模型视觉上到底看没看对
3. 模型端到端任务是否真做成了

如果这三层不分开，VLM 评测很容易被“会说话的错答案”误导。  
**你会看到很多模型**：

- 文风流畅
- 解释完整
- 答案听起来很自信

**但真正拆开后可能发现**：

- 视觉证据引用错了
- OCR 小字没看清
- 表格行列关系搞错了
- 屏幕代理点错了按钮

所以这条导读真正想建立的是一种评测分层意识。

!!! note "初学者先抓住"

    VLM 评测先拆层：看没看对、说没说清、对没对齐、做没做成。一个总分很高的模型，可能只是语言表达强，视觉证据、OCR、grounding 或工具执行仍然不可靠。

!!! note "难点解释：为什么“会说话的错答案”危险"

    LLM 语言能力强，会把错误视觉判断包装成流畅解释。评测时必须追问答案是否由图像证据支持，是否能定位到区域、文本、表格单元或操作结果，而不是只看回答像不像。

## 1. 先建立一个总框架：VLM 评测至少有四层

**建议把 VLM 评测拆成**：

1. **感知层**：图像细节到底看对没有？
2. **语言层**：输出文本是否像样、是否完整、是否结构化？
3. **对齐层**：回答是否真的针对问题和指令？
4. **执行层**：如果模型还要调用工具或输出动作，任务是否完成？

如果把这四层混成一个总分，很多 benchmark 会显得“很高”，但你不知道问题究竟出在哪里。

## 2. 第一阶段：先建立多模态问答的基本评测直觉

读早期 VQA 和图文问答工作时，重点不是 benchmark 数字，而是理解：

- 文本指标为什么不足以说明模型真的看懂图像
- 为什么同一个答案可能来自真正视觉证据，也可能来自语言先验
- 为什么开放式回答和选择题回答会产生不同的误导

读这类论文时，建议一直追问：

**如果不给这张图，只给问题，模型还能答出类似的话吗？**

这是区分“看图得出答案”和“靠常识瞎猜”的最好起点。

### 一个生动例子

**问题是**：“这张图里的人大概在做什么？”  
如果图里是厨房场景，模型回答“在做饭”，这很可能只是一种高频语言先验。  
**但如果问题改成**：

- “锅柄朝向左还是右？”
- “台面上总共有几个红色盖子？”
- “人物右手拿的是木勺还是金属铲？”

就更能逼近真实视觉证据。

## 3. 一个简单的概率分解，帮助你理解 VLM 为什么会“说得像真”

**可以把 VLM 输出粗略理解为**：

\[
p(y \mid x_{\text{img}}, x_{\text{text}})
\propto
p_{\text{visual}}
\cdot
p_{\text{language prior}}
\cdot
p_{\text{instruction}}.
\]

当图像证据弱、输入压缩过强或问题本身具有高共现答案时，语言先验很容易接管输出。  
因此评测设计的关键之一，是让语言先验很难蒙对。

这也是为什么很多 VLM 论文开始重视：

- 对抗样本
- 细粒度计数
- 局部区域问答
- 结构化表格和图表

## 4. 第二阶段：补 OCR、grounding 与文档理解

VLM 一进入真实业务，很快就会遇到：

- OCR
- 布局
- 目标定位
- 局部区域引用
- 表格结构
- 图表坐标与图例

因此建议补读文档理解、grounding 或 referring expression 相关代表工作。  
**阅读重点**：

- 为什么 `IoU`、`CER`、结构化抽取 `F1` 这类指标不可省
- 为什么“答案看起来合理”不足以覆盖定位任务
- 为什么 OCR 和布局错误会级联影响后续生成

### 4.1 为什么 grounding 任务能暴露很多生成式 VLM 的短板

如果要求模型指出“左上角第二行蓝色按钮”，那它必须同时做好：

- 局部视觉识别
- 空间关系理解
- 语言短语对齐

这类任务很难靠纯语言先验蒙对。

### 4.2 为什么文档理解比自然图像更苛刻

自然图像里，回答“在办公室”可能已经够有用。  
但文档场景里，错一个字符就会造成实质性错误，例如：

- 金额少一位小数
- 日期读错月份
- 表头和单元格错配
- 税号最后一位看错

因此文档 benchmark 的价值，往往不在“会不会描述这页文件”，而在“字段和结构是否精确”。

## 5. 第三阶段：进入工具使用与代理评测

当 VLM 输出不再只是自然语言，而是：

- 点击动作
- OCR 调用
- 工具链调用
- 浏览器操作
- 机器人动作提示

评测就从“答案质量”升级为“轨迹成功率”。

阅读这一类工作时，重点看：

- 中间状态是否记录
- 工具调用错误会如何传递
- 端到端 success rate 如何定义
- 是否区分“中途失败”和“最终失败”

### 5.1 为什么单步 accuracy 不够

屏幕代理最典型。  
模型可能第一步就点错，后面所有动作都建立在错误状态上。  
所以比起只看最终任务是否完成，更值得记录的是：

- 第一步命中率
- 第几个动作开始偏离
- 错误是否能恢复
- 是否发生循环点击

### 5.2 一个很实际的例子

在网页订票任务中，模型可能：

- 正确理解页面上“出发地”“目的地”的位置
- 但把日期框点成乘客数量框

这类错误如果只看最终自然语言解释，很可能被掩盖；  
如果看轨迹日志，它会非常明显。

## 6. 第四阶段：关注 robustness 与分布外评测

**真实系统最容易死在**：

- 模糊截图
- 低分辨率文档
- 奇怪布局
- 遮挡
- 图像压缩
- 屏幕缩放比例变化
- 多轮任务漂移

因此建议把 robustness 相关评测工作和主 benchmark 一起看，而不是只盯公开平均分。

### 6.1 为什么 robustness 不是“加分项”，而是部署前提

如果一个 VLM 在高清宣传图上很强，但在真实用户截图、扫描件和手机拍照文档上迅速退化，那么它对业务的价值会大打折扣。  
**评测时必须有意识加入**：

- 低光照
- 模糊
- 旋转
- 裁剪
- 异常布局

### 6.2 长尾场景尤其重要

例如：

- 医疗报告中的手写批注
- 财务单据中的盖章遮挡
- 手机截图里的弹窗遮挡
- 游戏 UI 的非标准按钮布局

这些场景在线下通用 benchmark 里往往并不充足。

## 7. 如何判断一个 benchmark 在测什么

读 VLM 评测论文时，建议先不看总分，先给 benchmark 做分类。

| 问题类型 | 更主要在测什么 | 常见伪成功 |
| --- | --- | --- |
| 开放式 VQA | 语言 + 感知 | 靠常识蒙对 |
| OCR / 文档抽取 | 局部识别 + 结构 | 粗略描述看似合理 |
| Grounding / Referring | 空间定位 + 对齐 | 会描述但不会指 |
| 图表问答 | 结构关系 + 数值理解 | 抓到大意却读错数值 |
| 屏幕代理 | 感知 + 规划 + 执行 | 解释正确但动作错 |
| 多工具代理 | 状态管理 + 执行 | 中间步骤错却文字圆回来 |

## 8. 阅读时最该记录的中间问题

读一篇 VLM 评测工作时，建议固定追问：

1. 这个 benchmark 测的是语言、感知、对齐，还是执行？
2. 它最容易被哪种“伪成功”欺骗？
3. 它和真实业务场景之间差了哪一层？
4. 它缺失了哪些中间日志或可解释性指标？
5. 它的输入分辨率、图像质量和工具设定是否接近真实世界？

如果能把这些问题持续记下来，你读 VLM 评测论文时就不容易只被总分牵着走。

## 9. 一份建议阅读包

若时间有限，建议至少建立如下阅读包：

1. 一组经典 VQA / 图文问答基准
2. 一组 OCR / 文档理解 / grounding 工作
3. 一组图表与结构化问答工作
4. 一组屏幕代理或工具使用评测工作
5. 一组鲁棒性或长尾场景评测工作

### 9.1 第一组：经典多模态问答

**目的**：建立“语言先验会骗你”的基本警觉。

### 9.2 第二组：OCR 和 grounding

**目的**：强迫自己关注局部证据，而不只是自然语言输出。

### 9.3 第三组：代理与工具使用

**目的**：建立端到端轨迹评测意识。

### 9.4 第四组：robustness

**目的**：把实验室好看结果拉回真实世界约束。

## 10. 读完后你应该能形成的判断

真正成熟的 VLM 评测观，不会把“总分高”直接等同于“看得懂、做得对、系统能上线”。  
**你应该能够明确地区分**：

- 这是语言表面强，还是视觉证据强；
- 这是开放问答强，还是局部定位强；
- 这是离线问答强，还是轨迹执行强；
- 这是理想数据强，还是真实长尾强。

一旦形成了这个判断，后面你读任何 VLM 论文或评测报告时，都会更容易看出它到底在解决哪一层问题。
