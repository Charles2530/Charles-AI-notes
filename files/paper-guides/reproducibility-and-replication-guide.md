# 论文复现与可复现性导读

在基础模型领域，“读懂一篇论文”和“复现出论文结果”常常是两件相距很远的事。很多方法在 paper 上看起来结构清晰、指标漂亮，但一旦真正动手，就会遇到代码缺失、数据口径不明、训练预算不现实、评测脚本口径不一致、随机种子波动巨大等问题。因此，论文复现不是一个附属技能，而是研究判断力的重要组成部分。它迫使你把方法、数据、系统、评测和资源约束全部重新过一遍。

!!! note "初学者先抓住"

    复现不是照着论文跑一次，而是审计方法、数据、系统和评测依赖。先明确目标是复现结论、复现指标，还是复现代码；目标不同，投入和验收标准完全不同。

!!! note "难点解释：为什么最小可运行版本很重要"

    大模型复现成本高，直接追全量表格容易浪费预算。先跑通最小路径，确认核心模块和评测口径，再逐步贴近原文，才能知道差距来自方法、数据、系统还是隐藏技巧。

## 1. 复现的三种层级

复现并不是只有“成功”或“失败”两种结果。更合理地说，可以分为：

1. **结论复现**：在近似条件下重现论文主结论。
2. **指标复现**：在相近设置下达到论文接近的数值。
3. **代码复现**：严格按照实现细节得到相同结果。

很多现实项目追求的是第一种或第二种，而不是百分百代码逐比特一致。先明确目标，能减少很多无效纠结。

## 2. 复现前先做“可行性审计”

在真正开始之前，建议先审计以下问题：

1. 论文是否提供代码、配置和数据处理说明？
2. 所需训练预算是否现实？
3. 关键数据是否公开可得？
4. 评测脚本和指标定义是否明确？
5. 论文中的关键收益是否可能来自未公开技巧？

如果这些问题中有多个答案是否定的，就应降低预期，把目标改成“验证核心思想”，而不是机械追平原始表格。

## 3. 复现路线：先最小可运行，再逐步贴近原文

最有效的复现方式通常不是一上来全量照搬，而是：

1. 跑通最小版本；
2. 确认关键模块工作；
3. 逐步补齐原文中的重要设定；
4. 每一步都记录指标变化。

这种方式能帮助你区分“根本没跑通”和“跑通但差在细节”，也更适合大模型时代昂贵的训练预算。

## 4. 把论文拆成四类依赖

### 4.1 方法依赖

新目标函数、新模块、新采样器、新路由器等。

### 4.2 数据依赖

训练集、清洗规则、分桶方式、标注协议、长尾构造。

### 4.3 系统依赖

并行策略、硬件类型、kernel、缓存、runtime、精度格式。

### 4.4 评测依赖

指标、归一化规则、基线实现、随机种子、采样参数。

很多“复现失败”的本质，并不是方法不对，而是某一类依赖没对齐。

## 5. 复现日志为什么必须结构化

**建议每次复现实验都记录**：

1. 代码版本
2. 配置版本
3. 数据版本
4. 资源配置
5. 实际 wall-clock
6. 关键指标
7. 异常现象

没有结构化日志，复现工作很快会变成“记忆驱动排错”，而这在多轮实验后几乎不可靠。

## 6. 评测口径是最容易被低估的问题

一些论文看似指标差距巨大，实际只是评测口径不一致。例如：

1. 文本归一化方式不同；
2. 图像分辨率或裁剪方式不同；
3. VLM 是否允许检索增强不同；
4. 机器人任务是否看首次成功率还是最终成功率；
5. 推理成本是否被算入。

因此复现中最先应锁定评测脚本，而不是先调模型。

## 7. 随机性与方差

即便方法完全一致，不同种子、不同 batch 顺序、不同 GPU kernel 也可能造成结果波动。若结果记为 $y_1,\dots,y_n$，则至少应报告均值和波动范围，而不是只盯着最好的那次：

$$
\bar{y}=\frac{1}{n}\sum_i y_i.
$$

对小提升尤其如此。否则你可能会把随机幸运当作方法增益。

## 8. 复现中的常见陷阱

### 8.1 复制超参但忽视预算差异

原论文在更大 batch、更长训练步数、更好硬件上有效，你在更小预算下直接照搬，结果往往不稳定。

### 8.2 只抄主文，不看附录和开源实现

很多真正关键的配方藏在附录、issue 或代码默认参数里。

### 8.3 用不同数据却期待相同数字

如果数据分布已经换了，最合理的目标应是复现趋势，而不是执着于表格中的绝对值。

### 8.4 发现偏差后一次改太多

这会让问题定位彻底失去因果线索。

## 9. 如何判断“复现失败但有价值”

**有价值的失败复现往往能回答**：

1. 方法对哪些假设极度敏感；
2. 论文结果中哪些部分靠资源堆叠，哪些部分是真机制；
3. 哪些改动对迁移到新场景是必要的；
4. 未来若要工程落地，真正要补的是什么。

这类失败并不比单纯“跑成了”价值低。

## 10. 一个扩散复现例子

假设你复现某个少步扩散蒸馏方法，结果发现 4 步采样远达不到论文质量。排查后可能发现：

1. 论文教师模型更强；
2. 训练时使用了特定时间采样分布；
3. 文中没强调的 EMA 对结果影响巨大；
4. 你自己的验证 prompt 分布与论文不一致。

这时真正学到的，不是“方法没用”，而是“该方法依赖高质量教师与特定配方”。

## 11. 一个 VLA 复现例子

若复现某个闭环恢复 VLA 论文，离线结果不错但实机没提升，可能意味着：

1. 训练数据中的失败轨迹质量不够；
2. 原论文默认有额外传感器或状态输入；
3. 评测环境的恢复定义不同；
4. 真实控制接口延迟比论文设定大。

这类偏差恰恰是复现最有价值的部分，因为它把论文假设暴露出来了。

## 12. 复现之后要写什么

**建议复现报告至少包含**：

1. 复现目标层级
2. 对齐了哪些条件，没对齐哪些条件
3. 结果与原文差异
4. 你认为差异最可能来自哪里
5. 对方法有效性的最终判断

这样你的复现才不是“一个运行日志”，而是一份真正能帮助团队做路线判断的材料。

## 13. 小结

论文复现不是机械劳动，而是一种把论文还原成真实系统的过程。它迫使你识别隐含假设、预算边界、评测口径和工程细节。长期做高质量复现的人，往往不只是更会跑代码，也更能判断一条研究路线到底是短期技巧、可迁移方法，还是值得长期投入的系统方向。
