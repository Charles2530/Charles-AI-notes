# VLM 的评测与工具使用

很多团队在做 VLM 时，训练阶段只盯 loss，评测阶段只看“答得像不像”，最后上线才发现问题集中爆发。  
**原因很简单**：VLM 真正面对的是一条链路，而不是一个单点模型。

!!! note "初学者先抓住"
    VLM 评测要先拆层：看没看清、对没对齐、推没推对、工具有没有用对。只看最终回答，很难知道错在视觉、语言、推理还是执行接口。

!!! example "有趣例子：看火车票订行程"
    如果模型把票上的时间读错，是感知错；把出发站和到达站对反，是对齐错；算错转车时间，是推理错；调用订票工具填错字段，是执行错。同样一句“订票失败”，背后可能是完全不同的问题。

## 1. 评测不是单一分数，而是分层能力画像

对 VLM，至少要分开评四类能力：

1. `感知层`
2. `对齐层`
3. `推理层`
4. `执行层`

**对应问题分别是**：

- 它有没有看清？
- 它有没有把图和字对上？
- 它有没有推对？
- 它有没有把动作或工具用对？

如果把所有任务混成一个总分，很容易出现假象：  
模型平均分不错，但真正部署时一塌糊涂。

## 2. 一个统一的评测抽象

给定样本 \((x_v, x_t)\)，模型输出 \(y\)。  
评测时，本质上比较预测分布与任务目标之间的偏差：

\[
\text{score} = \mathcal{M}(y, y^\star)
\]

其中 \(\mathcal{M}\) 可能是：

- `accuracy`
- `F1`
- `BLEU/ROUGE`
- `IoU`
- `edit distance`
- `success rate`

关键是：不同任务的 \(\mathcal{M}\) 完全不同，不能用一个文本相似度统一概括。

## 3. 感知层评测：先问它到底看没看见

**感知层常包括**：

- OCR
- 目标定位
- 细粒度属性识别
- 布局理解
- 图表元素读取

若模型需要输出框 \(b\)，则可用 IoU：

\[
\text{IoU}(b,\hat{b}) = \frac{|b \cap \hat{b}|}{|b \cup \hat{b}|}
\]

若模型需要读取文本，则可用字符错误率：

\[
\text{CER} = \frac{S + D + I}{N}
\]

其中：

- \(S\) 是替换数
- \(D\) 是删除数
- \(I\) 是插入数
- \(N\) 是真实字符数

### 例子：屏幕按钮定位

用户说“点右上角的保存”。  
如果模型回答“保存按钮在右上角”，语言上看似没问题，但真正执行时框偏了 30 个像素，点到“删除”。  
这说明该任务不能只评文本答案，必须评定位精度和真实点击成功率。

## 4. 对齐层评测：看图和看字是否指向同一个对象

对齐层关注的是图文映射是否正确，而不是语言是否流畅。  
**常见任务包括**：

- image-text retrieval
- grounding
- referring expression comprehension

检索场景中，常见指标有：

\[
\text{Recall@K}
\]

例如 `Recall@1` 表示正确图像或文本是否排在第一位。

### 一个直观例子：服装检索

查询“米白色 V 领针织衫”。  
如果模型把“米白色圆领毛衣”排第一，它可能语义大致对了，但关键属性没对上。  
**因此对齐评测必须覆盖**：

- 颜色
- 材质
- 版型
- 细节词

而不是只看粗类别。

## 5. 推理层评测：不是背模板，而是做多步判断

推理层最容易被误判。  
因为很多模型能用语言先验把答案“说得像对的”，但其实没有基于视觉证据推理。

**可把推理任务抽象为**：

\[
y = f_\theta(x_v, x_t, c)
\]

其中 \(c\) 表示中间上下文或多轮历史。  
评测时，应尽量设计那些只有看图后才能答对的题。

### 例子：图表因果解释

问题是“为什么二季度销量下降？”  
一个弱模型可能机械回答“因为市场需求下降”。  
**一个强模型会结合图表证据指出**：

- 哪个区域下降最明显
- 哪个产品线拖累最大
- 是否存在季节性波动

**这类任务最好评**：

- 答案正确性
- 证据引用是否准确
- 是否提到关键视觉实体

## 6. 执行层评测：真正的代理能力

当 VLM 开始调用工具时，问题就变成一个序列决策过程。设上下文状态为 \(c_t\)，模型在时刻 \(t\) 产生动作：

\[
a_t \sim \pi_\theta(a_t \mid c_t)
\]

**动作可能是**：

- 直接回答
- 调 OCR
- 调搜索
- 调计算器
- 点按钮
- 框选区域

工具返回结果 \(o_t\) 后，上下文更新为：

\[
c_{t+1} = U(c_t, a_t, o_t)
\]

最终目标不是单步动作对不对，而是整条轨迹是否成功：

\[
\text{Success}(\tau) \in \{0,1\}
\]

**或者平均成功率**：

\[
\text{SR} = \frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[\tau_i \text{ 成功}]
\]

## 7. 工具使用为什么比问答难很多

问答出错，往往只是答错一句话。  
工具使用出错，前一步的错误会污染后一步输入。

### 例子：报销单审核

用户上传模糊发票并问“税号和总金额是多少？”  
**一个靠谱的 VLM 代理通常会**：

1. 调 OCR。
2. 判断 OCR 结果中哪些字段可信。
3. 若金额和税额逻辑冲突，再调用规则校验或二次裁剪识别。
4. 最后输出结构化答案。

如果第一步 OCR 就把 `8` 识别成 `3`，后续所有推理都可能基于错误字段展开。  
**所以这类任务要同时评**：

- OCR 准确率
- 工具调用路径正确率
- 最终结构化结果正确率

## 8. 工具调用链可以写成规划问题

设工具集合为 \(\mathcal{A}\)，每一步选择动作 \(a_t \in \mathcal{A}\)，希望最大化任务完成收益：

\[
\max_\pi \mathbb{E}\left[\sum_{t=0}^{T}\gamma^t r_t\right]
\]

这里的奖励 \(r_t\) 可以不是显式训练信号，也可以只是离线评测时定义的成功奖励。  
这说明工具使用 VLM 在本质上已经不是静态感知模型，而是在做轻量 agent planning。

## 9. 三种典型评测场景

### 9.1 文档理解

**核心指标**：

- OCR 正确率
- 表格字段抽取 F1
- 多页问答准确率

**真实难点**：

- 版面复杂
- 图文混排
- 多页跨页引用

### 9.2 屏幕代理

**核心指标**：

- 元素定位精度
- 单步动作正确率
- 任务完成率
- 平均步数

**真实难点**：

- 按钮样式多变
- 相似控件很多
- 一步点错可能进入全新页面

### 9.3 多模态检索与客服

**核心指标**：

- Recall@K
- rerank precision
- 多轮任务完成率

**真实难点**：

- 用户描述含糊
- 图片质量差
- 商品属性细粒度区分难

## 10. 评测里最常见的假阳性

### 10.1 语言流畅掩盖视觉错误

模型答得很顺，但没真正看见图里的证据。

### 10.2 模板数据过拟合

如果测试集问题形式和训练集太像，模型可能只是记住问法。

### 10.3 只评最终答案，不评中间动作

工具调用场景里，最终答案对了，也可能只是运气好，中间步骤其实乱七八糟。

### 10.4 只看公开 benchmark

公开基准能反映共性，但企业部署往往栽在自己的长尾场景：

- 模糊截图
- 畸变扫描件
- 低光照监控画面
- 业务系统私有界面

## 11. 一个够用的评测表

若你要给 VLM 做一套真正能指导上线的评测，至少应包含：

- `感知`
  OCR、定位、细粒度识别
- `对齐`
  图文匹配、检索、grounding
- `推理`
  多跳问答、图表解释、文档跨页推理
- `执行`
  工具调用、UI 操作、任务完成率
- `稳定性`
  分辨率变化、噪声、裁剪、模糊、遮挡

## 12. 工程风险

- OCR 错误会被后续推理放大。
- 工具调用链过长会带来延迟和累计错误。
- 屏幕理解任务里，框选误差极易导致误点。
- 若没有中间日志，就很难判断失败发生在“看不清”“没对齐”“没推对”还是“工具没调对”。

## 13. 总结

VLM 的评测要从“模型答得像不像”升级到“整条任务链路是否可靠”。  
真正可部署的 VLM，不只是答题机器，而是一个多层系统：

- 底层要看清
- 中层要对齐
- 高层要推理
- 最后还要会行动

只要其中任何一层没单独测清楚，线上问题就迟早会集中爆发。

## 快速代码示例

```python
def eval_case(agent, sample):
    trace = agent.run(sample["task"], tools=sample["tools"])
    tool_ok = trace.tool_calls == sample["gold_tool_calls"]
    answer_ok = trace.final_answer == sample["gold_answer"]
    return {
        "tool_success": int(tool_ok),
        "task_success": int(tool_ok and answer_ok),
    }
```

这段代码展示了“工具成功率”和“任务成功率”分开统计的基本做法。它能避免只看最终答案导致的误判，帮助定位问题究竟出在工具调用链，还是出在推理与决策阶段。


## 实践补充与检查

### 用闭环任务重写 **VLM 的评测与工具使用**

涉及 VLM、VLA、世界模型和具身系统时，最容易出现的误区是：离线表征写得很完整，但真正和闭环任务、工具链、控制接口、风险治理放在一起时，内容支撑不够。围绕 **VLM 的评测与工具使用**，更扎实的组织方式应该首先回答：

1. **工具正确性、过程监督、任务完成率、格式稳定性、多轮反馈**。

这些坐标之所以关键，是因为这类系统通常横跨 **数据、模型、动作接口、在线约束和安全责任**。如果页面只讲模型结构，不讲这些接口，读者会很难判断一个方法是“研究上成立”还是“系统上可用”。

### 更实用的任务分解方式

围绕 **VLM 的评测与工具使用**，建议把问题拆成三层：

1. **理解层**：模型是否真的看懂了环境、界面、视频、状态或用户意图；
2. **决策层**：模型是否能基于这些理解做出正确下一步；
3. **闭环层**：一旦动作、工具或环境反馈回来，系统是否仍然稳定。

很多方法在第一层表现不错，但第二层和第三层并没有同步变强。页面若能把这三层明确拆开，读者就更容易看清楚：某些提升究竟是“更会看”，还是“更会做”，或者只是“更会在离线数据上得分”。

### 常见失败模式与误判

围绕 **VLM 的评测与工具使用**，真实失败常见于：

1. **只看最终答案**。
2. **工具成功率和任务成功率混淆**。
3. **judge 难以覆盖复杂流程**。

这些问题的共同点是：它们很少只靠平均准确率暴露出来，往往要靠 **回放、闭环任务、风险桶、人工抽检和高价值案例分析** 才能看清。也因此，这类页面尤其需要把“怎么验收”写得比“是什么方法”更扎实。

### 验收、上线与回流

对 **VLM 的评测与工具使用**，更合理的验收至少应覆盖：

1. **同时记录过程与终局**。
2. **对工具路径做 replay**。
3. **高风险任务保留人工抽检**。

进一步地，建议把这类主题都放进统一闭环：

1. **离线评测** 负责判断基础能力是否成立；
2. **回放与 shadow** 负责判断真实链路是否稳；
3. **线上/实机观察** 负责判断高价值场景是否真正受益；
4. **失败样本回流** 负责让系统下一轮变得更强。

一旦这条链路写清楚，**VLM 的评测与工具使用** 就不再只是“某个热门方向”，而会真正成为可操作的系统设计知识。
