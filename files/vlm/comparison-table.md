# VLM 方法对照表

这一页把视觉语言模型常见系统形态放在同一张地图里，不只回答“它们结构有什么不同”，更回答“为什么你在检索、OCR、文档问答、图表解析、屏幕代理和工具调用里会需要完全不同的 VLM 设计”。

VLM 很容易被一句“视觉编码器接 LLM”概括，但现实系统至少分成六层：

1. 感知特征怎么提。
2. 视觉 token 怎么送进语言模型。
3. 模型是否保留局部位置和空间关系。
4. 是否需要检索、OCR、框选、表格结构这些外部模块。
5. 评测到底看自然语言答案、定位结果，还是端到端任务成功率。
6. 部署时瓶颈是在图片分辨率、视觉 token 数，还是 agent 工具链。

如果这六层不拆开，VLM 的“对照”就会很容易只剩模型名。

!!! note "初学者先抓住"

    VLM 不是一种固定架构，而是一组系统取舍。检索型 VLM 擅长“找得到”，文档 VLM 擅长结构理解，屏幕代理型 VLM 要能执行动作；评测和部署瓶颈也必须随任务改变。

!!! note "难点解释：为什么通用视觉聊天不等于文档/屏幕能力"

    通用 VLM 可能能描述图片，却不一定能读小字、理解表格坐标、定位按钮或维护操作状态。文档和屏幕任务需要 OCR、布局、grounding、工具调用和端到端成功率评测，不能只看自然语言回答是否流畅。

## 1. 先把 VLM 拆成几种常见系统形态

| 类型 | 核心结构 | 更适合的任务 | 主要优点 | 主要短板 |
| --- | --- | --- | --- | --- |
| 双塔检索 | 图像塔 + 文本塔 | 图文检索、商品召回、粗粒度匹配 | 快、可索引、可离线建库 | 交互粒度粗，难做复杂推理 |
| Cross-attention 交互编码器 | 图文联合编码 | grounding、图文匹配、细粒度 rerank | 对齐细、局部证据更强 | 在线成本高，难大规模索引 |
| 视觉编码器 + LLM | encoder + projector + decoder-only LLM | 图像问答、聊天、工具使用 | 语言能力强，扩展方便 | 视觉细节容易丢，容易语言先验幻觉 |
| 文档 VLM | OCR / 布局编码 + LLM | 发票、表格、长 PDF、结构化问答 | 文档能力强，结构感知更好 | 输入长、系统复杂、成本高 |
| 图表与结构理解 VLM | 图像编码 + 图表结构归纳 | chart QA、表格解析、科学图理解 | 擅长结构关系 | 对坐标轴、图例、局部文字很敏感 |
| 屏幕代理型 VLM | 屏幕观察 + 工具/动作头 | UI 自动化、网页代理、软件操作 | 能执行而非只回答 | 定位偏差会直接变成错误操作 |

## 2. 用一句话抓住每类系统的本质

### 2.1 双塔检索

它的本质是把图像和文本都压进同一个 embedding 空间，让“相似内容靠近，不相似内容远离”。  
这类模型真正擅长的是召回，而不是解释。

如果用公式写，一个典型图文对比学习目标可以表示为：

\[
\mathcal{L}_{\text{InfoNCE}}
= - \log
\frac{\exp(\mathrm{sim}(v_i, t_i)/\tau)}
{\sum_j \exp(\mathrm{sim}(v_i, t_j)/\tau)}.
\]

它解决的是“找得到”，不是“说得清”。

### 2.2 Cross-attention 交互编码器

这类模型让视觉和文本 token 在内部充分交互，所以更适合：

- phrase grounding
- 细粒度匹配
- 需要看局部证据的任务

它本质上是“我要知道文本里这句话到底对应图像哪里”，而不是只判断整图和整句是否相关。

### 2.3 视觉编码器 + LLM

这是当前最常见的通用 VLM 结构。  
它的优点是语言生成和指令遵循能力强，但短板也很鲜明：

- 视觉 token 经过压缩后容易损失局部细节
- 语言模型太强时，会用语言先验补视觉证据
- 回答听起来很像真的，不代表它真看懂

### 2.4 文档 VLM

文档任务真正难的不是“图片里有什么”，而是：

- 哪块文字属于哪个区域
- 表头和单元格是什么关系
- 段落、表格、印章、注释如何共存

所以文档 VLM 往往必须把 OCR、布局树、坐标信息和语言建模一起放进系统里。

### 2.5 屏幕代理型 VLM

屏幕代理最容易被误解。  
它不是“对屏幕图片做问答”，而是“根据视觉状态决定下一步操作”。  
因此它的评测对象不是 BLEU 或 Rouge，而是：

- 点击是否点中
- 表单是否填对
- 工作流是否完成
- 出错后是否能恢复

## 3. 一张“任务到结构”的对照表

| 任务 | 优先结构 | 为什么 | 次选 |
| --- | --- | --- | --- |
| 商品图文检索 | 双塔检索 | 可离线建索引，召回快 | 双塔召回 + rerank |
| 图文 rerank | 双塔召回 + cross-encoder | 先快后准 | 纯交互编码器 |
| 普通图像问答 | 视觉编码器 + LLM | 指令能力强 | 交互编码器 |
| 文档问答 | 文档 VLM | 需要 OCR + 布局 | 通用 VLM + OCR 外挂 |
| 图表解析 | 图表专用 VLM | 结构和局部字很关键 | 通用 VLM |
| 屏幕代理 | 屏幕代理型 VLM | 需要动作与轨迹监督 | 通用 VLM + 工具脚本 |
| 多模态搜索代理 | 双塔检索 + 生成式 VLM | 召回与解释分开更稳 | 单一生成模型 |

## 4. 比结构更关键的是信息瓶颈

VLM 常见失真都和信息瓶颈有关。  
假设图像编码器输出 \(N_v\) 个视觉 token，每个 token 维度为 \(d\)，而 projector 再把它们压到 LLM 输入空间。  
如果高分辨率原图里存在大量局部文字或细碎结构，那么经过压缩后：

- 小字体可能消失
- 相邻表格线可能混淆
- 位置关系可能被弱化

**直观例子**：

- 一张风景图，压缩后仍能保留“海边、日落、人物”这类语义。
- 一张发票图，压缩后可能会直接丢掉金额小数点、日期格式、税号尾号。

这就是为什么“同一个 VLM 能做开放问答”并不意味着它适合做文档或 UI 代理。

## 5. 评测维度必须拆开，不然会误判

### 5.1 语言看起来流畅，不等于视觉证据正确

很多生成式 VLM 的错误不是胡言乱语，而是“有理有据地说错”。  
**因此评测至少要拆成**：

- 感知正确性
- 语言表达质量
- 任务完成度

### 5.2 局部定位任务要单独看

如果任务涉及框选、点击、OCR 片段引用或表格单元格定位，就不能只看最终答案文本。  
**应当同时看**：

- box IoU
- OCR CER/WER
- cell-level F1
- grounding recall

### 5.3 代理任务要看轨迹而不是一句话

屏幕代理里，一步点错就可能导致后续全错。  
因此端到端 success rate 之外，最好还要记录：

- 第一步是否正确
- 第几个动作开始偏离
- 是否发生重复点击
- 错误是否可恢复

## 6. 用一个公式理解“VLM 是否真的看图”

**可以把 VLM 的回答粗略分解成**：

\[
p(y \mid x_{\text{img}}, x_{\text{text}})
\propto
\underbrace{p_{\text{visual}}}_{\text{视觉证据}}
\cdot
\underbrace{p_{\text{language prior}}}_{\text{语言先验}}
\cdot
\underbrace{p_{\text{instruction}}}_{\text{指令遵循}}.
\]

实际问题在于，当视觉证据弱、图像压缩过强或任务本身答案常见时，语言先验会“接管”输出。  
所以真正好的评测设计，应该让语言先验很难蒙对。

例如：

- 不是问“厨房里通常会有什么”，而是问“图片左上角柜门是否打开”。
- 不是问“这张图大概是什么场景”，而是问“坐标轴纵轴最大值是多少”。

## 7. 三个具体场景说明为什么要选不同架构

### 场景 A：电商图搜文

**需求**：

- 几千万商品图离线建库
- 用户输入自然语言描述进行召回
- 结果需要快

最合适的是双塔检索，因为：

- 图像和文本 embedding 可以提前算好
- 查询时只需 ANN 检索
- 成本远低于逐项 cross-attention

此时若直接用生成式 VLM 做全库匹配，系统会极其低效。

### 场景 B：企业发票审核

**需求**：

- 识别抬头、金额、税号、日期
- 需要结构化输出
- 错一个字段就可能影响财务流程

这时“能聊天”的通用 VLM 不一定够，往往需要：

- 高分辨率输入
- OCR 模块
- 布局建模
- 结构约束输出

### 场景 C：网页代理自动下单

**需求**：

- 看屏幕
- 点击按钮
- 填表
- 滚动页面

这里最关键的不是最终文本答复，而是动作精度、时序和容错。  
**所以系统设计必须考虑**：

- 坐标表示
- 可点击区域建模
- 工具调用和状态同步
- 错误恢复策略

## 8. 一张“能力边界”对照表

| 能力 | 双塔检索 | 交互编码器 | 通用 VLM | 文档 VLM | 屏幕代理型 VLM |
| --- | --- | --- | --- | --- | --- |
| 大规模召回 | 很强 | 弱 | 弱 | 弱 | 弱 |
| 细粒度对齐 | 中 | 强 | 中 | 强 | 强 |
| 多轮对话 | 弱 | 弱 | 强 | 强 | 中 |
| OCR / 布局 | 弱 | 中 | 中 | 很强 | 中 |
| 端到端动作执行 | 弱 | 弱 | 中 | 弱 | 很强 |
| 部署简洁性 | 强 | 中 | 中 | 弱 | 弱 |

## 9. 常见误判

### 误判 1：一个模型能聊天，所以也能做 grounding

不对。  
能给出流畅解释，不代表能把“红色按钮”稳定定位到屏幕左下角的那个具体区域。

### 误判 2：检索效果强，所以问答能力也强

也不对。  
双塔 embedding 擅长召回相关项，但不擅长多步推理、局部证据解释和细节引用。

### 误判 3：文档只是特殊图像

错误很常见。  
文档里真正关键的是结构、顺序、局部字符和版面关系，不是普通视觉语义。

### 误判 4：总分高，说明系统适合产品

不一定。  
**如果产品最怕的是**：

- OCR 小字丢失
- 表单误填
- 错点删除按钮

那么通用 benchmark 总分未必有决定意义。

## 10. 工程部署时最该看的不是排行榜，而是 token 成本

以通用生成式 VLM 为例，真实成本经常由以下几项决定：

- 图像分辨率
- 视觉 patch/token 数
- 是否多图输入
- 是否长上下文对话
- 是否外挂 OCR、检索和工具调用

**一个简化成本模型可以写成**：

\[
\text{Cost} \approx
a \cdot N_{\text{vision tokens}}
 b \cdot N_{\text{text tokens}}
 c \cdot N_{\text{tool calls}}.
\]

这解释了为什么很多系统最终不是“全靠一个大一统 VLM”，而是拆成：

- 检索层
- OCR 层
- 生成层
- 工具层

因为分层后更容易控成本、做回退和记录日志。

## 11. 阅读和搭建 VLM 系统时的建议顺序

### 第一层：先理解表征

- CLIP 类双塔
- 图文对比学习

**目标**：搞清 embedding 空间、召回和基础对齐。

### 第二层：再理解生成式 VLM

- vision encoder + LLM
- projector / resampler
- instruction tuning

**目标**：搞清为什么它会“看图说话”。

### 第三层：补专项系统

- OCR / 文档
- 图表
- grounding
- 屏幕代理

**目标**：知道通用架构在哪些地方不够用。

### 第四层：回到部署

- 推理成本
- 评测体系
- 错误日志
- 工具链回退

**目标**：把模型能力真正变成系统能力。

## 12. 一个总判断

VLM 的方法对照，不能只看 backbone 或参数量。  
**更实用的判断方式是**：

- 这是做召回，还是做细粒度交互？
- 这是做回答，还是做动作？
- 这是看自然图像，还是看文档、图表和屏幕？
- 这是离线批量处理，还是在线低延迟系统？

只有把任务结构、信息瓶颈、评测指标和部署成本放在一起看，VLM 的“对照表”才真正有意义。
